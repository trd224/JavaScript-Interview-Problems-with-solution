What is the difference between mergeMap, switchMap, concatMap, and exhaustMap?

Definition

mergeMap, switchMap, concatMap, and exhaustMap are higher-order mapping operators. They are used when a source Observable emits a value and that value must be converted into another Observable, usually an HTTP request.

Main Difference

The main difference between them is how they handle multiple inner Observables when new values keep coming from the source Observable.

mergeMapRuns all inner Observables in parallel. Use it when every request must complete and order is not important.
switchMapCancels the previous inner Observable and keeps only the latest one. Use it for search, filters, and route-param API calls.
concatMapQueues inner Observables and runs them one by one. Use it when order must be preserved.
exhaustMapIgnores new source values while the current inner Observable is running. Use it for login, submit, or payment buttons.

Angular Interview Use Case

  • Use switchMap for typeahead search because old API calls should be cancelled.
  • Use mergeMap for independent parallel save operations.
  • Use concatMap when API calls must run in exact order.
  • Use exhaustMap when duplicate button clicks must be ignored while one request is running.

Key Points

  • They are used for flattening Observables returned from another Observable.
  • The correct operator depends on concurrency, cancellation, ordering, and duplicate-event handling.
  • Wrong choice can cause duplicate API calls, stale UI data, broken order, or ignored requests.
Remember: mergeMap = parallel, switchMap = cancel old and keep latest, concatMap = queue in order, exhaustMap = ignore new while busy.

Full Understandable Example / Examples

mergeMap-example.ts
import { from } from 'rxjs';
import { mergeMap } from 'rxjs/operators';

// Use case: save multiple independent records in parallel.
// Order is not guaranteed, but all requests are allowed to complete.

const userIds$ = from([101, 102, 103]);

userIds$.pipe(
  mergeMap(id => http.post(`/api/users/${id}/save`, {}))
).subscribe(result => {
  console.log('mergeMap save result:', result);
});
mergeMap: requests may complete in any order.
switchMap-example.ts
import { FormControl } from '@angular/forms';
import { debounceTime, distinctUntilChanged, switchMap } from 'rxjs/operators';

// Use case: typeahead search.
// Previous API call is cancelled when user types a new value.

const searchControl = new FormControl('');

searchControl.valueChanges.pipe(
  debounceTime(400),
  distinctUntilChanged(),
  switchMap(text => http.get(`/api/users?q=${text}`))
).subscribe(users => {
  console.log('Latest search users:', users);
});
switchMap: only latest active request result is used.
concatMap-example.ts
import { from } from 'rxjs';
import { concatMap } from 'rxjs/operators';

// Use case: ordered API calls.
// Next request starts only after previous request completes.

const orderIds$ = from([1, 2, 3]);

orderIds$.pipe(
  concatMap(id => http.post(`/api/orders/${id}/process`, {}))
).subscribe(result => {
  console.log('concatMap processed:', result);
});
concatMap: requests complete in source order.
exhaustMap-example.ts
import { fromEvent } from 'rxjs';
import { exhaustMap } from 'rxjs/operators';

// Use case: login button.
// Extra clicks are ignored while login request is running.

const loginButton = document.getElementById('loginBtn')!;

fromEvent(loginButton, 'click').pipe(
  exhaustMap(() => http.post('/api/login', {
    username: 'rizwan',
    password: 'secret'
  }))
).subscribe(response => {
  console.log('Login response:', response);
});
exhaustMap: extra emissions are ignored while one request is active.
