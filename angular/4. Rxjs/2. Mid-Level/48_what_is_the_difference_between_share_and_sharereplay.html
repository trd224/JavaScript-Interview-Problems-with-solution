What is the difference between share() and shareReplay()?

Definition

share and shareReplay both multicast a source Observable, but shareReplay also stores and replays previous values.

share

  • Shares active source execution.
  • Does not replay old values to late subscribers.
  • Late subscriber receives only future live emissions.

shareReplay

  • Shares source execution.
  • Caches previous values based on buffer size.
  • Replays cached values to late subscribers.

Key Points

  • share does not replay old values.
  • shareReplay replays cached values.
  • shareReplay is better for cached API data.
Remember: share = multicast only; shareReplay = multicast + cache.

Full Understandable Example / Examples

share-example.ts
import { interval } from 'rxjs';
import { share, take } from 'rxjs/operators';

// share multicasts active emissions.
// Late subscribers do not receive previous values.

const shared$ = interval(1000).pipe(
  take(5),
  share()
);

shared$.subscribe(value =>
  console.log('share A:', value)
);

setTimeout(() => {
  shared$.subscribe(value =>
    console.log('share B late:', value)
  );
}, 3500);
share A: 0
share A: 1
share A: 2

share A: 3
share B late: 3

share A: 4
share B late: 4
Why? B subscribes after 3500ms. Values 0, 1, and 2 were already emitted, so share() does not send them again. B receives only future values 3 and 4.

What Happens Without share()?

Without share(), every subscription creates a completely separate Observable execution. Each subscriber gets its own independent interval().

without-share-example.ts
import { interval } from 'rxjs';
import { take } from 'rxjs/operators';

// Without share(), each subscribe creates
// a separate interval execution.

const shared$ = interval(1000).pipe(
  take(5)
);

shared$.subscribe(value =>
  console.log('share A:', value)
);

setTimeout(() => {
  shared$.subscribe(value =>
    console.log('share B late:', value)
  );
}, 3500);
share A: 0
share A: 1
share A: 2

share B late: 0

share A: 3
share B late: 1

share A: 4
share B late: 2
share B late: 3
share B late: 4
Why? Without share(), both subscriptions run independently. B subscribes after 3500ms, so its own interval starts from 0 at that moment. That is why B receives 0, 1, 2, 3, 4 separately instead of continuing from A's live stream.
Easy Understanding: Without share() = two separate TVs playing different recordings.
With share() = everyone watching the same live TV channel together.
shareReplay-example.ts
import { interval } from 'rxjs';
import { shareReplay, take } from 'rxjs/operators';

// shareReplay multicasts and replays cached values to late subscribers.

const cached$ = interval(1000).pipe(
  take(5),
  shareReplay(2)
);

cached$.subscribe(value =>
  console.log('shareReplay A:', value)
);

setTimeout(() => {
  cached$.subscribe(value =>
    console.log('shareReplay B late:', value)
  );
}, 3500);
shareReplay A: 0
shareReplay A: 1
shareReplay A: 2

shareReplay B late: 1
shareReplay B late: 2

shareReplay A: 3
shareReplay B late: 3

shareReplay A: 4
shareReplay B late: 4
Why? shareReplay(2) stores the last 2 emitted values. When B subscribes at 3500ms, it immediately receives cached values 1 and 2, then receives future values 3 and 4.
Easy Memory Trick: share is like live TV broadcast — missed means missed. shareReplay is like YouTube replay/history — old values can be replayed.

Angular Real Use Case: Same API Call Used in Multiple Components

If multiple components need the same API data, use shareReplay(1) inside a service. Then the HTTP API call runs only once, and all components receive the same cached response.

user.service.ts
import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';
import { shareReplay } from 'rxjs/operators';

@Injectable({ providedIn: 'root' })
export class UserService {

  private users$?: Observable<any[]>;

  constructor(private http: HttpClient) {}

  getUsers(): Observable<any[]> {

    if (!this.users$) {
      this.users$ = this.http.get<any[]>('/api/users').pipe(
        shareReplay(1)
      );
    }

    return this.users$;
  }
}
user-list.component.ts
import { Component, OnInit } from '@angular/core';
import { UserService } from './user.service';

@Component({
  selector: 'app-user-list',
  template: `<p>User List Component</p>`
})
export class UserListComponent implements OnInit {

  constructor(private userService: UserService) {}

  ngOnInit() {
    this.userService.getUsers().subscribe(users => {
      console.log('UserListComponent:', users);
    });
  }
}
user-count.component.ts
import { Component, OnInit } from '@angular/core';
import { UserService } from './user.service';

@Component({
  selector: 'app-user-count',
  template: `<p>User Count Component</p>`
})
export class UserCountComponent implements OnInit {

  constructor(private userService: UserService) {}

  ngOnInit() {
    this.userService.getUsers().subscribe(users => {
      console.log('UserCountComponent count:', users.length);
    });
  }
}
API called only one time: GET /api/users
UserListComponent: [user data]
UserCountComponent count: 3
Remember: Without shareReplay(1), each component subscription can trigger a separate HTTP request. With shareReplay(1), the first component triggers the API call and later components reuse the cached response.
