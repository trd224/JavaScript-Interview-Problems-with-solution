RxJS Advanced · Question 105

What are the implications of calling next() on a Subject from multiple async sources simultaneously?

Advanced interview-level answer with headings, lists, full examples for every concept, colorful code, and expected behavior.

Interview-Level Answer

Definition

A Subject is a shared multicast source. Calling next() from multiple async sources can create ordering, consistency, and state-management issues.

Implications

  • Emission order depends on async timing.
  • State updates can overwrite each other.
  • Debugging becomes harder.
  • Race conditions may appear.
  • Errors in subscribers can affect flow if not handled.

Best Practice

Centralize writes, avoid public Subjects, expose asObservable(), and use scan/reducer-like state updates for consistency.

Key Points

  • Multiple async next calls can cause race conditions.
  • Keep Subjects private in services.
  • Use reducer patterns for state updates.
Remember: Subject writes should be controlled and predictable.

Full Understandable Example / Examples

bad-public-subject.ts
@Injectable({ providedIn: 'root' })
export class BadStore {
  // Bad: any component can call next()
  user$ = new Subject<any>();
}
better-private-subject.ts
@Injectable({ providedIn: 'root' })
export class GoodStore {
  private userSubject = new Subject<any>();

  user$ = this.userSubject.asObservable();

  updateUser(user: any) {
    this.userSubject.next(user);
  }
}
state-with-scan.ts
const actions$ = new Subject<any>();

const state$ = actions$.pipe(
  scan((state, action) => {
    switch (action.type) {
      case 'increment':
        return { count: state.count + 1 };
      default:
        return state;
    }
  }, { count: 0 })
);

Expected Output / Behavior

Bad version allows uncontrolled writes.
Good version centralizes writes.
scan version makes state transitions predictable.
