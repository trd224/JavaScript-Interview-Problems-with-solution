What are the pitfalls of using shareReplay() and how can it cause memory leaks?

Advanced with headings, lists, full examples for every concept, colorful code, and expected behavior.

Definition

shareReplay caches emitted values and can keep source subscriptions alive. This is useful for caching, but dangerous if used with infinite or long-running streams without proper lifecycle control.

Pitfalls

  • Can retain cached values in memory.
  • Can keep long-running source active.
  • Can cause stale data if cache invalidation is not planned.
  • Can hide duplicate subscription issues if used blindly.

How To Avoid

Use refCount when appropriate, limit bufferSize, avoid using it blindly on infinite streams, and design cache invalidation for HTTP data.

Key Points

  • Great for HTTP caching, risky for infinite streams.
  • Always think about lifecycle and cache invalidation.
  • Use bufferSize and refCount intentionally.
Remember: shareReplay is powerful, but it can retain both values and subscriptions.

Full Understandable Example / Examples

risky-shareReplay.ts
// Risky: interval never completes and replay can keep source alive
const live$ = interval(1000).pipe(
  shareReplay(1)
);

const sub = live$.subscribe(value => console.log(value));

sub.unsubscribe();
Risky version may keep long-running source active.
safer-shareReplay.ts
const live$ = interval(1000).pipe(
  shareReplay({
    bufferSize: 1,
    refCount: true
  })
);

const sub = live$.subscribe(value => console.log(value));

sub.unsubscribe();
refCount version disconnects when no subscribers remain.
