197. How do you audit and reduce bundle size in a large React application?

One-Line Definition

Auditing bundle size means finding what makes the JavaScript bundle large, and reducing it by removing, splitting, lazy loading, or replacing heavy code.

Simple Meaning

In a large React application, many teams add components, routes, utilities, icons, charts, editors, date libraries, UI libraries, and third-party packages.

Over time, the final production bundle can become very large.

A large bundle increases download time, parsing time, execution time, and slows down the first page load.

Audit vs Reduce

Step Meaning
Audit bundle size Find what is inside the bundle and what is making it large.
Reduce bundle size Remove, split, lazy load, or replace heavy code.

Step 1 - Create Production Build

Always audit the production build because development builds contain extra code and are not optimized. 

npm run build
Production build gives the real optimized files that users download in the browser.

Step 2 - Use Bundle Analyzer Tools

Tool Use
Webpack Bundle Analyzer Shows a visual map of packages inside the bundle.
Source Map Explorer Shows which source files and libraries are taking space.
Vite Bundle Visualizer Useful for Vite and Rollup based React apps.
Lighthouse Shows unused JavaScript and performance impact.

Example: Source Map Explorer

npm install --save-dev source-map-explorer
{
  "scripts": {
    "build": "react-scripts build",
    "analyze": "source-map-explorer 'build/static/js/*.js'"
  }
}
npm run build
npm run analyze

Example: Webpack Bundle Analyzer

npm install --save-dev webpack-bundle-analyzer
const { BundleAnalyzerPlugin } = require('webpack-bundle-analyzer');

module.exports = {
  plugins: [
    new BundleAnalyzerPlugin()
  ]
};

Step 3 - Find Heavy Dependencies

After opening the bundle report, check which packages are taking the most space.

Common Heavy Area Possible Fix
Chart libraries Lazy load chart pages or use smaller chart packages.
Rich text editors Load editor only when user opens editor screen.
Icon libraries Import only required icons.
Date libraries Use smaller alternatives like dayjs or date-fns.
Utility libraries Import only required functions.

Step 4 - Remove Unused Dependencies

Check whether a package is still used in the project. 

npm uninstall package-name

You can also inspect dependency usage: 

npm ls package-name
Removing unused packages is one of the safest ways to reduce bundle size.

Step 5 - Use Code Splitting

Code splitting divides the application into smaller JavaScript chunks.

Instead of loading everything at the first page load, we load code only when needed.

Route-Based Code Splitting

import React, { Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

const Dashboard = React.lazy(() => import('./pages/Dashboard'));
const Reports = React.lazy(() => import('./pages/Reports'));
const Settings = React.lazy(() => import('./pages/Settings'));

function App() {
  return (
    <Suspense fallback={<h2>Loading...</h2>}>
      <Routes>
        <Route path="/" element={<Dashboard />} />
        <Route path="/reports" element={<Reports />} />
        <Route path="/settings" element={<Settings />} />
      </Routes>
    </Suspense>
  );
}

export default App;
Each route gets its own bundle chunk and loads only when that route is visited.

Step 6 - Lazy Load Heavy Components

Some components are heavy but not needed immediately.

Examples:

  • Chart dashboard
  • PDF viewer
  • Rich text editor
  • Map component
  • Admin reports module

Example

import React, { Suspense } from 'react';

const RichTextEditor = React.lazy(() => import('./RichTextEditor'));

function EditPage() {
  return (
    <Suspense fallback={<p>Loading editor...</p>}>
      <RichTextEditor />
    </Suspense>
  );
}

export default EditPage;
Heavy code should load only when the user actually needs that feature.

Step 7 - Import Only Required Code

Bad:

import _ from 'lodash';

_.debounce(search, 300);

Better:

import debounce from 'lodash/debounce';

debounce(search, 300);

Or:

import { debounce } from 'lodash-es';

debounce(search, 300);
Small and specific imports help bundlers keep the final bundle smaller.

Step 8 - Ensure Tree Shaking Works

Tree shaking removes unused exports from the final bundle.

To benefit from it:

  • Use ES module syntax.
  • Avoid CommonJS require() where possible.
  • Use tree-shakable libraries.
  • Avoid unnecessary side effects.

Good

import { formatDate } from './dateUtils';

export function formatDate(date) {
  return date.toDateString();
}

Not Ideal

const dateUtils = require('./dateUtils');

Step 9 - Optimize Assets

Bundle size is not only JavaScript. Images, fonts, SVGs, and CSS can also make the app heavy.

  • Compress images.
  • Use WebP or AVIF formats.
  • Lazy load below-the-fold images.
  • Avoid importing very large assets into JavaScript.
  • Use only required font weights.
  • Remove unused CSS.

Step 10 - Check Duplicate Packages

Large applications sometimes contain multiple versions of the same package. 

npm ls react
npm ls lodash

If duplicate versions exist, fix dependency versions or deduplicate. 

npm dedupe
Duplicate dependencies increase bundle size without adding real value.

Step 11 - Set Bundle Budget

In a large team, bundle size can grow again after optimization.

So we should set a bundle budget in CI/CD.

A bundle budget means the build should warn or fail if bundle size crosses a fixed limit.

Example Concept

Maximum main bundle size: 250 KB
Maximum route chunk size: 150 KB
Maximum total JS size: 800 KB

Step 12 - Compare Before and After

Always compare bundle reports before and after optimization.

Before After
Main bundle: 900 KB Main bundle: 420 KB
Charts loaded on first page Charts lazy loaded
Full lodash imported Only debounce imported
All routes loaded initially Routes split into chunks

Large React App Optimization Checklist

  • Create production build.
  • Analyze with bundle analyzer.
  • Find largest packages.
  • Remove unused dependencies.
  • Replace heavy libraries where possible.
  • Apply route-based code splitting.
  • Lazy load heavy components.
  • Import only required functions.
  • Enable tree shaking.
  • Optimize images and fonts.
  • Remove duplicate packages.
  • Set bundle size budget in CI/CD.

Real Project Example

Suppose our admin application has Dashboard, Reports, Settings, Charts, and Rich Text Editor modules.

If all modules load at startup, first page load becomes slow.

The user may open only Dashboard, but browser still downloads Reports, Charts, Settings, and Editor code.

Better approach:

  • Load Dashboard initially.
  • Lazy load Reports route.
  • Lazy load Chart component inside Reports.
  • Lazy load Rich Text Editor only when editing starts.
  • Remove unused utilities and old packages.
This reduces initial JavaScript and improves first page load.

Important Interview Points

  • Audit production bundle, not development bundle.
  • Use bundle analyzer tools to find large packages.
  • Remove unused and duplicate dependencies.
  • Use route-based code splitting.
  • Lazy load heavy features.
  • Use tree shaking and ES modules.
  • Replace heavy libraries with smaller alternatives.
  • Optimize assets and fonts.
  • Set bundle budgets to prevent future bundle growth.

Short Interview Answer

In a large React application, I audit bundle size by creating a production build and using tools like Webpack Bundle Analyzer, Source Map Explorer, or Vite Visualizer. Then I reduce it by removing unused dependencies, fixing duplicate packages, using tree shaking, applying route-based code splitting, lazy loading heavy components, importing only required functions, optimizing assets, and setting bundle budgets in CI/CD.

One-Line Interview Answer

Audit bundle size with analyzer tools, then reduce it using tree shaking, code splitting, lazy loading, optimized imports, dependency cleanup, and bundle budgets.
