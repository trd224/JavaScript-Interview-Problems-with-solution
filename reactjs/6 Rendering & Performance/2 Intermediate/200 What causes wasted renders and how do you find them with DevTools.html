200. What causes wasted renders and how do you find them with DevTools?

One-Line Definition

A wasted render happens when a component renders again even though its visible output has not changed.

What Is a Wasted Render?

React rendering itself is not bad.

React is designed to re-render when state or props change.

The problem occurs when a component re-renders but produces exactly the same UI as before.

React spent time rendering, but the user sees no actual change.

Simple Example

function Parent() {

  const [count, setCount] = useState(0);

  return (
    <>
      <button onClick={() => setCount(count + 1)}>
        Increment
      </button>

      <Child />
    </>
  );
}

function Child() {
  console.log('Child Rendered');

  return <h2>Child Component</h2>;
}

Every time count changes:

  • Parent renders.
  • Child renders.

But Child does not use count.

Child rendered unnecessarily. This is a wasted render.

Common Causes of Wasted Renders

Cause Description
Parent Re-render Child re-renders because parent rendered.
New Object References Objects created on every render appear different.
New Array References Arrays recreated on every render trigger updates.
Inline Functions New function reference created every render.
Context Updates All consumers may re-render when context changes.
Incorrect State Placement State stored too high causes large tree re-renders.

Cause 1 - Parent Re-render

function Parent() {

  const [count, setCount] = useState(0);

  return (
    <>
      <button onClick={() => setCount(count + 1)}>
        Increment
      </button>

      <Child />
    </>
  );
}

Whenever Parent renders, Child renders too.

Fix

const Child = React.memo(function Child() {
  return <h2>Child Component</h2>;
});
React.memo skips rendering if props have not changed.

Cause 2 - Inline Functions

function Parent() {

  return (
    <Child
      onClick={() => console.log('Clicked')}
    />
  );
}

Every render creates a brand-new function.

React sees a new prop value.

Child renders again.

Fix With useCallback

const handleClick = useCallback(() => {
  console.log('Clicked');
}, []);

<Child onClick={handleClick} />
Function reference remains stable between renders.

Cause 3 - New Object References

function Parent() {

  const settings = {
    theme: 'dark'
  };

  return (
    <Child settings={settings} />
  );
}

settings is recreated every render.

React sees a new object reference.

Fix With useMemo

const settings = useMemo(() => {
  return {
    theme: 'dark'
  };
}, []);

We are using useMemo here because useCallback is for functions, while useMemo is for values (objects, arrays, calculations). Your example creates an object, not a function:

Cause 4 - New Array References

const users = [
  'Rizwan',
  'Ali',
  'Ahmed'
];

<UserList users={users} />

A new array is created on every render.

Fix

const users = useMemo(() => {
  return [
    'Rizwan',
    'Ali',
    'Ahmed'
  ];
}, []);

Cause 5 - Context Updates

const value = {
  user,
  theme
};

<AppContext.Provider value={value}>
  <Dashboard />
</AppContext.Provider>

If value changes:

  • All context consumers can re-render.
Large contexts often cause unnecessary re-renders.

Possible Fixes

  • Split context into smaller contexts.
  • Memoize context value.
  • Use selector-based state management.

Cause 5 - Context Updates

Problem: When a Context value changes, every component consuming that Context can re-render, even if it only uses a small part of the data. 
const value = {
  user,
  theme
};

<AppContext.Provider value={value}>
  <Dashboard />
</AppContext.Provider>

If value changes, all context consumers can re-render. Large contexts often cause unnecessary re-renders.

Possible Fixes

  • Split Context Into Smaller Contexts
  • Memoize Context Value
  • Use Selector-Based State Management

1. Split Context Into Smaller Contexts

Instead of storing unrelated data inside one large context, create separate contexts.

Problem Example

const AppContext = createContext();

function AppProvider({ children }) {

  const [user, setUser] = useState('Rizwan');
  const [theme, setTheme] = useState('dark');

  return (
    <AppContext.Provider value={{ user, theme }}>
      {children}
    </AppContext.Provider>
  );

}
function UserProfile() {

  const { user } = useContext(AppContext);

  return <h1>{user}</h1>;

}
function ThemeButton() {

  const { theme } = useContext(AppContext);

  return <button>{theme}</button>;

}
If theme changes, both components can re-render because they are subscribed to the same context.

Better Solution

const UserContext = createContext();
const ThemeContext = createContext();

function AppProvider({ children }) {

  const [user, setUser] = useState('Rizwan');
  const [theme, setTheme] = useState('dark');

  return (
    <UserContext.Provider value={user}>
      <ThemeContext.Provider value={theme}>
        {children}
      </ThemeContext.Provider>
    </UserContext.Provider>
  );

}
function UserProfile() {

  const user = useContext(UserContext);

  return <h1>{user}</h1>;

}
function ThemeButton() {

  const theme = useContext(ThemeContext);

  return <button>{theme}</button>;

}
Now changing theme only re-renders theme consumers. User consumers remain unaffected.

2. Memoize Context Value

Every render creates a new object reference. 

const value = {
  user,
  theme
};
Even if user and theme remain the same, the object reference is new on every render.

Fix With useMemo

const value = useMemo(() => {
  return {
    user,
    theme
  };
}, [user, theme]);

return (
  <AppContext.Provider value={value}>
    {children}
  </AppContext.Provider>
);

Short Version

const value = useMemo(
  () => ({ user, theme }),
  [user, theme]
);
If the parent re-renders but user and theme are unchanged, React reuses the previous object reference.
Important: This optimization only helps when context data has not changed. If user or theme changes, consumers should re-render.

3. Selector-Based State Management

Instead of subscribing to the entire state object, subscribe only to the piece of state you need.

Redux Example

const user = useSelector(
  state => state.user
);

const theme = useSelector(
  state => state.theme
);
function UserProfile() {

  const user = useSelector(
    state => state.user
  );

  return <h1>{user}</h1>;

}
function ThemeButton() {

  const theme = useSelector(
    state => state.theme
  );

  return <button>{theme}</button>;

}
If theme changes, only ThemeButton re-renders. UserProfile does not re-render because its selected state did not change.

Zustand Example

const user = useStore(
  state => state.user
);

const theme = useStore(
  state => state.theme
);
Redux and Zustand are efficient because components subscribe to specific slices of state instead of the entire state object.

Interview Summary

Split Context: Create multiple contexts so unrelated consumers do not re-render.
Memoize Context Value: Use useMemo to preserve object references between renders.
Selector-Based State Management: Subscribe only to required state so unrelated changes do not trigger re-renders.

Best Order of Optimization

1. Split Contexts
2. Memoize Context Value
3. Redux / Zustand Selectors

How To Find Wasted Renders Using React DevTools

Step 1 - Open React DevTools

Install React Developer Tools extension.

Open browser DevTools.

Step 2 - Open Profiler Tab

Select: 

Profiler

Then click: 

Start Profiling

Step 3 - Interact With The Application

Perform the action that feels slow.

Examples:

  • Type in a search box.
  • Open a modal.
  • Click a button.
  • Navigate between pages.

Step 4 - Stop Profiling

React DevTools will show:

  • Rendered components.
  • Render duration.
  • Commit duration.
  • Why components rendered.

Profiler Example

Component Render Count
Parent 10
Child 10

If Child does not use changed data:

Those 10 renders are likely wasted renders.

Use "Why Did This Render?"

Modern React DevTools can show: 

Why did this render?

Example output:

onClick

Previous:
ƒ()

Next:
ƒ()

This tells us that a new function reference triggered the render.

This feature makes finding wasted renders much easier.

Flamegraph View

The Flamegraph shows:

  • Which components rendered.
  • Which components took the most time.
  • How rendering propagated through the tree.
Large bars usually indicate expensive renders.

Real Project Example

Suppose our Aadhaar Authentication page contains:

  • Aadhaar input.
  • Consent section.
  • Authentication options.
  • Device information.
  • Error popup.

Typing one digit into Aadhaar input may trigger:

  • Consent section render.
  • Device panel render.
  • Error popup render.
  • Authentication list render.
Most of those renders are wasted because their UI did not change.

Profiler helps identify them.

Possible optimizations:

  • React.memo
  • useMemo
  • useCallback
  • Move state closer to where it is used
  • Split large components

Important Interview Points

  • A wasted render occurs when a component renders without changing visible output.
  • Common causes are parent renders, new objects, new arrays, inline functions, and context updates.
  • React DevTools Profiler helps detect wasted renders.
  • "Why did this render?" explains render causes.
  • React.memo, useMemo, and useCallback help reduce wasted renders.
  • State placement affects render frequency.
  • Always measure before optimizing.

Short Interview Answer

Wasted renders occur when components re-render even though their UI does not change. Common causes include parent re-renders, inline functions, recreated objects and arrays, context updates, and poor state placement. React DevTools Profiler helps identify these renders by showing render counts, render durations, and the reasons components rendered.
