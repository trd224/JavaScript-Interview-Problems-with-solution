201. What causes a "Can't perform a React state update on an unmounted component" warning?

One-Line Definition

This warning occurs when a component has already been removed from the UI, but some asynchronous code still tries to update its state.

What Does "Unmounted Component" Mean?

A component is mounted when it exists in the UI.

A component is unmounted when React removes it from the UI.

Component Created
↓
Mounted
↓
User Navigates Away
↓
Component Removed
↓
Unmounted

Why Does The Warning Happen?

Usually some asynchronous task is still running:

  • API request
  • setTimeout
  • setInterval
  • WebSocket
  • Event listener
  • Subscription

The component gets removed before that task finishes.

When the task completes, it tries to call: 

setState(...)

But the component no longer exists.

React warns because updating state on a removed component may cause memory leaks and wasted work.

Most Common Example - API Request

function UserProfile() {

  const [user, setUser] = useState(null);

  useEffect(() => {

    fetch('/api/user')
      .then(res => res.json())
      .then(data => {
        setUser(data);
      });

  }, []);

  return <div>{user?.name}</div>;

}

Suppose:

  1. Component mounts.
  2. API request starts.
  3. User navigates away.
  4. Component unmounts.
  5. API response arrives later.
  6. setUser() runs.
React may show: "Can't perform a React state update on an unmounted component".

Another Example - setTimeout

function Example() {

  const [message, setMessage] = useState('');

  useEffect(() => {

    setTimeout(() => {
      setMessage('Hello');
    }, 5000);

  }, []);

  return <h2>{message}</h2>;

}

If the component disappears before 5 seconds:

  • Timeout still runs.
  • setMessage executes.
  • Component no longer exists.
React may display the warning.

Another Example - setInterval

useEffect(() => {

  const id = setInterval(() => {

    setCount(c => c + 1);

  }, 1000);

}, []);

If the interval is not cleared:

  • Interval continues running.
  • State updates continue.
  • Memory leak occurs.

Correct Solution - Cleanup Function

React provides cleanup functions inside useEffect. 

useEffect(() => {

  const id = setInterval(() => {

    setCount(c => c + 1);

  }, 1000);

  return () => {

    clearInterval(id);

  };

}, []);
Cleanup runs when the component unmounts.

Cleaning Up setTimeout

useEffect(() => {

  const timeoutId = setTimeout(() => {

    setMessage('Hello');

  }, 5000);

  return () => {

    clearTimeout(timeoutId);

  };

}, []);

Cleaning Up Event Listeners

useEffect(() => {

  function handleResize() {
    console.log(window.innerWidth);
  }

  window.addEventListener(
    'resize',
    handleResize
  );

  return () => {

    window.removeEventListener(
      'resize',
      handleResize
    );

  };

}, []);
Always remove listeners when the component unmounts.

Cleaning Up API Requests

Modern fetch supports AbortController. 

useEffect(() => {
  const controller = new AbortController();

  fetch('/api/user', { signal: controller.signal })
    .then(res => res.json())
    .then(data => setUser(data))
    .catch(error => console.log(error));

  return () => controller.abort();
}, []);
The request is cancelled when the component unmounts.

React 18 Note

This warning was very common in React 17 and earlier.

React 18 removed many unnecessary versions of this warning because some cases were harmless.

However, cleanup is still important because memory leaks, subscriptions, timers, and network requests can continue running unnecessarily.

Real Project Example

Suppose our Aadhaar Authentication page:

  • Starts OTP request.
  • Starts SignalR connection.
  • Starts device capture process.

User closes the modal immediately.

The component unmounts.

Later:

  • OTP response arrives.
  • SignalR message arrives.
  • Device capture callback fires.

If they call: 

setOtp(...)
setLoader(...)
setUser(...)

after the component is gone, React may show this warning.

Always clean up subscriptions, timers, and async operations when the component unmounts.

Important Interview Points

  • The warning happens when state is updated after component unmount.
  • Usually caused by asynchronous operations.
  • Common sources are API calls, timers, intervals, subscriptions, and event listeners.
  • useEffect cleanup functions prevent these issues.
  • AbortController can cancel fetch requests.
  • React 18 reduced unnecessary versions of this warning.
  • Cleanup is still important to avoid memory leaks.

Short Interview Answer

This warning occurs when asynchronous code such as an API request, timer, subscription, or event listener tries to update state after the component has already been unmounted. The solution is to clean up those operations using the cleanup function returned from useEffect.

One-Line Interview Answer

The warning occurs when asynchronous code calls setState after a component has been removed from the UI.
