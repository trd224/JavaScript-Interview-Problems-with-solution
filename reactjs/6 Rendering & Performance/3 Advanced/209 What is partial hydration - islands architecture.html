209. What is Partial Hydration / Islands Architecture?

One-Line Answer

Partial Hydration means only the interactive parts of a server-rendered page are hydrated with JavaScript. Islands Architecture means the page is mostly static HTML, and only small interactive sections behave like JavaScript-powered islands.

First Understand The Problem

In traditional SSR React apps, the server sends HTML first, then React hydrates the page in the browser.

Server Sends HTML
↓ Browser Shows Page
↓ JavaScript Downloads
↓ React Hydrates App
↓ Whole App Becomes Interactive
Problem: even static parts like headings, paragraphs, footer, and article text may be included in the JavaScript/hydration process, although they do not need interactivity.

What Partial Hydration Does

Partial Hydration says: do not hydrate the whole page. Hydrate only the parts that need JavaScript.
Static HTML
↓ No Hydration Needed

Interactive Gallery
↓ Hydrate This

Add To Cart Button
↓ Hydrate This

Newsletter Form
↓ Hydrate This

What Is Islands Architecture?

Islands Architecture is a way to build pages where most of the page is static HTML, and interactive components are separate islands.

Static Ocean HTML
↓ Island 1: Image Gallery
↓ Island 2: Add To Cart
↓ Island 3: Newsletter Form
The static page is the ocean. Interactive components are islands.

Visual Example

Static Header
Static Product Text
🏝️ Interactive Gallery
🏝️ Add To Cart Button
Static Reviews
🏝️ Newsletter Form
Static Footer

Traditional Full Hydration

Whole App HTML
↓ Download Big JS Bundle
↓ Hydrate Whole App
↓ Everything Managed By JavaScript
Even static sections may pay hydration cost.

Partial Hydration

Whole Page HTML Visible
↓ Download JS Only For Interactive Islands
↓ Hydrate Gallery
↓ Hydrate Cart Button
↓ Hydrate Newsletter
↓ Static Sections Stay Static
Less JavaScript, less hydration work, faster page interactivity.

Code Style Example

This is a conceptual example. Frameworks like Astro use this style very clearly.

ProductPage.astro style idea
<Header />

<ProductContent product={product} />

<ImageGallery client:visible images={product.images} />

<AddToCart client:load productId={product.id} />

<Reviews reviews={reviews} />

<NewsletterForm client:idle />

<Footer />

Here only components with client directives become interactive islands.

DirectiveMeaning
client:loadLoad/hydrate immediately
client:visibleLoad/hydrate when visible
client:idleLoad/hydrate when browser is idle

React Conceptual Example

Concept
function ProductPage(){
  return (

Plain React SSR normally hydrates the React tree. Islands architecture needs framework-level support or special architecture to hydrate only selected components.

Why It Is Useful?

  • Less JavaScript sent to browser
  • Less hydration work
  • Faster Time To Interactive
  • Better performance on slow devices
  • Great for content-heavy websites
  • Static parts stay simple and fast

Where It Is Commonly Used?

Relation With React Server Components

React Server Components are related to this idea because they reduce client-side JavaScript.

Server Component
↓ Runs On Server
↓ No Client JS Needed

Client Component
↓ Runs In Browser
↓ Needs JS And Hydration
In Next.js App Router, Server Components are static/server-rendered by default, and Client Components are the interactive parts. This feels similar to islands, but it is not exactly the same architecture as Astro islands.

Partial Hydration vs Progressive Hydration vs Selective Hydration

ConceptMeaning
Partial HydrationHydrate only interactive parts
Islands ArchitectureMostly static HTML with isolated interactive islands
Progressive HydrationHydrate page gradually in stages
Selective HydrationHydrate important/user-interacted part first

Easy Difference

Progressive Hydration
↓ When should parts hydrate?

Selective Hydration
↓ Which part should hydrate first?

Partial Hydration
↓ Which parts need hydration at all?

Very Simple Example

Full Hydration

Header hydrated

Image hydrated

Image static

Only the button needs JavaScript, so only the button becomes an island.

Important Point

Partial hydration is not about delaying hydration of the whole app. It is about avoiding hydration for parts that never needed JavaScript.

When To Use It?

  • Blogs with a few interactive widgets
  • Marketing pages
  • Documentation websites
  • Product pages
  • News websites
  • Content-heavy pages with small interactive sections

When It May Not Help Much?

  • Complex dashboards
  • Apps where almost every section is interactive
  • Real-time collaborative apps
  • Large SPAs where client state controls most UI

Interview Answer

Partial Hydration is an SSR optimization technique where only the interactive parts of a page are hydrated with JavaScript, while static parts remain plain HTML. Islands Architecture is a design pattern based on this idea: the page is mostly static HTML, with small isolated interactive components called islands. This reduces JavaScript bundle size and hydration cost, improving performance and Time To Interactive. It is commonly associated with frameworks like Astro, and it is related to but different from progressive hydration and selective hydration.

Easy Way To Remember

Static Page = Ocean
↓ Interactive Components = Islands
↓ Only Islands Need JavaScript
↓ Only Islands Hydrate
↓ Partial Hydration / Islands Architecture

Final Summary

Partial Hydration asks: "Which parts need JavaScript?"

Answer: Only interactive islands.
