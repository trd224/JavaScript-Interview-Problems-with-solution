207. What is Selective Hydration?

One-Line Answer

Selective Hydration is a React 18 feature for SSR pages. After server HTML is shown and JavaScript downloads, React starts hydration automatically, but it can prioritize the part of the page that the user interacts with first.

First Understand SSR

In SSR, React renders HTML on the server and sends that HTML to the browser.

React Component On Server
↓ Server Generates HTML
↓ Browser Receives HTML
↓ User Can See Page Quickly

But this HTML is only visible content. It is not fully interactive yet.

SSR gives fast visible HTML, but JavaScript is still needed to attach React events like onClick, onChange, onSubmit, etc.

What Is Hydration?

Hydration means React connects JavaScript logic with the HTML that already came from the server.

Server HTML Already Exists
↓ React Loads In Browser
↓ React Reuses Existing HTML
↓ React Attaches Event Handlers
↓ Page Becomes Interactive
Hydration does not mean React creates the whole DOM again. React tries to reuse the server-rendered HTML and attach behavior to it.

Normal Hydration Flow

Server Sends HTML
↓ Browser Shows HTML
↓ JavaScript Bundle Downloads
↓ Hydration Starts Automatically
↓ React Attaches Events
↓ Components Become Interactive
Your correction is important: hydration starts automatically after JavaScript downloads. React does not wait for the user to click before starting hydration.

Problem With Big SSR Pages

Imagine an ecommerce page:

Navbar
↓ Search Box
↓ Filters
↓ ProductList
↓ Reviews
↓ Comments
↓ Footer

Some parts are small, but some parts are heavy.

ComponentHydration CostImportance
Navbar10msHigh
Search Box20msHigh
ProductList150msHigh
Reviews700msMedium
Comments2000msLow initially
Footer10msLow
If React spends too much time hydrating a heavy low-priority section, important UI like Search Box can feel delayed.

Old Style Thinking

In older SSR mental model, hydration feels like one big process.

Start Hydration
↓ Hydrate Big Page
↓ Wait Until Important Parts Are Ready
↓ Then User Interaction Works
The issue is not that the page is invisible. The page is visible. The issue is that some visible parts may not be interactive yet.

What Selective Hydration Changes

Selective Hydration allows React to treat parts of the page separately and hydrate higher-priority parts first.
Hydration Already Running
↓ User Clicks Search Box
↓ React Detects This Interaction
↓ React Gives Search/Product Area Higher Priority
↓ That Area Hydrates First
↓ User Interaction Is Handled Faster
↓ Other Areas Hydrate Later

Very Important Correction

Wrong

User Clicks Search Box
↓ Hydration Starts

Correct

JavaScript Downloads
↓ Hydration Starts Automatically
↓ User Clicks Search Box During Hydration
↓ React Prioritizes Search Box Hydration
So the click does not start hydration. The click changes hydration priority.

Example With Suspense Boundaries

Selective Hydration works best when the page is split using Suspense boundaries.

Page.jsx
function Page(){
  return (
    <>
      <Navbar />

      <Suspense fallback={<SearchSkeleton />}>
        <ProductSearch />
      </Suspense>

      <Suspense fallback={<ListSkeleton />}>
        <ProductList />
      </Suspense>

      <Suspense fallback={<CommentsSkeleton />}>
        <Comments />
      </Suspense>
    </>
  );
}

Each Suspense boundary gives React a separable area.

Boundary 1: ProductSearch
Boundary 2: ProductList
Boundary 3: Comments
↓ React Can Hydrate Them With Different Priorities

Real Scenario

SSR HTML Arrives
↓ Browser Shows Full Page
↓ JS Bundle Downloads
↓ React Starts Hydration
↓ Comments Section Is Heavy
↓ User Clicks Product Search
↓ React Pauses Lower Priority Work
↓ React Hydrates ProductSearch First
↓ Search onChange/onClick Works
↓ React Continues Hydrating Comments Later

What Happens To The User Click Before Hydration?

React can capture and replay some events during hydration.

User Clicks A Not-Yet-Hydrated Button
↓ React Captures The Event
↓ React Hydrates That Boundary With Priority
↓ React Replays/Handles The Event
This helps users feel that the app responds faster, even when the whole page is not fully hydrated yet.

Event Replay Simple Example

Search.jsx
function ProductSearch(){
  function handleChange(e){
    console.log(e.target.value);
  }

  return <input onChange={handleChange} placeholder="Search product" />;
}

If the user types while hydration is still happening:

User Types In Search
↓ React Knows Search Area Is Important
↓ React Hydrates ProductSearch First
↓ onChange Handler Gets Attached
↓ Typing Is Handled

Selective Hydration And Streaming SSR

Selective Hydration is very powerful with Streaming SSR.

Server Does Not Wait For Whole Page
↓ Server Streams HTML In Pieces
↓ Browser Shows Available Parts Early
↓ JavaScript Downloads
↓ React Hydrates Available Boundaries
↓ Important Boundaries Can Hydrate First
Streaming SSR helps send HTML in chunks. Selective Hydration helps make important chunks interactive earlier.

Streaming SSR Example

Server Page
function App(){
  return (
    <html>
      <body>
        <Navbar />

        <Suspense fallback={<LoadingProducts />}>
          <Products />
        </Suspense>

        <Suspense fallback={<LoadingComments />}>
          <Comments />
        </Suspense>
      </body>
    </html>
  );
}
Navbar HTML Streams First
↓ Products HTML Streams When Ready
↓ Comments HTML Streams Later
↓ React Can Hydrate Ready/Important Parts Selectively

Normal Hydration vs Selective Hydration

Normal HydrationSelective Hydration
Hydration feels like one big processHydration can be prioritized by area
Heavy parts may delay important interactionsImportant or clicked parts hydrate first
Less control over interaction priorityBetter priority handling
User may wait for interactivityUser gets key interactivity sooner
Not ideal for large SSR pagesBetter for large SSR + Suspense pages

Selective Hydration vs Lazy Loading

ConceptMeaning
Lazy LoadingLoad code/data later when needed
HydrationAttach React behavior to server HTML
Selective HydrationChoose which server-rendered part becomes interactive first
Do not confuse selective hydration with lazy loading. Lazy loading is about loading resources. Selective hydration is about prioritizing interactivity of already rendered SSR HTML.

Selective Hydration vs Time Slicing

FeatureWork TypeMain Idea
Time SlicingRendering workBreak render work into chunks and prioritize urgent updates
Selective HydrationHydration workPrioritize which SSR part becomes interactive first
Time Slicing
↓ Which render work should React finish first?

Selective Hydration
↓ Which server-rendered area should React hydrate first?

Why Suspense Is Important Here

Suspense boundaries give React natural checkpoints.

Page
↓ Suspense Boundary A
↓ Suspense Boundary B
↓ Suspense Boundary C
Because boundaries are separate, React can hydrate Boundary A before Boundary C if Boundary A is more important.

Without Suspense Boundaries

If the page is not split into good boundaries, React has less flexibility to hydrate separate areas efficiently.
One Large App Tree
↓ Harder To Prioritize Smaller Sections

With Suspense Boundaries

Smaller sections make prioritization easier.
Navbar
Product Search Boundary
Product List Boundary
Comments Boundary
↓ React Can Prioritize Important Boundary

Detailed Browser Timeline

1. User Requests Page
↓ 2. Server Renders React To HTML
↓ 3. Browser Receives HTML
↓ 4. Browser Paints HTML
↓ 5. User Can See Content
↓ 6. JavaScript Downloads
↓ 7. React Starts Hydration
↓ 8. User Interacts With ProductSearch
↓ 9. React Prioritizes ProductSearch Boundary
↓ 10. ProductSearch Becomes Interactive
↓ 11. Remaining Boundaries Hydrate Later

Why This Improves UX

Users usually do not interact with every part of the page immediately.

User Opens Page
↓ Most Likely Uses Navbar/Search/Button First
↓ Comments/Footer Are Less Urgent
↓ React Hydrates Urgent UI First
This improves perceived performance because the part the user needs becomes interactive earlier.

Before React 18 Mental Model

HTML Visible
↓ Wait For Hydration
↓ Page Becomes Interactive

React 18 Mental Model

HTML Visible
↓ Hydration Starts
↓ Important Areas Become Interactive Earlier
↓ Less Important Areas Continue Later

Where Do We See This?

  • React 18 SSR apps
  • Streaming SSR
  • Suspense-heavy pages
  • Next.js App Router style SSR/streaming architectures
  • Large pages where some sections are heavy

Full Easy Example

ProductSearch.jsx
function ProductSearch(){
  const [query,setQuery] = useState('');

  return (
    <input
      value={query}
      onChange={e => setQuery(e.target.value)}
      placeholder="Search products"
    />
  );
}

When this component comes from SSR, the input may be visible before hydration completes.

Input Visible
↓ JS Hydration Running
↓ User Types
↓ React Prioritizes This Input Area
↓ onChange Becomes Ready Faster

Interview Answer

Selective Hydration is a React 18 SSR feature that lets React prioritize hydration of specific parts of a server-rendered page. After the browser receives SSR HTML and downloads JavaScript, hydration starts automatically. During hydration, if the user interacts with a part of the page, React can give that Suspense boundary higher priority, hydrate it first, and handle the event sooner. This improves responsiveness because important UI becomes interactive before less important or heavy sections finish hydrating.

Easy Way To Remember

SSR HTML
↓ Visible But Not Fully Interactive
↓ JS Downloads
↓ Hydration Starts Automatically
↓ User Interacts With One Area
↓ React Hydrates That Area First
↓ Other Areas Later
↓ Selective Hydration

Final Short Summary

Selective Hydration does not mean hydration starts after click. Hydration starts automatically. Selective Hydration means React can change priority and finish hydration of the interacted/important part first.
