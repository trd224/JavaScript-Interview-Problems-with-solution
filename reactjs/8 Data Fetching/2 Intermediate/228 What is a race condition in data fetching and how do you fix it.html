228. What is a race condition in data fetching and how do you fix it?

One-Line Definition

A race condition in data fetching happens when multiple API requests run at the same time and an older response finishes after a newer response, causing old data to overwrite new data.

Simple Meaning

In React, users can change input, filters, tabs, routes, or search text quickly.

Each change may start a new API request.

The problem happens when responses do not come back in the same order they were sent.

Simple Example

User searches quickly:

Search: "r"
↓
API Request 1 Started

Search: "ri"
↓
API Request 2 Started

Request 2 Finishes First
↓
UI Shows "ri" Results

Request 1 Finishes Later
↓
Old "r" Results Overwrite UI
Now the UI shows old results even though the latest search is "ri".

Problem Code

useEffect(() => {
  fetch(`/api/users?search=${search}`)
    .then(res => res.json())
    .then(data => setUsers(data));
}, [search]);

Every time search changes, a new request starts.

But old requests are still allowed to finish and update state.

Why Is This a Race Condition?

Because multiple requests are racing to update the same state.

Request Search Value Finish Time Result
Request 1 r Slow Finishes last and overwrites UI
Request 2 ri Fast Finishes first and shows correct data

Fix 1 - Use AbortController

AbortController cancels the previous request when a new request starts or when the component unmounts. 

useEffect(() => {
  const controller = new AbortController();

  fetch(`/api/users?search=${search}`, { signal: controller.signal })
    .then(res => res.json())
    .then(data => setUsers(data))
    .catch(error => {
      if (error.name !== 'AbortError') console.log(error);
    });

  return () => controller.abort();
}, [search]);
When search changes, React runs cleanup first, aborts the old request, then starts the new request.

How Cleanup Fixes It

search = "r"
↓
Request 1 Starts
↓
search = "ri"
↓
Cleanup Runs
↓
Request 1 Aborted
↓
Request 2 Starts
↓
Only Latest Request Updates State

Fix 2 - Ignore Old Responses

Sometimes an API request cannot be cancelled.

In that case, we can ignore old responses. 

useEffect(() => {
  let ignore = false;

  async function loadUsers() {
    const res = await fetch(`/api/users?search=${search}`);
    const data = await res.json();

    if (!ignore) setUsers(data);
  }

  loadUsers();

  return () => {
    ignore = true;
  };
}, [search]);
Old request may finish, but it cannot update state after cleanup sets ignore to true.

Fix 3 - Track Latest Request Id

Another approach is to keep a request id and update state only for the latest request. 

const requestIdRef = useRef(0);

useEffect(() => {
  const requestId = ++requestIdRef.current;

  async function loadUsers() {
    const res = await fetch(`/api/users?search=${search}`);
    const data = await res.json();

    if (requestId === requestIdRef.current) setUsers(data);
  }

  loadUsers();
}, [search]);
This ensures only the latest request is allowed to update state.

Fix 4 - Use React Query / SWR

For real applications, libraries like React Query or SWR handle many data-fetching problems automatically.

  • Caching
  • Refetching
  • Loading state
  • Error state
  • Stale data handling
  • Request deduplication
const { data, isLoading } = useQuery({
  queryKey: ['users', search],
  queryFn: () => fetchUsers(search)
});
React Query helps prevent many manual data-fetching bugs in large applications.

Real Project Example

Suppose our Aadhaar Authentication dashboard has a department dropdown.

User quickly changes department:

Department A Selected
↓
Request A Started

Department B Selected
↓
Request B Started

Request B Finishes First
↓
UI Shows Department B Data

Request A Finishes Later
↓
Old Department A Data Overwrites UI
This is a race condition because old response updates the latest screen.

Fix it by aborting the old request: 

useEffect(() => {
  const controller = new AbortController();

  fetch(`/api/audit?department=${departmentId}`, { signal: controller.signal })
    .then(res => res.json())
    .then(data => setAuditData(data))
    .catch(error => {
      if (error.name !== 'AbortError') console.log(error);
    });

  return () => controller.abort();
}, [departmentId]);

Common Places Race Conditions Happen

Place Example
Search input User types quickly
Filter dropdown User changes filter multiple times
Tabs User switches tabs quickly
Route params User navigates between details pages
Pagination User clicks next pages quickly

Important Interview Points

  • Race condition happens when old async response overwrites newer state.
  • It commonly happens with search, filters, tabs, pagination, and route changes.
  • AbortController can cancel old fetch requests.
  • useEffect cleanup runs before the next effect and on unmount.
  • Ignoring old responses is useful when request cancellation is not available.
  • Request id tracking ensures only the latest request updates state.
  • React Query and SWR solve many data-fetching race issues.

Short Interview Answer

A race condition in data fetching happens when multiple requests are active and an older response arrives after a newer one, causing stale data to overwrite the latest state. We fix it by cancelling old requests with AbortController, ignoring stale responses in useEffect cleanup, tracking the latest request id, or using libraries like React Query or SWR.

One-Line Interview Answer

A data-fetching race condition happens when an old request finishes after a new request and overwrites fresh data; fix it by aborting or ignoring stale requests.
