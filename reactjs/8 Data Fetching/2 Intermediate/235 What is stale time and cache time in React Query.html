235. What is staleTime and cacheTime in React Query?

One-Line Definition

staleTime determines how long data is considered fresh, while cacheTime determines how long unused cached data remains in memory before being deleted.

Simple Meaning

staleTime
=
Freshness Timer

cacheTime
=
Deletion Timer

Example Code

useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers,
  staleTime: 60000,
  cacheTime: 300000
});

Meaning:

  • staleTime: 60000 means data is fresh for 1 minute.
  • cacheTime: 300000 means unused cache stays for 5 minutes.

Most Important Rule

When staleTime expires, data becomes stale. It does not mean cache is deleted and it does not mean API is called immediately.

Possibility 1 - User Returns Before staleTime Expires

Example: 

12:00 → API Call
12:00 - 12:01 → Data Fresh
12:00:30 → User Returns

Result:

Data Is Still Fresh
↓
Show Cached Data
↓
No API Call
Because staleTime has not expired, React Query trusts the cache and does not refetch.

Possibility 2 - User Returns After staleTime But Before cacheTime

Example: 

12:00 → API Call
12:01 → Data Becomes Stale
12:04 → User Returns
cacheTime has not expired yet

Result:

Cache Still Exists
↓
Show Cached Data Immediately
↓
Background API Refetch
↓
Fresh Data Arrives
↓
UI Updates Again
Yes, this can cause two UI updates: first cached data, then fresh data.

This is good for data where showing slightly old data is acceptable.

  • Department list
  • Categories
  • Country list
  • Product list
  • Static master data

But this can be bad UX for critical/live data.

  • Bank balance
  • Wallet amount
  • Payment status
  • Stock price
  • Live score

Possibility 3 - User Does Not Return Before cacheTime Expires

Example: 

12:00 → API Call
12:01 → Data Becomes Stale
12:01 → User Leaves
12:06 → cacheTime Expires
12:10 → User Returns

Result:

cacheTime Expired
↓
Cache Deleted
↓
User Returns
↓
No Cached Data
↓
Show Loading
↓
API Call
↓
Show Fresh Data
Cache expires only if no component uses that query until cacheTime finishes.

Possibility 4 - User Comes Back Before cacheTime Expires

Example: 

12:01 → User Leaves
12:04 → User Comes Back
cacheTime = 5 minutes

Result:

Cache Still Exists
↓
Query Becomes Active Again
↓
cacheTime Timer Stops / Resets
↓
Cached Data Can Be Used
Because the user came back before cacheTime expired, the cache is not deleted.

Possibility 5 - You Do Not Want Old Data To Show

For critical data, showing cached data first can be confusing.

In that case, you can choose a strategy where the user waits for fresh data. 

useQuery({
  queryKey: ['balance'],
  queryFn: fetchBalance,
  staleTime: 0,
  refetchOnMount: 'always'
});

UX:

User Opens Page
↓
Show Loading / Skeleton
↓
API Call
↓
Show Fresh Data
Use this type of behavior for sensitive or fast-changing data.

Possibility 6 - You Want To Avoid UI Fluctuation

If old cached data and fresh data are very different, UI can flicker or fluctuate.

For stable data, increase staleTime. 

useQuery({
  queryKey: ['departments'],
  queryFn: fetchDepartments,
  staleTime: 30 * 60 * 1000
});

Meaning:

For 30 Minutes
↓
Data Is Fresh
↓
No Background Refetch On Return
This avoids unnecessary UI changes for rarely-changing data.

Memory Trick

staleTime asks: "Should I trust this data?"
cacheTime asks: "Should I delete this unused data?"

Short Interview Answer

staleTime is the time during which React Query treats data as fresh. After it expires, data becomes stale but remains in cache. If the user returns while cache still exists, React Query may show cached data immediately and refetch fresh data in the background, causing another UI update. cacheTime is the time unused cached data stays in memory. If nobody uses the query until cacheTime expires, the cache is deleted and the next visit requires a fresh API call.

One-Line Interview Answer

staleTime controls data freshness; cacheTime controls how long unused cached data stays before deletion.
