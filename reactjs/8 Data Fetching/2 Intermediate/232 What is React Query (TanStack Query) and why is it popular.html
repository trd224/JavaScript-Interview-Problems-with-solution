232. What is React Query (TanStack Query) and why is it popular?

One-Line Definition

TanStack Query (formerly React Query) is a library that simplifies fetching, caching, synchronizing, and updating server data in React applications.

Simple Meaning

React itself does not provide a built-in solution for:

  • Fetching API data
  • Caching responses
  • Loading states
  • Error handling
  • Refetching data
  • Keeping data fresh

Without React Query, developers usually write lots of repetitive code.

Managing server data manually quickly becomes difficult in large applications.

What Is Server State?

Server state is data that comes from a backend API.

Examples:

  • User list
  • Products
  • Orders
  • Departments
  • Audit records
GET /api/users
GET /api/products
GET /api/orders
React Query is specifically designed for server state management.

The Traditional Way

const [users, setUsers] = useState([]);
const [loading, setLoading] = useState(false);
const [error, setError] = useState(null);

useEffect(() => {
  setLoading(true);

  fetch('/api/users')
    .then(res => res.json())
    .then(data => setUsers(data))
    .catch(err => setError(err))
    .finally(() => setLoading(false));
}, []);

Problems:

  • Manual loading state.
  • Manual error state.
  • No caching.
  • No retries.
  • No background updates.
  • No request deduplication.

React Query Way

const {data,isLoading,error}=useQuery({queryKey:['users'],queryFn:fetchUsers});
Loading, error handling, caching, and refetching are automatically managed.

Installation

npm install @tanstack/react-query

Setup QueryClient

import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

const queryClient = new QueryClient();

function App() {
  return (
    <QueryClientProvider client={queryClient}>
      <Dashboard />
    </QueryClientProvider>
  );
}

Basic Query Example

async function fetchUsers() {
  const res = await fetch('/api/users');

  return res.json();
}

function Users() {
  const {data, isLoading, error} = useQuery({
    queryKey: ['users'],
    queryFn: fetchUsers
  });

  if (isLoading) return <p>Loading...</p>;

  if (error) return <p>Error</p>;

  return (
    <ul>
      {data.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}

Automatic Refetching

React Query can automatically refetch when:

  • User returns to the tab.
  • Internet reconnects.
  • Query becomes stale.
useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers,
  staleTime: 60000
});

Data stays fresh for 60 seconds.

Mutations

Queries fetch data.

Mutations modify data. 

const mutation = useMutation({
  mutationFn: createUser
});

Create User

Mutations handle POST, PUT, PATCH, and DELETE requests.

Why Does React Query Need queryKey?

React Query stores API responses in a cache.

It needs a unique identifier to know:

  • Which data belongs to which request.
  • When cached data can be reused.
  • When a request should run again.
  • Which query should be invalidated or refetched.
Think of queryKey as a cache key.

Example 1

useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers
});

React Query stores: 

Cache = {
  users: [...]
};

Later another component calls: 

useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers
});

React Query says: 

I already have "users" in cache.

and returns cached data instantly.

Without queryKey

Imagine: 

useQuery({
  queryFn: fetchUsers
});

React Query would not know: 

What should I call this data?
Where should I store it?
How do I find it later?
That's why queryKey is required.

Example 2 - Dynamic User

useQuery({
  queryKey: ['user', userId],
  queryFn: () => fetchUser(userId)
});

If: 

userId = 1;

Cache becomes: 

['user', 1]

Data: 

{
  id: 1,
  name: 'Rizwan'
}

Later: 

userId = 2;

Now key becomes: 

['user', 2]

React Query sees: 

Different queryKey

and makes a new API request.

Visual

['user', 1]
    ↓
User 1 Data

['user', 2]
    ↓
User 2 Data

['user', 3]
    ↓
User 3 Data

Each key has its own cache.

Why Array Instead Of String?

Bad: 

queryKey: 'user'

Good: 

queryKey: ['user', userId]

Because arrays support parameters. 

['user', 1]
['user', 2]
['user', 3]

React Query can uniquely identify each request.

Real Search Example

useQuery({
  queryKey: ['users', search],
  queryFn: () => fetchUsers(search)
});

Search: 

search = 'ri';

Key: 

['users', 'ri']

Search: 

search = 'riz';

Key: 

['users', 'riz']

React Query understands: 

These are different queries.

So each search result gets its own cache.

Query Invalidation

Suppose you create a new user: 

mutation.mutate(newUser);

After success: 

queryClient.invalidateQueries({
  queryKey: ['users']
});

React Query finds: 

['users']

and refetches it.

Without queryKey it would not know which cache entry to refresh.

Simple Interview Answer

queryKey is a unique identifier for a query. React Query uses it to cache data, detect when query parameters change, reuse cached results, and know which queries should be refetched or invalidated. Think of it as the cache key for API data.

React Query vs Context

Context React Query
Client state Server state
No caching Built-in caching
Manual fetching Automatic fetching
Manual loading state Built-in loading state

React Query is not a replacement for Context. They solve different problems.

TanStack Query (React Query) is a server-state management library that simplifies fetching, caching, synchronizing, and updating API data. It is popular because it automatically handles caching, loading states, error handling, retries, background updates, and request deduplication, greatly reducing boilerplate code.
