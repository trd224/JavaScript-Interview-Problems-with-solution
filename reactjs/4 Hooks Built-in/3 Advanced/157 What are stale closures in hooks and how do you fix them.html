157. What are Stale Closures in Hooks and How Do You Fix Them?

One-Line Answer

A stale closure happens when a function remembers old state or props from an earlier React render and keeps using that old value even after newer values exist.

First Understand JavaScript Closure

Closure Example
function createCounter() {
  let count = 0;

  return function() {
    console.log(count);
  };
}

const counter = createCounter();

counter();
counter();
counter();

Output:

0
0
0

Here count never changes, so printing 0 again and again is normal.

Better Closure Flow

Returned Function
↓ Uses count
↓ JavaScript Sees Function Needs count
↓ If count Is Destroyed, Function Will Crash
↓ count Cannot Be Destroyed
↓ count Stored With Function
↓ Closure Created
"count stored with function" does not mean count is copied inside the function code.
It means the returned function is linked with a hidden closure environment that keeps count alive.
counter
│
├── Function Code
│
└── Hidden Closure Environment
      count = 0

Backpack Explanation

Think of the returned function as a person carrying a backpack.

Function
↓ Backpack
↓ count = 0

Even after createCounter() finishes, the backpack remains.

That backpack is the closure.

Why JavaScript Keeps count Alive?

If JavaScript destroyed count after createCounter() finished:

createCounter()
↓ count = 0
↓ Return Function
↓ Destroy count

Then this would fail: 

counter();
ReferenceError: count is not defined

So JavaScript keeps count alive because the returned function still needs it.

Closure With Updating count

Updated Closure Example
function createCounter() {
  let count = 0;

  return function() {
    count++;
    console.log(count);
  };
}

const counter = createCounter();

counter();
counter();
counter();

Output:

1
2
3
This proves count was not destroyed. The closure kept the same count variable alive.

Multiple Closures

const counter1 = createCounter();
const counter2 = createCounter();

counter1();
counter1();
counter2();

Output:

1
2
1

Because each call to createCounter() creates a separate closure.

counter1
↓ Closure A
count = 2

counter2
↓ Closure B
count = 1

Now Connect This To React

Counter.jsx
function Counter() {
  const [count, setCount] = useState(0);
}
React does NOT update the same count variable.

Every render creates a new count variable.

Render #1
count = 0

Render #2
count = 1

Render #3
count = 2

The Stale Closure Problem

Important correction: if there is no code that updates count, then expecting 0, 0, 0 is correct. To understand stale closure, we need one callback that logs count and another action that updates count.
Broken Example
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count);
    }, 1000);

    return () => clearInterval(id);
  }, []);

  return (
    <>
      <h2>UI Count: {count}</h2>
      <button onClick={() => setCount(count + 1)}>
        Increment
      </button>
    </>
  );
}

What Happens?

When you click the button, UI count changes:

UI Count
0 → 1 → 2 → 3

But console still prints:

0
0
0
0
The interval captured count = 0 from the first render.

Why This Happens?

Render #1
count = 0
↓ useEffect Runs
↓ setInterval Created
↓ Interval Function Gets Backpack
↓ count = 0

User Clicks Increment
↓ Render #2
count = 1

User Clicks Increment
↓ Render #3
count = 2

User Clicks Increment
↓ Render #4
count = 3

Interval still uses old backpack:
count = 0
That old backpack containing old React values is called a stale closure.

Why Empty Dependency Array Causes This

useEffect(() => {
  // runs only once after first render
}, []);
[] dependency
↓ Effect Runs Only Once
↓ Callback Created Once
↓ Callback Captures First Render Values
If that callback uses count, it will remember the first render count unless we fix it.

Fix #1 - Add Correct Dependencies

Correct Dependency Array
useEffect(() => {
  const id = setInterval(() => {
    console.log(count);
  }, 1000);

  return () => clearInterval(id);
}, [count]);
count Changes
↓ Effect Cleans Up
↓ Old Interval Removed
↓ New Interval Created
↓ New Closure Captures Latest count
Now the interval logs the latest count, but the interval is recreated whenever count changes.

Fix #2 - Functional State Update

Use this when the callback updates state based on previous state.

Broken Counter
useEffect(() => {
  const id = setInterval(() => {
    setCount(count + 1);
  }, 1000);

  return () => clearInterval(id);
}, []);

Because count is captured as 0, result becomes:

1
1
1
1

Correct Version

useEffect(() => {
  const id = setInterval(() => {
    setCount(prev => prev + 1);
  }, 1000);

  return () => clearInterval(id);
}, []);
Functional update works because React gives the latest state through prev.
Old Closure
↓ Does Not Matter
↓ React Gives Latest State
↓ prev => prev + 1

Fix #3 - useRef For Latest Value

Use ref when you want latest value without recreating the effect. 

function Counter() {
  const [count, setCount] = useState(0);
  const countRef = useRef(count);

  countRef.current = count;

  useEffect(() => {
    const id = setInterval(() => {
      console.log(countRef.current);
    }, 1000);

    return () => clearInterval(id);
  }, []);

  return (
    <button onClick={() => setCount(count + 1)}>
      Increment
    </button>
  );
}
Ref does not create a new closure value on every render. countRef.current always points to the latest value.

Real World Example - Search API

Buggy Search
useEffect(() => {
  fetch(`/api/search?q=${search}`);
}, []);

If search changes from React to Redux:

Initial Render
search = React
↓ Effect Captures React
↓ User Types Redux
↓ Effect Still Uses React
search is stale because it is missing from dependencies.

Correct Version

useEffect(() => {
  fetch(`/api/search?q=${search}`);
}, [search]);

Real World Example - Event Listener

Buggy Listener
useEffect(() => {
  function handleResize() {
    console.log(width);
  }

  window.addEventListener('resize', handleResize);

  return () => window.removeEventListener('resize', handleResize);
}, []);
If width changes later, handleResize may still use old width.

Fix With Dependencies

useEffect(() => {
  function handleResize() {
    console.log(width);
  }

  window.addEventListener('resize', handleResize);

  return () => window.removeEventListener('resize', handleResize);
}, [width]);

Or Fix With Ref

const widthRef = useRef(width);
widthRef.current = width;

useEffect(() => {
  function handleResize() {
    console.log(widthRef.current);
  }

  window.addEventListener('resize', handleResize);

  return () => window.removeEventListener('resize', handleResize);
}, []);

Common Places Where Stale Closures Happen

Place Problem Common Fix
setInterval Old state value Dependencies, functional update, or ref
setTimeout Old state value Dependencies or ref
useEffect Missing dependencies Add dependencies
Event listeners Old props/state Ref or recreate listener
WebSocket callbacks Old values Ref
Async requests Outdated query/data Dependencies + cleanup

Relationship With exhaustive-deps Rule

Function Uses Variable
↓ Variable Can Change
↓ Dependency Missing
↓ Stale Closure Risk
The exhaustive-deps ESLint rule exists mainly to prevent stale closure bugs.

Best Rules To Fix Stale Closures

Situation Best Fix
Effect uses changing value Add value to dependency array
Updating state based on previous state Use functional update
Long-running callback needs latest value Use useRef
Function depends on props/state Use useCallback with dependencies

Interview Answer

A stale closure occurs when a function captures state or props from an earlier React render and keeps using those outdated values after newer renders have created newer values. This happens because JavaScript closures preserve the variables they used when the function was created. In React, every render creates new variables, so callbacks inside useEffect, setInterval, setTimeout, event listeners, WebSocket handlers, and async functions can accidentally keep old values. You fix stale closures by adding correct dependencies, using functional state updates, or storing the latest value in a useRef when the callback should not be recreated.
