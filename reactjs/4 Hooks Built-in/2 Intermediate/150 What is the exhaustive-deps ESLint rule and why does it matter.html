150. What is the exhaustive-deps ESLint Rule and Why Does It Matter?

One-Line Definition

The exhaustive-deps ESLint rule checks whether all values used inside hooks such as useEffect, useMemo, and useCallback are correctly included in their dependency arrays.

Why Does This Rule Exist?

React hooks depend on dependency arrays. 

useEffect(() => {

  console.log(count);

}, [count]);

React uses the dependency array to decide:

Dependency Changed?
↓
Yes
↓
Run Effect Again

Dependency Changed?
↓
No
↓
Skip Effect

If we forget a dependency, React may use old values.

This creates bugs that are often difficult to find.

The Problem It Prevents

Example

function App() {

  const [count, setCount] = useState(0);

  useEffect(() => {

    console.log(count);

  }, []);

}

Inside the effect we use: 

count

But dependency array is: 

[]

ESLint shows: 

React Hook useEffect has a missing dependency: 'count'
The effect uses count but does not watch count.

Why Is This Dangerous?

First Render

count = 0

Effect runs: 

0

User Clicks Button

count = 1

Will effect run again? 

No

Because: 

[]

never changes.

The effect becomes stuck with old data.

Correct Version

useEffect(() => {

  console.log(count);

}, [count]);

Now:

count Changes
↓
Dependency Changes
↓
Effect Runs Again

What Is a Stale Closure?

This is the main bug exhaustive-deps prevents.

Example

function App() {

  const [count, setCount] = useState(0);

  useEffect(() => {

    const id = setInterval(() => {

      console.log(count);

    }, 1000);

    return () => clearInterval(id);

  }, []);

}

What Happens?

First render: 

count = 0

Interval remembers: 

0

User clicks: 

count = 1
count = 2
count = 3

Console still prints: 

0
0
0
0
The effect captured the old value of count. This is called a stale closure.

Correct Version

useEffect(() => {

  const id = setInterval(() => {

    console.log(count);

  }, 1000);

  return () => clearInterval(id);

}, [count]);

Now the effect updates when count changes.

The Rule Also Works For useCallback

Wrong

const handleClick = useCallback(() => {

  console.log(count);

}, []);

ESLint: 

Missing dependency: count

Correct

const handleClick = useCallback(() => {

  console.log(count);

}, [count]);
Every value used inside the callback should usually appear in the dependency array.

The Rule Also Works For useMemo

Wrong

const total = useMemo(() => {

  return price * quantity;

}, [price]);

Missing: 

quantity

Correct

const total = useMemo(() => {

  return price * quantity;

}, [price, quantity]);

What Does ESLint Actually Check?

ESLint looks inside the hook. 

useEffect(() => {

  console.log(user);
  console.log(theme);

}, [user]);

It sees: 

Used:
user
theme

Dependencies: 

user

Missing: 

theme

Warning appears.

Most Common Beginner Mistake

Developers often do this: 

useEffect(() => {

  fetchUsers();

}, []);

ESLint says: 

Missing dependency: fetchUsers

Many developers simply disable the warning.

Do not ignore exhaustive-deps warnings without understanding why they appear.

How To Fix Function Dependencies

Option 1 - Move Function Inside Effect

useEffect(() => {

  function fetchUsers() {

    console.log('Fetch');

  }

  fetchUsers();

}, []);

Option 2 - useCallback

const fetchUsers = useCallback(() => {

  console.log('Fetch');

}, []);

useEffect(() => {

  fetchUsers();

}, [fetchUsers]);

When Is It Okay To Ignore exhaustive-deps?

Very rarely.

Example

useEffect(() => {

  initializeAnalytics();

  // eslint-disable-next-line react-hooks/exhaustive-deps

}, []);

Only when we intentionally want: 

Run Once

and fully understand the consequences.

Ignoring the rule should be the exception, not the default solution.

Why This Rule Matters

  • Prevents stale closures.
  • Prevents missing updates.
  • Prevents difficult bugs.
  • Keeps hooks synchronized with state.
  • Encourages correct dependency arrays.

Comparison

Without exhaustive-deps With exhaustive-deps
Easy to miss dependencies Warnings appear immediately
More stale closure bugs Fewer stale closure bugs
Harder debugging Safer hooks

Value Used Inside Hook?

Interview Answer

The exhaustive-deps ESLint rule verifies that all values used inside hooks such as useEffect, useMemo, and useCallback are included in their dependency arrays. It helps prevent stale closures, missing updates, and synchronization bugs by ensuring hooks react to the values they depend on.

One-Line Interview Answer

The exhaustive-deps rule ensures hook dependency arrays include every value used inside the hook, helping prevent stale data and hidden bugs.
