155. What is the useTransition Hook?

One-Line Definition

useTransition is a React 18 hook that lets us mark non-urgent state updates as low-priority so the UI remains responsive.

Why Is It Used?

Sometimes a state update is expensive and causes the UI to freeze.

Examples

  • Searching a huge list.
  • Filtering thousands of records.
  • Rendering large tables.
  • Updating charts.

With useTransition, React can update important UI immediately and delay heavy rendering work.

The main benefit is: typing, clicking, and urgent UI updates remain smooth.

Basic Syntax

const [isPending, startTransition] = useTransition();

What Does It Return?

Value Meaning
isPending Tells us whether the transition update is still running.
startTransition Marks a state update as non-urgent or low-priority.

Real Project Use Case

Imagine our application has a search box and we need to filter 10,000 products.

  • The input text should update immediately.
  • The heavy product filtering can be delayed slightly.
Input typing is urgent. Large list filtering is non-urgent.

Example

import { useState, useTransition } from 'react';

function App() {

  const [input, setInput] = useState('');
  const [query, setQuery] = useState('');

  const [isPending, startTransition] = useTransition();

  const items = Array.from(
    { length: 10000 },
    (_, i) => `Product ${i + 1}`
  );

  const filteredItems = items.filter(item =>
    item.toLowerCase().includes(query.toLowerCase())
  );

  function handleChange(e) {

    const value = e.target.value;

    // Urgent update
    setInput(value);

    // Non-urgent update
    startTransition(() => {
      setQuery(value);
    });

  }

  return (
    <>

      <input
        type="text"
        value={input}
        onChange={handleChange}
        placeholder="Search products..."
      />

      {isPending && <p>Searching...</p>}

      <ul>
        {filteredItems.slice(0, 20).map(item => (
          <li key={item}>{item}</li>
        ))}
      </ul>

    </>
  );

}

export default App;

Why Do We Need Both input and query?

A very common question is: 

const [input, setInput] = useState('');
const [query, setQuery] = useState('');

Why do we need two states? Why not use only one state?

Can We Use input as query?

Yes, technically we can. 

const [input, setInput] = useState('');

const filteredItems = items.filter(item =>
  item.includes(input)
);

This works perfectly fine for small applications and small lists.

For normal forms and small datasets, using a single state is usually enough.

The Problem

Suppose filtering 50,000 products is expensive.

If the textbox and filtering both depend on the same state: 

const [input, setInput] = useState('');

const filteredItems = items.filter(item =>
  item.includes(input)
);
User Types
↓
input Changes
↓
Component Re-renders
↓
Heavy Filtering Runs
↓
Textbox May Feel Slow

The input update and heavy filtering are tied together.

Solution: Separate States

const [input, setInput] = useState('');
const [query, setQuery] = useState('');

Now each state has a different responsibility.

State Purpose
input Controls the textbox and updates immediately.
query Controls expensive filtering and can be delayed.

How It Works

function handleChange(e) {

  const value = e.target.value;

  setInput(value);

  startTransition(() => {
    setQuery(value);
  });

}
User Types
↓
input Updates Immediately
↓
Textbox Updates Immediately
↓
React Schedules query Update
↓
Heavy Filtering Runs Later

Temporary Difference Between States

For a brief moment, both values can be different. 

input = "React"
query = "Reac"

The textbox already shows the latest value, while React is still processing the expensive update.

After the transition finishes: 

input = "React"
query = "React"
The entire purpose of using two states is to separate an urgent update (typing) from a non-urgent update (expensive rendering).

Important Interview Point

If your filtering is small and there is no UI lag, you do not need a second state. The extra query state is mainly used to demonstrate the real benefit of useTransition for expensive rendering scenarios.

What Happens Here?

1. User Types

setInput(value);

This is an urgent update. React updates the textbox immediately.

2. Heavy Update

startTransition(() => {
  setQuery(value);
});

This is a non-urgent update. React can delay it if the browser needs to stay responsive.

3. Loading State with isPending

{isPending && <p>Searching...</p>}

While React is processing the transition, isPending becomes true. 

isPending === true

After the transition finishes, isPending becomes false. 

isPending === false

Flow

Without useTransition

setInput(value);
setQuery(value);

Both updates have the same priority.

If filtering or rendering is heavy, typing may become slow.

With useTransition

setInput(value);

startTransition(() => {
  setQuery(value);
});

The input update happens immediately, and the heavy update runs with lower priority.

This keeps the UI responsive while React handles expensive rendering work.

Important Interview Points

  • useTransition is used for non-urgent state updates.
  • It helps avoid UI blocking during expensive rendering.
  • isPending is useful for showing loading text or a spinner.
  • It does not make code asynchronous like setTimeout.
  • It does not replace API loading states.
  • It is mainly useful when rendering UI becomes heavy.

Common Confusion

Does useTransition delay input value?

No. We keep the input value outside the transition so typing updates immediately.

Should API calls be inside startTransition?

Usually no. useTransition is mainly for rendering updates, not for controlling API request loading.

When Should You Use useTransition?

  • Large list filtering.
  • Heavy table rendering.
  • Tab switching with expensive UI.
  • Search results rendering.
  • Updating charts or complex components.

When Should You NOT Use useTransition?

  • Simple forms.
  • Small UI updates.
  • Normal API loading state.
  • Basic button clicks.
  • When there is no visible UI lag.

Comparison Table

Without useTransition With useTransition
All updates have normal priority. Heavy updates can be low priority.
Input may lag during expensive rendering. Input remains responsive.
No pending state for transition work. isPending can show loading UI.
Good for simple updates. Good for expensive UI updates.

Short Interview Answer

useTransition allows us to mark state updates as non-urgent. React gives higher priority to important UI updates and performs heavy updates with lower priority. The isPending value tells us whether the transition is still running.

One-Line Interview Answer

useTransition helps keep the UI responsive by running expensive state updates with lower priority, while isPending shows whether that transition is currently in progress.
