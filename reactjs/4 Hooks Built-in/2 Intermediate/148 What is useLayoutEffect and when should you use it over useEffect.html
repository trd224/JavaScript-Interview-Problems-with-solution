148. What is useLayoutEffect and When Should You Use It Over useEffect?

One-Line Definition

useLayoutEffect is a React hook that runs synchronously after React updates the DOM but before the browser paints the screen, allowing us to read or modify the layout without visible flickering.

The Biggest Confusion

Most developers hear: 

useEffect()
useLayoutEffect()

and think: 

Both run after render
↓
Same thing

Not exactly.

The difference is:

useLayoutEffect
↓
Runs Before Browser Paint

useEffect
↓
Runs After Browser Paint

Browser Rendering Process

When React updates the UI:

React Render
↓
Update DOM
↓
useLayoutEffect
↓
Browser Paint
↓
useEffect
This timing difference is the entire reason useLayoutEffect exists.

What Does "Paint" Mean?

Paint means: 

Browser draws pixels on screen

The user can finally see the UI.

Example

DOM Updated
↓
Browser Paints Screen
↓
User Sees UI

How useEffect Works

useEffect(() => {
  console.log('Effect');
}, []);

Execution order:

React Render
↓
DOM Updated
↓
Browser Paint
↓
useEffect Runs

The user sees the UI first.

Then useEffect executes.

How useLayoutEffect Works

useLayoutEffect(() => {
  console.log('Layout Effect');
}, []);

Execution order:

React Render
↓
DOM Updated
↓
useLayoutEffect Runs
↓
Browser Paint

The browser waits for useLayoutEffect to finish.

useLayoutEffect blocks the browser from painting.

Real Problem: UI Flicker

Suppose we need to measure an element and reposition it.

Using useEffect

function Tooltip() {

  const ref = useRef();

  useEffect(() => {

    const height = ref.current.getBoundingClientRect().height;

    console.log(height);

  }, []);

  return (
    <div ref={ref}>
      Tooltip
    </div>
  );

}

Sequence:

Render
↓
Browser Paint
↓
Wrong Position Visible
↓
useEffect Runs
↓
Position Fixed

User may briefly see the wrong layout.

This visible jump is called flickering or layout shift.

Using useLayoutEffect

function Tooltip() {

  const ref = useRef();

  useLayoutEffect(() => {

    const height = ref.current.getBoundingClientRect().height;

    console.log(height);

  }, []);

  return (
    <div ref={ref}>
      Tooltip
    </div>
  );

}

Sequence:

Render
↓
DOM Updated
↓
Measure Layout
↓
Position Corrected
↓
Browser Paint

User only sees the final correct layout.

This is the primary use case of useLayoutEffect.

Common Real-World Uses

  • Measuring element width or height.
  • Reading getBoundingClientRect().
  • Positioning tooltips.
  • Positioning popups.
  • Scrolling before paint.
  • Preventing layout flicker.
  • Animation setup.

Example: Auto Scroll

Using useLayoutEffect

const listRef = useRef();

useLayoutEffect(() => {

  listRef.current.scrollTop = listRef.current.scrollHeight;

}, []);

Scroll position is updated before the user sees the page.

Example: Focus Input

const inputRef = useRef();

useLayoutEffect(() => {

  inputRef.current.focus();

}, []);

The input receives focus before paint.

Why Not Use useLayoutEffect Everywhere?

Because it blocks painting.

DOM Updated
↓
useLayoutEffect Runs
↓
Browser Must Wait
↓
Paint Delayed

Too many layout effects can make the UI feel slower.

useLayoutEffect is more powerful but more expensive than useEffect.

Most Common Mistake

Many developers write: 

useLayoutEffect(() => {

  fetch('/api/users');

}, []);

This is wrong.

Network requests do not need to block browser painting.

API calls should use useEffect.

When Should You Use useEffect?

  • API calls.
  • Subscriptions.
  • Timers.
  • Event listeners.
  • Analytics.
  • Logging.
  • Most side effects.

When Should You Use useLayoutEffect?

  • DOM measurements.
  • Reading element size.
  • Position calculations.
  • Preventing flicker.
  • Animation setup.
  • Scroll adjustments before paint.

Comparison Table

Easy Way To Remember

Need Data? (API, Timers, Events)
↓
useEffect

Need Layout? (Size, Position, Scroll)
↓
useLayoutEffect

Interview Answer

useLayoutEffect is similar to useEffect, but it runs synchronously after the DOM is updated and before the browser paints the screen. It is mainly used for DOM measurements, positioning, scrolling, and preventing visual flickering. For most side effects such as API calls and subscriptions, useEffect should be preferred.

One-Line Interview Answer

useLayoutEffect runs before the browser paints the screen, making it ideal for layout measurements and preventing UI flicker, while useEffect runs after paint.
