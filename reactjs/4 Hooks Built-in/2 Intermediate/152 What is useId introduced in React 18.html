152. What is useId Introduced in React 18?

One-Line Definition

useId is a React 18 hook that generates stable, unique IDs for accessibility attributes and works correctly with both client-side rendering and server-side rendering.

Why Was useId Introduced?

Many HTML elements need matching IDs.

Example

<label for="email">
  Email
</label>

<input id="email" />

The label and input must share the same ID.

In React we often generate IDs dynamically.

The Problem Before React 18

Developers commonly used: 

Math.random()

or: 

Date.now()

Example

const id = Math.random();

return (
  <>
    <label htmlFor={id}>
      Email
    </label>

    <input id={id} />
  </>
);

This works on the client.

But React 18 introduced more SSR features.

Random IDs can cause server and client HTML to generate different IDs.

The SSR Problem

Server Render

ID = 12345

Client Hydration

ID = 98765

React compares: 

12345 !== 98765

Result: 

Hydration Warning
The server HTML and client HTML no longer match.

Solution: useId

import { useId } from 'react';

function SignupForm() {

  const id = useId();

  return (
    <>
      <label htmlFor={id}>
        Email
      </label>

      <input id={id} />
    </>
  );

}
React generates a stable ID that matches between the server and client.

How useId Works

const id = useId();

React generates something similar to: 

:r0:

or: 

:r1:

The exact value is managed internally by React.

We should treat the ID as an opaque string and never rely on its format.

Basic Example

import { useId } from 'react';

export default function App() {

  const id = useId();

  return (
    <>
      <label htmlFor={id}>
        Username
      </label>

      <input id={id} />
    </>
  );

}

Real Accessibility Example

The most common use case is accessibility. 

import { useId } from 'react';

function InputField() {

  const id = useId();

  return (
    <>
      <label htmlFor={id}>
        Email Address
      </label>

      <input
        id={id}
        type="email"
      />
    </>
  );

}

Now screen readers correctly connect: 

Label
↓
Input

Multiple IDs

Sometimes we need multiple related IDs. 

const id = useId();

const inputId = `${id}-input`;
const errorId = `${id}-error`;

Usage

<input
  id={inputId}
  aria-describedby={errorId}
/>

<p id={errorId}>
  Invalid email
</p>
This creates multiple unique IDs from one useId value.

Common Interview Question

Can useId be used as a React key?

Wrong

{users.map(user => (

  <div key={useId()}>
    {user.name}
  </div>

))}
Never use useId for React keys.

Why Not?

React keys should come from data.

Correct

{users.map(user => (

  <div key={user.id}>
    {user.name}
  </div>

))}
useId is for accessibility IDs, not list keys.

Another Common Mistake

Wrong

const id = useId();

fetch(`/api/users/${id}`);

useId should not be used as:

  • Database IDs.
  • API IDs.
  • User IDs.
  • Primary keys.
useId only creates UI identifiers.

Most Common useId Use Cases

  • label + input linking.
  • aria-describedby.
  • aria-labelledby.
  • Form accessibility.
  • SSR-safe IDs.
  • Reusable form components.

Comparison

Method SSR Safe?
Math.random() ❌ No
Date.now() ❌ No
UUID Generated During Render ❌ Usually No
useId() ✅ Yes

When Should You Use useId?

  • Accessibility attributes.
  • Form labels.
  • ARIA relationships.
  • Reusable form components.
  • SSR applications.

When Should You NOT Use useId?

  • React list keys.
  • Database IDs.
  • User IDs.
  • API identifiers.
  • Business logic identifiers.

Interview Answer

useId is a React 18 hook that generates unique, stable IDs that work correctly across server-side rendering and client-side hydration. It is primarily used for accessibility features such as linking labels, inputs, and ARIA attributes.

One-Line Interview Answer

useId generates stable, SSR-safe unique IDs for accessibility and form elements.
