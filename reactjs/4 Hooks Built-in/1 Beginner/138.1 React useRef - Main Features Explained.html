138. What is useRef and What are its Use Cases?

React Hooks

One-Line Definition

useRef is a React hook that allows us to store mutable values and access DOM elements without causing component re-renders.

Simple Understanding

Before learning useRef, we first need to understand how React renders a component.

Once we understand React's rendering process, it becomes much easier to understand why useRef exists.

How React Rendering Works

Whenever React detects that something affecting the UI has changed, it executes the component function again.

Can We Store the Timer ID Inside useState?

Yes, but it creates another problem.

    const id = setInterval(() => {

But Do We Really Need a Re-render?

No. The timer ID is not displayed anywhere on the screen. It is only needed later when we call clearInterval().

This Is Why useRef Exists

React provides useRef for values that should survive re-renders but should not trigger another render when they change. 


const timerRef = useRef(null);

function startTimer(){

    timerRef.current = setInterval(() => {
        console.log("Running...");
    },1000);

}

The timer ID is preserved across renders, and updating timerRef.current does not trigger another render.

Main Features of useRef

  • Stores mutable values.
  • Accesses DOM elements.
  • Does not trigger component re-renders.
  • Value persists across renders.

1. Stores Mutable Values

What Does "Stores Mutable Values" Mean?

A mutable value is a value that can change over time.

Unlike a normal JavaScript variable, a value stored inside useRef is not recreated whenever the component re-renders.

Instead, React creates the ref object only once and keeps using the same object throughout the component's lifetime.

Why Do We Need It?

Remember how React works.

All Normal Variables Are Created Again

Suppose we write: 


export default function App(){

    let counter = 0;

    function increase(){
        counter++;
        console.log(counter);
    }

    return(
        <button onClick={increase}>
            Increase
        </button>
    );

}

When the button is clicked, the value changes.

Now imagine the component re-renders because some state changes.

The previous value is lost because the variable is created again.

Solution: useRef


import { useRef } from "react";

export default function App(){

    const counterRef = useRef(0);

    function increase(){

        counterRef.current++;

        console.log(counterRef.current);

    }

    return(
        <button onClick={increase}>
            Increase
        </button>
    );

}

React creates the ref object only once. During every render, React gives us the same object again.

Why Not Use useEffect?

Another common question is:

"Why aren't we using useEffect() here?"

Because the timer starts only when the user clicks the button. 


<button onClick={startTimer}>
    Start
</button>

This is a user action, not a rendering side effect.

If we wanted the timer to start automatically when the component mounted, then we would use useEffect(). 


useEffect(() => {

    const id = setInterval(() => {

        console.log("Running...");

    },1000);

    return () => clearInterval(id);

},[]);

Rule: Use useRef to store the timer ID. Use useEffect only when the timer should start because of rendering or dependency changes.

Key Points

  • Normal variables are recreated after every render.
  • useRef keeps the same object across renders.
  • Changing ref.current does not trigger a re-render.
  • Refs are perfect for storing timer IDs, previous values, API instances, and mutable objects.
  • Use useState only when changing the value should update the UI.

2. Accesses DOM Elements

What Does "Accesses DOM Elements" Mean?

One of the most common uses of useRef is to get a direct reference to a DOM element created by React.

Instead of searching the DOM using methods like document.getElementById() or querySelector(), React allows us to attach a ref directly to an element.

After the element is rendered, React automatically stores that DOM element inside ref.current.

How React Normally Renders an Element

3. Does Not Trigger Re-render

Why Doesn't useRef Trigger a Re-render?

To understand this feature, we first need to remember when React re-renders a component.

React re-renders a component only when something that affects the UI changes.

  • State changes
  • Props change
  • Context changes

A value stored inside useRef is not part of React's rendering system.

Therefore, changing ref.current does not tell React to render the component again.

Why Doesn't React Re-render?

Internally, React tracks state values. 


const [count,setCount] = useState(0);

When 


setCount(...)

is called, React knows that something affecting the UI has changed. So React schedules another render.

A ref works differently. 


countRef.current++;

This simply changes a property on an existing JavaScript object. React does not monitor changes to ref.current.

Updating a ref is just like changing the property of a normal object. React ignores it because it does not affect rendering.

When Should We Use useState Instead?

If the value is shown on the screen, use state.

<h2>{count}</h2>

Whenever count changes, the UI must update. Therefore React should re-render.

When Should We Use useRef?

If the value is used only internally and does not affect the UI, use a ref.

Examples include:

  • Timer IDs
  • Previous state values
  • DOM elements
  • WebSocket connections
  • API instances
  • Mutable objects

Rule to Remember:

UI changes → useState

Internal values that don't affect rendering → useRef

  • React does not track changes made to refs.
  • Refs are perfect for storing values that should survive renders but should not update the UI.
  • Using useRef avoids unnecessary renders and improves performance in these scenarios.

4. Value Persists Across Renders

What Does "Value Persists Across Renders" Mean?

When React re-renders a component, it executes the entire component function again.

Normally, every local variable inside the component is created again.

However, a value stored inside useRef is different. React creates the ref object only once and keeps reusing the same object on every render.

Because the same object is reused, the value stored in ref.current is preserved across renders.

Real-World Example

Suppose we store a timer ID.

Why Is This Important?

If React recreated the ref object on every render, 


{
    current: null
}

would be created every time. The timer ID would disappear after each render.

Instead, React preserves the same object.

This is why refs are useful for storing values that must survive renders.

Key Points

  • Normal variables are recreated during every render.
  • useRef creates its object only once.
  • React reuses the same ref object on every render.
  • Changing ref.current updates the stored value without creating a new object.
  • The value survives across renders, making refs ideal for timers, DOM elements, previous values, API instances, and other mutable data.

useState vs useRef

Now that we understand how React re-renders components, the difference between useState and useRef becomes much clearer.

useState useRef
Stores state that affects the UI. Stores mutable values that do not affect the UI.
Updating the value triggers a re-render. Updating ref.current does not trigger a re-render.
React tracks state changes. React does not track changes to ref.current.
The updated value is shown in the UI. The updated value is used internally by the component.
Best for counters, forms, loading states, themes, etc. Best for timer IDs, DOM elements, previous values, API instances, and WebSocket connections.

Interview Summary

What is useRef?

useRef is a React hook that creates a persistent object whose current property can store mutable values or DOM element references. Updating ref.current does not trigger a component re-render because React does not track changes to refs.

Final Conclusion

React re-renders a component whenever state, props, or context change because these values can affect the UI.

However, not every value inside a component is related to rendering. Some values are needed only internally, such as timer IDs, DOM references, previous values, WebSocket connections, or third-party library instances.

If we stored these values in useState, React would perform unnecessary re-renders even though the UI would remain unchanged.

To solve this problem, React provides useRef. A ref survives across renders, stores mutable values, gives direct access to DOM elements, and allows us to update its current property without causing another render.

Remember This Rule

  • If changing a value should update the UI, use useState.
  • If changing a value should not update the UI but the value should survive renders, use useRef.
