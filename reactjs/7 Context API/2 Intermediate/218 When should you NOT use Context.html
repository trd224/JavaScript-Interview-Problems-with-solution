218. When should you NOT use Context?

One-Line Definition

You should NOT use Context for local component state, frequently changing data, or complex global state where Context would cause unnecessary re-renders and maintenance problems.

A Common Misunderstanding

Many developers learn Context and think: 

Context = Global State Solution

Then they start putting everything into Context.

Examples:

  • Input values
  • Form state
  • Modal state
  • Cart state
  • Notifications
  • Search text
  • Table filters
Context is useful, but it is not the answer to every state management problem.

When Context Works Well

Context is best for data that:

  • Is needed in many places.
  • Changes infrequently.
  • Represents application-wide settings.

Examples:

  • Theme
  • Language
  • Authenticated user
  • Application configuration

When You Should NOT Use Context

1. For Local Component State

Bad: 

const SearchContext = createContext();
const [searchText, setSearchText] = useState('');

If only one page uses searchText, Context adds unnecessary complexity.

Better

function ProductPage() {
  const [searchText, setSearchText] = useState('');

  return (
    <SearchBox
      value={searchText}
      onChange={setSearchText}
    />
  );
}
Keep state close to where it is used.

2. For Parent → Child Data Passing

Some developers use Context immediately: 

App
 ↓
Header

But props are simpler.

Better

function App() {
  return <Header user={user} />;
}
If data only travels through a few levels, props are usually simpler than Context.

3. For Frequently Changing State

Imagine a search box: 

const [searchText, setSearchText] = useState('');

Every key press changes state.

If stored in Context: 

setSearchText('R');
setSearchText('Ri');
setSearchText('Riz');
setSearchText('Rizwa');
setSearchText('Rizwan');

Context updates repeatedly.

All Context consumers re-render on every keystroke.

Examples of Frequently Changing State

  • Search inputs
  • Mouse position
  • Drag and drop coordinates
  • Animation state
  • Live typing data
  • Real-time counters

4. For Large Complex Global State

Suppose we store: 

{
  user,
  cart,
  products,
  filters,
  notifications,
  orders,
  settings,
  permissions
}

inside one Context.

Every update can trigger many unnecessary re-renders.

For large applications, dedicated state libraries may be better:

  • Redux Toolkit
  • Zustand
  • Jotai
  • Recoil

5. For Server State

Server state means data coming from APIs. 

const users = await fetch('/api/users');

Many developers put API responses into Context.

This becomes difficult because:

  • Caching is manual.
  • Refetching is manual.
  • Loading state is manual.
  • Error handling is manual.

Better Tools

  • TanStack Query (React Query)
  • SWR
These libraries are designed specifically for server state.

Example - React Query Instead Of Context

Instead of: 

const UsersContext = createContext();

Use: 

const { data, isLoading } =
  useQuery({
    queryKey: ['users'],
    queryFn: fetchUsers
  });

6. For Temporary UI State

Examples:

  • Modal open/close
  • Dropdown open/close
  • Tooltip visibility
  • Accordion expansion

Bad

ModalContext

for a modal used only on one page.

Better

const [isOpen, setIsOpen] = useState(false);
Keep temporary UI state local whenever possible.

Real Project Example

Suppose our Aadhaar Authentication page contains:

  • Aadhaar number
  • OTP value
  • Captcha value
  • Loader state
  • Error popup state

Bad approach: 

AuthContext = {
  aadhaar,
  otp,
  captcha,
  loader,
  error
}

Typing one OTP digit: 

setOtp('1');

may trigger Context updates and unnecessary re-renders.

Better

  • Keep OTP local to the form.
  • Keep captcha local to the form.
  • Use Context only for shared authentication state.

Decision Guide

Important Interview Points

  • Do not use Context for local component state.
  • Do not use Context when props are sufficient.
  • Avoid Context for frequently changing values.
  • Context updates re-render all consumers.
  • Do not use Context as a replacement for React Query.
  • Large applications may benefit from Redux or Zustand.
  • Context is best for shared, infrequently changing state.

Short Interview Answer

You should avoid Context for local component state, frequently changing values, temporary UI state, and server state. Context is best for shared application-wide data such as theme, language, and authenticated user information. For complex global state use Redux or Zustand, and for server state use React Query or SWR.

One-Line Interview Answer

Do not use Context for local, frequently changing, or server state; use it only for shared application-wide state that changes infrequently.
