217. How do you split Contexts to avoid unnecessary re-renders?

One-Line Definition

We split Contexts by responsibility so that only components using the changed data re-render instead of the entire application.

Why Do We Need To Split Contexts?

A common mistake is putting all application state into one Context. 

const AppContext = createContext();
value={{
  user,
  theme,
  cart,
  notifications
}}

This seems simple initially.

But when any value changes, all Context consumers re-render.

Problem Example

Header

function Header() {
  const { user } = useContext(AppContext);

  return <h2>{user.name}</h2>;
}

Cart

function Cart() {
  const { cart } = useContext(AppContext);

  return <h2>{cart.length}</h2>;
}

Now suppose: 

setCart([...cart, product]);

Only cart changed.

However:

  • Cart re-renders.
  • Header re-renders.
  • Theme components re-render.
  • Notification components re-render.
Most of these renders are unnecessary.

Visual Problem

One Context
↓
User + Theme + Cart + Notifications
↓
Cart Changes
↓
Entire Context Updates
↓
All Consumers Re-render

Solution - Split Contexts

Instead of one huge Context: 

const AppContext = createContext();

Create multiple Contexts: 

const UserContext = createContext();
const ThemeContext = createContext();
const CartContext = createContext();
const NotificationContext = createContext();
Now each Context controls only one type of state.

Step 1 - Create User Context

import { createContext, useState } from 'react';

export const UserContext = createContext();

export function UserProvider({ children }) {
  const [user, setUser] = useState(null);

  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}
    </UserContext.Provider>
  );
}

Step 2 - Create Theme Context

import { createContext, useState } from 'react';

export const ThemeContext = createContext();

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');

  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

Step 3 - Create Cart Context

import { createContext, useState } from 'react';

export const CartContext = createContext();

export function CartProvider({ children }) {
  const [cart, setCart] = useState([]);

  return (
    <CartContext.Provider value={{ cart, setCart }}>
      {children}
    </CartContext.Provider>
  );
}

Step 4 - Combine Providers

function App() {
  return (
    <UserProvider>
      <ThemeProvider>
        <CartProvider>
          <Dashboard />
        </CartProvider>
      </ThemeProvider>
    </UserProvider>
  );
}

This is often called: 

Provider Composition

How Components Consume Context

Header Uses Only User Context

function Header() {
  const { user } = useContext(UserContext);

  return <h2>{user?.name}</h2>;
}

Cart Uses Only Cart Context

function Cart() {
  const { cart } = useContext(CartContext);

  return <h2>Items: {cart.length}</h2>;
}

What Happens Now?

When: 

setCart([...cart, product]);

Only:

  • CartContext changes.
  • Cart consumers re-render.

These do not re-render:

  • Header
  • Theme components
  • User components
This greatly reduces unnecessary renders.

Visual Result

Cart Context Changes
↓
Only Cart Consumers Notified
↓
Cart Components Re-render
↓
User Components Stay Untouched
↓
Theme Components Stay Untouched

Real Project Example

Suppose our Aadhaar Authentication application has:

  • User details
  • Department details
  • Theme
  • Language
  • Authentication state
  • Error popup state

Bad approach: 

AppContext = {
  user,
  department,
  theme,
  language,
  auth,
  error
};

Better approach: 

UserContext
DepartmentContext
ThemeContext
LanguageContext
AuthContext
ErrorContext
Changing OTP or authentication state will not force Theme or Language components to re-render.

Can We Use Custom Hooks?

Yes. This is the most common pattern.

useUser Hook

export function useUser() {
  return useContext(UserContext);
}

useCart Hook

export function useCart() {
  return useContext(CartContext);
}

Usage

const { user } = useUser();

const { cart } = useCart();
Custom hooks hide Context implementation details and make components cleaner.

How Many Contexts Are Too Many?

Having multiple Contexts is normal.

Don't create Contexts for every tiny value.

Group related state together.

Good Split Reason
UserContext User-related state
ThemeContext Theme-related state
CartContext Cart-related state
AuthContext Authentication state

Interview Difference

Question 216 explains why one giant Context is a problem.
Question 217 explains how to solve that problem by splitting Contexts by responsibility.

Important Interview Points

  • Split Contexts by responsibility.
  • Avoid one giant global Context.
  • Context updates trigger all consumers of that Context.
  • Smaller Contexts reduce unnecessary re-renders.
  • Provider composition is a common pattern.
  • Custom hooks make Context usage cleaner.
  • Frequently changing state benefits most from Context splitting.

Short Interview Answer

To avoid unnecessary re-renders, split Contexts based on responsibility such as UserContext, ThemeContext, CartContext, and AuthContext. This ensures that when one piece of state changes, only components consuming that specific Context re-render.

One-Line Interview Answer

Split Contexts by feature or responsibility so updates affect only relevant consumers instead of the entire application.
