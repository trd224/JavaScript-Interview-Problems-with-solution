216. What is the problem with putting everything in one Context?

One-Line Definition

Putting everything in one Context causes unnecessary re-renders because every consumer re-renders whenever any value inside the Context changes.

Why Developers Do This

When learning Context API, many developers create a single global context and put all application state inside it. 

const AppContext = createContext();
function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');
  const [cart, setCart] = useState([]);
  const [notifications, setNotifications] = useState([]);

  return (
    <AppContext.Provider
      value={{
        user,
        theme,
        cart,
        notifications,
        setUser,
        setTheme,
        setCart,
        setNotifications
      }}
    >
      {children}
    </AppContext.Provider>
  );
}

At first this looks convenient.

As the application grows, this becomes a performance problem.

The Main Problem

Suppose we have:

  • User information
  • Theme settings
  • Shopping cart
  • Notifications

All stored in one Context.

One Context
↓
User Theme Cart Notifications
↓
Any Value Changes
↓
All Consumers Re-render

Example

Header Component

function Header() {
  const { user } = useContext(AppContext);

  return <h2>{user.name}</h2>;
}

Cart Component

function Cart() {
  const { cart } = useContext(AppContext);

  return <h2>Items: {cart.length}</h2>;
}

Now imagine: 

setCart([...cart, product]);

Only cart changed.

But React creates a new Context value object.

Header re-renders even though user data did not change.

Why Does This Happen?

React compares the Context value by reference.

Every update creates a new object: 

value={{
  user,
  theme,
  cart,
  notifications
}}

React sees: 

Old Object !== New Object

Therefore all consumers are notified.

Visual Example

Cart Updated
↓
Provider Creates New Value Object
↓
Context Changed
↓
Header Re-renders
↓
ThemePanel Re-renders
↓
Notifications Re-render
↓
Cart Re-renders

Even though only Cart actually needed updating.

Real Project Example

Suppose our Aadhaar Authentication application stores:

  • User details
  • Department
  • OTP state
  • Theme
  • Language
  • Authentication status
  • Error popup state

inside one Context.

When OTP changes: 

setOtp('123456');

All components consuming the Context may re-render:

  • Header
  • Department panel
  • Language selector
  • Error popup
  • Theme panel
Most of these renders are wasted renders.

Better Approach - Split Contexts

Create multiple contexts based on responsibility. 

const UserContext = createContext();
const ThemeContext = createContext();
const CartContext = createContext();
const NotificationContext = createContext();
User Context
Theme Context
Cart Context
Notification Context
↓
Only Relevant Consumers Re-render

Example

function Header() {
  const user = useContext(UserContext);

  return <h2>{user.name}</h2>;
}
function Cart() {
  const cart = useContext(CartContext);

  return <h2>{cart.length}</h2>;
}

Now: 

setCart([...cart, product]);
Only CartContext consumers re-render.

Can React.memo Fix This?

Many developers think: 

export default React.memo(Header);

will solve the problem.

Unfortunately:

React.memo does not stop re-renders caused by Context updates.

If a component consumes Context: 

const { user } = useContext(AppContext);

it will re-render whenever the Context value changes.

Can useMemo Help?

You can memoize the Provider value: 

const value = useMemo(() => ({
  user,
  theme,
  cart
}), [user, theme, cart]);

This prevents unnecessary Provider updates.

However:

When cart changes, all consumers still receive the new Context value.

So splitting Contexts is usually the better solution.

When Is One Context Acceptable?

One Context is usually fine when data changes rarely.

Examples:

  • Theme
  • Language
  • Application settings

Because these values do not update frequently.

When Should You Split Contexts?

State Type Separate Context?
User Yes
Cart Yes
Notifications Yes
Theme Usually Yes
Language Usually Yes
Frequently Changing Data Definitely Yes

Important Interview Points

  • One large Context causes unnecessary re-renders.
  • All consumers are notified when Context value changes.
  • Context updates bypass React.memo.
  • Frequently changing state should be separated.
  • Split Contexts by responsibility.
  • Use Context for global state, but avoid putting everything into one Context.
  • Large applications often use multiple Contexts or state management libraries.

Short Interview Answer

The main problem with putting everything in one Context is that every consumer re-renders whenever any part of the Context value changes. This causes unnecessary renders and performance issues in large applications. The recommended solution is to split state into multiple Contexts based on responsibility.

One-Line Interview Answer

One large Context causes all consumers to re-render on every Context update, leading to unnecessary renders and performance problems.
