How to Use Custom Hooks Instead of HOC in React

One-Line Definition

A Custom Hook is a plain JavaScript function starting with use that encapsulates reusable stateful logic — replacing the need for HOC wrappers entirely.

What Problem Does This Solve?

Higher-Order Components (HOCs) were the go-to pattern for sharing stateful logic before React Hooks. But they introduced wrapper hell, prop collisions, and poor readability.

Custom Hooks extract that same logic into a simple function you call directly inside your component — no wrapping needed.

Reusable Stateful Logic
↓
HOC wraps a component (old way)
↓
Custom Hook called inside component (modern way)
↓
Cleaner, composable, no wrapper hell

Example: Auth Guard

HOC Approach (Old)

// withAuth.jsx
function withAuth(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const [user, setUser] = useState(null);
    const [loading, setLoading] = useState(true);

    useEffect(() => {
      fetchUser().then(u => {
        setUser(u);
        setLoading(false);
      });
    }, []);

    if (loading) return <Spinner />;
    if (!user) return <Redirect to="/login" />;

    return <WrappedComponent {...props} user={user} />;
  };
}

// Usage — must wrap the component
export default withAuth(Dashboard);

Problems: extra component in the tree, prop collision risk, hard to compose multiple HOCs, and tricky to type in TypeScript.

Custom Hook Approach (Modern)

// useAuth.js
function useAuth() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetchUser().then(u => {
      setUser(u);
      setLoading(false);
    });
  }, []);

  return { user, loading };
}

// Dashboard.jsx — call it directly, no wrapping needed
function Dashboard() {
  const { user, loading } = useAuth();

  if (loading) return <Spinner />;
  if (!user) return <Redirect to="/login" />;

  return <div>Welcome, {user.name}</div>;
}

How It Works

↓

↓

↓

Example: Data Fetching Comparison

HOC Way — Composing Two HOCs

// Two HOCs composed — messy
function withData(WrappedComponent, url) {
  return function(props) {
    const [data, setData] = useState(null);

    useEffect(() => {
      fetch(url).then(r => r.json()).then(setData);
    }, []);

    return <WrappedComponent {...props} data={data} />;
  };
}

// Nesting two HOCs gets ugly fast
export default withAuth(withData(UserList, '/api/users'));

Custom Hook Way — Composing Two Hooks

// useFetch.js
function useFetch(url) {
  const [data, setData] = useState(null);
  const [error, setError] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch(url)
      .then(r => r.json())
      .then(setData)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [url]);

  return { data, error, loading };
}

// UserList.jsx — clean hook composition
function UserList() {
  const { user } = useAuth();                         // hook 1
  const { data, loading } = useFetch('/api/users');   // hook 2

  if (loading) return <Spinner />;

  return (
    <ul>
      {data.map(u => <li key={u.id}>{u.name}</li>)}
    </ul>
  );
}

Another Example: Mouse Position Tracker

HOC Version

function withMousePosition(WrappedComponent) {
  return function(props) {
    const [position, setPosition] = useState({ x: 0, y: 0 });

    const handleMouseMove = e => {
      setPosition({ x: e.clientX, y: e.clientY });
    };

    return (
      <div onMouseMove={handleMouseMove}>
        <WrappedComponent {...props} position={position} />
      </div>
    );
  };
}

export default withMousePosition(MyComponent);

Custom Hook Version

// useMousePosition.js
function useMousePosition() {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handleMouseMove = e => {
      setPosition({ x: e.clientX, y: e.clientY });
    };

    window.addEventListener('mousemove', handleMouseMove);
    return () => window.removeEventListener('mousemove', handleMouseMove);
  }, []);

  return position;
}

// Usage — no wrapper, no prop injection
function MyComponent() {
  const { x, y } = useMousePosition();

  return <h2>Mouse: {x}, {y}</h2>;
}

Advantages of Custom Hooks over HOC

  • No extra component added to the React tree.
  • No prop name collision between layers.
  • Multiple hooks compose naturally inside a single component.
  • Much easier to read, debug, and type with TypeScript.
  • Logic is just a function — easily unit-testable in isolation.
  • No need to understand wrapping or injection patterns.

When to Still Use HOC

Situation Prefer
Pure logic reuse (auth, data fetching, timers) Custom Hook
Composing multiple behaviors Custom Hook
Logic + JSX wrapping (e.g. Error Boundary) HOC
Class components (legacy codebases) HOC
Third-party wrappers (React.memo, forwardRef) HOC

Important Interview Points

  • A Custom Hook is any function starting with use that calls other hooks.
  • It replaces HOC when the HOC does not render any JSX of its own.
  • Hooks compose cleanly — call multiple hooks side by side, no nesting.
  • No wrapper component is added to the React DevTools tree.
  • Hooks are easier to test in isolation as plain functions.
  • If a HOC only injects props via logic, convert it to a hook.

Short Interview Answer

Custom Hooks replace HOCs by moving shared stateful logic into a plain use-prefixed function that components call directly. This eliminates wrapper components, removes prop collision risk, and makes composition trivial — just call multiple hooks side by side inside one component.

One-Line Interview Answer

If a HOC only injects logic and not JSX, it should be a Custom Hook — a plain function starting with use that components call directly to access shared state or side effects.
