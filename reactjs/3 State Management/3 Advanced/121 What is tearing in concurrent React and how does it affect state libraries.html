121. What is tearing in concurrent React and how does it affect state libraries?

One-Line Answer

Tearing happens when different parts of the UI temporarily see different versions of the same state during Concurrent React rendering.

First Understand Normal React

Before React 18:

State Changes
↓ Render
↓ Commit UI

Rendering was mostly synchronous.

All components usually saw the same state version.

What Changed In React 18?

React introduced Concurrent Rendering.

Start Render
↓ Pause Render
↓ Resume Later
↓ Commit UI

React can now pause rendering and continue later.

The Problem

Imagine a Redux store:

count = 5

React starts rendering two components.

CounterA.jsx
function CounterA() {
  const count = useSelector(
    state => state.count
  );

  return <h2>A: {count}</h2>;
}
CounterB.jsx
function CounterB() {
  const count = useSelector(
    state => state.count
  );

  return <h2>B: {count}</h2>;
}

Normal Expected Result

A: 5
B: 5

Tearing Scenario

React starts rendering:

Render Starts
↓ CounterA Reads count=5

Before React finishes:

Redux Store Updates
↓ count=6

Now React continues rendering:

CounterB Reads count=6

UI may temporarily become:

A: 5
B: 6

Both components are showing different versions of the same state.

This inconsistency is called Tearing.

Visual Timeline

Store = 5
↓ CounterA Reads 5
↓ Store Changes To 6
↓ CounterB Reads 6
↓ UI Shows
A:5
B:6

Why Is This Bad?

Imagine an e-commerce checkout page:

Cart Total
₹500

Item Count
4 Items

Store updates while rendering.

UI could temporarily show:

Cart Total = ₹600
Item Count = 4

The numbers no longer match.

Why State Libraries Were Affected?

Libraries such as:

  • Redux
  • Zustand
  • MobX
  • Custom Stores

keep state outside React.

React State
↓ Safe

External Store
↓ Can Tear

Because React does not control external state.

Old Subscription Pattern

Before React 18
useEffect(() => {

  return store.subscribe(
    () => setState(
      store.getState()
    )
  );

}, []);
This pattern was not fully safe for Concurrent React.

React's Solution

React introduced:

useSyncExternalStore
useSyncExternalStore guarantees that all components read the same snapshot of external state during rendering.

How useSyncExternalStore Prevents Tearing

React Starts Render
↓ Take Snapshot
↓ All Components Read Same Snapshot
↓ Render Completes

Even if the store changes during render:

React Detects Change
↓ Restart Render
↓ Use New Snapshot

Real Redux Example

Modern React Redux internally uses:

Internal Concept
useSyncExternalStore(
  subscribe,
  getSnapshot
);

So when you write:

Redux Code
const user = useSelector(
  state => state.user
);

React Redux uses useSyncExternalStore internally.

This prevents tearing in Concurrent React.

Tearing vs No Tearing

Without Protection With useSyncExternalStore
A:5 B:6 A:6 B:6
Inconsistent UI Consistent UI
Tearing No Tearing

Relationship With Previous Question

React introduced useSyncExternalStore specifically to solve tearing and inconsistent snapshots when external stores are used with Concurrent React.
External Store
↓ Concurrent Rendering
↓ Tearing Risk
↓ useSyncExternalStore
↓ Consistent Snapshot

Interview Answer

Tearing is a rendering inconsistency where different components temporarily read different versions of the same state during Concurrent React rendering. It mainly affects external state libraries such as Redux, Zustand, MobX, and custom stores because their state exists outside React. React introduced useSyncExternalStore to ensure that all components read the same state snapshot during rendering, preventing tearing and keeping the UI consistent.
