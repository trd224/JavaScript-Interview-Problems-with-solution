118. What is the Flux Architecture Pattern?

One-Line Answer

Flux is an architectural pattern created by Facebook that enforces one-way data flow to make state changes predictable and easier to debug.

Why Flux Was Created?

Before Flux, large applications often had state flowing in multiple directions.

Component A
↔ Component B
↔ Component C
↔ Component D

As applications grew, it became difficult to know:

  • Who changed the state?
  • When was the state changed?
  • Why did the UI update?
Facebook created Flux to make state changes predictable.

Main Idea Of Flux

Data should always move in one direction.
User Interaction
↓ Action
↓ Dispatcher
↓ Store
↓ View
↓ User Interaction

Notice that data never moves backward.

Flux Components

Part Purpose
Action Describes what happened
Dispatcher Sends actions to stores
Store Holds application state
View Displays state to users

Why Dispatcher Exists?

Original Flux often had multiple stores.

Dispatcher
↓ UserStore
CartStore
ProductStore

Dispatcher's job was:

Receive Action
↓ Send To Correct Stores

Advantages Of Flux

  • Predictable state updates
  • Easier debugging
  • Clear data flow
  • Better scalability
  • Less accidental state mutations

Flux vs Traditional Two-Way Flow

Traditional Flux
State can change from many directions State changes follow one path
Hard to debug Easier to debug
Complex dependencies Predictable flow

Interview Answer

Flux is an architectural pattern created by Facebook that enforces one-way data flow using Actions, a Dispatcher, Stores, and Views. When a user performs an action, the action is sent to a Dispatcher, which forwards it to the appropriate Store. The Store updates its state and the View re-renders. Flux was designed to make state management predictable and easier to debug in large applications.
