119. What is redux-saga and how does it differ from redux-thunk?

One-Line Answer

Redux Thunk and Redux Saga are middleware for handling async work in Redux. Thunk runs async logic inside action creators, while Saga listens for actions and runs async logic in separate saga files.

Why Do We Need Thunk Or Saga?

Redux reducers must be pure. They should not call APIs directly.

Component
↓ Async API Call
↓ Success / Error
↓ Reducer Updates Store
Reducers cannot directly do API calls, setTimeout, localStorage, WebSocket, or other side effects.

Real World Example - Product Loading

We want to load products from API and show:

  • Loading state
  • Success data
  • Error message
Page Opens
↓ Show Loading
↓ Call Products API
↓ Success OR Error

Common Redux State

productReducer.js
const initialState = { products: [], loading: false, error: null };

function productReducer(state = initialState, action) {
  switch (action.type) {
    case 'PRODUCTS_LOADING':
      return { ...state, loading: true, error: null };

    case 'PRODUCTS_SUCCESS':
      return { ...state, loading: false, products: action.payload };

    case 'PRODUCTS_ERROR':
      return { ...state, loading: false, error: action.payload };

    default:
      return state;
  }
}

Redux Thunk Example

Thunk means the action creator itself does the async work.
productThunk.js
export const fetchProducts = () => async dispatch => {
  dispatch({ type: 'PRODUCTS_LOADING' });

  try {
    const products = await (await fetch('/api/products')).json();
    dispatch({ type: 'PRODUCTS_SUCCESS', payload: products });
  } catch {
    dispatch({ type: 'PRODUCTS_ERROR', payload: 'Failed to load products' });
  }
};
Products.jsx
useEffect(() => {
  dispatch(fetchProducts());
}, []);
dispatch(fetchProducts())
↓ Thunk Function Runs
↓ PRODUCTS_LOADING
↓ API Call
↓ PRODUCTS_SUCCESS / PRODUCTS_ERROR

Redux Saga Example

Saga means the component dispatches a normal action, and saga watches that action.
Products.jsx
useEffect(() => {
  dispatch({ type: 'FETCH_PRODUCTS' });
}, []);
productSaga.js
import { call, put, takeEvery } from 'redux-saga/effects';

function* fetchProducts() {
  try {
    yield put({ type: 'PRODUCTS_LOADING' });
    const products = yield call(api.getProducts);
    yield put({ type: 'PRODUCTS_SUCCESS', payload: products });
  } catch {
    yield put({ type: 'PRODUCTS_ERROR', payload: 'Failed to load products' });
  }
}

export function* productSaga() {
  yield takeEvery('FETCH_PRODUCTS', fetchProducts);
}
dispatch({ type:'FETCH_PRODUCTS' })
↓ Saga Watches Action
↓ PRODUCTS_LOADING
↓ API Call
↓ PRODUCTS_SUCCESS / PRODUCTS_ERROR

UI Component

Products.jsx
function Products() {
  const { products, loading, error } = useSelector(state => state.products);

  if (loading) return <h2>Loading...</h2>;
  if (error) return <h2>{error}</h2>;

  return (
    <ul>
      {products.map(product => <li key={product.id}>{product.name}</li>)}
    </ul>
  );
}

Thunk vs Saga Flow

Redux Thunk Redux Saga
dispatch(fetchProducts()) dispatch({ type:'FETCH_PRODUCTS' })
Thunk function runs Saga watches action
API logic inside action creator API logic inside saga file
Uses async/await Uses generator + yield
Simple API calls Complex async workflows

When Saga Is Better?

Saga is useful when async logic becomes complex.

  • Debounce search
  • Cancel previous API request
  • Retry failed request
  • Polling
  • Parallel API calls
Debounce Example
import { debounce } from 'redux-saga/effects';

export function* searchSaga() {
  yield debounce(500, 'SEARCH_USERS', searchUsers);
}

Interview Answer

Redux Thunk and Redux Saga are Redux middleware used to handle side effects such as API calls. Redux Thunk allows action creators to return functions, so async logic is written inside action creators. Redux Saga uses generator functions that listen for dispatched actions and run side effects separately in saga files. Thunk is simpler and commonly used for normal API calls, while Saga is more powerful for complex workflows like debounce, cancellation, retry, polling, and parallel tasks.
