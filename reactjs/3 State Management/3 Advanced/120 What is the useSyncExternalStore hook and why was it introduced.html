120. What is the useSyncExternalStore Hook and Why Was It Introduced?

One-Line Answer

useSyncExternalStore is a React 18 hook used to safely connect React components with external stores, especially in Concurrent React, so the UI does not show mixed or torn state.

First Clear The Biggest Confusion

External store does not mean only Redux.

External store means any state that lives outside React.

State Type Example
React State useState, useReducer, Context
External Store Redux, Zustand, MobX, custom store, browser API, WebSocket state
Redux is only one example of an external store.

React State vs External Store

React State
↓ React Owns It
↓ React Knows When It Changes

External Store
↓ State Lives Outside React
↓ React Does Not Automatically Know

Example Of React State

Counter.jsx
function Counter() {
  const [count, setCount] = useState(0);

  return <button onClick={() => setCount(count + 1)}>Count: {count}</button>;
}
Here React owns count, so React automatically re-renders when setCount runs.

Why Is This Called An External Store?

Many developers think external store means Redux only.

External Store does not mean Redux.

External store means any state that lives outside React.

External State Example

externalStore.js
let count = 0;

export const store = {
  getState: () => count,
  increment: () => { count++; }
};
Counter.jsx
function Counter() {
  const count = store.getState();

  return <h1>Count: {count}</h1>;
}
count Lives In externalStore.js
↓ React Does Not Own It
↓ React Only Reads It
Because count lives in externalStore.js instead of inside the React component, it is called external state or an external store.

Redux Is The Same Idea

Redux Store
const store = createStore(reducer);

Redux state lives inside the Redux store, not inside React components.

Redux Store
↓ State Lives Outside React
↓ External Store

Zustand Is Also An External Store

zustandStore.js
const useStore = create(() => ({
  count: 0
}));
State Lives Inside Zustand
↓ Outside React
↓ External Store

MobX Is Also An External Store

counterStore.js
class CounterStore {
  count = 0;
}
State Lives Inside MobX
↓ Outside React
↓ External Store

Real Definition Of External Store

Any state that lives outside React's useState, useReducer, or Context is considered an External Store.
Internal React State External Store
useState Redux
useReducer Zustand
Context MobX
React Owns State State Lives Outside React
Now the purpose of useSyncExternalStore becomes clearer: React created it to safely connect React components to any external store, not just Redux.

Why Subscription Is Needed?

React must be informed when external store changes.

External Store Changes
↓ Subscription Fires
↓ React Is Notified
↓ Component Re-renders

Wrong Direct Reading Example

If you directly read Redux or custom store without subscription:

Wrong Pattern
function Counter() {
  const count = store.getState().count;

  return <h1>Count: {count}</h1>;
}

Then the store may update: 

store.dispatch({ type: 'INCREMENT' });

But React will not automatically know.

Store changed, but React component may not re-render because React was not subscribed.

Old Way Before React 18

Before React 18, libraries commonly used useEffect + store.subscribe + setState.

Old Manual Subscription
function Counter() {
  const [count, setCount] = useState(store.getState().count);

  useEffect(() => {
    return store.subscribe(() => {
      setCount(store.getState().count);
    });
  }, []);

  return <h1>Count: {count}</h1>;
}

How Old Way Worked

Store Changes
↓ store.subscribe() Callback Runs
↓ setCount(store.getState().count)
↓ React Re-renders
Before React 18, this old pattern was mostly fine because React rendering was mostly synchronous.

Important: Old Pattern Was Not Created For Tearing

Old subscribe pattern was used only to tell React: "External store changed, please re-render."

Before React 18:

Store Changes
↓ setState()
↓ Render Starts
↓ Render Finishes
↓ Commit UI

React usually did not pause rendering in the middle.

What Changed In React 18?

React 18 introduced Concurrent Rendering.

Render Starts
↓ React Can Pause
↓ External Store Can Change
↓ React Can Resume
↓ Commit UI
Now external stores need stronger safety because components can read mixed state snapshots.

What Is Tearing?

Tearing means different components see different versions of the same external state during one render.

Real-World Shopping Cart Example

Store has: 

{
  cartItems: 4,
  totalPrice: 500
}

Correct UI:

Cart Items: 4
Total Price: ₹500

User adds one item. Store becomes: 

{
  cartItems: 5,
  totalPrice: 600
}

Without Tearing

Cart Items: 5
Total Price: ₹600

With Tearing

React Starts Render
↓ CartItems Reads Old State: 4
↓ Store Changes
↓ CartTotal Reads New State: ₹600

Bad UI:

Cart Items: 4
Total Price: ₹600
This is tearing because the UI is showing old and new state together.

Why useSyncExternalStore Was Introduced?

React introduced useSyncExternalStore to make external store subscriptions safe with Concurrent Rendering.

It gives React two things:

  • subscribe - how to listen when external store changes.
  • getSnapshot - how to read the current consistent value.

New Way In React 18

Syntax
const value = useSyncExternalStore(subscribe, getSnapshot);
Argument Meaning
subscribe Registers a listener for store changes
getSnapshot Returns the current value from the store
getServerSnapshot Optional snapshot for server rendering

How It Prevents Tearing

React Takes Snapshot
↓ All Components Read Same Snapshot
↓ If Store Changes During Render
↓ React Detects It
↓ React Re-renders With New Snapshot

Custom Store Example

This is not Redux. This is a small custom external store to understand the hook directly.

store.js
let count = 0;
const listeners = new Set();

export const store = {
  getSnapshot: () => count,

  subscribe: listener => {
    listeners.add(listener);
    return () => listeners.delete(listener);
  },

  increment: () => {
    count++;
    listeners.forEach(listener => listener());
  }
};
Counter.jsx
import { useSyncExternalStore } from 'react';
import { store } from './store';

function Counter() {
  const count = useSyncExternalStore(store.subscribe, store.getSnapshot);

  return (
    <>
      <h1>Count: {count}</h1>
      <button onClick={store.increment}>Increment</button>
    </>
  );
}
This example uses useSyncExternalStore directly because we created our own custom external store.

Why Not Use Redux Here?

If we use Redux, React Redux hides the internal mechanism behind useSelector.

Normal Redux Usage
const count = useSelector(state => state.count);

This is what you normally write in Redux apps.

In normal Redux, you do not write useSyncExternalStore manually.

Redux vs React Redux

Library Responsibility
Redux Creates store, dispatches actions, updates state, provides subscribe()
React Redux Connects Redux store to React using hooks like useSelector()
Redux itself does not use useSyncExternalStore because Redux is not React-specific.

Important Correction

Wrong: Redux uses useSyncExternalStore internally.
Correct: React Redux uses useSyncExternalStore internally.

React Redux Internal Idea

When you write:

Your Code
const count = useSelector(state => state.count);

React Redux conceptually does something like:

Simplified Internal Idea
function useSelector(selector) {
  const state = useSyncExternalStore(store.subscribe, store.getState);

  return selector(state);
}
Real React Redux is more advanced. It also has equality checks and optimizations.

What Happens After Redux Update?

User Click / API / WebSocket
↓ dispatch(action)
↓ Redux Store Updates
↓ store.subscribe() Notifies React Redux
↓ React Redux uses useSyncExternalStore
↓ useSelector Runs Selector
↓ Component Re-renders If Selected Value Changed

Example With useSelector

Redux state: 

{
  user: { name: 'Rizwan' },
  cart: { items: [] }
}

Component selects user: 

const user = useSelector(state => state.user);

If Cart Changes

Store Updates
↓ Subscription Fires
↓ Selector Runs
↓ user Same
↓ No Re-render

If User Changes

Store Updates
↓ Subscription Fires
↓ Selector Runs
↓ user Changed
↓ Component Re-renders

Old Way vs New Way

Old Way New Way
useEffect + store.subscribe + setState useSyncExternalStore
Manual subscription sync React-supported external store sync
Mostly okay before React 18 Safe for Concurrent Rendering
Possible tearing in React 18 Prevents tearing

When Do You Use useSyncExternalStore Directly?

  • When building your own external store
  • When building a state management library
  • When subscribing to browser APIs
  • When creating custom global state outside React
Normal Redux developers usually use useSelector(), not useSyncExternalStore().

Final Interview Answer

useSyncExternalStore is a React 18 hook introduced to safely subscribe React components to external stores. An external store is any state that lives outside React, such as Redux, Zustand, MobX, a custom store, or browser API state. Before React 18, libraries often used useEffect with store.subscribe and setState to update React when an external store changed. That worked mostly fine with synchronous rendering, but React 18 Concurrent Rendering can pause and resume renders, which may cause tearing where different components see different versions of the same store. useSyncExternalStore solves this by giving React a standard way to subscribe to external stores and read consistent snapshots. In Redux apps, developers normally use useSelector(); React Redux internally uses useSyncExternalStore and store.subscribe for safe updates.

Easy Way To Remember

Custom External Store
↓ You May Use useSyncExternalStore Directly

Redux Store
↓ You Use useSelector()
↓ React Redux Internally Uses useSyncExternalStore
