Part 1: Why Do We Need Redux?

One-Line Definition

Redux is a state management library that stores shared application state in one centralized store, so any component can read or update that state in a predictable way.

First Understand the Real Problem

Imagine we are building an e-commerce application like Amazon or Flipkart.

Our application has many pages and components: 

App
│
├── Navbar
├── ProductList
├── ProductDetails
├── CartPage
├── CheckoutPage
├── WishlistPage
├── ProfilePage
└── OrderHistoryPage

Now think about shared data:

  • User data is needed in Navbar, ProfilePage, CheckoutPage, and OrderHistoryPage.
  • Cart data is needed in Navbar, CartPage, CheckoutPage, and ProductDetails.
  • Wishlist data is needed in ProductDetails and WishlistPage.
  • Order data is needed in OrderHistoryPage and ProfilePage.
The problem is not only passing props. The real problem is that many components need the same shared data and many components can update that data.

Problem Without Redux

Suppose App owns shared state for an e-commerce application. 

const [cart, setCart] = useState([]);

const [user, setUser] = useState(null);

const [wishlist, setWishlist] = useState([]);

const [orders, setOrders] = useState([]);
The code above is the actual React state.

Now many components need to read this same state.

State Components That Need It
cart Navbar, ProductDetails, CartPage, CheckoutPage
user Navbar, ProfilePage, CheckoutPage, OrderHistoryPage
wishlist ProductDetails, WishlistPage
orders ProfilePage, OrderHistoryPage

If we do not use Redux, App often passes this state and update functions to other components through props. 

<Navbar cart={cart} user={user} />

<ProductDetails
  cart={cart}
  setCart={setCart}
  wishlist={wishlist}
  setWishlist={setWishlist}
/>

<CartPage
  cart={cart}
  setCart={setCart}
/>

<CheckoutPage
  cart={cart}
  user={user}
/>

<ProfilePage
  user={user}
  orders={orders}
/>
Important: cart, user, wishlist, and orders are state. When we pass them to components, they become props for those components.

State vs Props

Code What Is It?
const [cart, setCart] = useState([]) State inside App
const [user, setUser] = useState(null) State inside App
<Navbar cart={cart} /> Props passed to Navbar
<CartPage cart={cart} setCart={setCart} /> Props passed to CartPage

The real Redux problem is not simply that App has too much state.

The bigger problem is that the same shared state is needed by many components and updated by many components.

Same State, Many Readers

  • Navbar reads cart to show cart count.
  • CartPage reads cart to show cart items.
  • CheckoutPage reads cart to calculate total amount.
  • ProductDetails reads wishlist to show whether the product is wishlisted.

Same State, Many Writers

  • ProductDetails updates cart when the user clicks Add To Cart.
  • WishlistPage updates cart when the user clicks Move To Cart.
  • CartPage updates cart when the user increases quantity or removes an item.
  • CheckoutPage updates cart when the order is completed and cart is cleared.

Why Need One Predictable Store?

At this point, many developers get confused.

If many components can already read and update the same state, why do we need Redux?

The answer is not simply that Redux allows shared state.

The real reason is that when many components update the same state, we need one source of truth and one controlled update flow so that state changes remain predictable.

First Important Correction

If cart state is created in App and App passes cart and setCart to child components, then there is only one cart state. 

function App() {
  const [cart, setCart] = useState([]);

  return (
    <>
      <Navbar cart={cart} />
      <ProductDetails cart={cart} setCart={setCart} />
      <CartPage cart={cart} setCart={setCart} />
      <CheckoutPage cart={cart} setCart={setCart} />
    </>
  );
}

In this case, ProductDetails, CartPage, and CheckoutPage are not creating separate cart states.

They are all using the same cart state from App. 

App cart state
     ↓
Navbar reads cart
     ↓
ProductDetails updates cart using setCart
     ↓
CartPage reads latest cart
     ↓
CheckoutPage reads latest cart
So in this normal React example, sync is not the real problem. All components are connected to the same App state.

Then Why Redux?

The real problem is not that cart becomes out of sync.

The real problem is that cart update logic becomes scattered across many components. 

ProductDetails
setCart([...cart, product]);

WishlistPage
setCart([...cart, productFromWishlist]);

CartPage
setCart(cart.filter(item => item.id !== id));

CheckoutPage
setCart([]);

All components are updating the same cart, so the data stays synchronized.

But every component has its own cart update logic.

State is shared, but update logic is scattered. This is one important problem Redux solves in large applications.
Main Reason To Use Redux: Redux is useful when large applications have complex shared state that is read and updated from many places, and we need a centralized, predictable, and debuggable update flow.

Predictable Means One Controlled Update Flow

Redux says: do not write cart update logic separately inside every component.

Instead, keep cart update logic in one place. 

reducers: {
  addToCart,
  removeFromCart,
  increaseQuantity,
  clearCart
}

Now every component updates cart through the same controlled flow. 

Component
     ↓
dispatch(action)
     ↓
Reducer
     ↓
Redux Store Updated
     ↓
All Components Receive Latest Cart

Everyone sees the same data, and every update happens through the same reducer logic.

Redux Solution

Redux says:

  • Keep one shared store.
  • Keep one update flow.
  • Force all components to use the same update logic.
Component
     ↓
dispatch(action)
     ↓
Reducer
     ↓
Redux Store Updated
     ↓
All Components Receive Latest State
This is why Redux is called a predictable state container. We always know where the state lives and exactly how it was changed.
Redux is useful when many components need to read and update the same shared state from one reliable central place.
Shared State
↓
Many Components Read It
↓
Many Components Update It
↓
Update Logic Becomes Scattered
↓
Need One Controlled Update Flow
↓
Redux

Problem 1: Prop Drilling

Suppose only AddToCartButton needs setCart. 

App
│
└── ProductPage
      │
      └── ProductDetails
            │
            └── AddToCartButton

Without Redux:

App.jsx

<ProductPage cart={cart} setCart={setCart} />

ProductPage.jsx

<ProductDetails cart={cart} setCart={setCart} />

ProductDetails.jsx

<AddToCartButton cart={cart} setCart={setCart} />

AddToCartButton.jsx

function AddToCartButton({ cart, setCart, product }) {
  function handleAddToCart() {
    setCart([...cart, product]);
  }

  return <button onClick={handleAddToCart}>Add To Cart</button>;
}
ProductPage and ProductDetails may not need cart directly. They only pass it forward. This is prop drilling.

Problem 2: Scattered State

Scattered state means shared application state is stored in many different components and files.

At the beginning, this may look normal because each page manages its own data.

Small App Example

Navbar
ProductPage
CartPage
ProfilePage

In a small app, App can manage shared state. 

const [cart, setCart] = useState([]);
const [user, setUser] = useState(null);

This is still manageable.

Large E-Commerce App Problem

After the app grows, different developers may create state in different pages. 

Cart State
    ↓
CartPage.jsx

Wishlist State
    ↓
WishlistPage.jsx

User State
    ↓
ProfilePage.jsx

Orders State
    ↓
OrderHistoryPage.jsx

Now shared state is spread across many files.

This is called scattered state because shared application state is scattered across different components instead of living in one clear place.

Why Is This a Problem?

Suppose CheckoutPage needs cart, user, and orders.

Now we have to ask:

  • Where is cart stored?
  • Where is user stored?
  • Where are orders stored?
  • Which component updates them?
  • Which file should we modify?

The answers may be: 

Cart     → CartPage.jsx
User     → ProfilePage.jsx
Orders   → OrderHistoryPage.jsx

Now we must jump between multiple files just to understand one checkout flow.

The bigger the application becomes, the harder it becomes to locate, share, update, and debug scattered state.

Redux Solution

Redux says: keep shared state in one centralized store. 

{
  auth: {
    user: null
  },
  cart: {
    items: []
  },
  wishlist: {
    items: []
  },
  orders: {
    list: []
  }
}

Now every developer knows where shared state lives.

Redux Store
↓
auth
cart
wishlist
orders
With Redux, cart, user, wishlist, and orders are not hidden in different pages. They live in one predictable store.

Problem 3: Update Logic Becomes Scattered

Cart can be updated from many places:

  • ProductDetails → Add product to cart
  • WishlistPage → Move item to cart
  • CartPage → Increase/decrease quantity
  • CheckoutPage → Clear cart after order

Without Redux, cart update logic may be repeated in many components.

Redux keeps update logic inside reducers, so state changes become predictable and easier to debug.

The Biggest Reason Redux Exists

Many tutorials say Redux solves prop drilling.

That is true, but Context can also solve prop drilling.

The bigger reason Redux exists is that the same state is often read and updated from many different components.

Cart Example

  • ProductDetails → Add To Cart
  • WishlistPage → Move To Cart
  • CartPage → Increase Quantity
  • CartPage → Remove Item
  • CheckoutPage → Clear Cart After Order

All these components update the same cart state.

Redux provides one central store and one predictable update flow, so every component works with the same source of truth.
Component
↓
dispatch(action)
↓
Reducer
↓
Redux Store Updated
↓
All Components See Latest State

Redux Solution

Redux creates one central store. 

{
  auth: {
    user: null
  },
  cart: {
    items: []
  },
  wishlist: {
    items: []
  },
  orders: {
    list: []
  }
}

Now components do not need props from App.

Any component can read required data from Redux.

Component
↓
useSelector reads state
↓
dispatch sends action
↓
Reducer updates store
↓
Subscribed components update

Part 1 Final Summary

Redux is needed when our application has shared state that is used by many components, updated from many places, and becoming difficult to manage, debug, and keep organized with only props, Context, or local state.
