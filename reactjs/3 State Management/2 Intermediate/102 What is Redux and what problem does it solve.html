102. What is Redux and What Problem Does It Solve?

One-Line Definition

Redux is a centralized state management library that stores shared application state in a single store and allows any component to read or update that state predictably.

Before Learning Redux, Understand the Real Problem

Many developers think Redux exists only to solve prop drilling.

That is partially true, but Redux became popular because large applications faced much bigger state management problems.

Redux solves prop drilling, scattered state, scattered update logic, difficult debugging, and large-scale state management.

Problem #1 - Prop Drilling

Suppose we are building an e-commerce application.

The logged-in user is stored in App. 

const [user, setUser] = useState({
  id: 1,
  name: 'Rizwan'
});

Only UserInfo needs user data. 

App
│
└── ProductPage
      │
      └── ProductDetails
            │
            └── UserInfo

Without Redux:

App.jsx

<ProductPage user={user} />

ProductPage.jsx

<ProductDetails user={user} />

ProductDetails.jsx

<UserInfo user={user} />

ProductPage does not use user.

ProductDetails does not use user.

They only pass it forward.

Passing data through multiple intermediate components is called Prop Drilling.

Can Context Solve This?

Yes.

Context can solve prop drilling very easily.

So if prop drilling were the only problem, Redux would not be necessary.

Problem #2 - State Scattered Everywhere

Imagine a real e-commerce application.

The application contains:

  • User Authentication
  • Products
  • Cart
  • Wishlist
  • Orders
  • Coupons
  • Notifications
  • Addresses
  • Payments

Many screens need access to this data. 

Navbar

Home Page

Product Page

Cart Page

Checkout Page

Order History

Profile Page

Without Redux: 

Cart State In Navbar

Cart State In Product Page

Cart State In Checkout

Wishlist State In Product Page

User State In Profile

Orders State In History

Now state becomes scattered throughout the application.

As the application grows, it becomes difficult to know where state lives and who is responsible for updating it.

Why Scattered State Becomes Difficult

The previous example does not mean we create multiple copies of cart state in different components.

The real problem is that many parts of the application need access to the same data.

E-Commerce Example

Suppose a user adds a product to the cart: 

{
  id: 1,
  name: 'Laptop',
  price: 50000
}

Now several screens need cart information:

  • Navbar → Show cart count
  • Cart Page → Show cart items
  • Checkout Page → Calculate total amount

Similarly, user information is needed by:

  • Navbar
  • Profile Page
  • Order History Page

Wishlist information is needed by:

  • Product Page
  • Wishlist Page
The problem is not multiple copies of state. The problem is that the same state is required by many different parts of the application.

Without Redux

A common solution is to move everything to App. 

const [cart, setCart] = useState([]);

const [wishlist, setWishlist] = useState([]);

const [user, setUser] = useState(null);

const [orders, setOrders] = useState([]);

Now App becomes responsible for managing all shared state.

As the application grows, more and more state gets added: 

const [cart, setCart] = useState([]);

const [wishlist, setWishlist] = useState([]);

const [user, setUser] = useState(null);

const [orders, setOrders] = useState([]);

const [notifications, setNotifications] = useState([]);

const [addresses, setAddresses] = useState([]);
Eventually App becomes a giant state container and many components depend on it. This is where centralized state management tools such as Redux become useful.

Problem #3 - Shared State Everywhere

Suppose a user adds a product to the cart.

The following components must update:

  • Navbar Cart Count
  • Cart Page
  • Checkout Page
  • Mini Cart Popup

Without centralized state management, synchronizing all these components becomes difficult.

Problem #4 - Update Logic Becomes Scattered

Cart can be updated from:

  • Product Page
  • Wishlist Page
  • Cart Page
  • Checkout Page

Without Redux: 

Cart Logic In Product Page

Cart Logic In Wishlist

Cart Logic In Cart Page

Cart Logic In Checkout

Now business logic becomes duplicated across the application.

Problem #5 - Difficult Debugging

Imagine:

  • Cart Count Wrong
  • Order Total Incorrect
  • Coupon Not Applied
  • Wishlist Not Updated

Question:

Which component changed the state?

In large applications this becomes difficult to trace.

Redux Solution

Redux creates a centralized store.

Redux Store
↓
Single Source Of Truth
↓
All Components Read State
↓
All Components Update State Through Actions

Redux Store Example

{
  user: {},
  cart: [],
  wishlist: [],
  orders: [],
  notifications: []
}
Instead of storing state in many components, Redux stores shared state in one place.

How Redux Works

Component
↓
Dispatch Action
↓
Reducer
↓
Store Updated
↓
Subscribed Components Re-render

Real Cart Example

User clicks Add To Cart. 

dispatch({
  type: 'ADD_TO_CART',
  payload: product
});

Reducer

function cartReducer(state, action) {

  switch(action.type) {

    case 'ADD_TO_CART':

      return {
        ...state,
        cart: [
          ...state.cart,
          action.payload
        ]
      };

    default:
      return state;
  }

}

Store Updated

{
  cart: [
    {
      id: 101,
      name: 'Laptop'
    }
  ]
}

Now:

  • Navbar Cart Count Updates
  • Cart Page Updates
  • Checkout Updates
  • Mini Cart Updates

All automatically read the latest cart data from Redux.

Why Redux Is Better Than Context For Large Apps

Context re-renders all consumers whenever the Provider value changes.

Redux allows components to subscribe only to the state they need. 

const cart =
  useSelector(
    state => state.cart.items
  );

const user =
  useSelector(
    state => state.auth.user
  );

If cart changes:

Cart Updated
↓
Cart Components Re-render

User components do not re-render.

Redux Toolkit (Modern Redux)

Today we usually use Redux Toolkit instead of writing raw Redux. 

import {
  configureStore
} from '@reduxjs/toolkit';
Redux Toolkit is the official recommended way to use Redux because it reduces boilerplate and simplifies setup.

Important Interview Points

  • Redux is a centralized state management library.
  • Redux solves prop drilling, but that is not its main advantage.
  • Redux provides a single source of truth.
  • Redux centralizes update logic inside reducers.
  • Redux improves debugging and maintainability.
  • Redux allows selective subscriptions using useSelector.
  • Redux Toolkit is the modern recommended approach.

Short Interview Answer

Redux is a centralized state management library that stores shared application state in a single store. It solves problems such as prop drilling, scattered state, duplicated update logic, and difficult debugging by providing predictable and centralized state management.

One-Line Interview Answer

Redux solves large-scale state management problems by providing a centralized store and predictable state updates.
