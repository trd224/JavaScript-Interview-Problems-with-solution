Redux Thunk (Professional Architecture) — Product Load & Display

Redux is a predictable state container that allows us to manage global state inside a single centralized store. Redux Thunk is a middleware that lets action creators return a function instead of a plain object, so we can perform async work (like an API call) before dispatching the actual action.

Why Redux Thunk?

A plain Redux action creator can only return an object — it has no way to wait for an API response. When we need to fetch data from a server (like a list of products), we need something that can:

  • Dispatch a "loading" action immediately
  • Call the API
  • Dispatch a "success" action with the data, or a "failure" action with the error

Thunk middleware makes this possible by letting dispatch() accept a function, not just an object.

Step 2 — Create Action Constants

redux/constants/productConstants.js

export const PRODUCT_LOAD_REQUEST = "PRODUCT_LOAD_REQUEST";
export const PRODUCT_LOAD_SUCCESS = "PRODUCT_LOAD_SUCCESS";
export const PRODUCT_LOAD_FAIL = "PRODUCT_LOAD_FAIL";
We use three constants instead of one because an async call has three possible stages: started, succeeded, and failed. The reducer will react differently to each one.

Step 3 — Create the Thunk Action Creator

redux/actions/productActions.js

import {
  PRODUCT_LOAD_REQUEST,
  PRODUCT_LOAD_SUCCESS,
  PRODUCT_LOAD_FAIL
} from "../constants/productConstants";

// A normal action creator returns an object.
// A thunk action creator returns a FUNCTION that receives dispatch.
export const loadProducts = () => {
  return async (dispatch) => {
    try {
      dispatch({ type: PRODUCT_LOAD_REQUEST });

      const response = await fetch("https://fakestoreapi.com/products");
      const data = await response.json();

      dispatch({
        type: PRODUCT_LOAD_SUCCESS,
        payload: data
      });

    } catch (error) {
      dispatch({
        type: PRODUCT_LOAD_FAIL,
        payload: error.message
      });
    }
  };
};
Instead of writing fetch logic directly inside components, we simply write dispatch(loadProducts()) The thunk middleware sees that loadProducts() returned a function (not an object), so it calls that function with dispatch automatically, instead of passing it straight to the reducer.

Step 4 — Create Reducer

redux/reducers/productReducer.js

import {
  PRODUCT_LOAD_REQUEST,
  PRODUCT_LOAD_SUCCESS,
  PRODUCT_LOAD_FAIL
} from "../constants/productConstants";

const initialState = {
  loading: false,
  products: [],
  error: null
};

export default function productReducer(state = initialState, action) {
  switch (action.type) {

    case PRODUCT_LOAD_REQUEST:
      return { ...state, loading: true, error: null };

    case PRODUCT_LOAD_SUCCESS:
      return { ...state, loading: false, products: action.payload };

    case PRODUCT_LOAD_FAIL:
      return { ...state, loading: false, error: action.payload };

    default:
      return state;
  }
}
Reducers never modify the existing state. They always return a brand-new state object — even during async flows, the reducer itself stays a pure, synchronous function. All the async work happens in the thunk, not here.

Step 5 — Create Root Reducer

redux/reducers/rootReducer.js

import { combineReducers } from "redux";
import productReducer from "./productReducer";

const rootReducer = combineReducers({
  productState: productReducer
});

export default rootReducer;
Even if there is only one reducer today, using a Root Reducer makes scaling much easier.

Step 6 — Create Store with Thunk Middleware

app/store.js

import { createStore, applyMiddleware } from "redux";
import { thunk } from "redux-thunk";
import rootReducer from "../redux/reducers/rootReducer";

const store = createStore(rootReducer, applyMiddleware(thunk));

export default store;
applyMiddleware(thunk) is what teaches the Store to accept functions from action creators, not just plain objects. Without this line, dispatching loadProducts() would throw an error.

Step 7 — Connect Redux with React

index.js

import React from "react";
import ReactDOM from "react-dom/client";
import { Provider } from "react-redux";
import store from "./app/store";
import App from "./App";

const root = ReactDOM.createRoot(document.getElementById("root"));

root.render(
  <Provider store={store}>
    <App />
  </Provider>
);
The Provider makes the Redux Store (and the thunk-enabled dispatch) available to every React component.

Step 8 — Trigger the Thunk using useDispatch() + useEffect()

features/products/ProductLoader.js

import { useEffect } from "react";
import { useDispatch } from "react-redux";
import { loadProducts } from "../../redux/actions/productActions";

export default function ProductLoader({ children }) {
  const dispatch = useDispatch();

  useEffect(() => {
    dispatch(loadProducts());
  }, [dispatch]);

  return children;
}
useDispatch() gives us the dispatch function. We call dispatch(loadProducts()) once when the component mounts, which kicks off the request → success/fail flow inside the thunk.

Step 9 — Read Data using useSelector()

features/products/ProductList.js

import { useSelector } from "react-redux";

export default function ProductList() {
  const { loading, products, error } = useSelector(
    (state) => state.productState
  );

  if (loading) return <p>Loading products...</p>;
  if (error) return <p>Error: {error}</p>;

  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>
          {product.title} — ${product.price}
        </li>
      ))}
    </ul>
  );
}
useSelector() allows us to read the loading, products, and error fields from the Store. Whenever the reducer returns new state, React automatically re-renders this component — so the UI moves through "Loading..." → list of products (or an error message) without any manual wiring.

Step 10 — App.js

import ProductLoader from "./features/products/ProductLoader";

Why Three Action Types Instead of One?

A synchronous action (like incrementing a counter) finishes instantly, so one action type is enough. An async action (like calling an API) takes time and can fail, so Redux Thunk convention is to dispatch three actions across the lifecycle of a single request — this lets the UI show loading spinners and error messages instead of just a final result.

❌ Wrong (no loading/error state)

export const loadProducts = () => async (dispatch) => {
  const res = await fetch("/products");
  const data = await res.json();
  dispatch({ type: "PRODUCTS_LOADED", payload: data });
};

✅ Correct (full request/success/fail lifecycle)

export const loadProducts = () => async (dispatch) => {
  dispatch({ type: PRODUCT_LOAD_REQUEST });
  try {
    const res = await fetch("/products");
    const data = await res.json();
    dispatch({ type: PRODUCT_LOAD_SUCCESS, payload: data });
  } catch (err) {
    dispatch({ type: PRODUCT_LOAD_FAIL, payload: err.message });
  }
};

Advantages of Using Constants

  • Avoids spelling mistakes.
  • Provides auto-completion in editors.
  • Easy to reuse across multiple reducers.
  • Central place for action names.

Advantages of Thunk Action Creators

  • Keeps API/async logic out of components.
  • Reusable across multiple components.
  • Handles loading and error states consistently.
  • Easy to extend (e.g. add pagination, filters, retries) later.

Advantages of Root Reducer

  • Supports multiple reducers (products, cart, user, etc.).
  • Scales well for enterprise applications.
  • Keeps store configuration clean.

Redux Thunk Interview Questions

1. What is Redux Thunk?

Redux Thunk is a middleware that allows action creators to return a function instead of a plain action object, so that asynchronous logic (like API calls) can run before dispatching the real action.

2. Why can't we call an API directly inside a normal action creator?

Because a normal action creator must return a plain object synchronously — it has no way to "wait" for a promise to resolve before returning.

3. What are the three common action types in an async thunk?

REQUEST (call started), SUCCESS (call completed with data), and FAIL (call failed with an error).

4. What does applyMiddleware(thunk) do?

It configures the Store so that dispatch() can accept and execute functions, not only plain objects.

5. Where should the fetch/axios call live?

Inside the thunk action creator, not inside the React component and not inside the reducer.

6. Why must reducers stay synchronous even in an async flow?

Because reducers must be pure functions — all asynchronous work is handled by the thunk before any action reaches the reducer.

7. What does useSelector() return in this example?

The current loading, products, and error fields from state.productState.

8. What triggers the initial data load?

A useEffect() hook that dispatches loadProducts() once when the component mounts.

Redux Thunk Flow in One Line

Component → Dispatch(thunk) → Thunk Runs → REQUEST → fetch() → SUCCESS / FAIL → Reducer → New State → Store Update → useSelector → UI Re-render
