103. What Are the Three Principles of Redux?

One-Line Definition

Redux is based on three principles: one central store, state is read-only, and reducers are responsible for creating the next state.

Why These Principles Exist

Redux is used when our application has shared state that is read and updated by many components.

To keep state predictable, Redux follows a fixed rule-based flow.

Component
↓
dispatch(action)
↓
Reducer
↓
Redux Store Updated
↓
React UI Re-renders

Principle 1: Single Source of Truth

The entire shared state of our application is stored in one central Redux store.

Without Redux, state may be stored in many different components. 

ProductPage  → own cart state
CartPage     → own cart state
Navbar       → own cart count state

This can make the application harder to manage when it grows.

With Redux, shared state lives in one store. 

{
  auth: {
    user: null
  },
  cart: {
    items: []
  },
  products: {
    list: []
  },
  orders: {
    list: []
  }
}

Now every component reads the required data from the same Redux store. 

const cartItems = useSelector(state => state.cart.items);

Real Project Example

Component Data Needed Read From
Navbar Cart count Redux store
CartPage Cart items Redux store
CheckoutPage Cart items and user Redux store
ProfilePage User data Redux store
Easy Meaning: One application should have one reliable place for shared state.

Principle 2: State Is Read-Only

Redux state cannot be changed directly from components. Components must dispatch actions.

A component should not directly modify Redux state.

Wrong: Directly changing state is not allowed in Redux. 
state.cart.items.push(product);

Instead, the component dispatches an action.

Correct: Tell Redux what happened by dispatching an action. 
dispatch(addToCart(product));

The action describes what happened in the application. 

{
  type: 'cart/addToCart',
  payload: product
}

Why Is This Important?

Because every state change becomes easy to track.

User clicks Add To Cart
↓
dispatch(addToCart(product))
↓
Redux receives action
↓
Reducer updates state
Easy Meaning: Components do not directly change state. They only send actions to Redux.

Principle 3: Changes Are Made With Pure Reducers

Reducers decide how Redux state changes based on the current state and action.

A reducer receives two things: 

(state, action)

Then it returns the next state.

Normal Redux Reducer Example

const initialState = {
  items: []
};

function cartReducer(state = initialState, action) {
  switch (action.type) {
    case 'ADD_TO_CART':
      return {
        ...state,
        items: [...state.items, action.payload]
      };

    default:
      return state;
  }
}

Reducer Rules

  • A reducer should return new state.
  • A reducer should not directly mutate existing state in normal Redux.
  • A reducer should not call APIs.
  • A reducer should not use timers.
  • A reducer should produce the same output for the same input.

Redux Toolkit Reducer Example

In Redux Toolkit, we can write simpler reducer code. 

import { createSlice } from '@reduxjs/toolkit';

const initialState = {
  items: []
};

const cartSlice = createSlice({
  name: 'cart',
  initialState,
  reducers: {
    addToCart(state, action) {
      state.items.push(action.payload);
    },

    removeFromCart(state, action) {
      state.items = state.items.filter(item => item.id !== action.payload);
    }
  }
});

export const { addToCart, removeFromCart } = cartSlice.actions;
export default cartSlice.reducer;
Redux Toolkit uses Immer internally, so this mutation-style syntax is safe inside createSlice reducers.
Easy Meaning: Reducers are the only place where Redux decides how state should change.

Complete Simple Flow

Suppose the user clicks the Add To Cart button.

Button Click
↓
dispatch(addToCart(product))
↓
Action goes to reducer
↓
Reducer adds product to cart
↓
Redux store updates
↓
Navbar and CartPage show latest cart

Important Interview Points

  • Redux keeps shared application state in one central store.
  • Redux state should not be modified directly from components.
  • Components update state by dispatching actions.
  • Reducers receive current state and action, then produce the next state.
  • This makes state changes predictable, traceable, and easier to debug.

Short Interview Answer

Redux has three principles. First, the entire shared state is stored in one central store. Second, state is read-only, so components cannot directly change it and must dispatch actions. Third, reducers handle those actions and produce the next state in a predictable way.

One-Line Interview Answer

Redux is based on one central store, read-only state, and reducers that create the next state from actions.
