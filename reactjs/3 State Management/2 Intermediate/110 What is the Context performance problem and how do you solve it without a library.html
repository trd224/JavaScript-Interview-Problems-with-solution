110. What is the Context Performance Problem and How Do You Solve It Without a Library?

One-Line Definition

The Context performance problem happens when many components consume the same Context value, and every consumer re-renders whenever that Context value changes, even if the component only needs a small part of the data.

Simple Meaning

React Context is useful for sharing global data.

But if we put too much state inside one Context, then many components can re-render unnecessarily.

Context is not bad. The problem happens when one large Context stores many unrelated values together.

Problem Example

Suppose our application has one big AppContext. 

const AppContext = createContext();

function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [cart, setCart] = useState([]);
  const [theme, setTheme] = useState('dark');
  const [notifications, setNotifications] = useState([]);

  return (
    <AppContext.Provider value={{ user, setUser, cart, setCart, theme, setTheme, notifications, setNotifications }}>
      {children}
    </AppContext.Provider>
  );
}

Now many components use this same Context. 

const { user } = useContext(AppContext);

const { cart } = useContext(AppContext);

const { theme } = useContext(AppContext);

What Goes Wrong?

If cart changes, the Provider value becomes a new object. 

value={{ user, setUser, cart, setCart, theme, setTheme, notifications, setNotifications }}

Because this object changes, all components using AppContext may re-render.

Cart Changes
↓
AppContext value changes
↓
All AppContext consumers re-render
↓
Even components that only use user or theme
This is the main Context performance problem.

Real Example

Component Uses Should Re-render When Cart Changes?
Navbar cart Yes
ProfilePage user No
ThemeButton theme No
NotificationBell notifications No
Only Navbar needs the updated cart, but other components may also re-render because they consume the same large Context.

Solution 1: Split Contexts

Do not keep all global state inside one big Context.

Split Context based on feature. 

Bad:

AppContext
├── user
├── cart
├── theme
└── notifications


Good:

UserContext
CartContext
ThemeContext
NotificationContext

CartContext.jsx

import { createContext, useContext, useState } from 'react';

const CartContext = createContext();

export function CartProvider({ children }) {
  const [cart, setCart] = useState([]);

  return (
    <CartContext.Provider value={{ cart, setCart }}>
      {children}
    </CartContext.Provider>
  );
}

export function useCart() {
  return useContext(CartContext);
}

UserContext.jsx

import { createContext, useContext, useState } from 'react';

const UserContext = createContext();

export function UserProvider({ children }) {
  const [user, setUser] = useState(null);

  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}
    </UserContext.Provider>
  );
}

export function useUser() {
  return useContext(UserContext);
}

App.jsx

<UserProvider>
  <CartProvider>
    <ThemeProvider>
      <App />
    </ThemeProvider>
  </CartProvider>
</UserProvider>
Now cart changes only affect CartContext consumers. UserContext consumers are not affected.

Solution 2: Memoize Provider Value

If Provider value is an object, memoize it with useMemo.

Without useMemo

<CartContext.Provider value={{ cart, setCart }}>
  {children}
</CartContext.Provider>

A new object is created on every render.

With useMemo

const value = useMemo(() => ({ cart, setCart }), [cart]);

return (
  <CartContext.Provider value={value}>
    {children}
  </CartContext.Provider>
);
useMemo prevents creating a new Context value object unless cart changes.

Solution 3: Separate State and Actions Context

Sometimes components only need actions, not state.

Example: 

const { addToCart } = useCart();

This component only needs the function. It does not need cart data.

So we can create two Contexts: 

CartStateContext
CartActionsContext

CartProvider.jsx

import { createContext, useContext, useMemo, useState } from 'react';

const CartStateContext = createContext();
const CartActionsContext = createContext();

export function CartProvider({ children }) {
  const [cart, setCart] = useState([]);

  const actions = useMemo(() => ({
    addToCart(product) {
      setCart(prev => [...prev, product]);
    },

    clearCart() {
      setCart([]);
    }
  }), []);

  return (
    <CartStateContext.Provider value={cart}>
      <CartActionsContext.Provider value={actions}>
        {children}
      </CartActionsContext.Provider>
    </CartStateContext.Provider>
  );
}

export function useCartState() {
  return useContext(CartStateContext);
}

export function useCartActions() {
  return useContext(CartActionsContext);
}

ProductCard.jsx

function ProductCard({ product }) {
  const { addToCart } = useCartActions();

  return <button onClick={() => addToCart(product)}>Add To Cart</button>;
}

Navbar.jsx

function Navbar() {
  const cart = useCartState();

  return <h3>Cart Items: {cart.length}</h3>;
}
ProductCard uses only actions, so it does not re-render when cart state changes.

Solution 4: Keep Local State Local

Do not put every state in Context.

Only put truly shared state in Context.

State Where Should It Live?
Modal open/close Local component state
Input field value Local component state
Dropdown open/close Local component state
Logged-in user Context or global state
Cart items Context or global state
Putting unnecessary local UI state into Context creates unnecessary global re-renders.

Solution 5: Use React.memo for Child Components

React.memo can help when parent components re-render but child props are the same. 

const ProductCard = React.memo(function ProductCard({ product, onAdd }) {
  return <button onClick={() => onAdd(product)}>Add To Cart</button>;
});
React.memo does not stop Context consumers from re-rendering when Context value changes, but it can help protect child components receiving unchanged props.

Best Practical Solution

For most projects, use this approach:

  • Split large Context into smaller feature Contexts.
  • Memoize Provider values using useMemo.
  • Separate state Context and actions Context when needed.
  • Keep local UI state local.
  • Use React.memo for heavy child components.

Interview Answer

The Context performance problem happens when many components consume the same Context and all of them re-render whenever the Provider value changes. Without using a library, we can solve it by splitting Contexts by feature, memoizing Provider values, separating state and actions into different Contexts, keeping local state local, and using React.memo for heavy child components.

One-Line Interview Answer

Context performance issues are solved by avoiding one large Context, splitting state into smaller Contexts, memoizing Provider values, and keeping unnecessary state out of Context.
