042. What is the React Fiber architecture and how does it differ from the old stack reconciler?

One-Line Definition

React Fiber architecture is React's rewritten reconciliation system introduced in React 16. It represents rendering work as Fiber objects instead of relying only on the JavaScript call stack, making React more flexible for scheduling, prioritization, error recovery, and future concurrent rendering.

Simple Meaning

React needs to compare the old UI and the new UI to decide what should change in the DOM. This process is called reconciliation.

What Was The Old Stack Reconciler?

The old Stack Reconciler was React's earlier reconciliation system. It worked using recursive rendering and depended heavily on the JavaScript call stack.
Component A
↓
Component B
↓
Component C
↓
Component D
↓
Finish Entire Tree
Once the old stack reconciler started rendering, it generally had to finish the whole work before the browser could handle other important tasks.

Problem With Old Stack Reconciler

If the app was large, rendering could block the main thread.

Large State Update
↓
Old Reconciler Starts Work
↓
Processes Entire Component Tree
↓
Main Thread Blocked
↓
Typing / Clicking / Animation Can Feel Slow

What Is React Fiber Architecture?

Fiber architecture represents each component's work as a Fiber object. These objects form a Fiber tree that React can manage more flexibly.
Simplified Fiber Object
{
  type: Component,
  props: {},
  stateNode: {},
  child: Fiber,
  sibling: Fiber,
  return: Fiber,
  pendingProps: {},
  memoizedProps: {},
  memoizedState: {}
}

React Fiber architecture gives React its own internal structure to store and manage rendering work.

Main Difference

The old stack reconciler processed work like one continuous recursive task. Fiber breaks rendering work into units that React can track and manage.
Old Stack Reconciler React Fiber Architecture
Recursive call-stack based Fiber object based

Harder error recovery Enabled better error boundaries

Render Phase And Commit Phase

Fiber architecture separates work into phases.

Phase What Happens
Render Phase React calculates what needs to change
Commit Phase React applies changes to the real DOM
Render Phase
↓

↓
Commit Phase
↓

In modern React, render work can be interruptible in concurrent rendering cases, but commit phase must be completed because it changes the real DOM.

What Fiber Improved In React 16

1. Better Error Recovery

React 16 introduced Error Boundaries, which became possible because Fiber gave React better control over the component tree.

Error Boundary Example
class ErrorBoundary extends React.Component {
  state = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  render() {
    if (this.state.hasError) {
      return <h2>Something went wrong</h2>;
    }

    return this.props.children;
  }
}

Why Was It Rewritten From Scratch?

The old Stack Reconciler could not easily support React's future goals.

  • Interruptible rendering
  • Scheduling
  • Prioritized updates
  • Error boundaries
  • Suspense improvements
  • Concurrent rendering
  • Selective hydration
React needed a new foundation. That foundation was Fiber.

Interview Answer

React Fiber architecture is the rewritten reconciliation system introduced in React 16. The old Stack Reconciler used a recursive call-stack based approach where rendering work was processed as one continuous task. Fiber replaced this with Fiber objects that represent components and their work, forming a Fiber tree. This made React's internal rendering system more flexible, improved error recovery, and created the foundation for scheduling, prioritization, and React 18 concurrent rendering features.

One-Line Interview Answer

React Fiber differs from the old stack reconciler by representing rendering work as Fiber objects instead of one rigid recursive call-stack process, making React more flexible and ready for scheduling and concurrent rendering.
