039.1 What is Event Pooling in React?

One-Line Definition

Event Pooling was an old React optimization where React reused SyntheticEvent objects instead of creating a new event object for every browser event.

First Understand SyntheticEvent

When you write an event in React, React does not directly give you only the browser's native event. React wraps it inside its own event object called SyntheticEvent.

Button.jsx
function Button() {
  function handleClick(event) {
    console.log(event);
  }

  return <button onClick={handleClick}>Save</button>;
}
Here, event is React's SyntheticEvent. It gives a consistent event API across browsers.

How Event Pooling Worked in React 16 and Earlier

In React 16 and earlier, React reused event objects. After your event handler finished, React cleared the event object properties and kept the object for the next event.

User Clicks Button
↓
React Creates SyntheticEvent
↓
Your Handler Runs
↓
Handler Finishes
↓
React Clears Event Properties
↓
Same Event Object Is Reused Later

Why React Did This

React used event pooling to reduce memory allocation. Instead of creating many new event objects, React reused old objects.
Reason Meaning
Performance Fewer objects created again and again
Memory Saving Old browsers benefited from object reuse
Optimization React tried to make event handling cheaper

The Main Problem

Because React cleared the event object after the handler finished, you could not safely use the event later inside asynchronous code.

React 16 Problem
function Button() {
  function handleClick(event) {
    setTimeout(() => {
      console.log(event.target);
    }, 1000);
  }

  return <button onClick={handleClick}>Save</button>;
}
In React 16 and earlier, this could print null, because React already cleared the event properties before setTimeout executed.

Why event.target Became null

Click Event Happens
↓
handleClick(event) Runs
↓
setTimeout Is Registered
↓
Handler Finishes Immediately
↓
React Clears event.target
↓
1 Second Later setTimeout Runs
↓
event.target Is Already null

Old Solution: event.persist()

In old React, if you wanted to use the event later, you had to call event.persist().

React 16 Fix
function Button() {
  function handleClick(event) {
    event.persist();

    setTimeout(() => {
      console.log(event.target);
    }, 1000);
  }

  return <button onClick={handleClick}>Save</button>;
}
event.persist() told React: "Do not return this event object to the pool. Keep it alive."

Better Old Solution: Store Needed Value

Instead of keeping the whole event object, you could store the required value first.

Safer Pattern
function Button() {
  function handleChange(event) {
    const value = event.target.value;

    setTimeout(() => {
      console.log(value);
    }, 1000);
  }

  return <input onChange={handleChange} />;
}
This pattern is still good because you usually need the value, not the full event object.

React 17 and Later

React 17 removed event pooling. So in React 17, React 18, and React 19, SyntheticEvent objects are no longer pooled.
React 17+
function Button() {
  function handleClick(event) {
    setTimeout(() => {
      console.log(event.target);
    }, 1000);
  }

  return <button onClick={handleClick}>Save</button>;
}
In modern React, you do not need event.persist() for normal React DOM events.

React 16 vs React 17+

Feature React 16 and Earlier React 17+
Event Pooling Yes No
SyntheticEvent Reused Yes No
Event Properties Cleared Yes No
Need event.persist() Sometimes yes No
Async Event Access Unsafe without persist/value copy Safe

Important Point

Event pooling is mostly a historical React interview topic now. In modern React, you should know what it was, why it caused problems, and that it was removed in React 17.

Interview Answer

Event Pooling was an optimization used in React 16 and earlier where React reused SyntheticEvent objects to reduce memory allocation. After an event handler finished, React cleared the event properties and returned the object to the pool. Because of this, accessing event properties asynchronously could return null unless event.persist() was used or the needed value was copied first. React 17 removed event pooling, so event.persist() is no longer needed in modern React.

One-Line Interview Answer

Event Pooling means old React reused SyntheticEvent objects, cleared them after the handler, and required event.persist() for async access; modern React removed this behavior.
