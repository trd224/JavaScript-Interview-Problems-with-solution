430. What is React.FC and why do some people avoid it?

One-Line Definition

React.FC (React Functional Component) is a TypeScript type used to define React function components. Modern React projects often avoid it because it provides little benefit and can introduce unnecessary complexity.

What is React.FC?

File: src/components/UserCard.tsx
interface UserCardProps {
  name: string;
}

const UserCard: React.FC<UserCardProps> = ({ name }) => {
  return <h2>{name}</h2>;
};

export default UserCard;

Here: 

React.FC<UserCardProps>

means: 

This is a React Function Component
and its props are UserCardProps.

Modern Recommended Style

File: src/components/UserCard.tsx
interface UserCardProps {
  name: string;
}

function UserCard({ name }: UserCardProps) {
  return <h2>{name}</h2>;
}

export default UserCard;
This is the style recommended by most modern React TypeScript projects.

Benefit 1 - Explicit Component Type

File: src/components/UserCard.tsx
const UserCard: React.FC<UserCardProps> =
  ({ name }) => {
    return <h2>{name}</h2>;
  };

Immediately tells readers: 

This variable is a React component.

Benefit 2 - Return Type Checking

React.FC ensures the component returns valid JSX. 

const UserCard: React.FC = () => {
  return <div>Hello</div>;
};

The Main Problem

React.FC adds extra typing that often isn't needed.

Old React.FC Behavior

Historically React.FC automatically included children. 

const Card: React.FC = ({ children }) => {
  return <div>{children}</div>;
};

Many developers found this confusing because: 

children appeared automatically
even when not intended.
React 18 typings changed this behavior, but the reputation remained.

Explicit Children Is Better

File: src/components/Card.tsx
import { ReactNode } from 'react';

interface CardProps {
  children: ReactNode;
}

function Card({ children }: CardProps) {
  return <div>{children}</div>;
}

export default Card;
Much clearer. You can immediately see that children is supported.

Generic Components Become Harder

File: src/components/List.tsx
interface ListProps<T> {
  items: T[];
}

function List<T>({ items }: ListProps<T>) {
  return (
    <ul>
      {items.map((item, index) => (
        <li key={index}>{String(item)}</li>
      ))}
    </ul>
  );
}

This is simple with normal functions.

Using React.FC with generics becomes more complicated.

Should You Avoid React.FC?

Not necessarily.

  • React.FC is not wrong.
  • React.FC still works perfectly.
  • Many existing codebases use React.FC.
  • Most new codebases prefer normal functions.
You should be comfortable reading and writing both styles.

Interview Answer

React.FC is a TypeScript type for React function components. It provides component typing and return type checking, but many developers avoid it because it adds unnecessary abstraction, can complicate generic components, and historically caused confusion around children props. Modern React TypeScript projects typically prefer normal function components with explicitly typed props.

One-Line Interview Answer

React.FC is a type for React function components, but many developers prefer normal function components because they are simpler and work better with modern TypeScript patterns.
