441. What is XSS (Cross-Site Scripting) and How Can React Be Vulnerable?

One-Line Definition

XSS means an attacker submits malicious JavaScript as user input, and the browser later executes that code when the content is displayed on a page.

Understand From Starting

Imagine you have a website where users can write comments.

Article Page

Comment Box
[________________]

[Submit]

Normal User Flow

A normal user enters simple text.

Nice article

Your server stores it in the database.

Database Comment:
Nice article

Later your website displays it.

<div>Nice article</div>
Browser shows normal text. Everything is safe.

Attacker Flow

The attacker does not change your React source code. He enters dangerous code inside a normal input field like comment box, chat box, profile bio, or review box.

<script>alert('Hacked')</script>

If your server stores this input and your website later displays it as real HTML, the browser executes it.

<div>
  <script>alert('Hacked')</script>
</div>
The browser sees <script> and runs it as JavaScript. This is XSS.

Why Is XSS Dangerous?

In examples we use alert, but real attackers can do more dangerous things.

alert('Hacked');

Instead of alert, attacker may try:

stealCookies();
sendUserDataToAttacker();
redirectToFakeLoginPage();
  • Steal session data
  • Steal user information
  • Redirect users to fake login pages
  • Perform actions as the logged-in user

How React Protects You

React normally protects you when you render data inside JSX using curly braces.

Safe React Example
const comment = "<script>alert('Hacked')</script>";

function Comment() {
  return <div>{comment}</div>;
}

React converts HTML characters into text.

<div>
  &lt;script&gt;alert('Hacked')&lt;/script&gt;
</div>
Now the browser displays the script as plain text. It does not execute.

What User Sees

<script>alert('Hacked')</script>
React escaped the dangerous HTML, so it becomes text, not executable code.

Then How Can React Be Vulnerable?

React can become vulnerable when you use dangerouslySetInnerHTML with untrusted user data.

Unsafe React Example
const comment = "<script>alert('Hacked')</script>";

function Comment() {
  return <div dangerouslySetInnerHTML={{ __html: comment }} />;
}

Now React does not escape the HTML. It inserts it directly into the DOM.

<div>
  <script>alert('Hacked')</script>
</div>
The browser treats it as real JavaScript and executes it. XSS happens.

Why Is It Called dangerouslySetInnerHTML?

React intentionally uses the word dangerous because this API can create XSS problems if you pass untrusted user input into it.

Safe vs Unsafe

CodeResult
<div>{comment}</div>
Safe. React escapes HTML.
<div dangerouslySetInnerHTML={{ __html: comment }} />
Unsafe if comment comes from user input.

How To Prevent XSS In React

  • Render user input with JSX curly braces: {comment}
  • Avoid dangerouslySetInnerHTML for user-generated content.
  • If HTML is required, sanitize it before rendering.
  • Validate and sanitize input on backend also.
  • Use trusted libraries for sanitization, not manual string replace.

Interview Answer

XSS is a vulnerability where attackers inject malicious JavaScript through user input, and the browser later executes it when the content is displayed. React normally protects against XSS by escaping values rendered inside JSX. However, React can be vulnerable when developers use dangerouslySetInnerHTML with untrusted data because it inserts raw HTML directly into the DOM without escaping.

One-Line Interview Answer

React is safe by default because JSX escapes user input, but it can become vulnerable to XSS if untrusted data is rendered using dangerouslySetInnerHTML.
