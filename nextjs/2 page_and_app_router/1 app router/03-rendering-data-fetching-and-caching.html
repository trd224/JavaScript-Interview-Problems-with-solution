03-01Are getStaticProps() and getServerSideProps() Available in the App Router?

No. getStaticProps() and getServerSideProps() are only available in the Pages Router.

The App Router does not use getStaticProps(), getServerSideProps(), or getStaticPaths(). Instead, it uses Server Components, fetch caching options, and special exports such as revalidate and generateStaticParams().

Pages Router

pages/posts.js
export async function getStaticProps() {
  return {
    props: {}
  };
}
export async function getServerSideProps() {
  return {
    props: {}
  };
}

These APIs work only inside the Pages Router.

App Router

app/posts/page.js
export default async function Posts() {
  const res = await fetch(
    'https://api.example.com/posts'
  );
  const posts = await res.json();
  return (
    <h1>{posts.length}</h1>
  );
}

There is no getStaticProps() or getServerSideProps() here.

Why Were They Removed?

In the Pages Router, Next.js needed separate APIs:

Purpose API
SSG getStaticProps()
SSR getServerSideProps()
Dynamic SSG getStaticPaths() + getStaticProps()

In the App Router, data fetching is built directly into Server Components.

Instead of choosing special functions, we control rendering behavior through caching options.

App Router Replacement for SSG

Static generation happens automatically when data can be cached.

SSG Style
async function getPosts() {
  const res = await fetch(
    'https://api.example.com/posts'
  );
  return res.json();
}
export default async function Posts() {
  const posts = await getPosts();
  return (
    <h1>{posts.length}</h1>
  );
}

Next.js may generate this page during the build process.

App Router Replacement for SSR

SSR Style
async function getPosts() {
  const res = await fetch(
    'https://api.example.com/posts',
    { cache: 'no-store' }
  );
  return res.json();
}
export default async function Posts() {
  const posts = await getPosts();
  return (
    <h1>{posts.length}</h1>
  );
}

The cache: 'no-store' option forces fresh data on every request.

This is equivalent to SSR.

App Router Replacement for ISR

ISR Style
export const revalidate = 60;
async function getPosts() {
  const res = await fetch(
    'https://api.example.com/posts'
  );
  return res.json();
}
export default async function Posts() {
  const posts = await getPosts();
  return (
    <h1>{posts.length}</h1>
  );
}

This regenerates the page when a request arrives after 60 seconds.

App Router Replacement for getStaticPaths()

generateStaticParams()
export async function generateStaticParams() {
  return [
    { id: '1' },
    { id: '2' },
    { id: '3' }
  ];
}

This replaces getStaticPaths().

Mapping Old APIs to New APIs

getServerSideProps() Server Component + fetch(..., { cache: 'no-store' })
getStaticProps() Server Component + fetch() (default cache)
getStaticPaths() generateStaticParams()
ISR via revalidate Server Component + export const revalidate

03-02How does async data fetching work directly inside Server Components?

In the Next.js App Router, Server Components can be async functions. This allows us to fetch data directly inside the component before returning JSX.

Server Components run on the server, so we can use async and await directly inside them to fetch API data, database data, or session data before HTML is sent to the browser.

Basic Example

app/products/page.tsx
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products');
  const products = await res.json();
  return (
    <div>
      <h1>Products</h1>
      {products.map(product => <p key={product.id}>{product.name}</p>)}
    </div>
  );
}

Here, ProductsPage is a Server Component because it does not use 'use client'.

What Happens Internally?

Flow
User requests /products
        ↓
Server runs ProductsPage()
        ↓
fetch() waits for product data
        ↓
Server creates HTML with products
        ↓
Browser receives ready HTML

No useEffect Needed

In Client Components, we usually fetch data with useEffect().

Client Component Style
'use client';
import { useEffect, useState } from 'react';
export default function ProductsPage() {
  const [products, setProducts] = useState([]);
  useEffect(() => {
    fetch('/api/products').then(res => res.json()).then(data => setProducts(data));
  }, []);
  return <div>{products.map(product => <p key={product.id}>{product.name}</p>)}

But in Server Components, we can fetch before rendering, so the browser does not need to wait for JavaScript to load the data.

Server Component Style

Better App Router Pattern
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products');
  const products = await res.json();
  return <div>{products.map(product => <p key={product.id}>{product.name}</p>)}
The data is fetched on the server, and the browser receives HTML that already contains the product list.

Fetching From A Database

Because Server Components run on the server, we can also call a database directly.

app/users/page.tsx
import { db } from '@/lib/db';
export default async function UsersPage() {
  const users = await db.user.findMany();
  return (
    <div>
      <h1>Users</h1>
      {users.map(user => <p key={user.id}>{user.name}</p>)}
    </div>
  );
}

This is safe because database credentials and server logic are not sent to the browser.

Passing Data To A Client Component

A common pattern is to fetch data in a Server Component, then pass it to a Client Component for interactivity.

app/products/page.tsx
import ProductList from './ProductList';
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products');
  const products = await res.json();
  return <ProductList products={products} />;
}
app/products/ProductList.tsx
'use client';
export default function ProductList({ products }) {
  return (
    <div>
      {products.map(product => <button key={product.id}>{product.name}</button>)}
    </div>
  );
}

The Server Component fetches the data. The Client Component handles browser interaction.

Important: Async Component Means Server Component

This is valid:

Server Component
export default async function Page() {
  const data = await getData();
  return <h1>{data.title}</h1>;
}

But this is not the normal Client Component pattern:

Wrong Client Component Pattern
'use client';
export default async function Page() {
  const data = await getData();
  return <h1>{data.title}</h1>;
}
Client Components use useEffect(), event handlers, and state. Server Components can be async and fetch data before rendering.

Example With Revalidation

ISR-style Fetch
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products', {
    next: {
      revalidate: 60
    }
  });
  const products = await res.json();
  return <div>{products.map(product => <p key={product.id}>{product.name}</p>)}

This tells Next.js to reuse cached data and refresh it after 60 seconds.

Example With Fresh Data Every Request

Dynamic Fetch
export default async function DashboardPage() {
  const res = await fetch('https://api.example.com/stats', { cache: 'no-store' });
  const stats = await res.json();
  return <h1>Total Users: {stats.totalUsers}</h1>;
}

This fetch runs fresh for every request.

Slow Data and Suspense

If a Server Component is slow, it can block rendering until the data is ready.

Slow Component
export default async function UserStats() {
  const stats = await getSlowStats();
  return <h2>Users: {stats.users}</h2>;
}

We can wrap it in Suspense to show a fallback while it loads.

page.tsx
import { Suspense } from 'react';
import UserStats from './UserStats';
export default function Page() {
  return (
    <>
      <h1>Dashboard</h1>
      <Suspense fallback={<p>Loading stats...</p>}>
        <UserStats />
      </Suspense>
    </>
  );
}
Suspense allows the rest of the page to render while the slow Server Component finishes loading.

03-03What is the Extended fetch API in Next.js and what caching options does it add?

Next.js extends the standard JavaScript fetch() API by adding built-in caching, revalidation, deduplication, and data cache controls.

The browser's fetch API only downloads data. Next.js fetch can also control how long data is cached and when it should be refreshed.

Normal JavaScript fetch()

In plain React or Node.js:

Standard Fetch
const res = await fetch('https://api.example.com/products');
const products = await res.json();

This simply requests data.

There is no built-in:

  • ISR
  • Data Cache
  • Automatic Revalidation
  • Cache Tags

Next.js Extended fetch()

Next.js adds additional options:

Extended Fetch
const res = await fetch('https://api.example.com/products', {
  cache: 'force-cache',
  next: {
    revalidate: 60,
    tags: ['products']
  }
});

These options do not exist in the normal browser fetch API.

1. cache: 'force-cache'

This tells Next.js to cache the result.

Example
const res = await fetch('https://api.example.com/products', {
  cache: 'force-cache'
});

Flow

First Request
      ↓
Fetch API
      ↓
Save In Cache
Second Request
      ↓
Read From Cache

Good for:

  • Blog Posts
  • Documentation
  • Static Content
This is similar to SSG-style caching.

2. cache: 'no-store'

This disables caching completely.

Example
const res = await fetch('https://api.example.com/stats', {
  cache: 'no-store'
});

Flow

Request
   ↓
Fetch API
   ↓
Return Data
Next Request
   ↓
Fetch Again

Every request gets fresh data.

Good for:

  • User Dashboard
  • Live Statistics
  • Stock Prices
  • User Session Data
This makes the route dynamic (SSR).

3. next.revalidate

This enables ISR-style revalidation.

Example
const res = await fetch('https://api.example.com/products', {
  next: {
    revalidate: 60
  }
});

Meaning:

      ↓

      ↓
Refresh On Next Request

Good for:

  • Product Catalog
  • News Articles
  • Public Content
This is the most common ISR pattern in the App Router.

4. next.tags

Allows data to be grouped using tags.

Example
const res = await fetch('https://api.example.com/products', {
  next: {
    tags: ['products']
  }
});

Now this cache belongs to:

Next.js automatically deduplicates identical fetch requests.

Example

const users3 = fetch('https://api.example.com/users');

1 Network Request

3 Network Requests

This improves performance automatically.

03-04What is the difference between cache: 'force-cache' and cache: 'no-store'?

These are two caching modes provided by the Next.js extended fetch API.

force-cache tells Next.js to reuse cached data, while no-store tells Next.js to fetch fresh data on every request.

cache: 'force-cache'

Next.js fetches the data once and stores it in the Data Cache.

Example
const res = await fetch('https://api.example.com/products', {
  cache: 'force-cache'
});
const products = await res.json();

      ↓
Data Stored In Cache
Second Request
      ↓
Use Cached Data
Third Request
      ↓
Use Cached Data

Without Calling API Again
This is similar to SSG behavior because cached data is reused.

cache: 'no-store'

Next.js skips the cache completely.

Example
const res = await fetch('https://api.example.com/products', {
  cache: 'no-store'
});
const products = await res.json();

Request 1
     ↓
Call API
Request 2
     ↓
Call API Again
Request 3
     ↓
Call API Again

Result

Fresh Data
Every Request
This makes the route dynamic and behaves like SSR.

03-05What does next: { revalidate: N } do in a fetch call?

The next.revalidate option tells Next.js how long fetched data should stay in cache before being refreshed.

Products Page

}

03-06What is on-demand revalidation and how do we trigger it via an API route?

On-demand revalidation means manually clearing cached data when something changes, instead of waiting for revalidate: 60, revalidate: 3600, or another time-based interval.

In simple words, on-demand revalidation lets us refresh cached pages or cached data immediately after an event, such as product update, blog publish, CMS webhook, or admin action.

Problem Without On-Demand Revalidation

Suppose our product page uses ISR.

app/products/page.tsx
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products', { next: { revalidate: 3600 } });
  const products = await res.json();
  return <h1>Products</h1>;
}

This means cached data can remain old for 1 hour.

Problem
Product Updated Now
        ↓
Cache Still Valid For 1 Hour
        ↓
User May See Old Data

Solution: On-Demand Revalidation

When product data changes, we manually tell Next.js:

Meaning
Clear Cache Now
        ↓
Next Request Gets Fresh Data

This is called on-demand revalidation.

Two Main Functions

Function Purpose
revalidatePath() Refresh a specific page or route path
revalidateTag() Refresh all cached fetches using a specific tag

1. Revalidate By Path

Use revalidatePath() when we want to refresh a specific route.

Example
revalidatePath('/products');

This tells Next.js to invalidate the cache for the /products page.

API Route Example

app/api/revalidate/route.ts
import { revalidatePath } from 'next/cache';
import { NextRequest } from 'next/server';
export async function POST(request: NextRequest) {
  const secret = request.headers.get('x-revalidate-secret');
  if (secret !== process.env.REVALIDATE_SECRET) {
    return Response.json({ message: 'Invalid secret' }, { status: 401 });
  }
  revalidatePath('/products');
  return Response.json({ revalidated: true, path: '/products' });
}

How To Trigger This API Route

POST Request

  method: 'POST',
  headers: { 'x-revalidate-secret': 'my-secret-key' }
});

2. Revalidate By Tag

Use revalidateTag() when multiple pages use the same cached data.

Tagged Fetch
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products', { next: { tags: ['products'] } });
  const products = await res.json();
  return <h1>Products</h1>;
}

Now this fetch is connected to the products tag.

API Route Using revalidateTag()

app/api/revalidate-products/route.ts
import { revalidateTag } from 'next/cache';
import { NextRequest } from 'next/server';

  if (secret !== process.env.REVALIDATE_SECRET) {
    return Response.json({ message: 'Invalid secret' }, { status: 401 });
  }
  revalidateTag('products', 'max');
  return Response.json({ revalidated: true, tag: 'products' });
}

03-12How do you use React.cache for per-request memoization of non-fetch calls?

React.cache() is used to memoize the result of a function during a single server request.

In simple words, React.cache() prevents the same non-fetch function from running again and again during the same render request.

Why Do We Need React.cache?

Next.js automatically deduplicates identical fetch() calls, but it does not automatically memoize custom functions like database queries, Prisma calls, MongoDB calls, or session functions.

Fetch Deduplication Works Automatically

03-19What changed with cookies(), headers(), and params becoming async in Next.js 15?

In Next.js 15, cookies(), headers(), params, and searchParams became asynchronous APIs.

Before Next.js 15, these APIs could be accessed directly. In Next.js 15, we must use await because they now return Promises.

Why Did Next.js Make Them Async?

This change helps Next.js support:

  • Streaming
  • Partial Prerendering (PPR)
  • Server Components
  • Future rendering optimizations

Making these APIs asynchronous gives Next.js more flexibility when preparing and rendering pages.

cookies() Before Next.js 15

Old Version
import { cookies } from 'next/headers';
export default function Page() {
  const cookieStore = cookies();
  const token = cookieStore.get('token');
  return <h1>{token?.value}</h1>;
}

cookies() In Next.js 15

Next.js 15
import { cookies } from 'next/headers';
export default async function Page() {
  const cookieStore = await cookies();
  const token = cookieStore.get('token');
  return <h1>{token?.value}</h1>;
}

Or more commonly:

Compact Version
const token = (await cookies()).get('token')?.value;

Old Version
import { headers } from 'next/headers';
export default function Page() {

03-20What is the connection() API introduced in Next.js 15?

The connection() API is a new Next.js 15 function that allows us to explicitly wait for an incoming request before rendering a page.

The primary purpose of connection() is to tell Next.js that this page must be rendered dynamically at request time and cannot be prerendered ahead of time.

Why Was connection() Introduced?

Sometimes a page depends on request-specific information such as:

  • Cookies
  • Headers
  • User Session
  • Request Metadata
  • Random Values
  • Current Time

Next.js cannot prerender such pages during build time because the request does not exist yet.

Before Next.js 15

We commonly used:

Old Approach
export const dynamic = 'force-dynamic';

to force dynamic rendering.

The Pages Router does not have force-dynamic. Instead, use getServerSideProps(), which automatically renders the page on every request:

Next.js 15 Approach

connection()
import { connection } from 'next/server';
export default async function Page() {
  await connection();
  return Dynamic Page;
}

This tells Next.js: 

Wait For Request
      │
      ▼
Render Page

Real Example

Suppose we want to display the current time.

Example
import { connection } from 'next/server';
export default async function Page() {
  await connection();
  const now = new Date();
  return <h1>{now.toLocaleTimeString()}</h1>;
}

How Does connection() Relate to PPR?

Partial Prerendering (PPR) allows part of a page to be prerendered and part to wait for request-time data.

When Should We Use connection()?

  • Current time display.
  • Random values.
  • User-specific pages.
  • Cookie-based rendering.
  • Header-based rendering.
  • Request-dependent content.

When Should We Avoid connection()?

  • Static pages.
  • Blogs.
  • Marketing pages.
  • Documentation pages.
  • Pages that can be prerendered.

Important Note

Calling await connection() opts the route into request-time rendering because the page now explicitly depends on an incoming request.

Interview Tip

A common misconception is that connection() fetches data. In reality, connection() does not fetch anything—it simply waits for an incoming request and tells Next.js that rendering should happen at request time.
One-Line Definition:
The connection() API in Next.js 15 explicitly waits for an incoming request and marks a route as request-time dynamic, preventing prerendering when content depends on request-specific information.
