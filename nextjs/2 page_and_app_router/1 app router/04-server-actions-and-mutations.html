04 - Server Actions & Mutations

Part 04 of 7 · 10 topics · Next.js App Router Guide

04-01What are Server Actions in Next.js and what problem do they solve?

Server Actions are server-side functions that can be called directly from forms, Server Components, and Client Components without creating separate API routes.

Simple Meaning:
Server Actions allow us to execute backend code directly from our Next.js components.

What Problem Did Server Actions Solve?

Before Server Actions, every form submission usually required an API route.

Traditional Flow
User Clicks Save
       ↓
Client Component
       ↓
fetch('/api/users')
       ↓
API Route
       ↓
Database
       ↓
Response
       ↓
Update UI

Even a simple database insert required multiple files and extra boilerplate code.

Without Server Actions

API Route
// app/api/users/route.ts
export async function POST(req: Request) {
  const data = await req.json();
  await db.user.create({
    data
  });
  return Response.json({
    success: true
  });
}
Client Component
'use client';
export default function UserForm() {
  async function saveUser() {
    await fetch('/api/users', {
      method: 'POST',
      body: JSON.stringify({
        name: 'Rizwan'
      })
    });
  }
  return (
    <button onClick={saveUser}>
      Save User
    </button>
  );
}

We need:

  • Form
  • API Route
  • fetch()
  • Request Parsing
  • Response Handling

With Server Actions

Server Action
// app/actions.ts
'use server';
export async function createUser(formData: FormData) {
  await db.user.create({
    data: {
      name: formData.get('name')
    }
  });
}
Page
import { createUser } from './actions';
export default function Page() {
  return (
    <form action={createUser}>
      <input
        name="name"
        placeholder="Enter Name"
      />
      <button type="submit">
        Save User
      </button>
    </form>
  );
}

Flow: 

User Clicks Save
       ↓
Server Action
       ↓
Database
       ↓
Updated UI

No API route required.

Understanding "use server"

The 'use server' directive tells Next.js that this function must execute on the server.

Example
'use server';
export async function deleteUser(id: string) {
  await db.user.delete({
    where: { id }
  });
}

The browser cannot execute this function directly. Next.js automatically sends a request to the server and runs it there.

Real World Example

Suppose we have an Admin Panel where products can be added.

Server Action
'use server';
export async function addProduct(formData: FormData) {
  await db.product.create({
    data: {
      name: formData.get('name'),
      price: Number(formData.get('price'))
    }
  });
}
Form
<form action={addProduct}>
  <input name="name" />
  <input name="price" />
  <button>
    Add Product
  </button>
</form>

Flow: 

Submit Form
      ↓
Server Action
      ↓
Database Insert
      ↓
Success

Server Actions + Revalidation

Server Actions are commonly used with cache revalidation.

Example
'use server';
import { revalidatePath } from 'next/cache';
export async function addProduct(formData: FormData) {
  await db.product.create({
    data: {
      name: formData.get('name')
    }
  });
  revalidatePath('/products');
}

Flow: 

Add Product
      ↓
Database Updated
      ↓
revalidatePath('/products')
      ↓
Products Cache Cleared
      ↓
Next Request Gets Fresh Data

Can Client Components Use Server Actions?

Yes.

Client Component Example
'use client';
import { createUser } from './actions';
export default function Button() {
  return (
    <button onClick={() => createUser()}>
      Create User
    </button>
  );
}

Although the button is in a Client Component, the action still runs on the server.

When Should We Use Server Actions?

Use Case Use Server Action?
Create User ✅ Yes
Update Profile ✅ Yes
Delete Product ✅ Yes
Submit Contact Form ✅ Yes
Change Password ✅ Yes

When Should We Use API Routes Instead?

Use API Routes when external systems need access.

Example
Android App
      ↓
API Route
      ↓
Database
Third-Party Service
      ↓
API Route
      ↓
Database

Server Actions are mainly for communication between our Next.js frontend and backend.

Common Interview Doubt

If Server Components already run on the server, why do Server Actions need 'use server'?

Short Answer:
Because a Server Component and a Server Action are two different concepts.

Case 1: Normal Function Inside a Server Component

Example
export default async function Page() {
  async function getUsers() {
    return db.user.findMany();
  }
  const users = await getUsers();
  return (
    <div>
      {users.length}
    </div>
  );
}

Flow: 

Browser
   ↓
Page() runs on server
   ↓
getUsers() runs on server

Here, getUsers() is called by another server function (Page()), so Next.js already knows it will run on the server.

No 'use server' is required.

Case 2: Server Action

Server Action
'use server';
export async function createUser( formData: FormData ) {
  await db.user.create({
    data: {
      name: formData.get('name')
    }
  });
}
Form
<form action={createUser}>
  ...
</form>

Flow: 

Browser
   ↓
createUser()

Now the browser is trying to call createUser() directly.

Next.js must create a special server endpoint so that the browser can execute this function on the server.

That is why we add: 

'use server';

This tells Next.js:

Create a server endpoint for this function and execute it on the server whenever the browser calls it.

Easy Rule

Scenario Need 'use server'?
Function called inside a Server Component ❌ No
Function used in form action ✅ Yes
Function called from a Client Component ✅ Yes
Function called directly by the browser ✅ Yes
Remember:
The reason for 'use server' is not that the function runs on the server. The reason is that the browser needs a way to call that function on the server.

Server Components vs Server Actions

Feature Server Component Server Action
Main Purpose Read Data Write Data
Database Query ✅ ✅
Create Record ❌ ✅
Update Record ❌ ✅
Delete Record ❌ ✅

Easy Memory Trick

Remember
Server Component
       ↓
Read Data
Server Action
       ↓
Write Data
One-Line Definition:
Server Actions are server-side functions that allow us to perform create, update, and delete operations directly from Next.js components without creating separate API routes.

Server Actions vs Separate Backend (.NET, Java, Node)

Server Actions provide the biggest benefit when Next.js itself owns the backend logic and talks directly to the database.

Ideal Server Actions Architecture
Next.js Frontend
       ↓
Server Action
       ↓
Database

In this setup, we can directly create, update, or delete database records without creating API routes.

Best Use Case:
Next.js Frontend + Next.js Backend + Database

What If We Already Have a Separate Backend?

Many real-world applications use a separate backend such as .NET, Java, Node.js, Spring Boot, Laravel, etc.

Separate Backend Architecture
Next.js Frontend
       ↓
.NET API
       ↓
Database

In this case, the business logic already exists inside the backend API.

Therefore, we usually call the backend directly using fetch().

Example
const products = await fetch(
  'https://api.company.com/products'
);

or 

await fetch(
  'https://api.company.com/users',
  {
    method: 'POST'
  }
);

Can Server Actions Still Be Used?

Yes.

A Server Action can act as a middle layer between Next.js and the backend API.

Architecture
Browser
    ↓
Server Action
    ↓
.NET API
    ↓
Database
Example
'use server';
export async function createUser(formData: FormData) {
  await fetch(
    'https://api.company.com/users',
    {
      method: 'POST',
      body: JSON.stringify({
        name: formData.get('name')
      })
    }
  );
}

Here the Server Action does not access the database directly. Instead, it forwards the request to the backend API.

If We Can Use fetch(), Why Use a Server Action?

A common question is:

Question:
If we already have a .NET, Java, Node.js, or Laravel backend, why not simply call it directly using fetch()?

In many cases, that is exactly what we do.

Direct API Call
'use client';
async function saveUser() {
  await fetch(
    'https://api.company.com/users',
    {
      method: 'POST',
      body: JSON.stringify({
        name: 'Rizwan'
      })
    }
  );
}

Architecture: 

Browser
   ↓
fetch()
   ↓
.NET API
   ↓
Database

No Server Action is required.

Then Why Would We Use a Server Action?

1. Hide Secrets and Tokens

Suppose the backend API requires a secret token.

Server Action
'use server';
export async function createUser(
  formData: FormData
) {
  await fetch(
    'https://api.company.com/users',
    {
      headers: {
        Authorization:
          `Bearer ${process.env.API_TOKEN}`
      }
    }
  );
}

Because the code runs on the server, the token is never exposed to the browser.

2. Access Server-Only Features

Server Actions can use server-only APIs such as:

  • cookies()
  • headers()
  • Environment Variables
  • Database Access
  • File System Access
Example
'use server';
import { cookies } from 'next/headers';
export async function createUser() {
  const token =
    (await cookies()).get('token');
  await fetch(...);
}

3. Hide Backend URLs

Without a Server Action: 

Browser
   ↓
https://api.company.com/users

With a Server Action: 

Browser
   ↓
Server Action
   ↓
https://api.company.com/users

The browser never directly communicates with the backend API.

4. Simpler Form Handling

Server Action Form
<form action={createUser}>
  ...
</form>

Instead of:

Traditional Form
<form onSubmit={handleSubmit}>
  ...
</form>
async function handleSubmit(e) {
  e.preventDefault();
  await fetch(
    'https://api.company.com/users',
    {
      method: 'POST'
    }
  );
}

Server Actions reduce some of the client-side form handling code.

What Do Most Enterprise Applications Use?

Most enterprise applications already have a dedicated backend. 

React / Next.js
       ↓
.NET API
       ↓
Database

In these projects, developers commonly call the backend directly using fetch().

Example
await fetch(
  'https://api.company.com/users',
  {
    method: 'POST'
  }
);
Important:
If a separate backend already exists, Server Actions are optional, not mandatory. Many teams simply use fetch() to communicate directly with the backend API.

Easy Rule

Scenario Recommended Approach
Next.js owns the database ✅ Server Actions
Need secrets or server-only APIs ✅ Server Actions
Need to hide backend URLs ✅ Server Actions
Already have .NET / Java / Node backend ✅ Direct fetch() is usually enough
Interview Answer:
When a separate backend already exists, we can often call it directly using fetch(). Server Actions are mainly useful when we need server-side secrets, server-only APIs, hidden endpoints, or a simpler form handling model.

For Projects Like Aadhaar Authentication

Typical Architecture
Angular / Next.js
        ↓
.NET API
        ↓
Database

In such projects, it is usually simpler to call the backend API directly rather than introducing Server Actions.

Architecture Recommended Approach
Next.js + Database ✅ Server Actions
Next.js + .NET Backend ✅ Direct API Calls
Next.js + Java Backend ✅ Direct API Calls
Next.js + Node Backend ✅ Direct API Calls
Easy Rule:
If Next.js owns the database, Server Actions are usually the best choice.

If a separate backend already exists, direct API calls are usually simpler and more common.

04-02What is the 'use server' directive and where can you place it?

'use server' is a Next.js directive that marks a function or file to run only on the server.

Simple Meaning:
'use server' tells Next.js, "This code must execute on the server and never in the browser."

Why Do We Need 'use server'?

Some code should never be sent to the browser.

Examples:
  • Database queries
  • Secret API keys
  • Password hashing
  • File system access
  • Server-side business logic
Example
await db.user.create({
  data: {
    name: 'Rizwan'
  }
});

The browser cannot access the database directly, so this code must run on the server.

Function-Level use server

We can place 'use server' inside an individual function.

Example
export default function Page() {
  async function createUser(formData: FormData) {
    'use server';
    await db.user.create({
      data: {
        name: formData.get('name')
      }
    });
  }
  return (
    <form action={createUser}>
      <input name="name" />
      <button>Save</button>
    </form>
  );
}

Only the createUser() function runs on the server.

File-Level use server

We can place 'use server' at the top of a file.

actions.ts
'use server';
export async function createUser() {
  await db.user.create({
    data: {
      name: 'Rizwan'
    }
  });
}
export async function deleteUser(id: string) {
  await db.user.delete({
    where: { id }
  });
}

Now every exported function in this file becomes a Server Action.

How Next.js Treats It

Flow
Client Component
       ↓
Calls Server Action
       ↓
Next.js Internal Request
       ↓
Server
       ↓
Execute Function
       ↓
Return Result

The browser never executes the server code directly.

Using use server with Forms

Server Action
export async function createUser(
  formData: FormData
) {
  'use server';
  await db.user.create({
    data: {
      name: formData.get('name')
    }
  });
}
Form
<form action={createUser}>
  <input name="name" />
  <button>Save</button>
</form>

When the form is submitted: 

Submit Form
      ↓
Server Action
      ↓
Database
      ↓
Response

Using use server from a Client Component

actions.ts
'use server';
export async function createUser() {
  await db.user.create({
    data: {
      name: 'Rizwan'
    }
  });
}
Client Component
'use client';
import { createUser } from './actions';
export default function Button() {
  return (
    <button
      onClick={() => createUser()}
    >
      Create User
    </button>
  );
}

Although the button runs in the browser, the action executes on the server.

Where Can We Place use server?

Location Allowed?
Inside a Function ✅ Yes
Top of actions.ts File ✅ Yes
Inside Client Component File ❌ No
Inside Event Handler ❌ No
Inside useEffect() ❌ No

use server vs use client

Feature use server use client
Runs on Server ✅ ❌
Runs in Browser ❌ ✅
Database Access ✅ ❌
Access Secret Keys ✅ ❌
useState/useEffect ❌ ✅

Easy Memory Trick

Remember
use server
      ↓
Backend Code
use client
      ↓
Browser Code
One-Line Definition:
'use server' is a Next.js directive that marks functions or files to execute only on the server, allowing secure access to databases, secrets, and backend logic.

04-03Server Actions became stable in which Next.js version?

Server Actions became stable in Next.js 14.

Interview Answer:
Server Actions were introduced as an experimental feature in Next.js 13 and became stable in Next.js 14.

Timeline

Version Status
Next.js 13 ⚠️ Experimental
Next.js 14 ✅ Stable
Next.js 15 ✅ Improved and Enhanced

Next.js 13

When Server Actions were first introduced in Next.js 13, they were experimental and had to be enabled manually.

next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    serverActions: true
  }
};
module.exports = nextConfig;

Without this flag, Server Actions could not be used.

Next.js 14

In Next.js 14, Server Actions became stable and no experimental flag was required.

Server Action
'use server';
export async function createUser() {
  console.log('Server Action');
}

Everything works out of the box.

Next.js 15

Next.js 15 continues to support Server Actions and improves integration with:

  • React 19
  • Form Actions
  • useActionState()
  • useFormStatus()
  • Caching APIs
  • Revalidation APIs

Easy Memory Trick

Remember
Next.js 13
      ↓
Experimental
Next.js 14
      ↓
Stable
Next.js 15
      ↓
Enhanced
One-Line Definition:
Server Actions were introduced experimentally in Next.js 13 and became a stable production-ready feature in Next.js 14.

04-04How do Server Actions integrate with HTML elements?

Server Actions integrate directly with HTML forms using the form's action attribute.

Simple Meaning:
Instead of submitting a form to an API endpoint, we can submit it directly to a Server Action.

Traditional HTML Form

Before Server Actions, forms were submitted to a URL.

Traditional Form
<form action="/api/users" method="POST">
  <input name="name" />
  <button>Save</button>
</form>

Flow: 

Form Submit
      ↓
/api/users
      ↓
API Route
      ↓
Database
      ↓
Response

Server Actions Form

With Server Actions, the action attribute receives a function instead of a URL.

Server Action
'use server';
export async function createUser(
  formData: FormData
) {
  console.log(
    formData.get('name')
  );
}
Form
import { createUser } from './actions';
export default function Page() {
  return (
    <form action={createUser}>
      <input
        name="name"

Flow:

      ↓
Server Action
      ↓
Database
      ↓

When the form is submitted, Next.js automatically collects all form fields and passes them to the Server Action as a FormData object.

Form
<input name="name" />

Server Action

  formData: FormData

}

Values are automatically available inside formData.

04-05What is useActionState (React 19) and how does it replace useFormState in Next.js 15?

useActionState() is a React 19 hook that allows us to manage the state returned from a Server Action.

Simple Meaning:
useActionState() helps our form receive success messages, error messages, and validation results from a Server Action.

Why Do We Need useActionState?

When a form submits to a Server Action, the Server Action may return some result.

  • Success message
  • Error message
  • Validation error
  • Updated form state

useActionState() stores that returned result in component state.

Flow
User Submits Form
        ↓
Server Action Runs
        ↓
Server Action Returns Message
        ↓
useActionState Stores Message
        ↓
UI Shows Message

Old API: useFormState

Earlier, Next.js used useFormState() from react-dom.

Old Import
import { useFormState } from 'react-dom';

This was mainly focused on form state.

New API: useActionState

In React 19 and Next.js 15, useActionState() is used from react.

New Import
import { useActionState } from 'react';
Main Difference:
useFormState() was the older API, while useActionState() is the newer React 19 API.

Basic Syntax

Syntax
const [state, formAction] = useActionState(actionFunction, initialState);
Part Meaning
state Latest value returned from the Server Action
formAction Function passed to the form's action attribute
actionFunction Server Action function
initialState Default state before form submit

Complete Example

1. Server Action

app/actions.ts
'use server';
export async function createUser(previousState: string, formData: FormData) {
  const name = formData.get('name');
  if (!name) return 'Name is required';
  return 'User created successfully';
}

2. Client Component

app/UserForm.tsx
'use client';
import { useActionState } from 'react';
import { createUser } from './actions';
export default function UserForm() {
  const [message, formAction] = useActionState(createUser, '');
  return (
    <form action={formAction}>
      <input name="name" placeholder="Enter name" />
      <button>Save</button>
      <p>{message}</p>
    </form>
  );
}

How It Works

When the form is submitted, formAction calls the Server Action.

Important Line
const [message, formAction] = useActionState(createUser, '');

Here:

Value Meaning
message Stores the returned message from createUser()
formAction Used inside <form action={formAction}>

Important: Action Function Receives Previous State

When we use useActionState(), React automatically passes the previous state as the first argument.

Server Action Signature
export async function createUser(previousState: string, formData: FormData) {
  return 'Updated message';
}

So the action receives: 

1st argument → previousState
2nd argument → formData

Real Validation Example

app/actions.ts
'use server';
type State = {
  success: boolean;
  message: string;
};
export async function register(previousState: State, formData: FormData): Promise<State> {
  const email = formData.get('email');
  if (!email) return { success: false, message: 'Email is required' };
  return { success: true, message: 'Registration successful' };
}
app/RegisterForm.tsx
'use client';
import { useActionState } from 'react';
import { register } from './actions';
const initialState = { success: false, message: '' };
export default function RegisterForm() {
  const [state, formAction] = useActionState(register, initialState);
  return (
    <form action={formAction}>
      <input name="email" placeholder="Enter email" />
      <button>Register</button>
      <p>{state.message}</p>
    </form>
  );
}

useFormState vs useActionState

Feature useFormState useActionState
Import From react-dom react
Used In Older Next.js versions Next.js 15 / React 19
Purpose Manage form action state Manage action result state
Status Older API Recommended API

Easy Memory Trick

Remember
Server Action
      ↓
Returns Message / Error
      ↓
useActionState()
      ↓
Shows Message In UI

useActionState() is a React 19 hook that stores the result returned from a Server Action, and it replaces the older useFormState() API used in earlier Next.js versions.

04-06What is useFormStatus and what does it provide during a Server Action submission?

useFormStatus() is a React hook that provides information about the status of a form submission while a Server Action is running.

Simple Meaning:
useFormStatus() tells us whether a form is currently submitting so we can show loading indicators, disable buttons, and prevent duplicate submissions.

What Problem Does useFormStatus Solve?

Suppose a Server Action takes 3 seconds to complete.

Without useFormStatus
User Clicks Submit
         ↓
Nothing Changes
         ↓
User Clicks Again
         ↓
Duplicate Requests

The user cannot tell whether the form is still processing.

With useFormStatus(): 

User Clicks Submit
         ↓
pending = true
         ↓
Button Disabled
         ↓
Loading Text Shown
         ↓
Request Completes
         ↓
pending = false
         ↓
Button Enabled

Import

Import Hook
import { useFormStatus } from 'react-dom';

Unlike useActionState(), this hook is imported from react-dom.

Basic Example

1. Server Action

actions.ts
'use server';
export async function createUser(formData: FormData) {
  await new Promise(resolve => setTimeout(resolve, 3000));
  console.log(formData.get('name'));
}

2. Submit Button

SubmitButton.tsx
'use client';
import { useFormStatus } from 'react-dom';
export default function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button disabled={pending}>
      {pending ? 'Saving...' : 'Save'}
    </button>
  );
}

3. Form

UserForm.tsx
import { createUser } from './actions';
import SubmitButton from './SubmitButton';
export default function UserForm() {
  return (
    <form action={createUser}>
      <input name="name" />
      <SubmitButton />
    </form>
  );
}

How It Works

Flow

The hook returns an object describing the current form submission.

04-07How do you handle and surface validation errors from a Server Action?

Validation errors from a Server Action are handled by returning an error object from the Server Action and showing that object in the UI using useActionState().

Simple Meaning:
Do not throw validation errors for normal user mistakes. Return them as state and display them in the form.

Recommended Flow

Flow
User Submits Form
        ↓
Server Action Receives FormData
        ↓
Validate Data
        ↓
If Invalid → Return Error Object
        ↓
useActionState Receives Error
        ↓
Show Error In UI

1. Create Server Action

The Server Action validates the form data and returns a state object.

app/actions.ts
'use server';
type State = {
  success: boolean;
  message: string;
  errors?: {
    name?: string;
    email?: string;
  };
};
export async function createUser(previousState: State, formData: FormData): Promise<State> {
  const name = formData.get('name') as string;
  const email = formData.get('email') as string;
  const errors: State['errors'] = {};
  if (!name) errors.name = 'Name is required';
  if (!email) errors.email = 'Email is required';
  if (Object.keys(errors).length > 0) {
    return { success: false, message: 'Please fix the errors below', errors };
  }
  await db.user.create({ data: { name, email } });
  return { success: true, message: 'User created successfully' };
}

2. Use useActionState In Form

The Client Component receives the returned state from the Server Action.

app/UserForm.tsx
'use client';
import { useActionState } from 'react';
import { createUser } from './actions';
const initialState = { success: false, message: '', errors: {} };
export default function UserForm() {
  const [state, formAction] = useActionState(createUser, initialState);
  return (
    <form action={formAction}>
      <input name="name" placeholder="Enter name" />
      {state.errors?.name && <p>{state.errors.name}</p>}
      <input name="email" placeholder="Enter email" />
      {state.errors?.email && <p>{state.errors.email}</p>}
      <button>Save</button>
      {state.message && <p>{state.message}</p>}
    </form>
  );
}

Important Point

For expected validation errors, return values instead of throwing errors.

Situation Recommended Handling
User left name empty Return validation error
Email is invalid Return validation error
Password is too short Return validation error
Database server crashed Throw error or handle with error boundary

With Zod Validation

In real projects, we usually validate with a schema library like Zod.

app/actions.ts
'use server';
import { z } from 'zod';
const UserSchema = z.object({
  name: z.string().min(1, 'Name is required'),
  email: z.string().email('Invalid email address')
});
type State = {
  success: boolean;
  message: string;
  errors?: {
    name?: string[];
    email?: string[];
  };
};
export async function createUser(previousState: State, formData: FormData): Promise<State> {
  const result = UserSchema.safeParse({
    name: formData.get('name'),
    email: formData.get('email')
  });
  if (!result.success) {
    return { success: false, message: 'Validation failed', errors: result.error.flatten().fieldErrors };
  }
  await db.user.create({ data: result.data });
  return { success: true, message: 'User created successfully', errors: {} };
}

Client Form With Zod Errors

app/UserForm.tsx
'use client';
import { useActionState } from 'react';
import { createUser } from './actions';
const initialState = { success: false, message: '', errors: {} };
export default function UserForm() {
  const [state, formAction] = useActionState(createUser, initialState);
  return (
    <form action={formAction}>
      <input name="name" placeholder="Enter name" />
      {state.errors?.name && <p>{state.errors.name[0]}</p>}
      <input name="email" placeholder="Enter email" />
      {state.errors?.email && <p>{state.errors.email[0]}</p>}
      <button>Save</button>
      {state.message && <p>{state.message}</p>}
    </form>
  );
}

Why We Return Errors Instead Of Throwing

Validation errors are expected user mistakes, so they should be returned as normal state.

Good
return { success: false, message: 'Email is required' };
Avoid For Validation
throw new Error('Email is required');

Throwing is better for unexpected failures like database errors, server crashes, or permission problems.

useActionState vs useFormStatus

Hook Purpose
useActionState() Shows returned validation errors or success messages
useFormStatus() Shows loading/pending status while submitting

Final Best Practice

Best Practice Flow
Server Action
      ↓
Validate FormData
      ↓
Return { success, message, errors }
      ↓
useActionState()
      ↓
Show errors beside fields
One-Line Definition:
Validation errors from a Server Action should be returned as structured state and displayed in the UI using useActionState(), while unexpected system errors should be thrown or handled separately.

04-08Can Server Actions be called from Client Components? How?

Yes, Server Actions can be called from Client Components, but the Server Action must be defined in a separate file with 'use server' at the top.

Simple Meaning:
A Client Component can import a Server Action and call it from a form, button click, or event handler. The function still runs on the server.

1. Create Server Action In Separate File

First, create a separate server-only file.

app/actions.ts
'use server';
export async function createUser(name: string) {
  await db.user.create({ data: { name } });
  return { success: true, message: 'User created successfully' };
}

Because this file starts with 'use server', the exported function becomes a Server Action.

2. Call Server Action From Client Component

Now import that Server Action inside a Client Component.

app/UserButton.tsx
'use client';
import { createUser } from './actions';
export default function UserButton() {
  async function handleClick() {
    const result = await createUser('Rizwan');
    console.log(result.message);
  }
  return <button onClick={handleClick}>Create User</button>;
}

The button click happens in the browser, but createUser() executes on the server.

Flow

Internal Flow
User Clicks Button
        ↓
Client Component Calls createUser()
        ↓
Next.js Sends Internal Request
        ↓
Server Action Runs On Server
        ↓
Database Updated
        ↓
Result Returned To Client

3. Calling Server Action From Client Form

We can also use a Server Action with a form inside a Client Component.

app/actions.ts
'use server';
export async function saveUser(formData: FormData) {
  const name = formData.get('name') as string;
  await db.user.create({ data: { name } });
  return { success: true, message: 'Saved successfully' };
}
app/UserForm.tsx
'use client';
import { saveUser } from './actions';
export default function UserForm() {
  return (
    <form action={saveUser}>
      <input name="name" placeholder="Enter name" />
      <button>Save</button>
    </form>
  );
}

4. With useActionState

For showing success or error messages, use useActionState().

app/actions.ts
'use server';
type State = { success: boolean; message: string };
export async function createUser(previousState: State, formData: FormData): Promise<State> {

app/UserForm.tsx

04-09What are the security considerations and CSRF handling for Server Actions?

Server Actions run on the server, so they can safely access databases, cookies, sessions, and secret values. But they still need proper validation, authentication, authorization, and CSRF protection.

Simple Meaning:
Server Actions are server-side, but they are still callable from the browser, so every action must be treated like a protected backend endpoint.

1. Server Actions Are Not Public Browser Code

Code inside a Server Action is not shipped to the browser.

Server Action
'use server';
export async function createUser(formData: FormData) {
  await db.user.create({ data: { name: formData.get('name') as string } });
}

This code runs on the server, not inside the browser.

Good:
Database logic, secret keys, and server-side checks can stay on the server.

2. But Server Actions Still Need Authentication

Do not assume that only a valid user can call the action. Always check the current session inside the Server Action.

Check Session
'use server';
import { auth } from '@/lib/auth';
export async function updateProfile(formData: FormData) {
  const session = await auth();
  if (!session?.user) {
    return { success: false, message: 'Unauthorized' };
  }
  await db.user.update({
    where: { id: session.user.id },
    data: { name: formData.get('name') as string }
  });
  return { success: true, message: 'Profile updated' };
}
Important:
Never trust only the UI. Hiding a button is not security. Always check permission inside the Server Action.

3. Authorization Is Also Required

Authentication means the user is logged in. Authorization means the user is allowed to perform that specific action.

Authorization Example
'use server';
export async function deleteProduct(productId: string) {
  const session = await auth();
  if (!session?.user) {
    return { success: false, message: 'Unauthorized' };
  }
  if (session.user.role !== 'admin') {
    return { success: false, message: 'Forbidden' };
  }
  await db.product.delete({ where: { id: productId } });
  return { success: true, message: 'Product deleted' };
}
Check Meaning
Authentication Is the user logged in?
Authorization Is the user allowed to do this action?

4. Validate All Form Data

Never trust formData. A user can modify form values from the browser DevTools.

Validation Example
'use server';
export async function createUser(formData: FormData) {
  const name = formData.get('name') as string;
  const email = formData.get('email') as string;
  if (!name) return { success: false, message: 'Name is required' };
  if (!email.includes('@')) return { success: false, message: 'Invalid email' };
  await db.user.create({ data: { name, email } });
  return { success: true, message: 'User created' };
}

5. Do Not Trust Hidden Inputs

Hidden inputs are useful for sending IDs, but they are not secure. Users can change them.

Form
<form action={deleteProduct}>
  <input type="hidden" name="productId" value={product.id} />
  <button>Delete</button>
</form>

The Server Action must still check whether the logged-in user can delete that product.

Server Action
'use server';
export async function deleteProduct(formData: FormData) {
  const session = await auth();
  const productId = formData.get('productId') as string;
  if (session?.user.role !== 'admin') {
    return { success: false, message: 'Forbidden' };
  }
  await db.product.delete({ where: { id: productId } });
  return { success: true, message: 'Product deleted' };
}

6. CSRF Handling in Server Actions

CSRF means Cross-Site Request Forgery. It happens when another website tries to submit a request using the logged-in user's cookies.

CSRF Attack Idea
User logged in on our site
        ↓
User opens malicious website
        ↓
Malicious site tries to submit a request
        ↓
Browser may send cookies automatically

Next.js Server Actions include built-in CSRF protection by checking request origins. In simple terms, the request must come from the same origin as our application.

Simple Meaning:
Next.js checks where the Server Action request is coming from. Cross-site action requests are blocked unless explicitly allowed.

7. allowedOrigins for Proxies or Different Domains

Sometimes our app may be behind a reverse proxy, load balancer, or different domain. In that case, we can configure allowed origins.

next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    serverActions: {
      allowedOrigins: ['my-proxy.com', '*.my-proxy.com']
    }
  }
};
module.exports = nextConfig;
Important:
Only add trusted domains in allowedOrigins. Do not add random public domains.

8. Use SameSite Cookies

For cookie-based authentication, use secure cookie settings.

Cookie Security
cookies().set('session', token, {
  httpOnly: true,
  secure: true,
  sameSite: 'lax',
  path: '/'
});
Cookie Option Purpose
httpOnly JavaScript cannot read the cookie
secure Cookie is sent only over HTTPS
sameSite Helps reduce CSRF risk

9. Do Not Put Secrets in Client Components

Server Actions can access secrets, but Client Components cannot safely store secret values.

Good
'use server';
export async function callPrivateApi() {
  const apiKey = process.env.PRIVATE_API_KEY;
  await fetch('https://api.example.com/private', {
    headers: { Authorization: `Bearer ${apiKey}` }
  });
}
Bad
'use client';
const apiKey = 'secret-key';
export default function Page() {
  return <button>Call API</button>;
}

10. Avoid Returning Sensitive Data

A Server Action can return data to the browser. So do not return passwords, tokens, secret keys, or private database fields.

Bad
return {
  success: true,
  token: user.sessionToken,
  passwordHash: user.passwordHash
};
Good
return {
  success: true,
  message: 'Login successful'
};

11. Security Checklist

Security Rule Required?
Check logged-in user ✅ Yes
Check user permission ✅ Yes
Validate all form data ✅ Yes
Trust hidden input values ❌ No
Return secrets to browser ❌ No
Use secure cookies ✅ Yes
Add random allowedOrigins ❌ No

Easy Memory Trick

Remember
Server Action
      ↓
Still a Backend Endpoint
      ↓
Auth + Permission + Validation + CSRF Safety
One-Line Definition:
Server Actions are protected by server-side execution and built-in origin checks, but we must still validate input, check authentication, check authorization, secure cookies, and avoid exposing sensitive data.

04-10How do you redirect inside a Server Action without triggering a client-side navigation?

Inside a Server Action, we use redirect() from next/navigation.

Simple Meaning:
After the Server Action completes its work, Next.js can redirect the user from the server without using router.push().

Import redirect()

Import
import { redirect } from 'next/navigation';

Basic Example

Suppose we create a user and then want to redirect to the users page.

Server Action
'use server';
import { redirect } from 'next/navigation';
export async function createUser(formData: FormData) {
  const name = formData.get('name') as string;
  await db.user.create({ data: { name } });
  redirect('/users');
}
Page
import { createUser } from './actions';
export default function Page() {
  return (
    <form action={createUser}>
      <input name="name" />
      <button>Create User</button>
    </form>
  );
}

Flow

What Happens?
Submit Form
      ↓
Server Action Runs
      ↓
Database Updated
      ↓
redirect('/users')
      ↓
User Lands On /users

Important: redirect() Stops Execution

Once redirect() is called, no code below it will execute.

Example
'use server';
import { redirect } from 'next/navigation';
export async function createUser() {
  await db.user.create({ data: { name: 'Rizwan' } });
  redirect('/users');
  console.log('Never runs'); // ❌
}

Think of redirect() like throwing a special redirect exception.

Real World Login Example

Login Action
'use server';
import { redirect } from 'next/navigation';
export async function login(formData: FormData) {
  const email = formData.get('email') as string;
  const password = formData.get('password') as string;
  const user = await validateUser(email, password);
  if (!user) {
    return { success: false, message: 'Invalid credentials' };
  }
  await createSession(user.id);
  redirect('/dashboard');
}

After successful login, the user is redirected to the dashboard directly from the server.

redirect() vs router.push()

Feature redirect() router.push()
Runs On Server Client
Used In Server Actions ✅ ❌
Used In Client Components ❌ ✅
Navigation Type Server Driven Client Driven
Client Navigation Example
'use client';
import { useRouter } from 'next/navigation';
export default function Page() {
  const router = useRouter();
  return (
    <button onClick={() => router.push('/users')}>
      Go To Users
    </button>
  );
}

No. A single execution path can either return data or redirect.

Wrong
'use server';
import { redirect } from 'next/navigation';
export async function createUser() {
  redirect('/users');
  return { success: true }; // ❌ Never reached

Correct
'use server';
import { redirect } from 'next/navigation';
export async function createUser() {

}

redirect() With Validation
