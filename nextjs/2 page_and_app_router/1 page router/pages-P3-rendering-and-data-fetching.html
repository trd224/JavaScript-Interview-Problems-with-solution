P3-01Why Does Pages Router Hydrate Everything While App Router Doesn't?

The main difference is that the Pages Router was built around full-page hydration, while the App Router uses React Server Components to avoid sending unnecessary JavaScript to the browser.

Pages Router renders HTML on the server, but then still sends page JavaScript and React JavaScript to the browser so React can hydrate the whole page.

Step 1: What Happens in Pages Router?

In the Pages Router, even if the server already sends HTML, the browser still receives the page JavaScript and React JavaScript.

pages/index.js
export default function Page() {
  return <h1>Hello</h1>;
}

Pages Router Rendering Flow

Flow

   │
   ▼

   │

So in the Pages Router, the component runs on the server first and then runs again in the browser during hydration.

Result
Page() runs on Server  ✅
Page() runs on Browser ✅

Why Does Pages Router Send page.js and react.js?

Because the Pages Router architecture assumes that every page is a React application that must be hydrated in the browser.

It does not matter whether the current page is interactive or not. Pages Router was designed to hydrate the whole page.

What Is Hydration?

Hydration means React attaches browser-side JavaScript behavior to the HTML that was already generated on the server.

Server HTML
<button>Click Me</button>

This HTML is visible, but it does not know anything about React state or click handlers.

React Component
export default function Button() {
  return (
    <button onClick={() => alert('Hello')}>
      Click Me
    </button>
  );
}

The browser needs JavaScript to attach the onClick behavior.

Pages Router was built before React Server Components existed.

At that time, React had only one main server rendering model:

P3-02What is Server-Side Rendering (SSR) in Next.js?

Server-Side Rendering (SSR) is a rendering technique where Next.js generates the HTML for a page on the server for every request before sending it to the browser.

SSR allows us to fetch fresh data on every request and send a fully rendered HTML page to the browser.

Why Do We Need SSR?

Some pages require the latest data every time a user visits them.

Examples:
  • Stock Market Prices
  • Weather Data
  • Live Dashboard
  • News Websites
  • E-commerce Product Inventory

For such pages, generating HTML at build time is not enough because the data changes frequently.

How SSR Works

SSR Flow
User Requests Page
        │
        ▼
Next.js Server
        │
        ▼
Fetch Latest Data
        │
        ▼
Generate HTML
        │
        ▼
Send HTML To Browser
        │
        ▼
Hydration Happens

Real Example

Suppose our application displays the current Bitcoin price.

User Opens
/bitcoin-price

Next.js performs:

Server
1. Fetch Latest Bitcoin Price
2. Generate HTML
3. Send HTML To Browser

Every user gets the latest data.

SSR in the Pages Router

In the Pages Router, SSR is implemented using getServerSideProps().

pages/products.js
export async function getServerSideProps() {
  const res = await fetch(
    'https://api.example.com/products'
  );
  const products = await res.json();
  return {
    props: { products }
  };
}
export default function Products({ products }) {
  return (
    <h1>
      Total Products: {products.length}
    </h1>
  );
}

What Happens Internally?

Request Flow
Request #1
      │
      ▼
getServerSideProps()
      │
      ▼
Fresh Data
      │
      ▼
HTML Generated
Request #2
      │
      ▼
getServerSideProps()
      │
      ▼
Fresh Data Again
      │
      ▼
New HTML Generated

The function runs for every request.

SSR in the App Router

The App Router does not use getServerSideProps().

Instead, Server Components can directly fetch data.

app/products/page.js
async function getProducts() {
  const res = await fetch(
    'https://api.example.com/products',
    { cache: 'no-store' }
  );
  return res.json();
}
export default async function Products() {
  const products = await getProducts();
  return (
    <h1>
      Total Products: {products.length}
    </h1>
  );
}

Why cache: 'no-store'?

The cache: 'no-store' option tells Next.js:

Meaning
Do Not Cache
Fetch Fresh Data
For Every Request

This behavior is equivalent to SSR.

SSR vs CSR

Feature SSR CSR
HTML Generated Server Browser
Initial Load Fast Usually Slower
SEO Excellent Limited
Fresh Data Every Request Client Fetches Data

SSR vs SSG

Feature SSR SSG
HTML Generation Time Every Request Build Time
Performance Slower Faster
Fresh Data Always Only At Build Time
Server Required Yes No

Advantages of SSR

  • Always fresh data.
  • Excellent SEO.
  • Better social media previews.
  • Faster first content display.
  • Works well for dynamic content.

Disadvantages of SSR

  • Runs on every request.
  • Higher server load.
  • Slower than SSG.
  • Requires server resources.

When Should We Use SSR?

  • Frequently changing data.
  • User-specific dashboards.
  • Live statistics.
  • Stock prices.
  • Weather information.
  • Real-time inventory.

Interview Tip

A common misconception is that SSR means rendering on the server once. In reality, SSR generates fresh HTML on the server for every incoming request.
One-Line Definition:
Server-Side Rendering (SSR) is a rendering technique in Next.js where the server generates fresh HTML for every request before sending it to the browser.

P3-03What is Static Site Generation (SSG) in Next.js?

Static Site Generation (SSG) is a rendering technique where Next.js generates HTML pages during the build process and stores them as static files.

SSG allows us to generate pages once at build time and serve the same pre-generated HTML to all users.

Why Do We Need SSG?

Many pages do not change frequently.

Examples:
  • Blog Posts
  • Documentation Websites
  • Marketing Pages
  • Landing Pages
  • Company Information Pages

Generating these pages on every request would be unnecessary.

Instead, we can generate them once during the build process.

Dynamic SSG Pages

For dynamic routes, we use getStaticPaths().

pages/posts/[id].js

P3-04What is Client-Side Rendering (CSR) and When Should We Choose It?

Client-Side Rendering (CSR) is a rendering technique where the browser downloads JavaScript and generates the page UI on the client side.

In CSR, the server sends a minimal HTML page and JavaScript bundle, and the browser renders the application after the JavaScript executes.

Why Do We Need CSR?

Some applications are highly interactive and depend heavily on user actions.

Examples:
  • Admin Dashboards
  • Project Management Tools
  • Chat Applications
  • Internal Business Applications
  • Analytics Dashboards

For these applications, SEO is usually less important than user interaction.

P3-05What is getStaticProps and When is it Called?

getStaticProps() runs during the build process, generates static HTML, and provides the fetched data as props to the page component.

Why Do We Need getStaticProps()?

Some pages contain data that does not change frequently.

Examples:
  • Blog Posts
  • Documentation Pages
  • Landing Pages
  • Company Information Pages
  • Marketing Websites

P3-06What is getServerSideProps and How Does It Differ from getStaticProps?

getServerSideProps() is a special Next.js function used in the Pages Router to fetch data on every request and generate fresh HTML for each user request.

getServerSideProps() runs on every request, while getStaticProps() runs during the build process.

Why Do We Need getServerSideProps()?

Some pages require fresh data every time users visit them.

Examples:
  • Stock Prices
  • Weather Information
  • User Dashboards
  • Live Reports
  • Real-Time Analytics

In these cases, build-time data may become outdated.

Basic Example

pages/products.js
export async function getServerSideProps() {
  const res = await fetch(

P3-08What is the Difference Between fallback: true, false, and 'blocking'?

The fallback option is returned from getStaticPaths() in Next.js. It controls one specific thing: what should happen when a visitor requests a dynamic page whose path was NOT included in the paths array during build time.

Think of paths as "the list of pages I already know about." fallback answers the question: "What do I do about pages that AREN'T on that list?"

Why Does This Option Exist?

Imagine an e-commerce site with 50,000 products. Pre-building all 50,000 pages at deploy time would make builds extremely slow. So Next.js lets you pre-build only the important/popular pages (say, the top 20) and decide what to do for the other 49,980 when someone actually visits them.

That decision is exactly what fallback controls, and it has 3 possible values:

  • false — unknown paths are not allowed at all
  • true — unknown paths are generated on first visit, with a loading state
  • 'blocking' — unknown paths are generated on first visit, with no loading state (server waits)

P3-10How do you fetch data on the client side in a Next.js app?

Client-side data fetching means fetching data in the browser after the page has loaded.

In Next.js, we fetch data on the client side by using a Client Component with 'use client', then calling an API inside useEffect() or using a library like SWR or React Query.

When Do We Use Client-Side Fetching?

Client-side fetching is useful when data depends on browser interaction or user-specific state.

  • Dashboard data
  • User profile data
  • Search results
  • Filters
  • Pagination
  • Live updates
  • Data after button click

Basic Client-Side Fetching

In the App Router, we must use 'use client' because useEffect() runs only in the browser.

app/users/page.js
'use client';
import { useEffect, useState } from 'react';
export default function UsersPage() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    async function fetchUsers() {
      const res = await fetch('/api/users');
      const data = await res.json();
      setUsers(data);
      setLoading(false);
    }
    fetchUsers();
  }, []);
  if (loading) {
    return <h2>Loading...</h2>;
  }
  return (
    <div>
      <h1>Users</h1>
      {users.map(user => (
        <p key={user.id}>{user.name}</p>
      ))}
    </div>
  );
}

When Should We Choose Client-Side Fetching?

  • When data is not important for SEO.
  • When data depends on user interaction.
  • When data is private or user-specific.
  • When we need live updates.
  • When fetching should happen after the page loads.

When Should We Avoid Client-Side Fetching?

  • SEO-focused pages
  • Blogs
  • Marketing pages
  • Product pages that need search engine indexing
  • Pages where content should be visible immediately in HTML

Important Note

Client-side fetching means users may initially see a loading state because the data is fetched after JavaScript runs in the browser.

Interview Tip

A common misconception is that all data fetching in Next.js should happen on the server. In reality, client-side fetching is still useful for interactive, user-specific, and frequently changing data.
One-Line Definition:
Client-side data fetching in Next.js means fetching data in the browser using a Client Component with useEffect(), fetch(), or a library like SWR, usually for interactive or user-specific data.

P3-11What is SWR and how does it integrate with Next.js?

SWR is a React data fetching library created by Vercel that provides caching, automatic revalidation, request deduplication, polling, and real-time data synchronization.

SWR stands for Stale-While-Revalidate. It immediately shows cached data (stale data), then fetches fresh data in the background and updates the UI automatically.

Why Do We Need SWR?

Without SWR, we often write repetitive code for:

  • Loading states
  • Error handling
  • Caching
  • Refetching
  • Revalidation
Without SWR
'use client';
import { useEffect, useState } from 'react';
export default function Users() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    async function loadUsers() {
      const res = await fetch('/api/users');
      const data = await res.json();
      setUsers(data);
      setLoading(false);
    }
    loadUsers();
  }, []);
  return loading
    ? <h2>Loading...</h2>
    : <div>{users.length}</div>;
}

SWR handles most of this automatically.

Why Is SWR Fast?

Because cached data can be displayed instantly.

Users do not always need to wait for a new API request.

Automatic Revalidation

SWR automatically refetches data when:

  • User returns to the tab.
  • Network reconnects.
  • Component remounts.
Example
const {data}=useSWR(
  '/api/users',
  fetcher
);

These behaviors work automatically.

Polling Data

SWR can automatically refresh data at a specified interval.

Auto Refresh Every 5 Seconds
const {data}=useSWR(
  '/api/users',
  fetcher,
  {refreshInterval:5000}
);

Manual Revalidation

We can force SWR to refetch data.

Example
import useSWR,{mutate} from 'swr';
const refresh=()=>{
  mutate('/api/users');
};

SWR works especially well with Next.js because both are maintained by Vercel.

  • Supports Next.js caching strategies.
