P4 - API Routes (Backend in Next.js)

Part P4 of 7 · 2 topics · Next.js Pages Router Guide

P4-01What Are API Routes in Next.js and Where Do We Place Them?

API Routes are a way to build a backend API directly inside a Next.js Pages Router application. Any file placed inside the pages/api folder is automatically treated as an API endpoint instead of a page, and is never sent to the browser as part of the client bundle.

Simple Meaning:
Anything inside pages/api runs only on the server and gives us a ready-made backend without needing a separate Express or Node server.

Where Do API Routes Live?

Folder Structure
pages/
├── api/
│   ├── hello.js
│   ├── users.js
│   └── users/
│       └── [id].js

Basic API Route

pages/api/hello.js
export default function handler(req, res) {
  res.status(200).json({ message: 'Hello World' });
}

This file automatically becomes available at /api/hello. The exported default function receives Node-style req and res objects, the same pattern used by Express.

Dynamic API Routes

pages/api/users/[id].js
export default function handler(req, res) {
  const { id } = req.query;
  res.status(200).json({ userId: id });
}

Dynamic segments work the same way as page routes — a file named [id].js captures the URL segment as req.query.id.

Why Use API Routes?

  • Build a backend without a separate server
  • Hide API keys and secrets from the browser
  • Connect directly to a database
  • Handle form submissions and webhooks
  • Proxy requests to third-party APIs
API Routes only exist in the Pages Router. In the App Router, the equivalent concept is a Route Handler defined in a route.ts file.

P4-02How Do We Handle Different HTTP Methods in a Single API Route?

A single API Route file can handle multiple HTTP methods (GET, POST, PUT, DELETE, and so on) by checking req.method inside the handler function and branching accordingly.

Simple Meaning:
Since an API Route exports one default function, we read req.method ourselves and decide what to do for each type of request.

Handling Multiple Methods

pages/api/users.js
export default function handler(req, res) {
  switch (req.method) {
    case 'GET':
      return res.status(200).json({ users: [] });
    case 'POST':
      const newUser = req.body;
      return res.status(201).json(newUser);
    case 'PUT':
      return res.status(200).json({ updated: true });
    case 'DELETE':
      return res.status(200).json({ deleted: true });
    default:
      res.setHeader('Allow', ['GET', 'POST', 'PUT', 'DELETE']);
      return res.status(405).end(`Method ${req.method} Not Allowed`);
  }
}

Reading the Request Body

Next.js automatically parses JSON request bodies, so req.body is ready to use for POST and PUT requests without any extra middleware.

Example POST Body
export default function handler(req, res) {
  if (req.method === 'POST') {
    const { name, email } = req.body;
    res.status(201).json({ name, email });
  }
}

Setting the Response Status and Headers

res.status(404).json({ error: 'Not Found' });
res.setHeader('Content-Type', 'application/json');
res.setHeader('Cache-Control', 's-maxage=60');
Always return a 405 status with an Allow header for methods we don't explicitly handle — this tells clients exactly which methods are supported.
