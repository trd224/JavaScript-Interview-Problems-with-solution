P2-01What is File-Based Routing in Next.js?

File-Based Routing is a routing system in Next.js where routes are automatically created based on the file and folder structure of your project.

In Next.js, you do not manually configure routes like React Router. Instead, creating a file automatically creates a route.

Why is it Called File-Based Routing?

Because URLs are generated directly from files and folders.

For example, if you create:

Pages Router Example
pages/
├── index.js
├── about.js
└── contact.js

Next.js automatically creates:

File Route
pages/index.js /
pages/about.js /about
pages/contact.js /contact

No route configuration is needed.

How Routing Works in React

In a plain React application, you usually need React Router.

React Router Example
import { BrowserRouter, Routes, Route } from 'react-router-dom';
function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
      </Routes>
    </BrowserRouter>
  );
}

Routes must be configured manually.

How Routing Works in Next.js

Simply create files and Next.js automatically creates routes.

Next.js Pages Router
pages/
├── index.js
└── about.js

Generated Routes:

Routes
/
/about

File-Based Routing in App Router

In modern Next.js applications, routing is handled using the app directory.

App Router Structure
app/
├── page.js
├── about/
│   └── page.js
└── contact/
    └── page.js

Generated Routes:

File Route
app/page.js /
app/about/page.js /about
app/contact/page.js /contact

Nested Routes

Folders create nested routes automatically.

Nested Route Example

Generated Route:

Multiple Nested Routes

Folder Structure
app/
├── dashboard/
│   ├── analytics/
│   │   └── page.js

Generated Route:

Route
/dashboard/analytics

Dynamic Routes

Dynamic routes are created using square brackets.

Pages Router Dynamic Route

Possible URLs:

/products/1
/products/10
/products/100

App Router version:

App Router Dynamic Route

└── products/
    └── [id]/
        └── page.js

Catch-All Routes

Catch-All Route

└── docs/
    └── [...slug]/
        └── page.js

Can match:

Example URLs

Advantages of File-Based Routing

  • No route configuration required.
  • Cleaner project structure.
  • Easier to understand.
  • Automatic nested routing.
  • Better developer experience.
  • Less boilerplate code.

Pages Router vs App Router

Pages Router App Router

pages/products/[id].js app/products/[id]/page.js
Uses pages folder Uses app folder
Older system Modern system
One-Line Definition:
File-Based Routing in Next.js is a routing system where URLs are automatically generated from the project's file and folder structure, eliminating the need for manual route configuration.

P2-02What is _app.js and What is it Used For?

_app.js is a special file in the Pages Router of Next.js that allows us to customize and control the root component for all pages in our application.

Every page in the Pages Router is wrapped by _app.js. It is commonly used for global layouts, providers, global CSS, authentication, and shared application logic.

Where is _app.js Located?

Project Structure
pages/
├── _app.js
├── index.js
├── about.js
└── contact.js

The _app.js file is placed directly inside the pages directory.

Why Do We Need _app.js?

Without _app.js, every page renders independently.

Without _app.js
User
  │
  ▼
about.js
  │
  ▼
Rendered Page

When _app.js exists, all pages are wrapped by it.

With _app.js
User
  │
  ▼
_app.js
  │
  ▼
about.js
  │
  ▼
Rendered Page

Basic _app.js Example

pages/_app.js
export default function MyApp({ Component, pageProps }) {
  return (
    <Component {...pageProps} />
  );
}

What are Component and pageProps?

Property Description
Component The current page being rendered
pageProps The props passed to that page

Example Flow

pages/about.js
export default function About() {
  return (
    <h1>About Page</h1>
  );
}

When we visit /about, Next.js internally renders:

Internal Rendering
<MyApp Component={About} pageProps={{}}/>

Use Case 1: Global Layout

We can create a common layout for all pages.

Global Layout Example
import Header from '../components/Header';
import Footer from '../components/Footer';
export default function MyApp({
  Component,
  pageProps
}) {
  return (
    <>
      <Header />
      <Component {...pageProps} />
      <Footer />
    </>
  );
}

Now every page automatically displays the Header and Footer.

Use Case 2: Global CSS

Global CSS files are imported inside _app.js.

Import Global CSS
import '../styles/globals.css';
export default function MyApp({
  Component,
  pageProps
}) {
  return (
    <Component {...pageProps} />
  );
}

Use Case 3: Context Provider

We can provide React Context to the entire application.

Context Provider Example
import { UserProvider } from '../context/UserContext';
export default function MyApp({
  Component,
  pageProps
}) {
  return (
    <UserProvider>
      <Component {...pageProps} />
    </UserProvider>
  );
}

Use Case 4: Redux Provider

Redux Example
import { Provider } from 'react-redux';
import store from '../store';
export default function MyApp({
  Component,
  pageProps
}) {
  return (
    <Provider store={store}>
      <Component {...pageProps} />
    </Provider>
  );
}

Use Case 5: Authentication

Authentication checks can be centralized inside _app.js.

Authentication Example
export default function MyApp({
  Component,
  pageProps
}) {
  const isLoggedIn = true;
  if (!isLoggedIn) {
    return <Login />;
  }
  return (
    <Component {...pageProps} />
  );
}

Uses _app.js Uses layout.js

Older Routing System Modern Routing System

Modern Equivalent: layout.js

In Next.js 13+, most use cases of _app.js are handled by layout.js.

app/layout.js

  return (
    <html>
      <body>

      </body>

One-Line Definition:
_app.js is a special file in the Pages Router of Next.js that allows us to customize and control the root component for all pages in our application, making it useful for global layouts, providers, CSS, authentication, and shared application logic.

P2-03What is _document.js and When Would We Use It?

_document.js is a special file in the Pages Router of Next.js that allows us to customize the HTML document structure that Next.js sends to the browser.

Unlike _app.js, which controls React components, _document.js controls the root HTML structure such as <html>, <body>, and other document-level settings.

Where is _document.js Located?

Project Structure
pages/
├── _app.js
├── _document.js
├── index.js
└── about.js

Why Do We Need _document.js?

Normally, Next.js generates the HTML document automatically.

Generated HTML
<html>
  <body>
    <div id="__next"></div>
  </body>
</html>

Sometimes we need to customize this structure. That's when we use _document.js.

Basic Example

pages/_document.js
import { Html, Head, Main, NextScript } from 'next/document';
export default function Document() {
  return (
    <Html>
      <Head />
      <body>
        <Main />
        <NextScript />
      </body>
    </Html>
  );
}

Important Components

Component Purpose
Html Represents the root <html> element
Head Used for document-level head content
Main Renders the application content
NextScript Loads Next.js JavaScript files

Use Case 1: Set HTML Language

We can define the language of our application.

Language Example
<Html lang="en">
  ...
</Html>

Generated HTML:

Output
<html lang="en">

Use Case 2: Add Custom Fonts

Google Font Example

  <link
    rel="preconnect"
    href="https://fonts.googleapis.com"
  />
</Head>

Use Case 3: Add External Scripts

External Script Example
<Head>
  <script
    src="https://example.com/script.js"
  />
</Head>

Use Case 4: Add Global Body Classes

Body Class Example
<body className="dark-theme">
  <Main />
  <NextScript />
</body>

Technically yes, but it is generally better to add page-specific metadata using:

Page Metadata

  return (
    <>
      <Head>
        <title>Home Page</title>
      </Head>
    </>

Uses _document.js No _document.js
Uses _app.js Uses layout.js
Older routing system Modern routing system

What Replaces _document.js in App Router?

In Next.js 13+, the root HTML structure is usually defined inside app/layout.js.

app/layout.js
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>
        {children}
      </body>
    </html>
  );
}

When Would We Use _document.js?

  • Setting the HTML language attribute.
  • Adding custom fonts globally.
  • Adding external scripts.
  • Customizing the body element.
  • Customizing the root HTML structure.
  • Adding document-level settings.
One-Line Definition:
_document.js is a special file in the Next.js Pages Router that allows us to customize the HTML document structure sent to the browser, including the <html>, <head>, and <body> elements.

P2-04How does Dynamic Routing work in Next.js?

Dynamic Routing in Next.js allows us to create routes whose URL segments are determined at runtime instead of being fixed at development time.

Dynamic routes are created using square brackets []. The value inside the URL is captured as a route parameter.

Why Do We Need Dynamic Routing?

Suppose our application displays products.

Creating a separate file for every product would be impossible:

Bad Approach
products-1.js
products-2.js
products-3.js
products-100.js
products-1000.js

Instead, we create a single dynamic route that handles all product IDs.

Pages Router Dynamic Route

Folder Structure
pages/
└── products/
    └── [id].js

The [id] segment is dynamic.

This single file can handle:

URLs
/products/1
/products/5
/products/100
/products/999

Accessing Route Parameters (Pages Router)

pages/products/[id].js
import { useRouter } from 'next/router';
export default function Product() {
  const router = useRouter();
  return (
    <h1>
      Product ID: {router.query.id}
    </h1>
  );
}

If the URL is:

URL
/products/100

Output:

Output
Product ID: 100

App Router Dynamic Route

In modern Next.js applications, dynamic routes are created inside the app directory.

Folder Structure
app/
└── products/
    └── [id]/
        └── page.js

Accessing Route Parameters (App Router)

app/products/[id]/page.js
export default function Product({ params }) {
  return (
    <h1>
      Product ID: {params.id}
    </h1>
  );
}

For the URL:

URL
/products/100

Output:

Output
Product ID: 100

P2-05What is the Difference Between [id] and [...slug] in Next.js Routing?

Both [id] and [...slug] are used for dynamic routing in Next.js, but they behave differently.

[id] captures a single URL segment, while [...slug] captures multiple URL segments as an array.

1. Single Dynamic Route: [id]

The [id] syntax matches exactly one dynamic segment.

App Router Structure
app/
└── products/
    └── [id]/
        └── page.js

Supported URLs:

URLs

/products/10
/products/500

Not Supported:

Invalid URLs
/products
/products/10/reviews

  params
}) {
  return (
    <h1>
      {params.id}
    </h1>
  );
}

  id: "100"

The [...slug] syntax matches one or more URL segments.

P2-06What is the Difference Between [[...slug]] and [...slug]?

Both [...slug] and [[...slug]] are Catch-All Routes in Next.js, but they differ in whether the route parameter is required or optional.

[...slug] requires at least one URL segment, while [[...slug]] can match zero or more URL segments.

1. Catch-All Route: [...slug]

The [...slug] route matches one or more URL segments.

P2-07How do you Navigate Between Pages Using the Link Component?

The Link component in Next.js allows us to navigate between pages without performing a full page refresh.

The Link component enables client-side navigation, making page transitions faster and providing a smoother user experience.

Why Not Use a Normal <a> Tag?

A normal anchor tag reloads the entire page.

Normal HTML Link
<a href="/about">
  About
</a>

Navigation Flow:

Page Reload
Current Page
      │
      ▼
Browser Request
      │
      ▼
Server
      │
      ▼
Full Page Reload

This is slower because the browser reloads the entire page.

Using the Link Component

Next.js provides the Link component for client-side navigation.

Basic Example
import Link from 'next/link';
export default function Home() {
  return (
    <Link href="/about">
      About
    </Link>
  );
}

When we click the link, Next.js updates only the necessary content without reloading the entire page.

Navigation Flow with Link

Client-Side Navigation
Current Page
      │
      ▼
Link Click
      │
      ▼
Next.js Router
      │
      ▼
Update UI

P2-08What is the useRouter Hook and What Can We Do With It?

The useRouter hook allows us to access and control the Next.js router programmatically from our components.

The useRouter hook allows us to navigate between pages, access route parameters, read query strings, reload pages, and move backward or forward in browser history.

Important Note

The usage depends on which routing system our application uses.

Router Import
Pages Router import { useRouter } from 'next/router'
App Router import { useRouter } from 'next/navigation'

Pages Router Example

Import useRouter
import { useRouter } from 'next/router';
export default function Home() {
  const router = useRouter();
}

Use Case 1: Navigate Programmatically

Instead of clicking a Link component, we can navigate using JavaScript.

router.push()
import { useRouter } from 'next/router';
export default function Home() {
  const router = useRouter();
  return (
    <button onClick={() => router.push('/about')}>
      About
    </button>
  );
}

Clicking the button navigates to:

URL
/about

Why Do Both Return the Same Value?

Both examples return 100 because Next.js stores both dynamic route parameters and query string parameters inside router.query.

URL Source of id Value
/products/100 Dynamic Route Parameter ([id]) 100

Although both are accessed using router.query.id, they come from different parts of the URL.

Dynamic Route Parameter
/products/100
Query String Parameter
/products?id=100

Next.js combines route parameters and query string parameters into a single router.query object.

Example
/products/100?color=red
router.query
{
  id: '100',
  color: 'red'
}
Dynamic route parameters come from the URL path, while query parameters come from the query string. Both are available through router.query in the Pages Router.

What Happens If Both Use the Same Name?

Consider the following URL:

URL
/products/100?id=100

In this case, the URL contains:

  • A route parameter from [id].
  • A query string parameter from ?id=100.

Both use the same key name (id), so they are merged into the same router.query object.

Read Values
const router = useRouter();
console.log(router.query);
Output
{
  id: '100'
}

A more obvious example:

URL
/products/100?id=200
Output
{
  id: '200'
}

The query string value overrides the route parameter when both use the same key.

Recommended Approach

Use different names for route parameters and query parameters.

URL
/products/100?productId=200
router.query
{
  id: '100',
  productId: '200'
}
If a route parameter and a query parameter have the same name, the query parameter value takes precedence in router.query.

P2-11How do you Create a Custom 404 Page in Next.js?

A custom 404 page allows us to show our own page when a user visits a route that does not exist.

In the Pages Router, we create pages/404.js. In the App Router, we create app/not-found.js.

Pages Router: Custom 404 Page

In the Pages Router, create a file named 404.js inside the pages directory.

Folder Structure
pages/

├── about.js

import Link from 'next/link';
export default function Custom404() {
  return (
    <div>

      <p>The page we are looking for does not exist.</p>
      <Link href="/">Go Home</Link>
    </div>
  );
}

App Router: Custom 404 Page

In the App Router, create a file named not-found.js inside the app directory.

Folder Structure

├── page.js
├── layout.js
└── not-found.js

import Link from 'next/link';
export default function NotFound() {
  return (
    <div>
      <h1>404 - Page Not Found</h1>
      <p>The page we are looking for does not exist.</p>
      <Link href="/">Go Home</Link>
    </div>
  );
}

Show 404 Manually in App Router

If data is not found, we can manually show the 404 page using notFound().

app/products/[id]/page.js
import { notFound } from 'next/navigation';
export default async function Product({ params }) {
  const product = null;
  if (!product) {

P2-12What is Shallow Routing and When Would We Use It?

Shallow Routing is a feature in Next.js that allows us to change the URL without re-running data fetching methods or fully reloading the current page.

Shallow Routing updates the URL and router state while keeping the current page component mounted and avoiding unnecessary data fetching.

Why Do We Need Shallow Routing?

Suppose our application displays products with pagination, sorting, or filters.

When a user changes a filter or page number, we may want to update the URL so it can be bookmarked, refreshed, or shared with others.

Without Shallow Routing, changing the URL may trigger data fetching methods again.

Note:
Shallow Routing is most useful when our application data has already been loaded and we only want to update the URL or UI state. In such cases, re-fetching the same data from the server would be unnecessary.
Example
All Products Already Loaded
User Changes:
✔ View Mode
✔ Selected Tab
✔ Sort Order
✔ Filter State
Only URL Changes
No Need To Fetch Data Again

In these situations, Shallow Routing updates the URL without re-running page data fetching methods.

Current URL
/products?page=1

After clicking the next page:

Updated URL
/products?page=2

We may only want the URL to change without causing a complete page refresh cycle.

Basic Example

Pages Router
import { useRouter } from 'next/router';
export default function Products() {
  const router = useRouter();
  const nextPage = () => {
    router.push('/products?page=2', undefined, { shallow: true });  //router.push(url, as, options);
  };
  return (
    <button onClick={nextPage}>
      Next Page
    </button>
  );
}
| Parameter | Purpose                                                               |
| --------- | --------------------------------------------------------------------- |
| `url`     | Actual route to navigate to                                           |
| `as`      | URL shown in the browser (mainly for older dynamic routing use cases) |
| `options` | Extra options such as `shallow`, `scroll`, `locale`                   |

Navigating to a different page always performs normal routing.

This is a poor use case for Shallow Routing because the page content depends on new server data.
