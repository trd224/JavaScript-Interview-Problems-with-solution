P1-01What is Next.js and how does it differ from React?

React is a JavaScript library used to build user interfaces. It mainly helps us create reusable UI components.

Next.js is a React framework used to build complete web applications. It gives extra features like routing, server-side rendering, static pages, API routes, caching, and performance optimization.

React Example

React Component
function Welcome() {
  return <h1>Hello, world!</h1>;
}

In React, you create components and use them to build the UI. But React does not provide routing, SSR, SSG, or API routes by default.

Next.js Example

Next.js File-Based Routing
app/
  page.js
  about/
    page.js

In Next.js, routes are created automatically using files and folders. For example, app/about/page.js automatically creates the /about route.

Main Difference Between React and Next.js

React Next.js
JavaScript UI library React framework
Used mainly for building UI components Used for building complete web applications
Routing needs React Router File-based routing is built in
Client-side rendering by default Supports SSR, SSG, ISR, and client-side rendering
No built-in API routes API routes are built in
Needs extra setup for optimization Includes image, font, caching, and performance optimization

Simple Explanation

React is like an engine for building UI. Next.js is like a complete car built using that engine.

When to Use React?

Use React when you are building a small single page application, learning React basics, or adding UI components inside an existing project.

When to Use Next.js?

Use Next.js when you need SEO, routing, server rendering, static pages, API routes, better performance, and a production-ready project structure.

Final Answer: React is a UI library, while Next.js is a full React framework that provides routing, rendering options, APIs, caching, and optimization features.

P1-02What are the Main Features of Next.js?

Next.js is a React framework that provides many built-in features for building fast, scalable, SEO-friendly, and production-ready web applications.

1. File-Based Routing

Pages and routes are created automatically from files and folders.
File-Based Routing Example
app/
├── page.js
├── about/
│   └── page.js
└── contact/
    └── page.js

This automatically creates these routes:

  • /
  • /about
  • /contact

2. Server-Side Rendering (SSR)

Next.js can render pages on the server before sending them to the browser.

Benefits:

  • Better SEO
  • Faster first page load
  • Improved performance

3. Static Site Generation (SSG)

Next.js can generate static HTML pages during build time.

Benefits:

  • Very fast loading
  • Reduced server load
  • Good for blogs, landing pages, and documentation websites

4. Incremental Static Regeneration (ISR)

Static pages can be updated automatically after a fixed time without rebuilding the full app.
ISR Example
export const revalidate = 60;

This means the page can be regenerated after 60 seconds when a new request comes.

5. API Routes

You can create backend API endpoints inside the same Next.js project.
API Route Structure
app/
└── api/
    └── users/
        └── route.js
API Route Example
export async function GET() {
  return Response.json({
    users: []
  });
}

6. React Server Components

In the App Router, components are Server Components by default.

Benefits:

  • Less JavaScript sent to the browser
  • Better performance
  • Secure server-side data fetching

7. Client Components

For browser features like click events, state, and effects, we use "use client".
Client Component Example
'use client';
import { useState } from 'react';
export default function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}

8. Image Optimization

Next.js provides an Image component that optimizes images automatically.
Image Component Example
import Image from 'next/image';
export default function Home() {
  return (
    <Image
      src="/photo.jpg"
      width={500}
      height={300}
      alt="Photo"
    />
  );
}

9. Built-in SEO Support

Next.js makes it easy to add title, description, and metadata for pages.
Metadata Example
export const metadata = {
  title: 'My Website',
  description: 'Learning Next.js'
};

10. Automatic Code Splitting

Next.js loads only the JavaScript needed for the current page.

Benefits:

  • Smaller bundle size
  • Faster page load
  • Better performance

11. Built-in TypeScript Support

Next.js supports TypeScript with very little configuration.
TypeScript Example
type User = {
  id: number;
  name: string;
};
const user: User = {
  id: 1,
  name: 'Rizwan'
};

12. Fast Refresh

During development, changes appear instantly in the browser without losing component state.

P1-03What is the Difference Between Next.js and a Plain React App (Create React App)?

Both Next.js and Create React App (CRA) use React, but they serve different purposes.

Create React App creates a basic React Single Page Application (SPA), while Next.js is a full React framework with built-in routing, server rendering, API routes, SEO support, and performance optimizations.

What is Create React App (CRA)?

Create React App is a tool that sets up a React application with minimal configuration. The application runs entirely in the browser.

   ▼
React Renders UI

This approach is called Client-Side Rendering (CSR).

What is Next.js?

Next.js is a React framework that supports multiple rendering strategies and includes many production-ready features.

      │
      ▼
Next.js Server
      │
      ▼
Rendered HTML
      │
      ▼
Browser

Because HTML can be generated before reaching the browser, Next.js generally provides better SEO and faster initial page loads.

P1-05What is the Difference Between the Pages Router and the App Router?

The Pages Router and App Router are two routing systems in Next.js. The Pages Router uses the pages directory and traditional data-fetching methods, while the App Router uses the app directory and introduces Server Components, nested layouts, streaming, and the Metadata API.

The App Router is the modern routing system introduced in Next.js 13 and is now the recommended approach for new applications.

Pages Router

The Pages Router uses the pages folder to create routes.

Project Structure
pages
├── index.js
├── about.js
└── products
    └── [id].js
pages/about.js
export default function About() {
  return <h1>About Page</h1>;
}

This creates: 

/about

App Router

The App Router uses the app folder.

Project Structure
app
├── page.js
├── about
│   └── page.js
└── products
    └── [id]
        └── page.js
app/about/page.js
export default function About() {
  return <h1>About Page</h1>;
}

This also creates: 

/about

Data Fetching

Pages Router

pages/products.js
export async function getServerSideProps() {
  const products =
    await fetch('https://api.example.com/products')
      .then(res => res.json());
  return {
    props: { products: products }
  };
}
export default function Products({
  products
}) {
  return (
    <div>
      {products.length}
    </div>
  );
}

App Router

app/products/page.js
export default async function Page() {
  const products =
    await fetch('https://api.example.com/products')
      .then(res => res.json());
  return (
    <div>
      {products.length}
    </div>
  );
}

No getServerSideProps() is required.

P1-06What is the Purpose of the pages Directory in Next.js?

The pages directory is used for creating routes in traditional Next.js applications.

Every file inside the pages directory automatically becomes a route in your application.

Why is the pages Directory Important?

Instead of manually configuring routes using React Router, Next.js automatically creates routes based on the files inside the pages folder.

This feature is called File-Based Routing.

Basic Example

Pages Router Structure
pages/
├── index.js
├── about.js
└── contact.js
File Route
pages/index.js /
pages/about.js /about
pages/contact.js /contact

Home Page Example

pages/index.js
export default function Home() {
  return (
    <h1>Home Page</h1>
  );
}

Nested Routes

Nested Route Example
pages/
└── blog/
    └── index.js

This creates the route:

/blog

Dynamic Routes

Dynamic Route File
pages/
└── products/
    └── [id].js

This route can handle:

Example URLs

Special Files Explained

1. pages/_app.js

Acts as the root component for all pages.

pages/_app.js
export default function App({ Component, pageProps }) {
  return <Component {...pageProps} />;
}

Common use cases:

  • Global CSS
  • Redux Provider
  • Context API
  • Authentication Wrapper
  • Shared Layouts
Every page in the Pages Router passes through _app.js.

How _app.js Works Internally

Many developers get confused by this code:

pages/_app.js
export default function App({ Component, pageProps }) {
  return <Component {...pageProps} />;
}

The Component prop represents the currently active page component.

Next.js automatically changes the value of Component depending on the route being visited.

Project Structure
my-project/
├── pages/
│   ├── _app.js
│   ├── _document.js
│   ├── index.js
│   ├── about.js
│   └── contact.js
├── public/
├── styles/
└── package.json

When the user visits the home page:

Home Route
<App Component={Home} />  //You never write this yourself, Next.js creates it internally.
return <Home />;

When the user visits the About page:

About Route
<App Component={About} />
return <About />;

When the user visits the Contact page:

Contact Route
<App Component={Contact} />
return <Contact />;
Component = Current Page Component

Why _app.js is Useful

Because it allows us to wrap all pages with common UI.

Shared Layout Example
export default function App({ Component, pageProps }) {
  // Component, pageProps are passed by Next.js itself. Next.js calls it internally.
  // The pageProps value comes from the current page.
  return (
    <>
      <Header />
      <Component {...pageProps} />
      <Footer />
    </>
  );
}

Now every page automatically gets the Header and Footer.

About Page Render
<>
  <Header />
  <About />
  <Footer />
</>
Contact Page Render
<>
  <Header />
  <Contact />
  <Footer />
</>

What is pageProps?

The pageProps object contains props returned from getServerSideProps, getStaticProps, or getInitialProps.

Page Data
export async function getServerSideProps() {
  return {
    props: {
      name: 'Rizwan'
    }
  };
}

Next.js passes those props to _app.js.

Props Passed to _app.js
<App
  Component={Home}
  pageProps={{ name: 'Rizwan' }}
/>

Then:

Prop Forwarding
<Component {...pageProps} />

Becomes:

Final Render
<Home name="Rizwan" />
Interview Answer:
In _app.js, the Component prop represents the current page being rendered and pageProps contains the props returned from data-fetching methods. Next.js automatically injects both values, allowing _app.js to wrap every page with common layouts, providers, and global CSS.

2. pages/_document.js

Used to customize the HTML document structure.

pages/_document.js
import { Html, Head, Main, NextScript } from 'next/document';
export default function Document() {
  return (
    <Html lang="en">
      <Head />
      <body>
        <Main />
        <NextScript />
      </body>
    </Html>
  );
}

Common use cases:

  • Set HTML language
  • Add custom fonts
  • Modify body attributes
  • Customize document structure
_document.js runs only on the server.

3. pages/_error.js

Custom error page used for handling application errors.

pages/_error.js
function ErrorPage({ statusCode }) {
  return <h1>Error {statusCode}</h1>;
}
export default ErrorPage;

It can handle errors such as 404, 500, and other HTTP errors.

4. pages/404.js

Displayed when a route does not exist.

pages/404.js
export default function NotFound() {
  return <h1>404 - Page Not Found</h1>;
}

Example:

URL Request
/about      ✓ exists
/profile    ✗ does not exist

Visiting /profile displays the custom 404 page.

5. pages/500.js

Displayed when a server-side error occurs.

pages/500.js
export default function ServerError() {
  return <h1>500 - Server Error</h1>;
}

Example:

Server Error Example
export async function getServerSideProps() {
  throw new Error('Database Failed');
}

The user will see the custom 500 error page.

Pages Router Request Flow

Request Lifecycle
Request
   │
   ▼
pages/_document.js
   │
   ▼
pages/_app.js
   │
   ▼
Requested Page
   │
   ├── Success → Render Page

Interview Answer:
_app.js wraps all pages and is used for global layouts, providers, and global CSS. _document.js customizes the HTML document structure and runs only on the server. _error.js is a custom error page. 404.js handles not found routes, while 500.js handles server-side errors.
One-Line Definition:
The pages directory in Next.js is used to create application routes automatically, where each file becomes a URL route and pages/api can be used for API endpoints.

Pages Router vs App Router in Next.js

Next.js supports two routing systems:

  • Pages Router - older routing system
  • App Router - newer routing system introduced in Next.js 13
The App Router is recommended for new Next.js projects, but many existing projects still use the Pages Router.

1. Pages Router

Pages Router uses the pages directory. Every file inside the pages folder becomes a route.

Pages Router Structure
pages/
├── index.js
├── about.js
└── contact.js
File Route
pages/index.js /
pages/about.js /about
pages/contact.js /contact
pages/about.js

  return (
    <h1>About Page</h1>
  );
}

2. App Router

App Router uses the app directory. Each route is represented by a folder containing a page.js file.

├── page.js
├── about/
│   └── page.js
└── contact/
    └── page.js

app/page.js /

app/contact/page.js /contact
app/about/page.js
export default function About() {
  return (
    <h1>About Page</h1>
  );
}

Which One Should You Learn?

You should learn both because older projects may use Pages Router, while new Next.js projects mostly use App Router.

  • Use Pages Router when working on older Next.js projects.
  • Use App Router for new modern Next.js projects.
  • In interviews, questions can come from both routing systems.
One-Line Definition:
Pages Router uses the pages directory where each file becomes a route, while App Router uses the app directory where each route is represented by a folder containing a page.js file and supports modern features like Server Components, layouts, loading UI, and error UI.
