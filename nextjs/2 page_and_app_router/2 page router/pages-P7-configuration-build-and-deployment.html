P7-01What is next.config.js / next.config.ts and what can you configure in it?

next.config.js or next.config.ts is the main configuration file for a Next.js application.

It allows us to customize how Next.js builds, serves, optimizes, and runs our application.

Why Do We Need next.config?

By default, Next.js works with sensible defaults.

However, sometimes we need to customize behavior such as:

  • Remote image domains
  • Redirects
  • Rewrites
  • Environment variables
  • Webpack configuration
  • Headers
  • Build settings
  • Output configuration

These settings are configured in next.config.js or next.config.ts.

Project Structure

Example
my-app/
├── app/
├── public/
├── package.json
├── next.config.js
└── tsconfig.json

Basic Example

next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {};
module.exports = nextConfig;

TypeScript Version

next.config.ts
import type {
  NextConfig
} from 'next';
const nextConfig: NextConfig = {};
export default nextConfig;

Both files serve the same purpose.

The TypeScript version provides type safety and IntelliSense.

Configure Remote Image Domains

Example
const nextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'images.example.com'
      }
    ]
  }
};
module.exports = nextConfig;

This allows next/image to optimize images from external domains.

Configure Redirects

Example
const nextConfig = {
  async redirects() {
    return [
      {
        source: '/old-page',
        destination: '/new-page',
        permanent: true
      }
    ];
  }
};
module.exports = nextConfig;

Visitors to /old-page are redirected to /new-page.

Configure Rewrites

Example
const nextConfig = {
  async rewrites() {
    return [
      {
        source: '/api/users',
        destination: 'https://api.example.com/users'
      }
    ];
  }
};
module.exports = nextConfig;

The browser sees: 

/api/users

But the request actually goes to: 

https://api.example.com/users

Configure HTTP Headers

Example
const nextConfig = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-App',
            value: 'NextJS'
          }
        ]
      }
    ];
  }
};
module.exports = nextConfig;

Environment Variables

Example
const nextConfig = {
  env: {
    API_URL: 'https://api.example.com'
  }
};
module.exports = nextConfig;

Usage:

Component
console.log(
  process.env.API_URL
);

Custom Webpack Configuration

Example
const nextConfig = {
  webpack(config) {
    config.resolve.alias = {
      ...config.resolve.alias,
      '@': __dirname
    };
    return config;
  }
};
module.exports = nextConfig;

This allows: 

import Header from '@/components/Header';

Standalone Deployment Output

Example
const nextConfig = {
  output: 'standalone'
};
module.exports = nextConfig;

Useful for Docker deployments because Next.js generates a minimal production build.

Enable React Strict Mode

Example
const nextConfig = {
  reactStrictMode: true
};
module.exports = nextConfig;

Helps detect potential React issues during development.

P7-02How do you set environment variables in Next.js?

Environment variables allow us to store configuration values outside of our code.

They are commonly used for API URLs, database connections, secret keys, analytics IDs, and other environment-specific settings.

Why Use Environment Variables?

  • Keep sensitive values out of source code.
  • Use different values for development and production.
  • Avoid hardcoding configuration.
  • Improve security and maintainability.

Environment Files

Next.js automatically loads environment variables from special files.

Project Structure
my-app/
├── .env
├── .env.local
├── .env.development
├── .env.production
├── app/
└── package.json

Basic Example

.env.local
API_URL=https://api.example.com
DB_PASSWORD=my-secret-password

Using Environment Variables on the Server

Server Component
export default function Page() {
  console.log(
    process.env.API_URL
  );
  return <h1>Home</h1>;
}

Server Components can access all environment variables.

Client Components

Client Components cannot access private environment variables.

To expose a variable to the browser, it must start with: 

NEXT_PUBLIC_
.env.local
NEXT_PUBLIC_API_URL=https://api.example.com
Client Component
'use client';
export default function Home() {
  console.log(
    process.env.NEXT_PUBLIC_API_URL
  );
  return <h1>Home</h1>;
}

What Happens If NEXT_PUBLIC_ Is Missing?

.env.local
API_URL=https://api.example.com
Client Component
'use client';
console.log(
  process.env.API_URL
);

Output: 

undefined

Private variables are not exposed to the browser.

Using Environment Variables in Route Handlers

app/api/users/route.js
export async function GET() {
  const apiUrl =
    process.env.API_URL;
  return Response.json({
    apiUrl
  });
}

Using Environment Variables in Server Actions

Example
"use server";
export async function saveUser() {
  const secret =
    process.env.DB_PASSWORD;
}

Environment-Specific Files

File Purpose
.env Default variables
.env.local Local overrides
.env.development Development only
.env.production Production only

Loading Order

Priority
.env.local
      │
      ▼
.env.development
or
.env.production
      │
      ▼
.env

Files higher in the list override files below them.

What Happens Internally?

Flow
npm run dev
      │
      ▼
Read .env Files
      │
      ▼
Load Variables
      │
      ▼
Populate process.env
      │
      ▼
Application Uses Variables

Important Notes

Variables without the NEXT_PUBLIC_ prefix are available only on the server.
Variables with the NEXT_PUBLIC_ prefix are included in the browser bundle and can be viewed by users.
Whenever we change .env files, we must restart the development server because environment variables are loaded only during startup.

Server vs Client Access

Variable Server Client
API_URL ✅ ❌
NEXT_PUBLIC_API_URL ✅ ✅

Real-World Example

.env.local
DATABASE_URL=mongodb://localhost:27017/mydb
NEXT_PUBLIC_API_URL=https://api.example.com
NEXT_PUBLIC_APP_NAME=My App
Usage

  process.env.NEXT_PUBLIC_APP_NAME;
const apiUrl =
  process.env.NEXT_PUBLIC_API_URL;

Interview Tip

A common mistake is storing secrets in variables that begin with NEXT_PUBLIC_. Any variable with this prefix becomes part of the client-side JavaScript bundle and is visible to users.
One-Line Definition:
Environment variables in Next.js are configuration values loaded from .env files and accessed through process.env, with only NEXT_PUBLIC_ variables exposed to the browser.

P7-03What Is the Difference Between NEXT_PUBLIC_ and Non-Prefixed Env Variables?

Next.js exposes two kinds of environment variables: variables prefixed with NEXT_PUBLIC_, which are inlined into the browser JavaScript bundle at build time, and non-prefixed variables, which stay server-only and are never sent to the browser.

Simple Meaning:
Add NEXT_PUBLIC_ only when a value is safe to be public, like an analytics ID; leave the prefix off for anything secret, like a database password or API key.

Example .env.local

.env.local
DATABASE_URL=postgres://user:pass@host/db
NEXT_PUBLIC_API_URL=https://api.example.com
NEXT_PUBLIC_GA_ID=G-XXXXXXX

Where Each One Can Be Used

VariableServer code (getServerSideProps, API routes)Browser code (components)
DATABASE_URL✅ Available❌ Undefined
NEXT_PUBLIC_API_URL✅ Available✅ Available
pages/index.js
console.log(process.env.NEXT_PUBLIC_API_URL); // works in browser
console.log(process.env.DATABASE_URL); // undefined in browser
Never put secrets like API keys, tokens, or database credentials behind NEXT_PUBLIC_ — anything with that prefix is bundled into client-side JavaScript and visible to anyone who views the page source.

P7-04What is Turbopack and Why is it the Default Dev Bundler in Next.js 15?

Turbopack is a modern JavaScript bundler created by the team behind Next.js. It is written in Rust and is designed to provide significantly faster development performance than Webpack.

Turbopack is the default development bundler in Next.js 15 because it provides faster startup times, faster rebuilds, better caching, and a smoother developer experience.

What is a Bundler?

A bundler analyzes our application's files and dependencies, then combines everything into optimized bundles that can be loaded by the browser.

Bundler Process
React Components
       │
CSS Files
       │
Images
       │
Libraries
       │
       ▼
    Bundler
       │
       ▼
 Optimized Bundles
       │
       ▼
    Browser

Why Was Turbopack Created?

As applications grow larger, Webpack can become slower during development.

The Next.js team created Turbopack to solve problems such as:

  • Slow startup times
  • Slow rebuild times
  • Large project compilation delays
  • Memory consumption issues
  • Developer productivity bottlenecks

Webpack vs Turbopack

Feature Webpack Turbopack
Language JavaScript Rust
Development Speed Good Much Faster
Incremental Builds Supported Highly Optimized
Caching Good Advanced
Large Projects Can Become Slower Optimized for Large Projects
Default in Next.js 15 No Yes

     ▼

     ▼

How Development Works with Turbopack

Development Flow

     │
     ▼
Rebuild Only What Changed
     │
     ▼
Instant Update

Turbopack updates only the affected modules instead of rebuilding everything.

Incremental Compilation

One of Turbopack's biggest advantages is incremental compilation.

Suppose our project contains:

app/
├── page.js
├── dashboard/
│   ├── page.js
│   └── analytics.js
├── users/
│   └── page.js
└── settings/
    └── page.js

If we modify only:

Modified File
dashboard/analytics.js

Turbopack rebuilds only the affected dependency graph instead of reprocessing the entire application.

Why Rust?

Turbopack is written in Rust because Rust offers:

  • High performance
  • Memory safety
  • Low memory usage
  • Excellent concurrency support

This allows Turbopack to perform many tasks faster than JavaScript-based tooling.

Running Next.js with Turbopack

In modern Next.js versions, Turbopack is used automatically during development.

Start Development Server
npm run dev

Internally, Next.js uses Turbopack by default in development mode.

Explicit Turbopack Command

Using Turbopack Explicitly
next dev --turbopack

Why is Turbopack the Default Dev Bundler in Next.js 15?

The Next.js team made Turbopack the default development bundler because it improves developer productivity.

  • Faster startup times
  • Faster hot reloading
  • Faster rebuilds
  • Better scaling for large projects
  • Efficient caching
  • Lower waiting time during development

Important Note

Turbopack is primarily focused on improving the development experience. It does not automatically reduce our production bundle size. Its main goal is to make development faster.

Real-World Example

Imagine our application contains 5,000+ files.

  • Webpack may need more work when files change.
  • Turbopack updates only the necessary parts.
  • This results in a much faster feedback loop during development.

P7-05What is Fast Refresh and how does it improve DX?

Fast Refresh is a Next.js and React development feature that instantly updates the browser when code changes without performing a full page reload.

Fast Refresh improves Developer Experience (DX) by allowing us to see code changes immediately while preserving component state whenever possible.

What is DX?

DX stands for Developer Experience.

It refers to how easy, fast, and enjoyable it is for developers to build and maintain applications.

Before Fast Refresh

In traditional web development, every code change required a full browser refresh.

Old Workflow
Change Code
     │
     ▼
Rebuild
     │
     ▼
Reload Entire Page
     │
     ▼
Lose Current State
     │
     ▼
Test Again

With Fast Refresh

Modern Workflow
Change Code
     │
     ▼
Save File
     │
     ▼
Update Browser Instantly
     │
     ▼
Keep Current State
     │
     ▼
Continue Working

Example

Counter Component
import { useState } from 'react';
export default function Counter() {
  const [count, setCount] = useState(5);
  return (
    <button onClick={() => setCount(count + 1)>
      {count}
    </button>
  );
}

Suppose the button currently shows:

Current State
Count = 5

Now we modify the component:

Updated Component
import { useState } from 'react';
export default function Counter() {
  const [count, setCount] = useState(5);
  return (
    <button onClick={() => setCount(count + 1)>
      Counter: {count}
    </button>
  );
}

Fast Refresh updates the UI immediately:

Result
Counter: 5

Notice that the count state remains 5.

Without Fast Refresh

Full Reload
Count = 5
     │
     ▼
Save File
     │
     ▼
Reload Page
     │
     ▼
Count = 0

The component state would be lost.

What Changes Can Fast Refresh Handle?

Change Fast Refresh
Component JSX ✅
Styles ✅
Hooks Logic ✅
React Components ✅
State Preservation ✅ (Usually)

When Does Fast Refresh Trigger a Full Reload?

Sometimes React cannot safely preserve state.

Example
export default function Counter() {
  return <h1>Counter</h1>;
}

Changed to: 

export default function User() {
  return <h1>User</h1>;
}

Since the component identity changed, React may perform a full reload.

Fast Refresh in Next.js

Fast Refresh is enabled automatically in development mode.

Development Server
npm run dev

No additional configuration is required.

How It Works Internally

Flow
Edit File
    │
    ▼
Save File
    │
    ▼
Webpack / Turbopack Detect Change
    │
    ▼
Recompile Module
    │
    ▼
Send Update To Browser
    │
    ▼
Update UI Instantly

Fast Refresh vs Full Reload

Feature Fast Refresh Full Reload
Reload Entire Page ❌ ✅
Preserve State ✅ ❌
Faster Updates ✅ ❌
Better DX ✅ ❌

Benefits of Fast Refresh

  • Instant feedback.
  • Faster development.
  • Preserves component state.
  • Reduces manual testing steps.
  • Improves productivity.
  • Provides a better developer experience.

Important Note

Fast Refresh works only in development mode. It is not used in production builds.

Interview Tip

Fast Refresh is an enhanced version of Hot Module Replacement (HMR) that updates React components instantly while preserving state whenever possible.
One-Line Definition:
Fast Refresh is a React and Next.js development feature that instantly updates the browser after code changes while preserving component state, providing a faster and better developer experience (DX).

P7-07What is Automatic Code Splitting in Next.js?

Automatic Code Splitting is a Next.js optimization that automatically splits JavaScript into smaller chunks so users download only the code needed for the current page.

Instead of sending the entire application bundle to the browser, Next.js sends only the JavaScript required for the requested route.

Why Do We Need Code Splitting?

Imagine an application with many pages:

Project Structure
app
├── page.js
├── about
│   └── page.js
├── products
│   └── page.js
├── dashboard
│   └── page.js
└── settings
    └── page.js

If all page code were bundled together, every user would download JavaScript for every page, even pages they never visit.

Without Code Splitting

Single Bundle
Home Code
About Code
Products Code
Dashboard Code
Settings Code
      │
      ▼
One Large Bundle
      │
      ▼
Download Everything

Even users visiting only the Home page must download code for all pages.

With Automatic Code Splitting

Route-Based Splitting
Home Route
    │
    ▼
Download Home Chunk
About Route
    │
    ▼
Download About Chunk
Products Route
    │
    ▼
Download Products Chunk

Users download only the JavaScript needed for the current page.

Example

app/page.js
export default function Home() {
  return <h1>Home Page</h1>;
}
app/about/page.js
export default function About() {
  return <h1>About Page</h1>;
}

When a user visits: 

/

Next.js sends only the Home page JavaScript.

When a user visits: 

/about

Next.js sends only the About page JavaScript.

Build Output Concept

Generated Chunks
.next
└── static
    └── chunks
        ├── home.js
        ├── about.js
        ├── products.js
        └── dashboard.js

Each route gets its own JavaScript chunk.

How It Works

Flow
User Visits /about
        │
        ▼
Next.js Detects Route
        │
        ▼
Load About Chunk
        │
        ▼
Do Not Load
Home/Dashboard Chunks
        │
        ▼
Faster Page Load

Automatic Route Splitting

No configuration is required.

Example
app/page.js
app/about/page.js
app/products/page.js

Next.js automatically creates separate bundles for these routes.

Dynamic Imports for Additional Splitting

We can split large components even further using dynamic imports.

Example
import dynamic from 'next/dynamic';
const Chart = dynamic(() =>
  import('./Chart')
);
export default function Dashboard() {
  return <Chart />;
}

The Chart component is loaded only when needed.

Benefits of Automatic Code Splitting

  • Smaller JavaScript bundles.
  • Faster initial page load.
  • Better performance.
  • Reduced bandwidth usage.
  • Improved user experience.
  • No manual configuration required.

Automatic Code Splitting vs Traditional SPA

Feature Next.js Traditional SPA
Route-Based Splitting ✅ Automatic ⚠️ Manual Setup
Smaller Initial Bundle ✅ ❌ Often Large
Performance ✅ Better ⚠️ Depends
Configuration None Usually Required

Real-World Example

E-Commerce Website
Home Page
    │
    ▼
Download Home Code
Product Page
    │
    ▼
Download Product Code
Admin Dashboard
    │
    ▼
Download Dashboard Code

Customers visiting the Home page do not download Dashboard code.

Important Note

Automatic code splitting happens at the route level by default. Every page gets its own JavaScript bundle automatically.

Interview Tip

One of the biggest performance benefits of Next.js is automatic route-based code splitting, where users download only the code required for the page they visit.
One-Line Definition:
Automatic code splitting is a Next.js optimization that automatically creates separate JavaScript bundles for routes so users download only the code needed for the current page.

P7-08How do you add and configure ESLint in a Next.js project?

ESLint is used in a Next.js project to find code issues, enforce coding rules, and improve code quality during development.

Next.js supports ESLint out of the box, and we can configure it using an ESLint configuration file such as eslint.config.mjs.

What is ESLint?

ESLint is a linting tool that checks JavaScript, TypeScript, React, and Next.js code for mistakes and style issues.

Install ESLint Packages

Terminal
npm install -D eslint eslint-config-next

Create ESLint Config File

eslint.config.mjs
import next from 'eslint-config-next';
export default [
  ...next
];

Alternative Config for CommonJS Projects

.eslintrc.json
{
  "extends": "next/core-web-vitals"
}
In newer ESLint versions, eslint.config.mjs is the modern flat config style. Older projects commonly use .eslintrc.json.

Add Lint Script

package.json
