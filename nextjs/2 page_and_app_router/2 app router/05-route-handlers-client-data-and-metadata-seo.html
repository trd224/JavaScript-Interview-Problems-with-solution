Part 05 of 7 · 11 topics · Next.js App Router Guide

05-01Route Handlers (route.ts) and How They Replace API Routes

Route Handlers are the App Router's replacement for API Routes. They live in a route.ts (or route.js) file inside the app directory and let us define server-side endpoints using the standard Web Request and Response APIs instead of the older req, res style used in pages/api.

Simple Meaning:
A Route Handler is a server function tied to a URL segment, defined by exporting an HTTP-method-named function from a route.ts file.

Basic Route Handler

app/api/users/route.ts
export async function GET() {
  const users = await db.user.findMany();
  return Response.json(users);
}
export async function POST(request: Request) {
  const body = await request.json();
  const user = await db.user.create({ data: body });
  return Response.json(user, { status: 201 });
}

Each exported function name (GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS) maps directly to that HTTP method. A request that doesn't match any exported method automatically receives a 405 response.

Why Route Handlers Replace API Routes

pages/api (API Routes)app/.../route.ts (Route Handlers)
Single default export handling all methodsSeparate named export per HTTP method
Uses Node-style req, res objectsUses Web standard Request / Response
No built-in fetch-style cachingGET handlers can opt into caching like fetch
Lives under pages/apiLives anywhere inside app, colocated with routes

Dynamic Segments and Request Data

app/api/users/[id]/route.ts
export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;
  const user = await db.user.findUnique({ where: { id } });
  if (!user) return new Response("Not Found", { status: 404 });
  return Response.json(user);
}

Query parameters are read from the request's URL using new URL(request.url).searchParams, and cookies/headers are read with the cookies() and headers() functions from next/headers.

A route.ts file cannot coexist with a page.tsx file in the same route segment — a segment is either a UI page or an API endpoint, not both.

Caching Behavior

GET Route Handlers are static by default when they don't read dynamic request data (like cookies(), headers(), or search params) and can be revalidated with the same { next: { revalidate } }fetch. As soon as a Route Handler uses a dynamic API, it automatically switches to dynamic (uncached) rendering.

05-02How do you fetch data on the client side in a Next.js app?

Client-side data fetching means fetching data in the browser after the page has loaded.

In Next.js, we fetch data on the client side by using a Client Component with 'use client', then calling an API inside useEffect() or using a library like SWR or React Query.

When Do We Use Client-Side Fetching?

Client-side fetching is useful when data depends on browser interaction or user-specific state.

  • Dashboard data
  • User profile data
  • Search results
  • Filters
  • Pagination
  • Live updates
  • Data after button click

Basic Client-Side Fetching

In the App Router, we must use 'use client' because useEffect() runs only in the browser.

app/users/page.js
'use client';
import { useEffect, useState } from 'react';
export default function UsersPage() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    async function fetchUsers() {
      const res = await fetch('/api/users');
      const data = await res.json();
      setUsers(data);
      setLoading(false);
    }
    fetchUsers();
  }, []);
  if (loading) {
    return <h2>Loading...</h2>;
  }
  return (
    <div>
      <h1>Users</h1>
      {users.map(user => (
        <p key={user.id}>{user.name}</p>
      ))}
    </div>
  );
}

Fetching Data After Button Click

Sometimes we do not want to fetch data immediately. We may fetch data only after a user action.

Button Click Fetch
'use client';
import { useState } from 'react';
export default function ProductsPage() {
  const [products, setProducts] = useState([]);
  const loadProducts = async () => {
    const res = await fetch('/api/products');
    const data = await res.json();
    setProducts(data);
  };
  return (
    <div>
      <button onClick={loadProducts}>Load Products</button>
      {products.map(product => (
        <p key={product.id}>{product.name}</p>
      ))}
    </div>
  );
}

Using SWR

SWR is a popular data fetching library from Vercel.

It handles caching, loading states, revalidation, and refetching automatically.

Install SWR
npm install swr
SWR Example
'use client';
import useSWR from 'swr';
const fetcher = url => fetch(url).then(res => res.json());
export default function UsersPage() {
  const { data, error, isLoading } = useSWR('/api/users', fetcher);
  if (isLoading) {
    return <h2>Loading...</h2>;
  }
  if (error) {
    return <h2>Failed to load users</h2>;
  }
  return (
    <div>
      {data.map(user => (
        <p key={user.id}>{user.name}</p>
      ))}
    </div>
  );
}

When Should We Choose Client-Side Fetching?

  • When data is not important for SEO.
  • When data depends on user interaction.
  • When data is private or user-specific.
  • When we need live updates.
  • When fetching should happen after the page loads.

When Should We Avoid Client-Side Fetching?

  • SEO-focused pages
  • Blogs
  • Marketing pages
  • Product pages that need search engine indexing
  • Pages where content should be visible immediately in HTML

Important Note

Client-side fetching means users may initially see a loading state because the data is fetched after JavaScript runs in the browser.

Interview Tip

A common misconception is that all data fetching in Next.js should happen on the server. In reality, client-side fetching is still useful for interactive, user-specific, and frequently changing data.
One-Line Definition:
Client-side data fetching in Next.js means fetching data in the browser using a Client Component with useEffect(), fetch(), or a library like SWR, usually for interactive or user-specific data.

05-03What is SWR and how does it integrate with Next.js?

SWR is a React data fetching library created by Vercel that provides caching, automatic revalidation, request deduplication, polling, and real-time data synchronization.

SWR stands for Stale-While-Revalidate. It immediately shows cached data (stale data), then fetches fresh data in the background and updates the UI automatically.

Why Do We Need SWR?

Without SWR, we often write repetitive code for:

  • Loading states
  • Error handling
  • Caching
  • Refetching
  • Revalidation
Without SWR
'use client';
import { useEffect, useState } from 'react';
export default function Users() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    async function loadUsers() {
      const res = await fetch('/api/users');
      const data = await res.json();
      setUsers(data);
      setLoading(false);
    }
    loadUsers();
  }, []);
  return loading
    ? <h2>Loading...</h2>
    : <div>{users.length}</div>;
}

SWR handles most of this automatically.

Install SWR

Terminal
npm install swr

Basic SWR Example

Client Component
'use client';
import useSWR from 'swr';
const fetcher = (url) => fetch(url).then(res=>res.json());

    <div>
      {data.map(user=>(
        <p key={user.id}>
          {user.name}
        </p>

    </div>

}

Why Is SWR Fast?

Because cached data can be displayed instantly.

Users do not always need to wait for a new API request.

Automatic Revalidation

SWR automatically refetches data when:

  • User returns to the tab.
  • Network reconnects.
  • Component remounts.
Example
const {data}=useSWR(
  '/api/users',
  fetcher
);

These behaviors work automatically.

Polling Data

SWR can automatically refresh data at a specified interval.

Auto Refresh Every 5 Seconds
const {data}=useSWR(
  '/api/users',
  fetcher,
  {refreshInterval:5000}
);

Manual Revalidation

We can force SWR to refetch data.

Example
import useSWR,{mutate} from 'swr';
const refresh=()=>{
  mutate('/api/users');
};

How Does SWR Integrate With Next.js?

SWR works especially well with Next.js because both are maintained by Vercel.

  • Works in Client Components.
  • Works with API Routes.
  • Works with Route Handlers.
  • Works with Server Components + Client Components.
  • Supports Next.js caching strategies.

05-04What is the metadata export in the App Router?

The metadata export is a special App Router feature that allows us to define page metadata such as title, description, Open Graph tags, Twitter cards, icons, and other SEO-related information without using the Head component.

The metadata export automatically generates HTML head tags and is the recommended way to manage SEO metadata in the App Router.

Static Metadata Example

Use metadata when metadata is known in advance and does not depend on fetched data.

app/about/page.js
export const metadata = {
  title: 'About Us',
  description: 'Learn more about our company.'
};
export default function About() {
  return <h1>About Page</h1>;
}

Generated HTML

HTML Output
<head>
  <title>About Us</title>
  <meta name="description" content="Learn more about our company." />
</head>

Layout Metadata Example

Metadata can also be defined inside layout.js. This metadata is inherited by all child pages unless a page overrides it.

app/layout.js
export const metadata = {
  title: 'My Website',
  description: 'Welcome to my website'
};
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}

Do We Really Fetch Data Twice?

At first glance, it looks like getProduct() is called twice:

  • Once inside generateMetadata().
  • Once inside the page component.
Example

    title: product.title,
    description: product.description

This appears to call getProduct() twice, but Next.js automatically deduplicates identical fetch requests.

When Will It Fetch Twice?

If the requests are different, Next.js cannot deduplicate them.

Different Fetch Options
fetch(`/products/${id}`);
fetch(`/products/${id}`, {
  cache: 'no-store'
});
Different URLs
fetch(`/products/${id}`);
fetch(`/products/${id}?details=true`);
Deduplication works only when the requests are identical.

05-05How do you generate dynamic metadata using generateMetadata()?

The generateMetadata() function is used when metadata depends on dynamic data such as products, blog posts, user profiles, CMS content, or database records.

Unlike the static metadata export, generateMetadata() can fetch data and generate metadata dynamically for each page.

Why Use generateMetadata()?

Sometimes metadata cannot be known ahead of time because it depends on the current route or fetched data.

For example:

  • Product Detail Pages
  • Blog Detail Pages
  • User Profile Pages
  • CMS Content
  • Database Records

Complete Example

05-07What is metadataBase and why is it important for Open Graph URLs?

metadataBase is a Metadata API property that defines the base URL for all relative metadata URLs such as Open Graph images, canonical URLs, Twitter images, and alternates.

Without metadataBase, relative URLs like /og-image.jpg cannot be converted into complete URLs such as https://example.com/og-image.jpg.

Why Do We Need metadataBase?

Social media platforms and search engines require absolute URLs.

Invalid Open Graph URL
openGraph: {
  images: ['/og-image.jpg']
}

This only contains a relative path.

Result
/og-image.jpg

Facebook, LinkedIn, WhatsApp, and Twitter cannot reliably use this path because the domain is missing.

Using metadataBase

app/layout.js
export const metadata = {
  metadataBase: new URL('https://example.com'),
  openGraph: {
    images: ['/og-image.jpg']
  }
};
export default function RootLayout({
  children
}) {
  return (
    <html>
      <body>{children}</body>
    </html>
  );
}

Generated Open Graph URL

Result
https://example.com/og-image.jpg

05-08How do you generate a sitemap.xml using the App Router file convention?

In the App Router, we create a special sitemap.js (or sitemap.ts) file. Next.js automatically generates a sitemap.xml file from the data returned by this file.

The sitemap.js file is a built-in App Router file convention used to generate SEO-friendly sitemap.xml files.

What is a Sitemap?

A sitemap helps search engines discover and index pages on our website.

Common search engines that use sitemaps:

  • Google
  • Bing
  • Yahoo
  • DuckDuckGo

Basic Sitemap Example

app/sitemap.js
export default function sitemap() {
  return [
    {
      url: 'https://example.com',
      lastModified: new Date()
    },
    {
      url: 'https://example.com/about',
      lastModified: new Date()
    },
    {
      url: 'https://example.com/contact',
      lastModified: new Date()
    }
  ];
}

Generated sitemap.xml

<urlset>

  </url>
  <url>
    <loc>https://example.com/about</loc>
  </url>
  <url>
    <loc>https://example.com/contact</loc>

Dynamic Sitemap Example

For dynamic routes such as blog posts or products, we can fetch data and generate sitemap entries dynamically.

app/sitemap.js
async function getProducts() {
  const res = await fetch(
    'https://dummyjson.com/products'
  );
  const data = await res.json();
  return data.products;
}
export default async function sitemap() {
  const products = await getProducts();
  const productUrls = products.map(product => ({
    url: `https://example.com/products/${product.id}`,
    lastModified: new Date()
  }));
  return [
    {
      url: 'https://example.com',
      lastModified: new Date()
    },
    ...productUrls
  ];
}

05-09How do you configure robots.txt in Next.js?

In Next.js App Router, we configure robots.txt by creating a special robots.js file inside the app folder.

Next.js automatically converts the return value of app/robots.js into a valid robots.txt file.

What is robots.txt?

The robots.txt file tells search engine crawlers which pages or paths they are allowed or not allowed to crawl.

Example robots.txt
User-agent: *
Allow: /
Disallow: /admin
Disallow: /dashboard
Sitemap: https://example.com/sitemap.xml

Basic App Router Example

app/robots.js
export default function robots() {
  return {
    rules: {
      userAgent: '*',
      allow: '/',
      disallow: ['/admin', '/dashboard']
    },
    sitemap: 'https://example.com/sitemap.xml'
  };
}

Generated robots.txt

Output
User-agent: *
Allow: /
Disallow: /admin
Disallow: /dashboard
Sitemap: https://example.com/sitemap.xml

Project Structure

App Router File Convention

├── page.js

├── sitemap.js
├── robots.js
└── admin
    └── page.js

      userAgent: '*',

Block All Crawlers

export default function robots() {
  return {
    rules: {
      userAgent: '*',
      disallow: '/'
    }
  };

Block Private Routes

We usually block private or user-specific routes from crawling.

app/robots.js
export default function robots() {
  return {
    rules: {
      userAgent: '*',
      allow: '/',
      disallow: ['/admin', '/dashboard', '/profile', '/checkout']
    },
    sitemap: 'https://example.com/sitemap.xml'
  };
}

Multiple Rules Example

We can also define different rules for different crawlers.

app/robots.js
export default function robots() {
  return {
    rules: [
      {
        userAgent: 'Googlebot',
        allow: '/',
        disallow: ['/private']
      },
      {
        userAgent: '*',
        allow: '/',
        disallow: ['/admin']
      }
    ],
    sitemap: 'https://example.com/sitemap.xml'
  };
}

Using Environment Variable

In real projects, we should avoid hardcoding the domain and use an environment variable.

.env.local
NEXT_PUBLIC_SITE_URL=https://example.com

export default function robots() {
  const baseUrl = process.env.NEXT_PUBLIC_SITE_URL;
  return {
    rules: {
      userAgent: '*',
      allow: '/',
      disallow: ['/admin', '/dashboard']
    },
    sitemap: `${baseUrl}/sitemap.xml`
  };
}

Pages Router Alternative

In the Pages Router, we commonly place a static robots.txt file inside the public folder.

public/robots.txt
User-agent: *
Allow: /
Disallow: /admin
Disallow: /dashboard

robots.js vs public/robots.txt

Method Use Case
app/robots.js Best for App Router and dynamic configuration
public/robots.txt Simple static robots.txt file

Important Note

robots.txt controls crawling, not security. Do not use it to protect private data. Private pages must still be protected with authentication and authorization.

In Next.js, robots.txt is configured using app/robots.js in the App Router or public/robots.txt for a static file, and it tells search engines which routes they can or cannot crawl.
