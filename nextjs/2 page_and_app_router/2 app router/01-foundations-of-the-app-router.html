01-01What is the app Directory Introduced in Next.js 13?

The app directory is a new routing system introduced in Next.js 13 that uses the App Router.

The app directory is used to create routes, layouts, loading states, error pages, and Server Components using a folder-based structure.

Why Was the app Directory Introduced?

The older pages directory worked well, but it had limitations when building large applications.

The app directory was introduced to provide:

  • Server Components
  • Nested Layouts
  • Streaming
  • Loading UI
  • Error UI
  • Better Data Fetching
  • Improved Performance

Basic app Directory Structure

App Router Structure
app/
├── page.js
├── layout.js
├── about/
│   └── page.js
└── contact/
    └── page.js

  return (
    <h1>About Page</h1>
  );
}

layout.js

A layout is shared UI that appears across multiple pages.

export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        {children}
      </body>
    </html>
  );
}

The layout wraps every page inside the application.

Server Components

Components inside the app directory are Server Components by default.
Server Component Example

  const response = await fetch(
    'https://api.example.com/users'
  );
  const users = await response.json();
  return (
    <div>{users.length}</div>
  );

This code runs on the server, not in the browser.

01-02What is the Difference Between the Pages Router and the App Router?

The Pages Router and App Router are two routing systems in Next.js. The Pages Router uses the pages directory and traditional data-fetching methods, while the App Router uses the app directory and introduces Server Components, nested layouts, streaming, and the Metadata API.

The App Router is the modern routing system introduced in Next.js 13 and is now the recommended approach for new applications.

The Pages Router uses the pages folder to create routes.

01-03What are React Server Components (RSC) and how does Next.js leverage them?

React Server Components (RSC) are React components that execute on the server instead of the browser. They can fetch data, access backend resources, and generate UI without sending their JavaScript code to the client.

In the App Router, all components are React Server Components by default unless we add the 'use client' directive.

Why Were Server Components Introduced?

Traditional React applications send component JavaScript to the browser.

Traditional React
Server
   │
   ▼
Send HTML + JavaScript
   │
   ▼
Browser Downloads JS
   │
   ▼
Hydration
   │
   ▼
Interactive UI

Large applications can send a lot of JavaScript to the browser.

React Server Components reduce the amount of JavaScript sent to users.

Server Component Example

app/products/page.js
export default async function Page() {
  const products =
    await fetch(
      'https://dummyjson.com/products'
    ).then(res => res.json());
  return (
    <ul>
      {products.products.map(product => (
        <li key={product.id}>
          {product.title}
        </li>
      ))}
    </ul>
  );
}

This component runs on the server.

01-04Why Does Pages Router Hydrate Everything While App Router Doesn't?

The main difference is that the Pages Router was built around full-page hydration, while the App Router uses React Server Components to avoid sending unnecessary JavaScript to the browser.

Pages Router renders HTML on the server, but then still sends page JavaScript and React JavaScript to the browser so React can hydrate the whole page.

Step 1: What Happens in Pages Router?

In the Pages Router, even if the server already sends HTML, the browser still receives the page JavaScript and React JavaScript.

pages/index.js
export default function Page() {
  return <h1>Hello</h1>;
}

Pages Router Rendering Flow

So in the Pages Router, the component runs on the server first and then runs again in the browser during hydration.

Page() runs on Server  ✅

Why Does Pages Router Send page.js and react.js?

Because the Pages Router architecture assumes that every page is a React application that must be hydrated in the browser.

It does not matter whether the current page is interactive or not. Pages Router was designed to hydrate the whole page.

What Is Hydration?

Hydration means React attaches browser-side JavaScript behavior to the HTML that was already generated on the server.

Server HTML
<button>Click Me</button>

This HTML is visible, but it does not know anything about React state or click handlers.

React Component
export default function Button() {
  return (
    <button onClick={() => alert('Hello')}>
      Click Me
    </button>
  );
}

The browser needs JavaScript to attach the onClick behavior.

01-05What is the Difference Between Server Components and Client Components?

Server Components run only on the server and do not send their component JavaScript to the browser, while Client Components run in the browser and support interactivity such as state, effects, and event handlers.

In the App Router, components are Server Components by default. A component becomes a Client Component only when we add the 'use client' directive.

01-06How do you mark a component as a Client Component with 'use client'?

The 'use client' directive is used at the top of a file to mark a component as a Client Component in the Next.js App Router.

The primary purpose of 'use client' is to enable browser-side interactivity such as React hooks, event handlers, browser APIs, and client-side state management.

Why Is 'use client' Needed?

By default, all components in the App Router are Server Components.

Server Components cannot use:

  • useState()
  • useEffect()
  • useRef()
  • Event Handlers
  • Browser APIs
  • Client-side Libraries

When we need any of these features, we must mark the component as a Client Component.

Basic Example

Client Component
'use client';
export default function Counter() {
  return <h1>Client Component</h1>;
}

The directive must be placed at the top of the file before imports and other code.

Using useState()

Counter Example
'use client';
import { useState } from 'react';
export default function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button
      onClick={() => setCount(count + 1)}
    >
      Count: {count}
    </button>
  );
}

This component requires 'use client' because it uses useState() and a click event.

Using useEffect()

Effect Example
'use client';
import { useEffect } from 'react';
export default function Page() {
  useEffect(() => {
    console.log('Component Mounted');
  }, []);
  return <h1>Hello World</h1>;
}

The useEffect() hook runs in the browser, so it requires a Client Component.

Using Browser APIs

Window Example
'use client';
export default function Page() {
  const width = window.innerWidth;
  return <h1>Width: {width}</h1>;
}

Browser objects such as window, document, and localStorage are only available in Client Components.

Incorrect

}

The directive must always be the first statement in the file.

What Gets Sent To The Browser?

When a component is marked with 'use client', Next.js sends:

  • Rendered HTML
  • Component JavaScript
  • React Runtime Code
  • Hydration Instructions

This allows React to take control of the component after the page loads.

When Should We Use 'use client'?

  • Forms.
  • Buttons with click handlers.
  • Interactive UI.
  • State management.
  • Animations.
  • Browser APIs.
  • Third-party client-side libraries.

When Should We Avoid 'use client'?

  • Static content.
  • Database queries.
  • Server-side data fetching.
  • SEO-focused pages.
  • Components without interactivity.

Important Note

A common misconception is that Client Components render only in the browser. In reality, Next.js still generates their initial HTML on the server, then sends JavaScript to the browser for hydration and interactivity.

Interview Tip

Adding 'use client' does not disable server rendering. It simply tells Next.js to include the component's JavaScript in the browser bundle and enable client-side React features.
One-Line Definition:
The 'use client' directive marks a component as a Client Component, allowing us to use React hooks, event handlers, browser APIs, and other interactive features in the browser.
