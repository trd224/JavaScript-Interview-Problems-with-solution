07-01What is next/dynamic and what options (ssr, loading) does it accept?

next/dynamic is a Next.js utility that allows us to dynamically import components instead of including them in the initial JavaScript bundle.

Simple Meaning:
next/dynamic lets us load a component only when it is needed, reducing the initial bundle size and improving page performance.

Why Do We Need next/dynamic?

Normally, imported components become part of the page bundle.

Normal Import
import Chart from './Chart';
export default function Page() {
  return <Chart />;
}

Flow: 

Page Loads
      ↓
Chart Downloaded
      ↓
Page Rendered

Even if the user never views the chart, its JavaScript is still downloaded.

Using next/dynamic

Dynamic Import
import dynamic from 'next/dynamic';
const Chart = dynamic(() => import('./Chart'));
export default function Page() {
  return <Chart />;
}

Flow: 

Page Loads
      ↓
Chart Not Downloaded Yet
      ↓
Chart Needed
      ↓
Download Chart
      ↓
Render Chart

Basic Syntax

Syntax
const Component = dynamic(
  () => import('./Component'),
  options
);

Option 1: ssr

The ssr option controls whether the component should be rendered on the server.

Disable SSR
import dynamic from 'next/dynamic';
const Map = dynamic(() => import('./Map'), {
  ssr: false
});

Flow: 

Server Render
      ↓
Map Skipped
      ↓
Browser Loads
      ↓
Map Rendered
Use Case:
Browser-only libraries such as Google Maps, Leaflet, Chart.js, or components that access window, document, or localStorage.

Without ssr: false

Problem Example
const Map = dynamic(() => import('./Map'));

If the component contains: 

window.location.href

The server will throw an error because window does not exist on the server.

Option 2: loading

The loading option displays a fallback component while the dynamic component is being downloaded.

Loading Example
import dynamic from 'next/dynamic';
const Chart = dynamic(() => import('./Chart'), {
  loading: () => <p>Loading Chart...</p>
});

Flow: 

Page Loads
      ↓
Loading Chart...
      ↓
Chart Bundle Downloaded
      ↓
Chart Rendered

Using Both Options Together

Example
import dynamic from 'next/dynamic';
const Map = dynamic(() => import('./Map'), {
  ssr: false,
  loading: () => <p>Loading Map...</p>
});

Meaning:

Option Effect
ssr: false Do not render on server
loading Show fallback while downloading

Real World Example

Suppose our dashboard contains a heavy chart library.

Without Dynamic Import
Dashboard Page
      ↓
Download Chart Library
      ↓
Download Table
      ↓
Download Filters
      ↓
Render

The page becomes slower.

With Dynamic Import
Dashboard Page
      ↓
Render Immediately
      ↓
Chart Downloads Separately
      ↓
Chart Appears Later

The dashboard becomes faster because the chart code is split into a separate chunk.

Common Use Cases

Use Case Use next/dynamic?
Google Maps ✅ Yes
Chart.js ✅ Yes
Video Player ✅ Yes
Rich Text Editor ✅ Yes
Small Header Component ❌ Usually No

next/dynamic vs React.lazy

Feature next/dynamic React.lazy
Works In Next.js ✅ ✅
Supports SSR Control ✅ ❌
loading Option ✅ ❌ (Uses Suspense)
Next.js Optimized ✅ ❌

Easy Memory Trick

Remember
next/dynamic
      ↓
Load Later
ssr: false
      ↓
Client Only
loading
      ↓
Show Placeholder
One-Line Definition:
next/dynamic dynamically loads components to reduce initial bundle size, while ssr controls server rendering and loading provides a temporary fallback UI during download.

07-02How do you lazy-load a heavy Client Component to avoid hydration cost?

We lazy-load a heavy Client Component by using next/dynamic, so that the heavy component's JavaScript is not included in the initial page bundle.

Simple Meaning:
Load the heavy Client Component only when needed, instead of downloading and hydrating it immediately on page load.

Problem: Heavy Client Component

A Client Component needs JavaScript in the browser and must be hydrated.

Heavy Component
'use client';
export default function Chart() {
  return <div>Heavy Chart Component</div>;
}

If we import it normally, it becomes part of the initial JavaScript bundle.

Normal Import
import Chart from './Chart';
export default function Page() {
  return <Chart />;
}
Flow
Page Loads
      ↓
Chart JavaScript Downloads Immediately
      ↓
Chart Hydrates Immediately
      ↓
Initial Load Becomes Heavier

Solution: Lazy Load With next/dynamic

Use dynamic() from next/dynamic.

Lazy Loaded Component
import dynamic from 'next/dynamic';
const Chart = dynamic(() => import('./Chart'), {
  loading: () => <p>Loading chart...</p>
});
export default function Page() {
  return <Chart />;
}
Flow
Page Loads
      ↓
Main Page Hydrates
      ↓
Chart Bundle Loads Separately
      ↓
Chart Hydrates Later

Disable SSR For Browser-Only Heavy Component

If the component uses browser-only APIs like window, document, localStorage, maps, charts, or editors, use ssr: false.

Client Only Dynamic Import
import dynamic from 'next/dynamic';
const RichEditor = dynamic(() => import('./RichEditor'), {
  ssr: false,
  loading: () => <p>Loading editor...</p>
});
export default function Page() {
  return <RichEditor />;
}
Meaning:
ssr: false means this component will not render on the server. It will only load and render in the browser.

Real World Example: Dashboard Chart

Suppose our dashboard has normal static content plus one heavy chart.

HeavyChart.tsx
'use client';
import Chart from 'chart.js/auto';
export default function HeavyChart() {
  return <canvas id="sales-chart"></canvas>;
}
page.tsx
import dynamic from 'next/dynamic';
const HeavyChart = dynamic(() => import('./HeavyChart'), {
  ssr: false,
  loading: () => <p>Loading sales chart...</p>
});
export default function DashboardPage() {
  return (
    <main>
      <h1>Dashboard</h1>
      <p>This content loads fast.</p>
      <HeavyChart />
    </main>
  );
}

Now the dashboard content can load first, and the chart JavaScript loads separately.

Lazy Load Only After User Click

For maximum saving, we can load the heavy component only after user interaction.

ReportSection.tsx
'use client';
import { useState } from 'react';
import dynamic from 'next/dynamic';
const HeavyReport = dynamic(() => import('./HeavyReport'), {
  loading: () => <p>Loading report...</p>
});
export default function ReportSection() {
  const [showReport, setShowReport] = useState(false);
  return (
    <div>
      <button onClick={() => setShowReport(true)}>Show Report</button>
      {showReport && <HeavyReport />}
    </div>
  );
}
Flow
Page Loads
      ↓
HeavyReport Not Downloaded
      ↓
User Clicks Show Report
      ↓
HeavyReport Downloads
      ↓
HeavyReport Hydrates

When Should We Lazy Load?

Component Lazy Load?
Chart component ✅ Yes
Map component ✅ Yes
Rich text editor ✅ Yes
Video player ✅ Yes
Payment widget ✅ Yes
Small button ❌ Usually No
Header navigation ❌ Usually No

Important: Lazy Loading Reduces Initial JS

Lazy loading does not remove hydration completely. It delays hydration until the component is loaded and rendered.

Normal Import Dynamic Import
Component JS loads immediately Component JS loads later
Hydrates immediately Hydrates later
Higher initial cost Lower initial cost

Best Practice

Best Pattern
import dynamic from 'next/dynamic';
const HeavyClientComponent = dynamic(() => import('./HeavyClientComponent'), {
  ssr: false,
  loading: () => <p>Loading...</p>
});
export default function Page() {
  return <HeavyClientComponent />;
}
Important:
Use ssr: false only for components that do not need server rendering or depend on browser-only APIs.

Easy Memory Trick

Remember
Heavy Client Component
        ↓
dynamic()
        ↓
Separate JS Chunk
        ↓
Hydrate Later
        ↓
Faster Initial Load
One-Line Definition:
We lazy-load a heavy Client Component with next/dynamic so its JavaScript loads and hydrates later instead of increasing the initial page hydration cost.

07-03What strategies does the next/script component offer and when do you use each?

next/script allows us to control when third-party JavaScript files are loaded so they do not negatively impact page performance.

Simple Meaning:
The strategy prop decides when a script should be downloaded and executed.

Why Use next/script?

Normally, a script tag can block page rendering.

Traditional Script
<script src="https://example.com/script.js"></script>
Problem
Browser Reads HTML
        ↓
Downloads Script
        ↓
Executes Script
        ↓
Continues Rendering

Large scripts can slow down page loading.

next/script provides optimized loading strategies.

Basic Usage

Example
import Script from 'next/script';
export default function Page() {
  return (
    <>
      <Script
        src="https://example.com/script.js"
        strategy="afterInteractive"
      />
    </>
  );
}

Available Strategies

Strategy When It Runs
beforeInteractive Before Next.js hydration
afterInteractive After hydration begins
lazyOnload During browser idle time
worker Inside a web worker

1. beforeInteractive

Loads and executes before Next.js hydrates the page.

Example
<Script
  src="https://example.com/script.js"
  strategy="beforeInteractive"
/>
Flow
HTML Arrives
      ↓
Script Loads
      ↓
Script Executes
      ↓
Hydration Starts
Use For:
Critical scripts that must run before the application becomes interactive.

Examples:

  • Bot detection
  • Security scripts
  • Consent management
  • Critical monitoring scripts
Warning:
Use sparingly because it can delay page interactivity.

2. afterInteractive (Default)

This is the default strategy.

Example
<Script
  src="https://example.com/script.js"
  strategy="afterInteractive"
/>
Flow
HTML Loads
      ↓
Hydration Starts
      ↓
Script Downloads
      ↓
Script Executes
Use For:
Scripts needed soon after page interaction begins.

Examples:

  • Google Analytics
  • Tag Manager
  • Customer chat widgets
  • A/B testing tools

3. lazyOnload

Loads only when the browser is idle.

Example
<Script
  src="https://example.com/script.js"
  strategy="lazyOnload"
/>
Flow
Page Loads
      ↓
Hydration Completes
      ↓
Browser Becomes Idle
      ↓
Script Downloads
      ↓
Script Executes
Use For:
Low-priority scripts that are not immediately needed.

Examples:

  • Social sharing widgets
  • Heatmaps
  • Feedback widgets
  • Marketing tools

4. worker

Loads the script inside a Web Worker instead of the main thread.

Example
<Script
  src="https://example.com/script.js"
  strategy="worker"
/>
Flow
Script Loads
      ↓
Runs In Worker
      ↓
Main Thread Remains Free
Use For:
Heavy third-party scripts that support running in a worker environment.
Important:
Not all third-party scripts work inside a Web Worker.

Real World Examples

Script Recommended Strategy
Cookie Consent beforeInteractive
Google Analytics afterInteractive
Google Tag Manager afterInteractive
Hotjar lazyOnload
Feedback Widget lazyOnload
Heavy Analytics SDK worker (if supported)

Which Strategy Should We Choose?

Decision Flow
Needed Before Page Works?
           ↓
         Yes
           ↓
beforeInteractive
Needed Soon After Load?
           ↓
         Yes
           ↓
afterInteractive
Can Wait?
           ↓
         Yes
           ↓
lazyOnload
Heavy CPU Work?
           ↓
         Yes
           ↓
worker

Complete Example

Layout
import Script from 'next/script';
export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        {children}
        <Script
          src="https://www.googletagmanager.com/gtag/js"
          strategy="afterInteractive"
        />
        <Script
          src="https://example.com/feedback.js"
          strategy="lazyOnload"
        />
      </body>
    </html>
  );
}

Easy Memory Trick

Remember
beforeInteractive
       ↓
Before Hydration
afterInteractive
       ↓
After Hydration Starts
lazyOnload
       ↓
When Browser Is Idle
worker
       ↓
Run In Worker Thread
One-Line Definition:
The strategy prop of next/script controls when a script loads and executes, with beforeInteractive, afterInteractive, lazyOnload, and worker providing different performance and loading behaviors.

07-04How does prefetching work in the Link component and how did it change in v13?

Prefetching allows Next.js to download a route before the user clicks the link, making navigation much faster.

Simple Meaning:
When a link becomes visible on the screen, Next.js can quietly prepare the destination page in the background.

What Is Prefetching?

Without prefetching:

Normal Navigation
User Clicks Link
        ↓
Download Route Code
        ↓
Fetch Route Data
        ↓
Render New Page

There is a delay because everything starts after the click.

With prefetching:

Prefetch Navigation
Page Loads
      ↓
Link Becomes Visible
      ↓
Route Prefetched
      ↓
User Clicks Link
      ↓
Instant Navigation

Basic Link Example

Link
import Link from 'next/link';
export default function Page() {
  return (
    <Link href="/products">
      Products
    </Link>
  );
}

In production mode, Next.js automatically prefetches this route when the link enters the viewport.

How Prefetching Works

Next.js uses an Intersection Observer internally.

Flow
Link Appears In Viewport
          ↓
Intersection Observer Detects It
          ↓
Next.js Starts Prefetching
          ↓
Route Stored In Cache
          ↓
User Clicks Link
          ↓
Fast Navigation

Automatic Prefetching

By default:

Automatic Prefetch
<Link href="/products">
  Products
</Link>
Environment Prefetch
Development ❌ Disabled
Production ✅ Enabled

This prevents unnecessary network requests during development.

Disable Prefetching

Sometimes we do not want prefetching.

Disable
<Link href="/products" prefetch={false}>
  Products
</Link>

Now the route loads only after the user clicks.

What Changed In Next.js 13?

1. No More <a> Tag Required

Before v13:

Next.js 12
<Link href="/products">
  <a>Products</a>
</Link>

Starting with v13:

Next.js 13+
<Link href="/products">
  Products
</Link>

The Link component automatically renders the anchor element.

Version Syntax
v12 <Link><a>...</a></Link>
v13+ <Link>...</Link>

2. App Router Prefetching Became Smarter

In the App Router, Next.js prefetches React Server Component (RSC) payloads instead of just JavaScript bundles.

Pages Router
Prefetch JavaScript Bundle
          ↓
User Click
          ↓
Render Page
App Router
Prefetch Route Segments
          ↓
Prefetch RSC Payload
          ↓
User Click
          ↓
Faster Render
Main v13 Improvement:
The App Router can prefetch route segments and React Server Component payloads, making navigation more efficient.

Real World Example

Navigation Menu
import Link from 'next/link';
export default function Navigation() {
  return (
    <nav>
      <Link href="/">Home</Link>
      <Link href="/products">Products</Link>
      <Link href="/about">About</Link>
    </nav>
  );
}
Flow
Home Page Loads
        ↓
Links Become Visible
        ↓
Routes Prefetched
        ↓
User Clicks Products
        ↓
Products Page Opens Quickly

When Should We Disable Prefetch?

Scenario Disable Prefetch?
Main Navigation ❌ No
Frequently Visited Pages ❌ No
Thousands Of Links ✅ Sometimes
Rarely Used Admin Pages ✅ Sometimes
Huge Dynamic Routes ✅ Sometimes

Link vs Traditional Anchor

Feature Link <a>
Client Navigation ✅ ❌
Prefetching ✅ ❌
SPA Experience ✅ ❌
Full Page Reload ❌ ✅

Important Interview Point

Important:
Prefetching works only in production builds. During development (next dev), automatic prefetching is disabled.

Easy Memory Trick

Remember
Link Visible
      ↓
Prefetch Route
      ↓
User Clicks
      ↓
Fast Navigation
v12
 ↓
<Link><a></a></Link>
v13+
 ↓

One-Line Definition:
The Link component automatically prefetches visible routes in production, and in Next.js 13 it became smarter by prefetching App Router data and no longer requiring a nested <a> element.

07-05What are Core Web Vitals and which Next.js features directly improve each?

Core Web Vitals are Google's user experience metrics that measure loading speed, responsiveness, and visual stability.

Simple Meaning:
Core Web Vitals tell us how fast, responsive, and stable our application feels to real users.

The Three Core Web Vitals

Metric Measures Good Score
LCP Loading Performance ≤ 2.5 seconds
INP User Interaction Responsiveness ≤ 200 ms
CLS Visual Stability ≤ 0.1

1. LCP (Largest Contentful Paint)

LCP measures how quickly the largest visible content appears on the screen.

Example
User Opens Page
       ↓
Hero Image Appears
       ↓
LCP Recorded

Usually the LCP element is:

  • Hero image
  • Banner image
  • Large heading
  • Featured product image

Next.js Features That Improve LCP

Feature How It Helps
next/image Image optimization
priority Preloads LCP image
SSG HTML already generated
ISR Cached HTML served quickly
Server Components Less JavaScript sent
next/font Eliminates font delays
LCP Example
import Image from 'next/image';
export default function Home() {
  return (
    <Image
      src="/hero.jpg"
      alt="Hero"
      width={1200}
      height={600}
      priority
    />
  );
}
Why?
The priority prop preloads the hero image, helping the largest visible content appear faster.

2. INP (Interaction to Next Paint)

INP measures how quickly the page responds after a user interaction.

Example
User Clicks Button
        ↓
Browser Processes Event
        ↓
UI Updates
        ↓
INP Recorded

Slow JavaScript usually causes poor INP.

Next.js Features That Improve INP

  return <Chart />;

Heavy chart code is downloaded later, reducing the amount of JavaScript that blocks interaction.

Bad Experience

      ↓
Button Appears
      ↓

Next.js Features That Improve CLS

      src="/product.jpg"
      alt="Product"

Because dimensions are known before loading, space is reserved and no layout shift occurs.

07-06How do you analyze and reduce bundle size in a Next.js app?

Bundle size means the amount of JavaScript sent to the browser. In Next.js, we analyze bundle size using bundle analyzer tools and reduce it by sending less client-side JavaScript.

Simple Meaning:
Smaller bundle size means the browser downloads, parses, and hydrates less JavaScript, so our application becomes faster.

Why Bundle Size Matters

Large Bundle Problem
User Opens Page
      ↓
Browser Downloads Large JS
      ↓
Browser Parses JS
      ↓
React Hydrates Page
      ↓
Page Becomes Interactive Slowly

Large JavaScript mainly hurts interaction performance, especially on slow devices.

1. Analyze Bundle Size

Use @next/bundle-analyzer to see which packages are increasing the bundle.

Install
npm install @next/bundle-analyzer
next.config.js
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true'
});
const nextConfig = {};
module.exports = withBundleAnalyzer(nextConfig);
Run Analyzer
ANALYZE=true npm run build

On Windows PowerShell:

Windows PowerShell
$env:ANALYZE="true"; npm run build

2. Check Build Output

Next.js build output also shows route sizes.

Command
npm run build
Example Output
Route                          Size      First Load JS
┌ ○ /                          5 kB      95 kB
├ ○ /about                     2 kB      92 kB
└ ○ /dashboard                 50 kB     180 kB
Column Meaning
Size JavaScript specific to that route
First Load JS Total JS needed when opening that route first time

3. Prefer Server Components

In the App Router, components are Server Components by default. Server Components do not send their JavaScript to the browser.

Good
export default async function ProductsPage() {
  const products = await getProducts();
  return (
    <ul>
      {products.map(product => <li key={product.id}>{product.name}</li>)}
    </ul>
  );
}
Why?
Because this component runs on the server, its JavaScript is not added to the client bundle.

4. Do Not Add 'use client' Everywhere

Adding 'use client' makes that component and its imports part of the browser bundle.

Bad
'use client';
import HeavyChart from './HeavyChart';
import ProductList from './ProductList';
export default function Page() {
  return <ProductList />;
}

Now unnecessary code may go to the browser.

Better
import ProductList from './ProductList';
import AddToCartButton from './AddToCartButton';
export default function Page() {
  return (
    <>
      <ProductList />
      <AddToCartButton />
    </>
  );
}

Only AddToCartButton should be a Client Component if it needs browser interaction.

5. Lazy Load Heavy Client Components

Use next/dynamic for heavy browser components like charts, maps, editors, and video players.

Dynamic Import
import dynamic from 'next/dynamic';
const Chart = dynamic(() => import('./Chart'), {
  ssr: false,
  loading: () => <p>Loading chart...</p>
});
export default function Page() {
  return <Chart />;
}
Flow
Initial Page Load
        ↓
Chart JS Not Included Immediately
        ↓
Chart Loads Separately
        ↓
Initial Bundle Smaller

Main Difference Between Lazy Loading and Suspense

Many developers get confused between lazy loading and Suspense, but both have different responsibilities.

Suspense

Suspense is used to show a fallback UI while a component or data is not ready.

Suspense Example
<Suspense fallback={<Loading />}>
  <Chart />
</Suspense>
Meaning:
If Chart is not ready yet, show Loading.

Suspense does not decide whether the component should be server-rendered or client-rendered. It only controls what the user sees while waiting.

Lazy Loading

Lazy loading means the component code is not downloaded immediately. It is downloaded only when the component is needed.

React Lazy Example
import { lazy, Suspense } from 'react';
const Chart = lazy(() => import('./Chart'));
export default function Page() {
  return (
    <Suspense fallback={<p>Loading...</p>}>
      <Chart />
    </Suspense>
  );
}
Meaning:
lazy() delays the download of the component, and Suspense shows the loading UI while waiting.
Next.js dynamic()

In Next.js, dynamic() is commonly used instead of React.lazy() because it supports lazy loading, loading UI, and optional SSR control.

Next.js dynamic Example
import dynamic from 'next/dynamic';
const Chart = dynamic(() => import('./Chart'), {
  loading: () => <p>Loading...</p>
});
export default function Page() {
  return <Chart />;
}
Feature Suspense lazy() dynamic()
Shows loading UI ✅ Yes ❌ No ✅ Yes
Delays component download ❌ No ✅ Yes ✅ Yes
Controls SSR ❌ No ❌ No ✅ Yes, using ssr:false
Next.js optimized ❌ No ❌ No ✅ Yes
Easy Memory Trick
Suspense
    ↓
Shows loading UI
lazy()
    ↓
Delays component download
dynamic()
    ↓
Lazy loading
+ Loading UI
+ Optional SSR control
One-Line Interview Answer:
lazy() or dynamic() decides when the component code is downloaded, while Suspense decides what the user sees while waiting.

6. Import Only What We Need

Avoid importing a full library when only one function is needed.

Bad
import _ from 'lodash';
const name = _.capitalize('rizwan');
Better
import capitalize from 'lodash/capitalize';
const name = capitalize('rizwan');

Sometimes a small native JavaScript solution can replace a heavy package.

Heavy Usage Better Option

07-10How do you implement granular Suspense boundaries to avoid render waterfalls?

Granular Suspense boundaries mean splitting a page into multiple smaller <Suspense> sections, so slow parts can load independently without blocking the whole page.

Simple Meaning:
Do not wrap the whole page in one Suspense boundary. Wrap each slow section separately so fast sections can render first.

What Is A Render Waterfall?

A render waterfall happens when one async part waits for another async part unnecessarily.

Waterfall Flow
Fetch User
    ↓
Then Fetch Orders
    ↓
Then Fetch Analytics
    ↓
Then Render Page

This makes the page slower because work happens one after another.

Bad Pattern: One Big Suspense Boundary

If we wrap the whole dashboard in one Suspense boundary, the complete page waits for the slowest section.

Bad Example
import { Suspense } from 'react';
import Profile from './Profile';
import Orders from './Orders';
import Analytics from './Analytics';
export default function Page() {
  return (
    <Suspense fallback={<p>Loading dashboard...</p>}>
      <Profile />
      <Orders />
      <Analytics />
    </Suspense>
  );
}
Problem
Profile Ready Fast
Orders Ready Medium
Analytics Ready Slow
        ↓
Whole Page Waits For Analytics

Good Pattern: Granular Suspense Boundaries

Instead, wrap each independent async section in its own Suspense boundary.

Good Example
import { Suspense } from 'react';
import Profile from './Profile';
import Orders from './Orders';
import Analytics from './Analytics';
export default function Page() {
  return (
    <main>
      <Suspense fallback={<p>Loading profile...</p>}>
        <Profile />
      </Suspense>
      <Suspense fallback={<p>Loading orders...</p>}>
        <Orders />
      </Suspense>
      <Suspense fallback={<p>Loading analytics...</p>}>
        <Analytics />
      </Suspense>
    </main>
  );
}
Result
Profile Ready
    ↓
Show Profile Immediately
Orders Ready
    ↓
Show Orders
Analytics Ready Later
    ↓
Show Analytics Later

Example Async Components

Profile.tsx
export default async function Profile() {
  const profile = await getProfile();
  return <section>Welcome {profile.name}</section>;
}
Orders.tsx
export default async function Orders() {
  const orders = await getOrders();
  return <section>Orders: {orders.length}</section>;
}
Analytics.tsx
export default async function Analytics() {
  const analytics = await getAnalytics();
  return <section>Revenue: {analytics.revenue}</section>;
}

Each component can fetch its own data independently.

Parallel Data Fetching

If data belongs to the same component, start requests together instead of awaiting them one by one.

Bad: Sequential Fetching
export default async function Dashboard() {
  const user = await getUser();
  const orders = await getOrders();
  const analytics = await getAnalytics();
  return <div>Dashboard</div>;
}
Good: Parallel Fetching
export default async function Dashboard() {
  const [user, orders, analytics] = await Promise.all([getUser(), getOrders(), getAnalytics()]);
  return <div>Dashboard</div>;
}
Important:
Suspense helps streaming UI, but we should also avoid sequential await chains when data can be fetched in parallel.

Granular Suspense In Real Dashboard

Dashboard Page

  return (
    <main>
      <h1>Dashboard</h1>

The heading renders immediately, and each section streams when ready.

07-11How do you share state between Server and Client Components?

Server Components and Client Components do not share live React state directly. Instead, we pass server data to Client Components through props, and Client Components manage interactive state in the browser.

Simple Meaning:
Server Components can send initial data to Client Components, but live state like useState() belongs only inside Client Components.

1. Pass Server Data To Client Component Using Props

import Counter from './Counter';

  const initialCount = await getCartCount();
  return <Counter initialCount={initialCount} />;
}

'use client';
import { useState } from 'react';

  return (
    <button onClick={() => setCount(count + 1)}>
      Cart: {count}
    </button>
  );
}

If the user changes data, call a Server Action and then update UI.

07-12Why can't you use React Context directly in a Server Component?

React Context is designed for React trees running in the browser. Server Components run on the server, do not maintain interactive state, and cannot consume Client Context values directly.

Simple Meaning:
Server Components render once on the server and disappear. React Context depends on a live React tree, so it only works in Client Components.

How React Context Normally Works

In a traditional React application, Context shares state between components running in the browser.

Client Side Flow
ThemeProvider
       ↓
ThemeContext
       ↓
Component A
Component B
Component C

All components exist inside the browser React tree and can access the same Context value.

Why It Fails In Server Components

Server Components run on the server and generate HTML/RSC payloads.

Server Component Flow
Request Arrives
       ↓
Server Component Runs
       ↓
HTML/RSC Generated
       ↓
Response Sent
       ↓
Server Component Ends

There is no persistent React state living on the server between renders.

Important:
React Context relies on a live React state tree. Server Components do not have a live browser state tree.

Example That Does Not Work

ThemeProvider.tsx
'use client';
import { createContext } from 'react';
export const ThemeContext = createContext('dark');

What Can Server Components Use Instead?

Server Components should use:

  • Props
  • Database queries
  • API requests
  • cookies()
  • headers()
  • searchParams
Server Component Example
import { cookies } from 'next/headers';
export default async function Page() {
  const cookieStore = await cookies();
  const theme = cookieStore.get('theme')?.value;
  return <h1>Theme: {theme}</h1>;
}

Instead of Context, the Server Component gets data directly from the server.

Correct Pattern

Read data in the Server Component and pass it to a Client Component as props.

Server Component
import ThemeSwitcher from './ThemeSwitcher';
export default async function Page() {
  const theme = 'dark';
  return <ThemeSwitcher initialTheme={theme} />;
}
Client Component
'use client';
import { useState } from 'react';
export default function ThemeSwitcher({ initialTheme }) {
  const [theme, setTheme] = useState(initialTheme);
  return (
    <button onClick={() => setTheme('light')}>
      {theme}
    </button>
  );
}

07-13Zustand Integration With the App Router (Per-Request Store Pattern)

Zustand is a small, unopinionated client-state library. Because App Router components are Server Components by default, Zustand stores can only be created and used inside Client Components — but with a twist: on the server, a shared module-level store would leak state between different users' requests, so the App Router needs a per-request store pattern instead of Zustand's usual singleton store.

Simple Meaning:
Instead of one global store shared by everyone, each request/user gets its own store instance, created fresh and handed down through React Context.

The Problem With a Global Store

Traditional SPA:  one store, one browser tab, one user
Next.js Server:    one Node process, many concurrent users
                    -> a module-level store would mix user A's
                       and user B's state together

Step 1: Define a Store Factory

stores/counter-store.ts
import { createStore } from 'zustand/vanilla';
export type CounterState = { count: number };
export type CounterActions = { increment: () => void };
export type CounterStore = CounterState & CounterActions;
export const createCounterStore = (init: CounterState = { count: 0 }) => {
  return createStore()((set) => ({
    ...init,
    increment: () => set((s) => ({ count: s.count + 1 })),
  }));
};

Step 2: Provide It Once per Request via a Client Component

providers/counter-provider.tsx
'use client';
import { createContext, useRef, useContext } from 'react';
import { useStore } from 'zustand';
import { createCounterStore, CounterStore } from '@/stores/counter-store';
const CounterContext = createContext | null>(null);
export function CounterProvider({ children }: { children: React.ReactNode }) {
  const storeRef = useRef>();
  if (!storeRef.current) storeRef.current = createCounterStore();
  return (
    
      {children}
    
  );
}
export function useCounterStore(selector: (state: CounterStore) => T): T {
  const store = useContext(CounterContext);
  if (!store) throw new Error('Missing CounterProvider');
  return useStore(store, selector);
}

Step 3: Wrap the Segment in a Server Component

app/layout.tsx
import { CounterProvider } from '@/providers/counter-provider';
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    
      
        {children}
      
    
  );
}

The RootLayout stays a Server Component; only the provider that owns the store is a Client Component. Because useRef creates the store during that component's first render, every request gets a brand-new store instance instead of sharing one across users.

Never create the store at module scope (const store = createCounterStore() outside a component) in App Router code — that single instance would be reused across every request on the server, leaking state between users.
