02-01What is File-Based Routing in Next.js?

File-Based Routing is a routing system in Next.js where routes are automatically created based on the file and folder structure of your project.

In Next.js, you do not manually configure routes like React Router. Instead, creating a file automatically creates a route.

Why is it Called File-Based Routing?

Because URLs are generated directly from files and folders.

For example, if you create:

Pages Router Example
pages/
├── index.js
├── about.js
└── contact.js

Next.js automatically creates:

File Route
pages/index.js /
pages/about.js /about
pages/contact.js /contact

No route configuration is needed.

How Routing Works in React

In a plain React application, you usually need React Router.

React Router Example
import { BrowserRouter, Routes, Route } from 'react-router-dom';
function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
      </Routes>
    </BrowserRouter>
  );
}

Routes must be configured manually.

How Routing Works in Next.js

Simply create files and Next.js automatically creates routes.

  • Cleaner project structure.
  • Easier to understand.
  • Automatic nested routing.
  • Better developer experience.
  • Less boilerplate code.

02-02What are layouts in the App Router and how do they persist across navigations?

Layouts are special components in the Next.js App Router that wrap pages and other layouts. They allow us to share UI such as headers, sidebars, navigation menus, and footers across multiple routes.

The primary purpose of a layout is to provide a shared UI structure that remains mounted while navigating between pages inside the same layout segment.

Why Do We Need Layouts?

Many applications have UI elements that appear on every page.

  • Header
  • Navigation Menu
  • Sidebar
  • Footer
  • Dashboard Shell

Instead of repeating this code in every page, we place it inside a layout.

Basic Layout Example

app/layout.tsx
export default function RootLayout({
  children
}) {
  return (
    <html>
      <body>
        <Header />
        {children}
        <Footer />
      </body>
    </html>
  );
}

The {children} prop represents the current page being rendered.

Page Example

app/page.tsx
export default function Home() {
  return <h1>Home Page</h1>;
}

Rendered Output

<Header />
<h1>Home Page</h1>
<Footer />

├── page.tsx
│
├── dashboard/
│   ├── layout.tsx
│   ├── page.tsx
│   └── settings/
│       └── page.tsx

Each route segment can have its own layout.

Nested Layout Example

app/dashboard/layout.tsx
export default function DashboardLayout({
  children
}) {
  return (
    <>
      <Sidebar />
      {children}
    </>
  );
}

All dashboard pages automatically receive the sidebar.

How Do Layouts Persist Across Navigations?

When navigating between pages that share the same layout, Next.js does not unmount and recreate that layout.

      ▼
/dashboard/settings

Both routes use the same dashboard layout.

When Should We Use Layouts?

  • Headers.
  • Footers.
  • Navigation Bars.
  • Sidebars.
  • Dashboard Shells.
  • Shared Route UI.

Important Note

Layouts persist only when navigating between routes that share the same layout segment. Navigating to a route with a different layout causes the old layout to unmount and the new layout to mount.

Interview Tip

A common misconception is that layouts re-render on every navigation. In reality, shared layouts remain mounted and only the page content inside the layout is replaced.

02-03What is the root layout and why is it required in the App Router?

The Root Layout is the top-level layout of a Next.js App Router application. It is defined in app/layout.tsx and wraps every page and layout in the application.

The primary purpose of the Root Layout is to provide the application's global HTML structure and shared UI that should exist across the entire application.

Why Is the Root Layout Required?

Every App Router application must have exactly one Root Layout because Next.js needs a top-level component to define:

  • HTML Structure
  • Body Element
  • Global UI
  • Shared Providers
  • Fonts
  • Application Shell

Without a Root Layout, Next.js would not know how to construct the application's HTML document.

Basic Root Layout Example

app/layout.tsx
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>
        {children}
      </body>
    </html>
  );
}

This is the minimum required Root Layout.

Why Must It Contain html and body?

The Root Layout is responsible for generating the application's document structure.

Required Structure
<html lang="en">
  <body>
    Application Content
  </body>
</html>

Unlike nested layouts, the Root Layout must contain both the html and body tags.

Where Is the head Tag?

The Root Layout must contain the html and body tags, but you might notice that it does not contain a head tag.

Root Layout
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}

In the App Router, Next.js automatically generates the head element. We usually use the Metadata API instead of manually creating a head tag.

Metadata API
export const metadata = {
  title: 'My App',
  description: 'Learning Next.js'
};

Next.js converts this metadata into HTML inside the head element.

Generated HTML
<html>
  <head>
    <title>My App</title>
  </head>
  <body>...</body>
</html>

Although uncommon, a head tag can also be added manually.

Manual head Tag
export default function RootLayout({ children }) {
  return (
    <html>
      <head>
        <meta name="author" content="Rizwan" />
      </head>
      <body>{children}</body>
    </html>
  );
}
The Root Layout must define the html and body tags. The head tag is usually generated automatically by Next.js from the Metadata API.

02-04What is a template.js file and how is it different from a layout?

A template.js file is a special file in the Next.js App Router that works like a wrapper around a page, but unlike a layout, it is recreated on every navigation.

In simple words, layout.js keeps the same wrapper during navigation, but template.js creates a fresh wrapper every time the route changes.

Basic Difference

Layout Template
Persists across navigation Remounts on navigation
Preserves state Resets state
Good for Header, Footer, Sidebar Good for animations, reset state, re-running effects
Same component instance is reused New component instance is created

Main Confusion: Page Already Remounts, Then Why Need Template?

Yes, the page already remounts during navigation.

Page Behavior
Page
  → Always recreated on navigation

If page already remounts, why do we need template.js?

The answer is:

Answer
Page remounts itself.
Template remounts the wrapper around the page.

So template.js is useful when we want something around the page to also restart.

02-05How does loading.js work with React Suspense?

The loading.js file is a special App Router file that automatically creates a React Suspense boundary around a route segment and displays a loading UI while that segment is being rendered.

In simple words, loading.js is Next.js's automatic Suspense fallback for a route.

Why Do We Need loading.js?

Sometimes a page takes time to render because it is waiting for:

  • Database Queries
  • API Requests
  • Server Components
  • Dynamic Data

Instead of showing a blank screen, we can show a loading UI.

Basic Example

Folder Structure
app/
│
├── dashboard/
│   ├── page.tsx
│   └── loading.tsx
loading.tsx
export default function Loading() {
  return <h2>Loading Dashboard...</h2>;
}
page.tsx
export default async function Page() {
  await new Promise(resolve =>
    setTimeout(resolve, 3000)
  );
  return <h1>Dashboard</h1>;
}

        ▼

How Does Suspense Fit In?

Internally, Next.js automatically wraps the route in a Suspense boundary.

Conceptual Implementation
<Suspense fallback={<Loading />}>
  <Page />
</Suspense>

Next.js creates this Suspense boundary automatically.

You do not manually write Suspense when using loading.js. Next.js creates it for you.

Dynamic routes are created using square brackets []. The value inside the URL is captured as a route parameter.

Creating a separate file for every product would be impossible:

Why Do Both Return the Same Value?

Both examples return 100 because Next.js stores both dynamic route parameters and query string parameters inside router.query.

URL Source of id Value
/products/100 Dynamic Route Parameter ([id]) 100
/products?id=100 Query String Parameter (?id=100) 100

Although both are accessed using router.query.id, they come from different parts of the URL.

Dynamic Route Parameter
/products/100
Query String Parameter
/products?id=100

Next.js combines route parameters and query string parameters into a single router.query object.

Example
/products/100?color=red
router.query
{
  id: '100',
  color: 'red'
}
Dynamic route parameters come from the URL path, while query parameters come from the query string. Both are available through router.query in the Pages Router.

What Happens If Both Use the Same Name?

Consider the following URL:

URL
/products/100?id=100

In this case, the URL contains:

  • A route parameter from [id].
  • A query string parameter from ?id=100.

Both use the same key name (id), so they are merged into the same router.query object.

Read Values
const router = useRouter();
console.log(router.query);
Output
{
  id: '100'
}

A more obvious example:

URL
/products/100?id=200
Output
{
  id: '200'
}

The query string value overrides the route parameter when both use the same key.

Recommended Approach

Use different names for route parameters and query parameters.

URL
/products/100?productId=200
router.query
{
  id: '100',
  productId: '200'
}
If a route parameter and a query parameter have the same name, the query parameter value takes precedence in router.query.

02-17How do you Create a Custom 404 Page in Next.js?

A custom 404 page allows us to show our own page when a user visits a route that does not exist.

In the Pages Router, we create pages/404.js. In the App Router, we create app/not-found.js.

Pages Router: Custom 404 Page

In the Pages Router, create a file named 404.js inside the pages directory.

Folder Structure
pages/
├── index.js
├── about.js
└── 404.js
pages/404.js
import Link from 'next/link';
export default function Custom404() {
  return (
    <div>
      <h1>404 - Page Not Found</h1>
      <p>The page we are looking for does not exist.</p>
      <Link href="/">Go Home</Link>
    </div>
  );
}

App Router: Custom 404 Page

In the App Router, create a file named not-found.js inside the app directory.

Folder Structure
app/
├── page.js
├── layout.js
└── not-found.js
app/not-found.js
import Link from 'next/link';
export default function NotFound() {
  return (
    <div>
      <h1>404 - Page Not Found</h1>
      <p>The page we are looking for does not exist.</p>
      <Link href="/">Go Home</Link>
    </div>
  );
}

Show 404 Manually in App Router

If data is not found, we can manually show the 404 page using notFound().

app/products/[id]/page.js
import { notFound } from 'next/navigation';
export default async function Product({ params }) {
  const product = null;
  if (!product) {
    notFound();
  }
  return <h1>Product Details</h1>;
}

02-18What is Shallow Routing and When Would We Use It?

Shallow Routing is a feature in Next.js that allows us to change the URL without re-running data fetching methods or fully reloading the current page.

Shallow Routing updates the URL and router state while keeping the current page component mounted and avoiding unnecessary data fetching.

Why Do We Need Shallow Routing?

Suppose our application displays products with pagination, sorting, or filters.

When a user changes a filter or page number, we may want to update the URL so it can be bookmarked, refreshed, or shared with others.

Without Shallow Routing, changing the URL may trigger data fetching methods again.

Note:
Shallow Routing is most useful when our application data has already been loaded and we only want to update the URL or UI state. In such cases, re-fetching the same data from the server would be unnecessary.

Current URL
/products?page=1

After clicking the next page:

Updated URL
/products?page=2

We may only want the URL to change without causing a complete page refresh cycle.

Basic Example

Pages Router
import { useRouter } from 'next/router';
export default function Products() {
  const router = useRouter();
  const nextPage = () => {
    router.push('/products?page=2', undefined, { shallow: true });  //router.push(url, as, options);
  };
  return (
    <button onClick={nextPage}>
      Next Page
    </button>
  );
}
