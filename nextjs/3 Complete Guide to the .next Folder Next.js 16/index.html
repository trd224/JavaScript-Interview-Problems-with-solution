NEXT.JS BUILD SYSTEM

The Complete Guide to the .next Folder

Everything Next.js generates when you run next dev or next build — what each file does, how the pieces fit together, and how to work with it safely.

01Overview

The .next folder is the build output directory that Next.js generates automatically. It's created the moment you run next dev for local development, or next build to produce a production build. Nothing inside it is written by hand — every file is compiled, bundled, or generated by the Next.js toolchain itself.

Simple meaning: think of .next as the "compiled output" of your app — the same relationship dist/ has to a typical JavaScript project, or target/ has to a Java project. Your source code lives in app/ or pages/; the runnable, optimized version of it lives in .next.

It holds compiled server code (Server Components, Route Handlers, Server Actions), optimized client-side JavaScript and CSS chunks, prerendered HTML for static pages, and a set of JSON "manifest" files that tell the Next.js server how to map an incoming URL to the right compiled output.

Safe to delete: because everything in .next is derived from your source code, you can always delete the folder and regenerate it with next dev or next build. Deleting it is also the standard fix for stale-cache or "weird build" issues.

02When It's Created and Updated

CommandWhat happens to .next
next devCreated on first run; updated incrementally on every file save (Fast Refresh / Turbopack recompiles just the changed module).
next buildFully regenerated from scratch — compiles every route, generates all manifests, and prerenders static/ISR pages.
next startDoesn't modify .next — it just reads the production build already sitting there and serves it.

03Full Folder Structure

A production build (next build) produces a tree that looks roughly like this:

.next/ ├── cache/ # Turbopack/webpack, fetch & image cache ├── server/ # compiled Server Components, Route Handlers, Actions ├── static/ # hashed JS/CSS chunks served to the browser ├── types/ # generated TypeScript route types ├── diagnostics/ # framework/build diagnostic data ├── trace / trace-build # performance trace data (next build --profile) ├── app-build-manifest.json ├── app-path-routes-manifest.json ├── build-manifest.json ├── BUILD_ID ├── export-marker.json ├── fallback-build-manifest.json ├── images-manifest.json ├── next-minimal-server.js.nft.json ├── next-server.js.nft.json ├── package.json ├── prerender-manifest.json ├── react-loadable-manifest.json ├── required-server-files.json └── routes-manifest.json
The exact file list varies by Next.js version, whether you're on the App Router or Pages Router, and whether Turbopack or webpack built the project — treat this as a representative snapshot, not a fixed spec.

04Top-Level Folders

FolderContains
server/The compiled server-side output: your Server Components, Route Handlers, Server Actions, and the App/Pages Router route tree, all turned into runnable Node.js/Edge code. This is what next start actually executes.
static/Every hashed, browser-facing asset: JS chunks, CSS files, the React/Next.js runtime, and font files. Everything here is content-hashed so it can be cached forever by a CDN.
cache/Incremental build cache — SWC/Turbopack compilation cache, the fetch cache used by fetch() and unstable_cache, and the image optimization cache. This is what makes repeated next dev/next build runs fast; deleting it just costs you a slower next rebuild, nothing more.
types/Auto-generated .d.ts files that give typed Link href values and typed route params when typedRoutes is enabled.
diagnostics/Structured diagnostic output used by Next.js tooling and error overlays to explain build failures.
trace / trace-buildChrome-trace-format performance data, useful for profiling slow builds with next build --profile.

05Manifest & Metadata Files

Manifests are small JSON files the Next.js server reads at request time. They exist so the server never has to re-inspect the filesystem on every request — instead it looks up an already-computed answer.

FilePurpose
BUILD_IDA unique, per-build identifier (a hash) that tags a specific production build. Used to detect when a client's cached assets belong to an outdated deployment.
build-manifest.jsonMaps every page/route to the exact list of JS files it needs, so the server can inject only the required scripts for each request.
app-build-manifest.jsonThe App Router's equivalent of build-manifest.json — per-route JS chunk mapping for routes defined in app/.
app-path-routes-manifest.jsonMaps App Router URL paths to their underlying file-system route, including route groups and parallel/intercepting route segments.
routes-manifest.jsonThe master routing table: static routes, dynamic routes, rewrites, redirects, and headers defined in next.config.js.
prerender-manifest.jsonTracks every statically prerendered page — which pages are fully static, which use ISR, and their revalidate intervals.
fallback-build-manifest.jsonA build manifest used specifically for the fallback: true/blocking loading state of dynamic SSG pages.
images-manifest.jsonThe resolved next/image configuration — allowed remote domains, device sizes, and formats — used by the image optimizer at request time.
react-loadable-manifest.jsonMaps dynamically-imported (next/dynamic) components to their JS chunks, so lazy-loaded code can be found and served correctly.
export-marker.jsonA small marker file recording whether the build used next export-style static export.
required-server-files.jsonLists every file the production server needs to boot — used by minimal/standalone deployments to know exactly what to ship.
package.jsonA minimal package.json generated for standalone output mode, so a deployment can run with just node server.js without the full source tree.
*.nft.json (Node File Trace)Per-entry-point dependency traces — the exact set of files a given server file needs at runtime. Used to build minimal deployment bundles.

06How It's Used at Runtime

Build → Serve Flow
app/ or pages/ (your source code)
        │
        ▼
  next dev / next build
        │
        ▼
Compile with Turbopack or SWC/webpack
        │
   ┌────┼─────────────┐
   ▼    ▼              ▼
.next/server   .next/static   manifest *.json files
   │               │                 │
   └──────┬────────┴─────────────────┘
          ▼
   next start (Node server)
          │
          ▼
  Incoming request → routes-manifest.json
  resolves the route → server/ runs the
  matching Server Component / Route Handler
  → HTML + RSC payload sent to the browser,
    which then fetches static/ for JS/CSS

In short: manifests answer "which route is this, and what does it need?" while server/ and static/ hold the actual compiled code that answers the request.

07Dev vs Production Output

next dev

Optimized for rebuild speed, not output size. Code is compiled on-demand per route as you navigate, source maps are verbose, and nothing is minified. The cache/ folder here is what powers Fast Refresh.

next build + next start

Optimized for runtime performance. Every route is compiled ahead of time, JS/CSS is minified and content-hashed, static/ISR pages are prerendered into HTML, and required-server-files.json enables minimal "standalone" deploys.

08Best Practices

  • Always .gitignore it. create-next-app already adds .next/ to .gitignore by default — generated build output has no place in version control.
  • Delete it to force a clean rebuild. rm -rf .next (or rmdir /s /q .next on Windows) followed by next dev/next build is the standard fix for stale-cache or corrupted-build symptoms.
  • Don't hand-edit anything inside it. Any change will be silently overwritten on the next build.
  • Don't ship .next alone as your deployment artifact. A self-hosted deploy also needs node_modules (or a standalone bundle), public/, package.json, and next.config.js — unless you're using output: 'standalone', which bundles the minimal server for you.
  • Use next build --profile when you need the trace/trace-build files to diagnose a slow build.
Force a Clean Rebuild
# macOS / Linux
rm -rf .next
npm run dev

# Windows
rmdir /s /q .next
npm run dev

09FAQ

Can I delete .next while the dev server is running?
Stop the dev server first — Next.js writes to this folder continuously while running, so deleting it mid-session can cause errors. Stop, delete, then restart.
Why is .next so large?
Most of the size comes from cache/ (build caches that speed up future builds) and per-route compiled output in server/. It's expected to be much larger than your source code — none of it needs to be small since it's never committed to git.
Does .next exist in the App Router and Pages Router the same way?
The overall folder is shared, but some manifests differ — for example app-build-manifest.json and app-path-routes-manifest.json only exist for App Router routes, while a Pages Router project relies more heavily on build-manifest.json.
What's the difference between .next and out/?
out/ only appears when using static export (output: 'export') — it's a folder of plain HTML/CSS/JS with no Node server involved. .next is the standard build output used by next start and most hosting platforms, including serverless/edge deployments.
