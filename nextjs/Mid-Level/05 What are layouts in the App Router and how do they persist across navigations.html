What are layouts in the App Router and how do they persist across navigations?

Layouts are special components in the Next.js App Router that wrap pages and other layouts. They allow us to share UI such as headers, sidebars, navigation menus, and footers across multiple routes.

The primary purpose of a layout is to provide a shared UI structure that remains mounted while navigating between pages inside the same layout segment.

Why Do We Need Layouts?

Many applications have UI elements that appear on every page.

  • Header
  • Navigation Menu
  • Sidebar
  • Footer
  • Dashboard Shell

Instead of repeating this code in every page, we place it inside a layout.

Basic Layout Example

app/layout.tsx
export default function RootLayout({
  children
}) {

  return (
    <html>
      <body>
        <Header />

        {children}

        <Footer />
      </body>
    </html>
  );

}

The {children} prop represents the current page being rendered.

Page Example

app/page.tsx
export default function Home() {

  return <h1>Home Page</h1>;

}

Rendered Output

<Header />

<h1>Home Page</h1>

<Footer />

Folder Structure

App Router Structure
app/
│
├── layout.tsx
├── page.tsx
│
├── dashboard/
│   ├── layout.tsx
│   ├── page.tsx
│   └── settings/
│       └── page.tsx

Each route segment can have its own layout.

Nested Layout Example

app/dashboard/layout.tsx
export default function DashboardLayout({
  children
}) {

  return (
    <>
      <Sidebar />

      {children}
    </>
  );

}

All dashboard pages automatically receive the sidebar.

How Do Layouts Persist Across Navigations?

When navigating between pages that share the same layout, Next.js does not unmount and recreate that layout.

Navigation Example
/dashboard
      │
      ▼
/dashboard/settings

Both routes use the same dashboard layout.

What Happens?
Sidebar Layout
      │
      ├── Dashboard Page
      │
Navigate
      │
      ▼
Sidebar Layout
      │
      └── Settings Page

Only the page content changes. The layout remains mounted.

Without Layout Persistence

Traditional Behavior
Navigate
   │
   ▼
Destroy Header
Destroy Sidebar
Destroy Footer
   │
   ▼
Create Everything Again

This causes unnecessary re-rendering and loss of UI state.

With Layout Persistence

App Router Behavior
Navigate
   │
   ▼
Keep Layout Mounted
   │
   ▼
Replace Page Content Only

This makes navigation faster and preserves layout state.

Real Example

Suppose the sidebar contains a search box.

Sidebar
'use client';

import { useState } from 'react';

export default function Sidebar() {

  const [search, setSearch] = useState('');

  return (
    <input
      value={search}
      onChange={e => setSearch(e.target.value)}
    />
  );

}

When navigating between dashboard pages, the sidebar stays mounted and the search value remains intact.

What Happens Internally?

Navigation Flow
Current Layout
      │
      ▼
User Navigates
      │
      ▼
Reuse Existing Layout
      │
      ▼
Load New Page
      │
      ▼
Update Children

Layout vs Page

Feature Layout Page
Shared UI ✅ ❌
Receives children ✅ ❌

Can Be Nested ✅ ❌

When Should We Use Layouts?

  • Headers.
  • Footers.
  • Navigation Bars.
  • Sidebars.
  • Dashboard Shells.
  • Shared Route UI.

Important Note

Layouts persist only when navigating between routes that share the same layout segment. Navigating to a route with a different layout causes the old layout to unmount and the new layout to mount.

Interview Tip

A common misconception is that layouts re-render on every navigation. In reality, shared layouts remain mounted and only the page content inside the layout is replaced.
One-Line Definition:
Layouts in the App Router are shared UI wrappers that persist across navigations, allowing Next.js to reuse the layout and update only the page content inside it.
