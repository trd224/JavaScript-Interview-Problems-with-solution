22. What is unstable_cache and when would you use it over fetch caching?

unstable_cache() is a Next.js API that allows us to cache the result of any async function, not just fetch() requests.

fetch caching caches API requests.
unstable_cache caches custom server-side functions such as database queries, expensive calculations, CMS SDK calls, and third-party libraries.

The Problem With fetch Caching

Next.js caching works automatically when using:

Fetch Caching
const res = await fetch('https://api.myshop.com/products', {
  next: { revalidate: 60 }
});

const products = await res.json();

This is great because Next.js can cache the HTTP request.

But what if there is no fetch?

Example: Database Query

Direct Database Access
import { db } from '@/lib/db';

export default async function ProductsPage() {
  const products = await db.product.findMany();

  return <h1>Products</h1>;
}

There is no fetch() here.

So: 

No HTTP Request

No Fetch Cache

Database Runs Every Request
This is where unstable_cache() becomes useful.

Basic unstable_cache Example

Cached Database Query
import { unstable_cache } from 'next/cache';
import { db } from '@/lib/db';

const getProducts = unstable_cache(
  async () => {
    return db.product.findMany();
  },
  ['products'],
  { revalidate: 60 }
);

export default async function ProductsPage() {
  const products = await getProducts();

  return <h1>Products</h1>;
}

Now: 

First Request
      ↓
Query Database
      ↓
Save Result In Cache

Second Request
      ↓
Use Cache

60 Seconds Later
      ↓
Refresh Cache

Why Not Use fetch?

Sometimes data comes from:

  • Prisma
  • MongoDB Driver
  • Mongoose
  • Redis
  • Firebase SDK
  • Stripe SDK
  • CMS SDK

These do not use HTTP fetch requests.

unstable_cache() allows us to cache any async function regardless of where the data comes from.

Real World Example

Suppose we have:

Database
Products Table
   ↓

Every request executes:

Expensive Query
const products = await db.product.findMany();

Without caching:

   ↓
Database Query

   ↓
Database Query

Database Query

Database gets hit every time.

With unstable_cache

Cached Query
const getProducts = unstable_cache(
  async () => db.product.findMany(),
  ['products'],
  { revalidate: 60 }
);

Now: 

User 1
   ↓
Database Query
   ↓
Cache Result

User 2
   ↓
Use Cache

User 3
   ↓
Use Cache

Using Tags

Just like fetch caching, unstable_cache can use tags.

Tagged Cache
const getProducts = unstable_cache(
  async () => db.product.findMany(),
  ['products'],
  {
    tags: ['products'],
    revalidate: 60
  }
);

Now we can clear the cache:

On-Demand Revalidation
import { revalidateTag } from 'next/cache';

revalidateTag('products', 'max');

Fetch Cache vs unstable_cache

Feature fetch Cache unstable_cache
Caches HTTP Requests ✅ ❌
Caches Database Queries ❌ ✅
Caches Custom Functions ❌ ✅
Supports Revalidation ✅ ✅
Supports Tags ✅ ✅
Works With Prisma ❌ ✅
Works With MongoDB Driver ❌ ✅

When To Use fetch Cache?

Use fetch Cache
const res = await fetch('https://api.myshop.com/products', {
  next: { revalidate: 60 }
});

When data comes from:

  • REST APIs
  • External Services
  • Backend APIs

When To Use unstable_cache?

Use unstable_cache
const users = await db.user.findMany();

When data comes from:

  • Prisma
  • MongoDB
  • Redis
  • Firebase SDK
  • Custom Server Functions
  • Expensive Computations

One-Line Definition:
unstable_cache() is used to cache the result of custom async functions such as database queries and expensive server-side operations, while fetch caching is used specifically for HTTP requests made with fetch().
