34. How do you handle and surface validation errors from a Server Action?

Validation errors from a Server Action are handled by returning an error object from the Server Action and showing that object in the UI using useActionState().

Simple Meaning:
Do not throw validation errors for normal user mistakes. Return them as state and display them in the form.

Recommended Flow

Flow
User Submits Form
        ↓
Server Action Receives FormData
        ↓
Validate Data
        ↓
If Invalid → Return Error Object
        ↓
useActionState Receives Error
        ↓
Show Error In UI

1. Create Server Action

The Server Action validates the form data and returns a state object.

app/actions.ts
'use server';

type State = {
  success: boolean;
  message: string;
  errors?: {
    name?: string;
    email?: string;
  };
};

export async function createUser(previousState: State, formData: FormData): Promise<State> {
  const name = formData.get('name') as string;
  const email = formData.get('email') as string;

  const errors: State['errors'] = {};

  if (!name) errors.name = 'Name is required';
  if (!email) errors.email = 'Email is required';

  if (Object.keys(errors).length > 0) {
    return { success: false, message: 'Please fix the errors below', errors };
  }

  await db.user.create({ data: { name, email } });

  return { success: true, message: 'User created successfully' };
}

2. Use useActionState In Form

The Client Component receives the returned state from the Server Action.

app/UserForm.tsx
'use client';

import { useActionState } from 'react';
import { createUser } from './actions';

const initialState = { success: false, message: '', errors: {} };

export default function UserForm() {
  const [state, formAction] = useActionState(createUser, initialState);

  return (
    <form action={formAction}>
      <input name="name" placeholder="Enter name" />
      {state.errors?.name && <p>{state.errors.name}</p>}

      <input name="email" placeholder="Enter email" />
      {state.errors?.email && <p>{state.errors.email}</p>}

      <button>Save</button>

      {state.message && <p>{state.message}</p>}
    </form>
  );
}

Important Point

For expected validation errors, return values instead of throwing errors.

Situation Recommended Handling
User left name empty Return validation error
Email is invalid Return validation error
Password is too short Return validation error
Database server crashed Throw error or handle with error boundary

With Zod Validation

In real projects, we usually validate with a schema library like Zod.

app/actions.ts
'use server';

import { z } from 'zod';

const UserSchema = z.object({
  name: z.string().min(1, 'Name is required'),
  email: z.string().email('Invalid email address')
});

type State = {
  success: boolean;
  message: string;
  errors?: {
    name?: string[];
    email?: string[];
  };
};

export async function createUser(previousState: State, formData: FormData): Promise<State> {
  const result = UserSchema.safeParse({
    name: formData.get('name'),
    email: formData.get('email')
  });

  if (!result.success) {
    return { success: false, message: 'Validation failed', errors: result.error.flatten().fieldErrors };
  }

  await db.user.create({ data: result.data });

  return { success: true, message: 'User created successfully', errors: {} };
}

Client Form With Zod Errors

app/UserForm.tsx
'use client';

import { useActionState } from 'react';
import { createUser } from './actions';

const initialState = { success: false, message: '', errors: {} };

export default function UserForm() {
  const [state, formAction] = useActionState(createUser, initialState);

  return (
    <form action={formAction}>
      <input name="name" placeholder="Enter name" />
      {state.errors?.name && <p>{state.errors.name[0]}</p>}

      <input name="email" placeholder="Enter email" />
      {state.errors?.email && <p>{state.errors.email[0]}</p>}

      <button>Save</button>

      {state.message && <p>{state.message}</p>}
    </form>
  );
}

Why We Return Errors Instead Of Throwing

Validation errors are expected user mistakes, so they should be returned as normal state.

Good
return { success: false, message: 'Email is required' };
Avoid For Validation
throw new Error('Email is required');

Throwing is better for unexpected failures like database errors, server crashes, or permission problems.

useActionState vs useFormStatus

Hook Purpose
useActionState() Shows returned validation errors or success messages
useFormStatus() Shows loading/pending status while submitting

Final Best Practice

Best Practice Flow
Server Action
      ↓
Validate FormData
      ↓
Return { success, message, errors }
      ↓
useActionState()
      ↓
Show errors beside fields
One-Line Definition:
Validation errors from a Server Action should be returned as structured state and displayed in the UI using useActionState(), while unexpected system errors should be thrown or handled separately.
