How does Next.js on-demand revalidation work with another backend like .NET or Java?

In a real-world project, we can use Next.js as our frontend and use another backend like .NET, Java, PHP, or Node.js for business logic and database work.

Next.js is not only frontend. In the App Router, Next.js also has a small server layer for Server Components, SSR, caching, and revalidation.

Real Architecture

https://api.myshop.com
      ↓

Here:

  • https://myshop.com is our Next.js website.
  • https://api.myshop.com is our .NET backend API.

Next.js Product Page

app/products/page.tsx
export default async function ProductsPage() {
  const res = await fetch('https://api.myshop.com/products', { next: { tags: ['products'] } });
  const products = await res.json();

  return <div>{products.map(product => <p key={product.id}>{product.name} - ₹{product.price}</p>)}</div>;
}

Next.js fetches product data from the .NET backend and stores it in the Next.js Data Cache using the tag products.

The product data is coming from .NET, but the cache belongs to Next.js.

Why Cache Problem Happens?

Suppose the product price is:

Old Price
{
  "name": "Shoes",
  "price": 100
}

Next.js caches this data.

Now admin changes the price in the .NET backend:

New Price In Database
{
  "name": "Shoes",
  "price": 120
}

But Next.js cache may still have the old price:

Next.js Cache
{
  "name": "Shoes",
  "price": 100
}
So the database has fresh data, but Next.js may still show old cached data until the cache is cleared.

Solution: On-Demand Revalidation

After the .NET backend updates the product, it makes a backend-to-backend HTTP call to Next.js.

Backend To Backend Call
.NET Backend
     ↓
POST https://myshop.com/api/revalidate-products
     ↓
Next.js API Route
     ↓
revalidateTag('products')
     ↓
Next.js Cache Cleared

Important Point

https://myshop.com/api/revalidate-products belongs to Next.js only if myshop.com is hosted by Next.js. It is different from https://api.myshop.com, which belongs to .NET.

Correct Domain Separation

URL Technology Purpose
https://myshop.com Next.js Frontend, SSR, Server Components, Cache
https://myshop.com/api/revalidate-products Next.js Small revalidation API
https://api.myshop.com/products .NET Main backend API
Database SQL Server / MongoDB Actual data storage

Next.js Revalidate API Route

app/api/revalidate-products/route.ts
import { revalidateTag } from 'next/cache';

export async function POST(request: Request) {
  const secret = request.headers.get('x-revalidate-secret');

  if (secret !== process.env.REVALIDATE_SECRET) {
    return Response.json({ message: 'Invalid secret' }, { status: 401 });
  }

  revalidateTag('products', 'max');

  return Response.json({ revalidated: true, tag: 'products' });
}

This API route exists only to clear the Next.js cache.

.NET Backend Calls Next.js API

.NET Example
using var client = new HttpClient();

client.DefaultRequestHeaders.Add("x-revalidate-secret", "my-secret-key");

await client.PostAsync("https://myshop.com/api/revalidate-products", null);

This is a normal HTTP request from one backend to another backend.

        ↓

        ↓

        ↓
12. Next.js fetches fresh data from .NET backend
        ↓
13. User sees product price = 120

Why .NET Cannot Directly Run revalidateTag()

Because revalidateTag() is a Next.js function.

Runtime Difference
.NET Backend
   Can update database
   Can run C# code
   Cannot run revalidateTag()


Next.js Server
   Can render Server Components
   Can manage Next.js cache
   Can run revalidateTag()
The cache belongs to Next.js, so only Next.js can clear it.

Why This Is Not Two Main Backends

The .NET backend is still the main backend.

Responsibility
.NET Backend
   → Business logic
   → Product update
   → Database
   → Authentication APIs
   → Payment APIs


Next.js Server
   → Frontend rendering
   → Server Components
   → Data Cache
   → Revalidation API

The Next.js API route is just a small helper endpoint for cache clearing.

If /api Already Belongs To .NET

If your domain is configured like this:

Problem Setup
https://myshop.com/api/products
      ↓
.NET Backend

Then this URL: 

https://myshop.com/api/revalidate-products

will also go to .NET, not Next.js.

Solution

Use a separate route for Next.js revalidation:

Possible Next.js Revalidate URLs
https://myshop.com/internal/revalidate-products

https://myshop.com/next-revalidate/products

https://frontend.myshop.com/api/revalidate-products

The important thing is that the URL must reach the Next.js server, not the .NET server.

One-Line Definition:
When using Next.js with a .NET or Java backend, the main backend updates the database, then sends a backend-to-backend HTTP request to a small Next.js revalidation endpoint so Next.js can clear its own cache.
