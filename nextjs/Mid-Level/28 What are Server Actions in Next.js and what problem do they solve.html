What are Server Actions in Next.js and what problem do they solve?

Server Actions are server-side functions that can be called directly from forms, Server Components, and Client Components without creating separate API routes.

Simple Meaning:
Server Actions allow us to execute backend code directly from our Next.js components.

What Problem Did Server Actions Solve?

Before Server Actions, every form submission usually required an API route.

Traditional Flow
User Clicks Save
       ↓
Client Component
       ↓
fetch('/api/users')
       ↓
API Route
       ↓
Database
       ↓
Response
       ↓
Update UI

Even a simple database insert required multiple files and extra boilerplate code.

Without Server Actions

API Route
// app/api/users/route.ts

export async function POST(req: Request) {

  const data = await req.json();

  await db.user.create({
    data
  });

  return Response.json({
    success: true
  });

}
Client Component
'use client';

export default function UserForm() {

  async function saveUser() {

    await fetch('/api/users', {
      method: 'POST',
      body: JSON.stringify({
        name: 'Rizwan'
      })
    });

  }

  return (
    <button onClick={saveUser}>
      Save User
    </button>
  );

}

We need:

  • Form
  • API Route
  • fetch()
  • Request Parsing
  • Response Handling

With Server Actions

Server Action
// app/actions.ts

'use server';

export async function createUser(formData: FormData) {

  await db.user.create({
    data: {
      name: formData.get('name')
    }
  });

}
Page
import { createUser } from './actions';

export default function Page() {

  return (
    <form action={createUser}>

      <input
        name="name"
        placeholder="Enter Name"
      />

      <button type="submit">
        Save User
      </button>

    </form>
  );

}

Flow: 

User Clicks Save
       ↓
Server Action
       ↓
Database
       ↓
Updated UI

No API route required.

Understanding "use server"

The 'use server' directive tells Next.js that this function must execute on the server.

Example
'use server';

export async function deleteUser(id: string) {

  await db.user.delete({
    where: { id }
  });

}

The browser cannot execute this function directly. Next.js automatically sends a request to the server and runs it there.

Real World Example

Suppose we have an Admin Panel where products can be added.

Server Action
'use server';

export async function addProduct(formData: FormData) {

  await db.product.create({
    data: {
      name: formData.get('name'),
      price: Number(formData.get('price'))
    }
  });

}
Form
<form action={addProduct}>

  <input name="name" />

  <input name="price" />

  <button>
    Add Product
  </button>

</form>

Flow: 

Submit Form
      ↓
Server Action
      ↓
Database Insert
      ↓
Success

Server Actions + Revalidation

Server Actions are commonly used with cache revalidation.

Example
'use server';

import { revalidatePath } from 'next/cache';

export async function addProduct(formData: FormData) {

  await db.product.create({
    data: {
      name: formData.get('name')
    }
  });

  revalidatePath('/products');

}

Flow: 

Add Product
      ↓
Database Updated
      ↓
revalidatePath('/products')
      ↓
Products Cache Cleared
      ↓
Next Request Gets Fresh Data

Can Client Components Use Server Actions?

Yes.

Client Component Example
'use client';

import { createUser } from './actions';

export default function Button() {

  return (
    <button
      onClick={() => createUser()}
    >
      Create User
    </button>
  );

}

Although the button is in a Client Component, the action still runs on the server.

One-Line Definition:
Server Actions are server-side functions that allow us to perform create, update, and delete operations directly from Next.js components without creating separate API routes.

Server Actions vs Separate Backend (.NET, Java, Node)

Server Actions provide the biggest benefit when Next.js itself owns the backend logic and talks directly to the database.

Ideal Server Actions Architecture
Next.js Frontend
       ↓
Server Action
       ↓
Database

In this setup, we can directly create, update, or delete database records without creating API routes.

Best Use Case:
Next.js Frontend + Next.js Backend + Database

What If We Already Have a Separate Backend?

Many real-world applications use a separate backend such as .NET, Java, Node.js, Spring Boot, Laravel, etc.

Separate Backend Architecture
Next.js Frontend
       ↓
.NET API
       ↓
Database

In this case, the business logic already exists inside the backend API.

Therefore, we usually call the backend directly using fetch().

Example
const products = await fetch(
  'https://api.company.com/products'
);

or 

await fetch(
  'https://api.company.com/users',
  {
    method: 'POST'
  }
);

Can Server Actions Still Be Used?

Yes.

A Server Action can act as a middle layer between Next.js and the backend API.

Architecture
Browser
    ↓
Server Action
    ↓
.NET API
    ↓
Database
Example
'use server';

export async function createUser(formData: FormData) {

  await fetch(
    'https://api.company.com/users',
    {
      method: 'POST',
      body: JSON.stringify({
        name: formData.get('name')
      })
    }
  );

}

Here the Server Action does not access the database directly. Instead, it forwards the request to the backend API.

For Projects Like Aadhaar Authentication

Typical Architecture
Angular / Next.js
        ↓
.NET API
        ↓
Database

In such projects, it is usually simpler to call the backend API directly rather than introducing Server Actions.

Architecture Recommended Approach
Next.js + Database ✅ Server Actions
Next.js + .NET Backend ✅ Direct API Calls
Next.js + Java Backend ✅ Direct API Calls
Next.js + Node Backend ✅ Direct API Calls
Easy Rule:
If Next.js owns the database, Server Actions are usually the best choice.

If a separate backend already exists, direct API calls are usually simpler and more common.
