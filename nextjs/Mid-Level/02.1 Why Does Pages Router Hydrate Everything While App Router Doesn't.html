Why Does Pages Router Hydrate Everything While App Router Doesn't?

The main difference is that the Pages Router was built around full-page hydration, while the App Router uses React Server Components to avoid sending unnecessary JavaScript to the browser.

Pages Router renders HTML on the server, but then still sends page JavaScript and React JavaScript to the browser so React can hydrate the whole page.

Step 1: What Happens in Pages Router?

In the Pages Router, even if the server already sends HTML, the browser still receives the page JavaScript and React JavaScript.

pages/index.js
export default function Page() {
  return <h1>Hello</h1>;
}

So in the Pages Router, the component runs on the server first and then runs again in the browser during hydration.

Why Does Pages Router Send page.js and react.js?

Because the Pages Router architecture assumes that every page is a React application that must be hydrated in the browser.

It does not matter whether the current page is interactive or not. Pages Router was designed to hydrate the whole page.

What Is Hydration?

Hydration means React attaches browser-side JavaScript behavior to the HTML that was already generated on the server.

Server HTML
<button>Click Me</button>

This HTML is visible, but it does not know anything about React state or click handlers.

React Component
export default function Button() {
  return (
    <button onClick={() => alert('Hello')}>
      Click Me
    </button>
  );
}

The browser needs JavaScript to attach the onClick behavior.

Hydration Flow

      ▼

      │
      ▼

      │
      ▼
Button Becomes Interactive

Why Was Pages Router Built Like This?

Pages Router was built before React Server Components existed.

At that time, React had only one main server rendering model:

Old React Model
Server Generates HTML
      │
      ▼
Browser Hydrates Entire React App

So React's older assumption was:

Old Assumption
Any Page Might Become Interactive
        │
        ▼
Send JavaScript For The Whole Page

Example of Interactive Page

Counter Component
import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}

For this page, the browser definitely needs JavaScript.

Without JavaScript
Server Sends:
<button>0</button>

User Clicks Button
      │
      ▼
Nothing Happens ❌
With JavaScript
Server Sends HTML
      +
Browser Gets React.js
      +
Browser Gets Counter.js
      │
      ▼
Click Updates Count ✅

The Problem With Pages Router

The problem is that Pages Router sends JavaScript even when the page does not need browser interaction.

Static Content Example
export default function BlogPost() {
  return (
    <article>
      <h1>Next.js Guide</h1>
      <p>This is a static blog article.</p>
    </article>
  );
}

This content is readable HTML. It does not need React in the browser.

But Pages Router Still Sends
BlogPost.js
React.js
Next.js Runtime

Step 2: What Changed in App Router?

The App Router uses React Server Components.

React Server Components allow components to execute only on the server and not send their component JavaScript to the browser.
app/page.js
export default function Page() {
  return <h1>Hello</h1>;
}

In the App Router, this component is a Server Component by default because there is no 'use client'.

App Router Server Component Flow

Flow
Request
   │
   ▼
Server Runs Page()
   │
   ▼

   │
   ▼
Send HTML
   │
   ▼
Browser Displays HTML

No page.js for this component
No hydration for this component
Result
Page() runs on Server  ✅
Page() runs on Browser ❌

App Router Rule

App Router follows a smarter rule:

App Router Rule
Need Browser Features?

 │         │
 ▼         ▼

Server Component Example

app/products/page.js
export default async function Products() {
  const res = await fetch('https://dummyjson.com/products');
  const data = await res.json();

  return (
    <ul>
      {data.products.map(product => (
        <li key={product.id}>{product.title}</li>
      ))}
    </ul>
  );
}

This component fetches data and renders HTML on the server.

Sent To Browser
HTML Output Only

No Products.js
No React hydration for Products component

Client Component Example

If our component needs browser interaction, we mark it with 'use client'.

app/Counter.js
'use client';

import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}

Now Next.js sends JavaScript for this component because it needs browser features.

Sent To Browser
Counter.js
React.js
Required Client JavaScript

Mixing Server and Client Components

app/page.js
import Counter from './Counter';

export default function Page() {
  return (
    <>
      <h1>Product Page</h1>
      <p>This content is server-rendered.</p>
      <Counter />
    </>
  );
}
app/Counter.js
'use client';

import { useState } from 'react';

export default function Counter() {

Why React Does Not Need Control in App Router?

React does not need browser control for components that have no browser behavior.

For example:

Server-Only Content
Header
Footer
Blog Content
Product Details
Article Content
FAQ Section

These components usually only display data. They do not need:

Browser Features
useState
useEffect
onClick
window
localStorage

So App Router keeps them on the server and avoids sending their JavaScript.

Simple Memory Trick

Pages Router
HTML + JS for whole page
App Router
HTML for server parts
JS only for interactive parts

Final Comparison

Component Type Server Browser
Pages Router Page ✅ Runs ✅ Runs Again
App Router Server Component ✅ Runs ❌ Does Not Run
App Router Client Component ✅ Pre-rendered ✅ Hydrated

Interview Tip

Pages Router was built before React Server Components, so it hydrates the entire page in the browser. App Router uses React Server Components, so non-interactive components can stay on the server and only interactive Client Components are hydrated.
One-Line Definition:
Pages Router sends HTML plus JavaScript for the whole page because it hydrates every page, while App Router sends JavaScript only for Client Components and keeps Server Components on the server.
