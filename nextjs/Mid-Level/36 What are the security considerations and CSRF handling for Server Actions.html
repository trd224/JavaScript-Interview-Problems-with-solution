36. What are the security considerations and CSRF handling for Server Actions?

Server Actions run on the server, so they can safely access databases, cookies, sessions, and secret values. But they still need proper validation, authentication, authorization, and CSRF protection.

Simple Meaning:
Server Actions are server-side, but they are still callable from the browser, so every action must be treated like a protected backend endpoint.

1. Server Actions Are Not Public Browser Code

Code inside a Server Action is not shipped to the browser.

Server Action
'use server';

export async function createUser(formData: FormData) {
  await db.user.create({ data: { name: formData.get('name') as string } });
}

This code runs on the server, not inside the browser.

Good:
Database logic, secret keys, and server-side checks can stay on the server.

2. But Server Actions Still Need Authentication

Do not assume that only a valid user can call the action. Always check the current session inside the Server Action.

Check Session
'use server';

import { auth } from '@/lib/auth';

export async function updateProfile(formData: FormData) {
  const session = await auth();

  if (!session?.user) {
    return { success: false, message: 'Unauthorized' };
  }

  await db.user.update({
    where: { id: session.user.id },
    data: { name: formData.get('name') as string }
  });

  return { success: true, message: 'Profile updated' };
}
Important:
Never trust only the UI. Hiding a button is not security. Always check permission inside the Server Action.

3. Authorization Is Also Required

Authentication means the user is logged in. Authorization means the user is allowed to perform that specific action.

Authorization Example
'use server';

export async function deleteProduct(productId: string) {
  const session = await auth();

  if (!session?.user) {
    return { success: false, message: 'Unauthorized' };
  }

  if (session.user.role !== 'admin') {
    return { success: false, message: 'Forbidden' };
  }

  await db.product.delete({ where: { id: productId } });

  return { success: true, message: 'Product deleted' };
}
Check Meaning
Authentication Is the user logged in?
Authorization Is the user allowed to do this action?

4. Validate All Form Data

Never trust formData. A user can modify form values from the browser DevTools.

Validation Example
'use server';

export async function createUser(formData: FormData) {
  const name = formData.get('name') as string;
  const email = formData.get('email') as string;

  if (!name) return { success: false, message: 'Name is required' };
  if (!email.includes('@')) return { success: false, message: 'Invalid email' };

  await db.user.create({ data: { name, email } });

  return { success: true, message: 'User created' };
}

5. Do Not Trust Hidden Inputs

Hidden inputs are useful for sending IDs, but they are not secure. Users can change them.

Form
<form action={deleteProduct}>
  <input type="hidden" name="productId" value={product.id} />
  <button>Delete</button>
</form>

The Server Action must still check whether the logged-in user can delete that product.

Server Action
'use server';

export async function deleteProduct(formData: FormData) {
  const session = await auth();
  const productId = formData.get('productId') as string;

  if (session?.user.role !== 'admin') {
    return { success: false, message: 'Forbidden' };
  }

  await db.product.delete({ where: { id: productId } });

  return { success: true, message: 'Product deleted' };
}

6. CSRF Handling in Server Actions

CSRF means Cross-Site Request Forgery. It happens when another website tries to submit a request using the logged-in user's cookies.

CSRF Attack Idea
User logged in on our site
        ↓
User opens malicious website
        ↓
Malicious site tries to submit a request
        ↓
Browser may send cookies automatically

Next.js Server Actions include built-in CSRF protection by checking request origins. In simple terms, the request must come from the same origin as our application.

Simple Meaning:
Next.js checks where the Server Action request is coming from. Cross-site action requests are blocked unless explicitly allowed.

7. allowedOrigins for Proxies or Different Domains

Sometimes our app may be behind a reverse proxy, load balancer, or different domain. In that case, we can configure allowed origins.

next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    serverActions: {
      allowedOrigins: ['my-proxy.com', '*.my-proxy.com']
    }
  }
};

module.exports = nextConfig;
Important:
Only add trusted domains in allowedOrigins. Do not add random public domains.

8. Use SameSite Cookies

For cookie-based authentication, use secure cookie settings.

Cookie Security
cookies().set('session', token, {
  httpOnly: true,
  secure: true,
  sameSite: 'lax',
  path: '/'
});
Cookie Option Purpose
httpOnly JavaScript cannot read the cookie
secure Cookie is sent only over HTTPS
sameSite Helps reduce CSRF risk

9. Do Not Put Secrets in Client Components

Server Actions can access secrets, but Client Components cannot safely store secret values.

Good
'use server';

export async function callPrivateApi() {
  const apiKey = process.env.PRIVATE_API_KEY;

  await fetch('https://api.example.com/private', {
    headers: { Authorization: `Bearer ${apiKey}` }
  });
}
Bad
'use client';

const apiKey = 'secret-key';

export default function Page() {
  return <button>Call API</button>;
}

10. Avoid Returning Sensitive Data

A Server Action can return data to the browser. So do not return passwords, tokens, secret keys, or private database fields.

Bad
return {
  success: true,
  token: user.sessionToken,
  passwordHash: user.passwordHash
};
Good
return {
  success: true,
  message: 'Login successful'
};

11. Security Checklist

Security Rule Required?
Check logged-in user ✅ Yes
Check user permission ✅ Yes
Validate all form data ✅ Yes
Trust hidden input values ❌ No
Return secrets to browser ❌ No
Use secure cookies ✅ Yes
Add random allowedOrigins ❌ No

Easy Memory Trick

Remember
Server Action
      ↓
Still a Backend Endpoint
      ↓
Auth + Permission + Validation + CSRF Safety
One-Line Definition:
Server Actions are protected by server-side execution and built-in origin checks, but we must still validate input, check authentication, check authorization, secure cookies, and avoid exposing sensitive data.
