What is the use cache directive introduced in Next.js 15 and how does it differ from unstable_cache?

Next.js 15 introduced the 'use cache' directive as a simpler and more modern way to cache Server Components and async functions.

Think of 'use cache' as the future replacement for many unstable_cache() use cases.

The Old Way: unstable_cache()

Before Next.js 15, if we wanted to cache a database query, we wrote:

unstable_cache Example
import { unstable_cache } from 'next/cache';

const getProducts = unstable_cache(
  async () => {
    return db.product.findMany();
  },
  ['products'],
  { revalidate: 60 }
);

This works, but the syntax is quite verbose.

The New Way: use cache

Next.js 15 allows us to write:

use cache Example
async function getProducts() {
  'use cache';

  return db.product.findMany();
}

Much simpler.

Instead of wrapping functions with unstable_cache(), we place 'use cache' inside the function body.

Server Component Example

Products Page
async function getProducts() {
  'use cache';

  return db.product.findMany();
}

export default async function ProductsPage() {
  const products = await getProducts();

  return <h1>Products</h1>;
}

Now Next.js can cache the result of getProducts().

What Problem Does It Solve?

Suppose:

Database Query
await db.product.findMany();

Without caching: 

Request 1
    ↓
Database Query

Request 2
    ↓
Database Query

Request 3
    ↓
Database Query

Database gets hit every time.

With:

Cached Function
async function getProducts() {
  'use cache';

  return db.product.findMany();
}

Next.js can reuse cached results.

How Is It Different From React.cache?

Many developers confuse these two.

React.cache()
import { cache } from 'react';

export const getProducts = cache(async () => {
  return db.product.findMany();
});

Lifetime: 

Current Request Only
use cache
async function getProducts() {
  'use cache';

  return db.product.findMany();
}

Lifetime: 

Across Requests
React.cache() = request memoization.
use cache = persistent Next.js cache.

How Is It Different From unstable_cache()?

Both create persistent cache entries.

The main difference is syntax and future direction.

unstable_cache
const getProducts = unstable_cache(
  async () => {
    return db.product.findMany();
  },
  ['products'],
  { revalidate: 60 }
);
use cache
async function getProducts() {
  'use cache';

  return db.product.findMany();
}

The second version is much easier to read.

Using Cache Tags

use cache can work with cache tags.

Tagged Function
import { cacheTag } from 'next/cache';

async function getProducts() {
  'use cache';

  cacheTag('products');

  return db.product.findMany();
}

Now we can invalidate:

Revalidate Tag
import { revalidateTag } from 'next/cache';

revalidateTag('products', 'max');

Using Cache Lifetime

Example
import { cacheLife } from 'next/cache';

async function getProducts() {
  'use cache';

  cacheLife('minutes');

  return db.product.findMany();
}

This controls how long the cached value remains valid.

When Should We Use use cache?

  • Database queries.
  • CMS data.
  • Expensive computations.
  • Third-party SDK calls.
  • Server-side helper functions.

Basically anywhere we previously considered: 

unstable_cache(...)

One-Line Definition:
'use cache' is a Next.js 15 caching directive that creates persistent server-side cache entries for functions and Server Components, providing a simpler and more modern alternative to unstable_cache().
