50. What is the connection() API introduced in Next.js 15?

The connection() API is a new Next.js 15 function that allows us to explicitly wait for an incoming request before rendering a page.

The primary purpose of connection() is to tell Next.js that this page must be rendered dynamically at request time and cannot be prerendered ahead of time.

Why Was connection() Introduced?

Sometimes a page depends on request-specific information such as:

  • Cookies
  • Headers
  • User Session
  • Request Metadata
  • Random Values
  • Current Time

Next.js cannot prerender such pages during build time because the request does not exist yet.

Before Next.js 15

We commonly used:

Old Approach
export const dynamic = 'force-dynamic';

to force dynamic rendering.

The Pages Router does not have force-dynamic. Instead, use getServerSideProps(), which automatically renders the page on every request:

Next.js 15 Approach

connection()
import { connection } from 'next/server';

export default async function Page() {

  await connection();

  return 
Dynamic Page
;

}

This tells Next.js: 

Wait For Request
      │
      ▼
Render Page

Real Example

Suppose we want to display the current time.

Example
import { connection } from 'next/server';

export default async function Page() {

  await connection();

  const now = new Date();

  return <h1>{now.toLocaleTimeString()}</h1>;

}

Every request gets a fresh time value.

Why Is connection() Needed Here?

Without it, Next.js might try to prerender:

Build Time
10:00 AM

Then every user would see: 

10:00 AM

even hours later.

Using Request-Specific Data

Cookie Example
import { connection } from 'next/server';
import { cookies } from 'next/headers';

export default async function Page() {

  await connection();

  const token = (await cookies()).get('token')?.value;

  return <h1>{token}</h1>;

}

This page depends on the current request's cookies.

How Does connection() Relate to PPR?

Partial Prerendering (PPR) allows part of a page to be prerendered and part to wait for request-time data.

PPR Flow
Static Content
      │
      ▼
Prerendered

Dynamic Content
      │
      ▼
await connection()
      │
      ▼
Rendered Per Request

This helps Next.js determine which parts depend on the request.

When Should We Use connection()?

  • Current time display.
  • Random values.
  • User-specific pages.
  • Cookie-based rendering.
  • Header-based rendering.
  • Request-dependent content.

When Should We Avoid connection()?

  • Static pages.
  • Blogs.
  • Marketing pages.
  • Documentation pages.
  • Pages that can be prerendered.

Important Note

Calling await connection() opts the route into request-time rendering because the page now explicitly depends on an incoming request.

Interview Tip

A common misconception is that connection() fetches data. In reality, connection() does not fetch anything—it simply waits for an incoming request and tells Next.js that rendering should happen at request time.
One-Line Definition:
The connection() API in Next.js 15 explicitly waits for an incoming request and marks a route as request-time dynamic, preventing prerendering when content depends on request-specific information.
