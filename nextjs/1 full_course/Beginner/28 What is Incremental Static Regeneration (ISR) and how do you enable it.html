28. What is Incremental Static Regeneration (ISR) and How Do You Enable It?

Incremental Static Regeneration (ISR) is a Next.js feature that combines the speed of Static Site Generation (SSG) with the ability to update static pages after deployment.

ISR allows us to regenerate static pages in the background at a specified interval without rebuilding the entire application.

Why Do We Need ISR?

SSG generates pages during the build process.

The problem is that data may change after deployment.

Without ISR:

Problem
Build Application
       │
       ▼
Generate HTML
       │
       ▼
Deploy

Data Changes Later

       │
       ▼
HTML Becomes Outdated

To get fresh data, we would need to rebuild and redeploy the application.

ISR solves this problem.

How ISR Works

ISR Flow
Build Application
       │
       ▼
Generate Static HTML
       │
       ▼
Deploy

After Revalidate Time

       │
       ▼
Next Request Arrives
       │
       ▼
Regenerate Page
In Background
       │
       ▼
Update Cached Page

Basic Example

pages/posts.js
export async function getStaticProps() {

  const res = await fetch(
    'https://api.example.com/posts'
  );

  const posts = await res.json();

  return {
    props: { posts },
    revalidate: 60
  };

}

export default function Posts({ posts }) {

  return (
    <h1>
      Total Posts: {posts.length}
    </h1>
  );

}

How Do We Enable ISR?

ISR is enabled by returning the revalidate property from getStaticProps().

Enable ISR
return {
  props: { posts },
  revalidate: 60
};

Meaning: 

Regenerate Page
Every 60 Seconds

Important Point

The user who triggers regeneration still receives the old cached page. Regeneration happens in the background, and future users receive the updated page.

Real Example

Suppose our application displays product information.

Route
/products

Products rarely change, but updates may occur every few minutes.

ISR Configuration
return {
  props: { products },
  revalidate: 300
};

Meaning: 

Regenerate Page
Every 5 Minutes

We get:

  • Static page performance.
  • Automatic updates.
  • No manual rebuilds.

ISR with Dynamic Routes

pages/products/[id].js
export async function getStaticPaths() {

  return {
    paths: [
      { params: { id: '1' } }
    ],
    fallback: 'blocking'
  };

}

export async function getStaticProps({ params }) {
  const res = await fetch(`https://api.example.com/products/${params.id}`);
  const product = await res.json();
  return {
    props: { product },
    revalidate: 60
  };
}

Each generated page can be regenerated independently.

SSR vs SSG vs ISR

Feature SSR SSG ISR
HTML Generated Every Request Build Time Build Time + Regeneration
Fresh Data Always Build Time Data Periodic Updates
Performance Slower Fastest Very Fast
Server Load High Very Low Low

When Should We Use ISR?

  • Blogs
  • News Websites
  • Product Catalogs
  • E-commerce Websites
  • Documentation Sites
  • Marketing Websites

These pages need occasional updates but do not require fresh data on every request.

When Should We Avoid ISR?

  • User Dashboards
  • Stock Prices
  • Live Sports Scores
  • Real-Time Analytics

These pages require SSR because data must always be fresh.

App Router Equivalent

In the App Router, ISR is enabled using the revalidate export.

app/posts/page.js
export const revalidate = 60;

async function getPosts() {

  const res = await fetch(
    'https://api.example.com/posts'
  );

  return res.json();

}

export default async function Posts() {

  const posts = await getPosts();

  return (
    <h1>
      {posts.length}
    </h1>
  );

}

Meaning: 

Regenerate Page
Every 60 Seconds

Interview Tip

A common misconception is that ISR regenerates pages exactly when the revalidate time expires. In reality, regeneration happens only when a new request arrives after the revalidate period has passed.
One-Line Definition:
Incremental Static Regeneration (ISR) is a Next.js feature that allows statically generated pages to be automatically regenerated in the background after a specified interval without rebuilding the entire application.

Does ISR Regenerate Automatically?

Important:
A common misconception is that ISR automatically regenerates pages exactly when the revalidate time expires. This is not true.

ISR does not regenerate pages automatically when the timer expires.

Instead, Next.js waits for a new request after the revalidate period has passed.

Actual ISR Rule
Revalidate Time Expires
          +
New Request Arrives
          =
Regeneration Happens

Not:

Incorrect Assumption
Revalidate Time Expires
          =
Regeneration Happens ❌

The first user after the revalidate period usually receives the old cached page while regeneration happens in the background.

ISR regeneration is triggered by a request after the revalidate period has passed, not by the timer itself.
