27. What is the Difference Between fallback: true, false, and 'blocking'?

The fallback option is returned from getStaticPaths() in Next.js. It controls one specific thing: what should happen when a visitor requests a dynamic page whose path was NOT included in the paths array during build time.

Think of paths as "the list of pages I already know about." fallback answers the question: "What do I do about pages that AREN'T on that list?"

Why Does This Option Exist?

Imagine an e-commerce site with 50,000 products. Pre-building all 50,000 pages at deploy time would make builds extremely slow. So Next.js lets you pre-build only the important/popular pages (say, the top 20) and decide what to do for the other 49,980 when someone actually visits them.

That decision is exactly what fallback controls, and it has 3 possible values:

  • false — unknown paths are not allowed at all
  • true — unknown paths are generated on first visit, with a loading state
  • 'blocking' — unknown paths are generated on first visit, with no loading state (server waits)

What happens next depends entirely on the value of fallback in getStaticPaths(). Let's walk through all 3 cases with complete, runnable code.

fallback: false

"Strict mode." Only the pages listed in paths exist. Anything else is a 404, no matter what.

pages/products/[id].js — Complete Example
export async function getStaticPaths() {
  return {
    paths: [
      { params: { id: '1' } },
      { params: { id: '2' } }
    ],
    fallback: false
  };
}

export async function getStaticProps({ params }) {
  const res = await fetch(`https://api.example.com/products/${params.id}`);
  const product = await res.json();
  return {
    props: { product: product }
  };
}

Common misconception: many developers assume Next.js will "just generate the page anyway" when a route is missing. With fallback: false that never happens — getStaticProps() is never even called for an unknown route.
Use fallback: false when: the full list of valid pages is small and fully known ahead of time — e.g. a blog with 30 posts, a documentation site, a marketing site with a fixed set of pages.

fallback: true

"Generate on demand, with a loading state." Unknown pages are built the first time someone visits them, and while that build is happening, the browser shows a fallback/loading version of the page.

pages/products/[id].js — Complete Example
import { useRouter } from 'next/router';

export async function getStaticPaths() {
  return {
    paths: [
      { params: { id: '1' } },
      { params: { id: '2' } }
    ],
    fallback: true
  };
}

export async function getStaticProps({ params }) {
  const res = await fetch(`https://api.example.com/products/${params.id}`);
  const product = await res.json();
  return {
    props: { product },
    revalidate: 60
  };
}

export default function Product({ product }) {
  const router = useRouter();
  if (router.isFallback) {
    return <h1>Loading...</h1>;
  }
  return (
    <div>
      <h1>{product.name}</h1>
      <p>Price: ${product.price}</p>
    </div>
  );
}
router.isFallback is required with fallback: true. On the very first visit to an ungenerated page, product is not available yet — the component renders BEFORE getStaticProps() finishes. Skipping this check causes a crash like "Cannot read property 'name' of undefined."

t=1ms     Next.js has no pre-built page for "3"
t=2ms     Sends back a "fallback" HTML shell instantly
t=2ms     React hydrates, router.isFallback = true
t=2ms     Page shows "Loading..."
t=350ms   getStaticProps() finishes fetching product 3
t=351ms   Page re-renders with real data
t=351ms   router.isFallback = false
t=351ms   Page now shows "Product 3"
t=351ms   This HTML is cached on the server

Every Visit After That

/products/3  →  Product 3 page served instantly (from cache)
                No loading state, no delay
Use fallback: true when: you have thousands of pages, most of them are rarely visited, and a brief "Loading..." flash is an acceptable trade-off — e.g. a huge product catalog or user-generated content site.

fallback: 'blocking'

"Generate on demand, no loading state." Unknown pages are still built on first visit, but instead of sending a loading shell to the browser, the server holds the response and waits until the full page is ready before sending anything at all.

pages/products/[id].js — Complete Example
export async function getStaticPaths() {
  return {
    paths: [
      { params: { id: '1' } },
      { params: { id: '2' } }
    ],
    fallback: 'blocking'
  };
}

export async function getStaticProps({ params }) {
  const res = await fetch(`https://api.example.com/products/${params.id}`);
  const product = await res.json();
  return {
    props: { product },
    revalidate: 60
  };
}

export default function Product({ product }) {
  return (
    <div>
      <h1>{product.name}</h1>
      <p>Price: ${product.price}</p>
    </div>
  );
}
Notice there is no router.isFallback check here — it isn't needed. The component only ever renders once product is already available, because the browser doesn't get a response until the full HTML is ready.

First Visit to /products/3 — Timeline

t=0ms     Browser requests /products/3
t=1ms     Next.js has no pre-built page for "3"
t=1ms     Server does NOT respond yet — browser tab shows a spinner
t=350ms   getStaticProps() finishes fetching product 3
t=351ms   Full HTML is generated on the server
t=352ms   Complete "Product 3" HTML is sent to the browser
t=352ms   This HTML is cached on the server

Every Visit After That

/products/3  →  Product 3 page served instantly (from cache)
Use fallback: 'blocking' when: SEO matters and you don't want search engine crawlers or users to ever see a "Loading..." state — e.g. news articles, product pages that get shared on social media, anything where the crawler needs finished HTML immediately.

When to Use Which — Decision Guide

Situation Recommended fallback
You know every page ahead of time (30 blog posts, 10 marketing pages) false
Millions of product pages, most rarely visited, UX can tolerate a brief loading flash true
Pages get shared on social media / indexed by Google and must never show "Loading..." 'blocking'
You want unknown pages to behave like a normal 404 for security/product reasons false

Interview Tips

Key misconception to call out: fallback does NOT affect pages already listed in paths — those are always pre-built at compile time regardless of the fallback value. It only changes what happens for paths NOT in that list.
One-line definitions to memorize:
  • fallback: false → unknown pages return a 404, getStaticProps never runs for them
  • fallback: true → unknown pages show a loading state first, then render once data arrives
  • fallback: 'blocking' → unknown pages wait server-side and only render fully-formed HTML, no loading state
Both true and 'blocking' cache the generated page after the first request, just like revalidate in ISR (Incremental Static Regeneration). The difference between them is purely about what the FIRST visitor experiences — not about long-term caching behavior.
