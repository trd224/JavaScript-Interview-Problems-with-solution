24. How do you use React.cache for per-request memoization of non-fetch calls?

React.cache() is used to memoize the result of a function during a single server request.

In simple words, React.cache() prevents the same non-fetch function from running again and again during the same render request.

Why Do We Need React.cache?

Next.js automatically deduplicates identical fetch() calls, but it does not automatically memoize custom functions like database queries, Prisma calls, MongoDB calls, or session functions.

Fetch Deduplication Works Automatically
await fetch('https://api.myshop.com/user');
await fetch('https://api.myshop.com/user');
await fetch('https://api.myshop.com/user');

Next.js can make only one request for the same fetch call during one render.

But Non-Fetch Calls Are Different
await getUser();
await getUser();
await getUser();

Without React.cache(), this custom function can run multiple times.

Basic Example Without React.cache

lib/user.ts
export async function getUser() {
  console.log('Database query running...');
  return db.user.findFirst();
}
Header.tsx
import { getUser } from './lib/user';

export default async function Header() {
  const user = await getUser();
  return <h1>Hello {user.name}</h1>;
}
Sidebar.tsx
import { getUser } from './lib/user';

export default async function Sidebar() {
  const user = await getUser();
  return <p>{user.email}</p>;
}
Profile.tsx
import { getUser } from './lib/user';

export default async function Profile() {
  const user = await getUser();
  return <p>{user.role}</p>;
}

If all three components render in the same request, getUser() can run three times.

Without cache()
Header calls getUser()
        ↓
Database query


Sidebar calls getUser()
        ↓
Database query


Profile calls getUser()
        ↓
Database query

Using React.cache

lib/user.ts
import { cache } from 'react';

export const getUser = cache(async () => {
  console.log('Database query running...');
  return db.user.findFirst();
});

Now, during the same server request, repeated calls to getUser() reuse the first result.

With cache()
Header calls getUser()
        ↓
Database query runs once


Sidebar calls getUser()
        ↓
Reuses cached result


Profile calls getUser()
        ↓
Reuses cached result

Full Page Example

app/dashboard/page.tsx
import Header from './Header';
import Sidebar from './Sidebar';
import Profile from './Profile';

export default function DashboardPage() {
  return (
    <>
      <Header />
      <Sidebar />
      <Profile />
    </>
  );
}

All components can call getUser(), but the database query runs only once per request.

Important: Per-Request Memoization

React.cache() does not cache forever.

React invalidates memoized cache for each new server request. Each request gets its own cache scope.
Request Flow
Request 1
   ↓
getUser() runs once
   ↓
Other calls reuse result


Request 2
   ↓
getUser() runs once again
   ↓
Other calls reuse result


Request 3
   ↓
getUser() runs once again
   ↓
Other calls reuse result

So React.cache() is not long-term caching like unstable_cache() or fetch(..., { next: { revalidate } }).

Example With Arguments

React.cache() memoizes based on function arguments.

lib/product.ts
import { cache } from 'react';

export const getProduct = cache(async (id: string) => {
  console.log('Product query running:', id);
  return db.product.findUnique({ where: { id } });
});
Usage
await getProduct('1');
await getProduct('1');
await getProduct('2');
Result
getProduct('1') → runs once
getProduct('1') → reuses result
getProduct('2') → runs separately

When Should We Use React.cache?

  • Same database query is used in multiple Server Components.
  • Same session check is needed in layout, page, and components.
  • Same custom async function may run multiple times in one request.
  • We want request-level memoization, not long-term caching.

Session Example

lib/session.ts
import { cache } from 'react';
import { cookies } from 'next/headers';

export const getSession = cache(async () => {
  const token = (await cookies()).get('token')?.value;
  if (!token) return null;

  return verifyToken(token);
});
Usage In Multiple Components
const session = await getSession();

Now layout, page, and child components can call getSession() without repeating the same verification work during the same request.

React.cache vs Fetch Memoization vs unstable_cache

Feature React.cache Fetch Memoization unstable_cache
Works With fetch() Possible, but usually not needed ✅ Automatic Possible
Works With DB Queries ✅ ❌ ✅
Lifetime One request One render/request Across requests
Good For Session Checks ✅ ❌ Sometimes
Long-Term Cache ❌ ❌ ✅
Manual Revalidation ❌ ❌ ✅

When Not To Use React.cache?

  • When using normal fetch(); Next.js already memoizes identical fetch requests.
  • When we need persistent cache across users or requests.
  • When data should be shared for minutes or hours.
  • When we need tag-based revalidation.

One-Line Definition:
React.cache() memoizes non-fetch function results during a single server request, so repeated calls with the same arguments reuse the same result instead of running the function again.

React documentation notes that cached functions are invalidated for each server request, and Next.js documentation also describes React.cache as scoped to the current request only.
