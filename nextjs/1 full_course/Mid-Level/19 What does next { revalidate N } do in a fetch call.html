19. What does next: { revalidate: N } do in a fetch call?

The next.revalidate option tells Next.js how long fetched data should stay in cache before being refreshed.

next: { revalidate: N } enables ISR (Incremental Static Regeneration). Data is cached and automatically refreshed after N seconds.

Basic Example

Products Page
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products', { next: { revalidate: 60 } });
  const products = await res.json();

  return <h1>Products</h1>;
}

This means: 

Cache Product Data
        ↓
Keep Cache For 60 Seconds
        ↓
Refresh Cache Automatically

How It Works

Timeline
0 Seconds
    ↓
Fetch API
    ↓
Save Data In Cache

30 Seconds
    ↓
Use Cached Data

50 Seconds
    ↓
Use Cached Data

60 Seconds
    ↓
Cache Becomes Stale

Next Request
    ↓
Fetch New Data
    ↓
Update Cache

Real Example

Suppose the API initially returns:

Initial API Response
{
  "price": 100
}

After 30 seconds the database changes:

Database Update
{
  "price": 120
}

User Visits At 45 Seconds

Result
Price = 100

Because the cache is still valid.

User Visits At 61 Seconds

Result
Price = 120

Because Next.js regenerates the cache after it becomes stale.

Visual Comparison

Option Behavior Rendering Type
cache: 'force-cache' Cache Forever SSG
next: { revalidate: 60 } Refresh Every 60 Seconds ISR
cache: 'no-store' Fetch Every Request SSR

Difference From force-cache

force-cache
const res = await fetch(url, { cache: 'force-cache' });

Data remains cached indefinitely until a rebuild or manual revalidation occurs.

Difference From no-store

no-store
const res = await fetch(url, { cache: 'no-store' });

Data is fetched again on every request.
