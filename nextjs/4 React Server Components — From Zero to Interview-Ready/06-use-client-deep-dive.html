'use client' Deep Dive

What changes (and what doesn't) when you convert a page to a Client Component — the RSC Payload doesn't disappear.
Explained in plain language, with real code examples and diagrams.

01Converting a Page to a Client Component

app/dashboard/page.tsx
'use client';

export default function Dashboard() {
  console.log("Dashboard Loaded");
  return <h1>Dashboard</h1>;
}

Adding that one directive at the top completely changes how this file is treated by Next.js — even though the JSX looks almost identical to a Server Component.

02Does the RSC Payload Disappear Now?

No — and this surprises a lot of people. Even a page that's entirely a Client Component still goes through the RSC system. Next.js still fetches an RSC Payload for it on navigation; the payload just now describes "mount this Client Component here" instead of describing static HTML output.

03Why the App Router Still Requests RSC for a Client Page

  • Consistency. Every route — Server or Client — flows through the exact same navigation mechanism. There's no special-case "skip RSC" path for Client Component pages.
  • The layout around it might still be a Server Component. Even a fully client-side Dashboard likely sits inside a Server Component layout (navbar, sidebar) — the RSC Payload is what ties both pieces together correctly.
  • The router needs to know what to mount, and where. The payload still carries the routing information — which segment changed, what needs to be placed into the page.

04What the Payload Looks Like Now — Client References

Instead of inlining rendered output, the payload for a Client Component page contains a reference, roughly like this (simplified):

Simplified payload for a 'use client' page
2:I[39756,["/_next/static/chunks/dashboard-abc123.js"],"default"]
0:["$","$2",null,{}]

I[...] is the import reference — it tells the browser exactly which JS file contains the compiled Dashboard component, so it knows what to fetch.

05JS Chunk Generation for a Client Page

Dashboard marked 'use client'
        │
        ▼
Next.js compiles Dashboard into a JS chunk
   (bundled together with anything it imports and shares
    with other Client Components on the same route)
        │
        ▼
Chunk gets a generated, hashed filename
   e.g. dashboard-abc123.js  (NOT dashboard.js — more on this in Part 7)
        │
        ▼
RSC Payload references this chunk by its real name

06The Full Client Rendering Flow

1. Navigate to /dashboard (now a Client Component)
2. Browser sends GET /dashboard?_rsc=...   (still an RSC fetch!)
3. Server responds with a payload containing a client reference
4. Browser checks: is dashboard's JS chunk already downloaded?
     No  → fetch it now
     Yes → skip straight to step 5 (already cached from earlier)
5. React mounts  in the browser using that JS
6. console.log("Dashboard Loaded") fires — this time in the
   BROWSER console, not the terminal, because the whole
   component now runs client-side
Notice the flip: back when Dashboard was a Server Component, its console.log printed in your terminal (server-side). The moment you add 'use client', the exact same line of code starts printing in the browser console instead — because the entire function now executes there.

07Part 6 — Interview Questions

Q: If I add 'use client' to a page, does Next.js stop sending an RSC Payload for it?
No. It still sends one — the payload now contains a client reference pointing to the compiled JS chunk, instead of describing static rendered output.
Q: Where does a console.log inside a Server Component print? What about a Client Component?
Server Component: in your terminal (it ran on the server). Client Component: in the browser console (it runs there after hydration/mounting).
Q: Can a 'use client' page still be inside a Server Component layout?
Yes — this is extremely common. The layout (navbar/sidebar) stays a Server Component while the page content underneath is a Client Component.
Q: Why does the payload use a "reference" instead of embedding the Client Component's real code?
Because the code itself needs to be downloaded and run by the browser as actual JavaScript — the payload's job is just to describe the tree structure and point to where that code lives.
