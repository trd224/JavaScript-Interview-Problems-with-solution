Initial Load vs Client Navigation

Hard navigation vs soft navigation, side by side — where HTML comes from, and what those ?_rsc= requests really are.
Explained in plain language, with real code examples and diagrams.

01Initial Page Load — the Full Picture

This is a "hard navigation" — typing a URL, hitting refresh, or clicking a link that comes from outside your app. It's the only case that behaves like the traditional web. 

Browser types URL and hits Enter
        │
        ▼
Full document request sent
        │
        ▼
Server renders the whole page: HTML + embedded RSC Payload
        │
        ▼
Browser replaces the entire document with the new one

02Browser → Server, Step by Step

  1. Browser sends GET /about with normal headers (Accept: text/html).
  2. Next.js matches the route, runs every Server Component along the path (layouts, then the page).
  3. Any Client Components found are also rendered on the server for their initial HTML — but marked for hydration.
  4. The finished HTML + embedded RSC Payload is streamed back as the response.

03The HTML Response

What comes back looks like a completely normal HTML document — you could view-source it like any website. The difference is what's tucked inside: those __next_f.push(...) script tags carrying the RSC Payload, sitting quietly at the bottom of the body.

04Why the HTML Contains Flight Data

The HTML contains the Flight (RSC) data so the browser gets everything it needs in a single response. It immediately displays the page using the HTML, and at the same time already has the Flight data that React will use later for hydration and making the page interactive. If the Flight data came in a separate request, the browser would have to wait for another network call before React could continue, making the first page load slower. By including both HTML and Flight data together, Next.js makes the initial page load faster and more efficient.

05Clicking a <Link> — Where Everything Changes

Now for the interesting part. When you're already inside the app and click a <Link href="/dashboard">, none of the above happens again.

Click 
        │
        ▼
App Router's client-side router intercepts the click
        │  (the browser itself never reloads anything)
        ▼
A background fetch request is sent for the RSC Payload ONLY
        │
        ▼
Server runs Dashboard's Server Component(s)
        │
        ▼
Returns RSC Payload (no HTML this time)
        │
        ▼
React updates the DOM directly, in place

06What Are Those ?_rsc= Requests?

Open your Network tab during a client-side navigation and you'll see something like:

Network tab entry
GET /dashboard?_rsc=k34K8mlpxEk9AWlg
  • _rsc — a marker Next.js adds itself, meaning "this is a client-side RSC fetch, don't send HTML."
  • The random value — a short hash used as a cache key, so the client-side Router Cache can store multiple payload variants for the same path without mixing them up.
Seeing the same path (e.g. /dashboard) with several different _rsc hashes over time doesn't mean the page reloaded repeatedly — each was a separate lightweight payload fetch, possibly triggered by prefetching, not a full page reload.

07It's Just a Fetch/XHR Request

Technically, this navigation request is a plain fetch() call made by Next.js's client-side router code — the same kind of network request your own code could make. In DevTools it's usually labeled fetch, not document. That label is the easiest way to tell hard navigation apart from soft navigation at a glance.

08Why There's No New Document Request

Restaurant analogy: you don't ask the whole restaurant to be rebuilt just because you want a second course — you just ask the waiter to bring the next dish. React already "owns" the page in the browser; it just needs the recipe card for the new dish (the RSC Payload), not an entirely new restaurant (a new HTML document).

09How the DOM Actually Gets Updated

  1. The new RSC Payload arrives, describing the Dashboard route's tree.
  2. React compares it against what's currently on screen.
  3. Only the parts that actually changed are patched into the DOM — shared layouts (like a navbar) are left completely untouched.
  4. Any Client Component state elsewhere on the page (an open modal, scroll position, a half-filled form) survives, because nothing was torn down.

10This Is What "SPA Navigation" Means Here

Put simply: the App Router behaves like a Single Page Application after the very first load. One real page load happens per visit; everything after that is small, targeted data fetches that patch the existing page instead of replacing it.
