JS Bundles, Turbopack & Chunking

Why there's no file named after your component, how chunks are built and shared, and how to find your code inside one.
Explained in plain language, with real code examples and diagrams.

01Why dashboard.js Doesn't Exist

A very common moment of confusion: you go looking in _next/static/chunks for a file named after your component, and it's nowhere to be found. Instead you see things like 1rxncug86bump.js or 25jzwbvzz6ghc.js.

The reason: Next.js doesn't create one JS file per page or per component. It runs your code through a build process that reorganizes everything based on what's actually used together — the result has no obligation to match your file names at all.

02Turbopack's Chunking Approach

If you're running npm run dev, you're likely using Turbopack — visible from chunk names like turbopack-0p5b0djru7djj.js in your page source. Turbopack builds and names chunks dynamically and more granularly than the older webpack bundler did, which is part of why the generated names look so random.

03Code Splitting

Code splitting means breaking your app into smaller pieces that load only when actually needed, instead of bundling everything into one giant file every visitor has to download upfront. 

Without code splitting:
   One huge bundle.js containing EVERY page's code
   → slow first load, even for a visitor who only sees Home

With code splitting:
   home-chunk.js     → loaded when visiting Home
   dashboard-chunk.js → loaded only when visiting Dashboard
   → each visitor downloads only what they actually need

04Shared Chunks

Components used on more than one route often get grouped into a shared chunk, downloaded once and reused everywhere it's needed.

Home     uses  
Dashboard uses  
        │
        ▼
Counter's code goes into a SHARED chunk
        │
        ▼
Visiting Home downloads it once.
Navigating to Dashboard afterward needs NO new download for Counter —
it's already sitting in memory.

05Dynamic Imports

You can explicitly tell Next.js to split a component into its own separate chunk, loaded only when it's actually rendered, using next/dynamic:

Lazy-loading a heavy component
import dynamic from 'next/dynamic';

const HeavyChart = dynamic(() => import('./HeavyChart'), {
  loading: () => <p>Loading chart...</p>,
});

This is useful for big, rarely-used pieces of UI (like a complex chart library) that you don't want slowing down every page's initial load.

06Analyzing the Network Tab for Chunks

When you navigate to a new route, watch the Network tab for new .js files appearing that weren't there before — that's your signal a new chunk was needed (usually because a new Client Component showed up that wasn't loaded yet).

Development mode is noisy — Hot Reload, HMR, React Refresh, and the Turbopack dev runtime all add extra requests that won't exist in production. Don't judge real bundle size from a dev-mode Network tab.

07How to Actually Find Your Component Inside a Chunk

  1. Open Chrome DevTools → Sources tab (not Network).
  2. Expand localhost:3000 → _next → static → chunks.
  3. Open the chunk file that loaded when you navigated to your route.
  4. Use Ctrl+F inside that file to search for a string unique to your component — some text you rendered, or a distinctive variable/function name.
You'll usually find it — minified as it is, your compiled component's source is genuinely in there, the chunk's name just doesn't reveal that on its own.

08Production vs Development Chunking

npm run devnpm run build + npm start
Chunks compiled on-demand, per route visitedEvery route's chunks pre-built ahead of time
Extra dev-only runtime code includedMinified, no dev tooling included
Noisier Network tabMuch cleaner, more predictable chunk loading

09A Practical Experiment You Can Try

Make a component deliberately heavy to spot its chunk easily
'use client';

export default function Dashboard() {
  console.log("Dashboard Loaded");
  const arr = Array.from({ length: 10000 });
  return <h1>Dashboard</h1>;
}
  1. Refresh Home first, so your Network tab starts clean.
  2. Click into Dashboard and watch for a new JS chunk appearing.
  3. It still won't be named dashboard.js — but you've now confirmed, with your own eyes, exactly when and why a new chunk gets downloaded.
