React Server Components (RSC) Fundamentals

The default rendering model of the App Router — what a Server Component is, why it exists, and how it turns into HTML.
Explained in plain language, with real code examples and diagrams.
The one analogy we'll use for this entire book: think of your Next.js app as a restaurant. The server is the kitchen — it has the ingredients (your database, APIs, secret keys) and the chef (your code). The browser is the dining table. HTML is the plated food — ready to look at and eat immediately. The RSC Payload is the order ticket / recipe card that travels with the food, so the waiter (React) knows exactly what's on the plate and can rearrange it later without going back to the kitchen for everything.

01What Is RSC?

RSC stands for React Server Component. In simple words: it's a React component that only ever runs on the server — never in the browser.

Easiest possible definition: a Server Component is a component whose job finishes on the server. It reads data, builds some UI, and hands off the result. It never runs again inside the browser.
app/page.tsx — this is a Server Component by default
export default async function Home() {
  const res = await fetch("https://api.example.com/users");
  const users = await res.json();
  return 
We have {users.length} users
;
}

Notice something important: you never wrote 'use client' anywhere. In the App Router, every component is a Server Component unless you say otherwise. That's the whole trick — RSC isn't something you opt into, it's the default you opt out of.

02Why RSC Was Introduced

Before RSC, every single component — even ones that just displayed static text — had to be turned into JavaScript, sent to the browser, and run there. That caused three real problems:

  • Too much JavaScript. Even a simple "About Us" paragraph shipped as JS code the browser had to download, parse, and execute.
  • Secrets leaked easily. Database queries or API keys used inside a component could accidentally end up visible in the browser bundle.
  • Slower first paint. The browser had to download JS, run it, THEN show you anything — instead of just showing you ready-made content.
Restaurant analogy: imagine every customer had to be handed raw ingredients and a recipe, and cook the meal themselves at the table. That's the old way (Client Components doing everything). RSC lets the kitchen cook the meal fully and just serve you the plate — much faster, and the kitchen's secret recipes stay in the kitchen.
In one line: RSC exists so that only the parts of your UI that truly need interactivity (buttons, forms, dropdowns) ship JavaScript to the browser — everything else is just delivered as ready-made output.

Simple rule of thumb: if a component needs useState, useEffect, onClick, or any browser-only API (like window), it must be a Client Component. If it only reads data and displays it, keep it a Server Component.

04How RSC Works (Under the Hood)

When a request comes in for a page, Next.js doesn't just run your function and grab the return value like a normal JavaScript function call. It runs your Server Component through React's special server renderer, which does two things at once:

  1. Turns your JSX into real HTML (so the browser has something to paint instantly).
  2. Turns your JSX into a compact, serialized description called the RSC Payload (so React in the browser knows the exact shape of the tree — more on this in Part 2).
Restaurant analogy: the kitchen doesn't just hand you food — it plates the food (HTML) and staples a ticket to the tray (RSC Payload) describing exactly what's on it, in case the waiter needs to swap out just one item later without re-cooking the whole meal.

06The Complete Lifecycle, Start to Finish

1. Browser requests a page              (GET /)
2. Server runs your Server Component(s)  (fetches data)
3. Server produces HTML + RSC Payload
4. Browser paints the HTML               (page visible!)
5. Browser downloads JS — ONLY for Client Components
6. React hydrates                        (Client Components become interactive)
7. Page is fully ready
Common confusion: step 4 (page visible) happens before step 6 (interactive). This gap is why a page can look ready but a button might not respond to a click for a few milliseconds — it hasn't hydrated yet.

07Walking Through Your Home Page Example

app/page.tsx
export default async function Home() {
  const res = await fetch("https://api.example.com/users");
  const users = await res.json();
  return 
We have {users.length} users
;
}

Let's trace exactly what happens when someone visits this page:

  1. The browser sends GET /.
  2. Next.js runs Home() — this code executes on the server, in Node.js, not in any browser.
  3. fetch() calls the API — this network request happens from the server, so your API key (if you had one here) never touches the browser.
  4. The function returns JSX — React turns this into both HTML and an RSC Payload.
  5. The browser receives a finished <h1>We have 10 users</h1> — no loading spinner needed, no client-side fetch required.

09How the HTML Is Generated

The HTML isn't written by you directly — React's server renderer walks your component tree and converts every JSX element into its matching HTML tag. <h1>We have 10 users</h1> becomes literally that string of HTML, byte for byte, ready for the browser to display without running a single line of JavaScript first.

Why this matters: this is what makes Next.js pages appear fast, and it's also why search engines can read your content easily — they see real HTML immediately, not an empty page waiting for JavaScript to fill it in.

10Why the Browser Receives HTML (Not Just Data)

You might wonder — if React can rebuild the UI from the RSC Payload alone, why bother sending HTML too on the first load?

  • Instant paint. The browser can display real HTML the moment it arrives — no JavaScript needs to run first.
  • Works even if JS is slow or fails. The content is already visible even before React "wakes up."
  • Good for SEO and crawlers. Search engines see fully-formed content immediately.
Restaurant analogy: the kitchen could just hand you a recipe card and let you cook it yourself at the table — but that's slow and you'd sit there hungry. Instead they hand you the finished plate and the recipe card together, so you can eat right away, and the waiter can still make small edits later using that card.

11Part 1 — Interview Summary

Q: What is a React Server Component?
A component that runs only on the server, never ships JavaScript to the browser, and produces static output that can't be made interactive.
Q: Is every component in the App Router a Server Component by default?
Yes — unless the file (or something it imports) starts with the 'use client' directive.
Q: Why was RSC introduced?
To reduce the amount of JavaScript sent to the browser, keep secrets/database access safely on the server, and make the first paint faster.
Q: What two things does rendering a Server Component actually produce?
HTML (for instant display) and an RSC Payload (a serialized description of the component tree, used by React on the client).
