Server Execution of Client Components & Why Navigation Skips Hydration

A worked example straight from real HTML output, followed by why the server renders Client Components at all, and why a client-side navigation never needs hydration.
Explained in plain language, with real code examples and diagrams.

01A Real Example — Reading Actual Output

Take a Client Component like this:

Users.jsx (Client Component)
'use client';

export default function Users() {
  const [users] = useState([ /* ...1 user... */ ]);

  return (
    <div className="abcd">
      {users.length}
    </div>
  );
}

Now look at the actual HTML the server sends back for this component, and what it tells us about everything happening behind the scenes.

02The HTML Line That Proves Server Execution

<div class="abcd">1</div> — this is the single most important line in the whole response.

It means the server already executed Users(), calculated users.length, got 1, and generated that exact <div> — all before any JavaScript reached the browser. So before the browser has downloaded a single script, it already has 1 painted on the screen.

03Why useEffect Didn't Run on the Server

If the response contained 10 instead of 1, that would mean a data-fetching effect had already run somewhere. But the HTML only shows 1 — the initial state — which proves that this never executed on the server:

useEffect(() => {
  loadUsers();
}, []);
This is exactly what React promises: effects are a browser-only concept. The server can render output, but it never runs useEffect.

04Reading the Script Tags

Further down the response there are tags like <script src="/_next/static/chunks/...js"></script> — these are the Client Component JavaScript bundles. Notice the order they appear in: the HTML already exists on the page before these scripts are even mentioned. It's never "download JS, then create HTML" — it's "HTML already exists, then JS starts downloading."

There's also a block like this:

self.__next_f.push(...)

with entries conceptually shaped like 5:I[69231, ["/_next/static/chunks/..."], "default"]. This is the RSC Payload. It is not rendered HTML — the HTML already exists above it. This block only tells React that a Client Component exists, which JS file its code lives in, and how to rebuild the React tree around it.

05After Hydration — useEffect Finally Fires

Eventually the browser downloads the JS, runs Users() again, gets users = [1 user] again, and compares its own render (<div>1</div>) against the existing DOM (<div>1</div>). It's a perfect match, so hydration succeeds. Only after hydration does React finally run useEffect — the fetch(...) call executes for the first time, ever, in the browser. The server never ran it. The response comes back with 10 users, setUsers(users) runs, and the page updates from 1 to 10.

07What the Browser Knows Before JS Runs

Before any JavaScript executes, the browser knows only <div class="abcd">1</div>. It does not know about useState, setUsers, useEffect, loadUsers, fetch, the users variable, any event handlers, React hooks, or the React Fiber tree.

Those all exist only after the browser downloads the Client Component JavaScript and executes Users() during hydration. The HTML shows what to display; the JavaScript teaches the browser how that UI actually works.

08Why Execute Client Components on the Server At All?

Take this page:

app/users/page.jsx (Server Component)
import Users from "./Users";

export default function Page() {
  return (
    <>
      <h1>Users</h1>
      <Users />
    </>
  );
}

When the browser requests /users, it has nothing yet. The server starts executing <Page />. It immediately knows how to render <h1>Users</h1> since that's a Server Component. Then it reaches <Users /> — a Client Component — and now the server has a choice.

09Option A vs Option B — SPA vs Next.js

Option A — don't execute it (classic SPA)

The server sends <h1>Users</h1> plus an empty <div id="root"></div>. The browser sees "Users" followed by blank space, then downloads JS, executes Users(), and only then creates <div>1</div>. This is exactly how Create React App works.

Option B — execute it once on the server (Next.js)

The server runs Users() itself. Inside, useState(1) gives count = 1, and Users returns <div>1</div>. The server now has the complete <h1>Users</h1><div>1</div> and sends it immediately — the browser sees "Users 1" with zero JavaScript downloaded yet.

This is the biggest reason the user sees the finished content immediately, instead of a blank placeholder followed by a delayed pop-in once JS finishes loading.

10Why Execute the Component Twice?

HTML is just text — it can't carry logic. The browser's copy of <div>1</div> doesn't know about useState, setCount, onClick, useEffect, or that React even exists. So the browser downloads Users.js and executes Users() a second time, gets count = 1 again, and produces <div>1</div> again. React compares that against the existing HTML, finds a perfect match, and attaches its hooks, state, effects, and event listeners — this whole process is hydration.

First (server)Generate HTML quickly and send it to the browser<div>1</div>
Second (browser)Create the Fiber tree, restore hooks/state, attach events, hydrateInteractive React application

12The RSC Payload, Revisited

The HTML tells the browser what to display. But React also needs to know which parts are Server Components, which are Client Components, which JS file contains which component, and how to reconstruct the component tree — all of that travels separately as the RSC Payload.

ArtifactJob
HTMLWhat to display
RSC PayloadHow this page was built
Client JSHow to make it interactive

13Complete Mental Model — 10 Steps

1. Browser requests page
2. Server executes Server Components
3. Server executes Client Components (only to generate HTML)
4. Server sends HTML + RSC Payload
5. Browser immediately displays HTML
6. Browser downloads Client Component JS
7. Browser executes Client Components again
8. React rebuilds its Fiber tree
9. React hydrates the existing HTML (state, hooks, events)
10. The page becomes interactive
Two different goals, one component: server execution exists only to generate HTML quickly so the user isn't stuck waiting on JavaScript; browser execution exists to recreate React's internal state — Fiber tree, hooks, event handlers, effects — so the already-visible HTML becomes a live, interactive app.

14Client Navigation — React Is Already Running

Everything above describes the initial load, where the browser starts with nothing. A click on <Link href="/users"> from an already-hydrated page is a completely different situation, because the browser already has React, React DOM, the Next.js router, and a hydrated application running.

Click <Link>
  ↓
Next.js intercepts the click
  ↓
Prevents a full page reload
  ↓
Fetches the /users RSC Payload  (NOT HTML)

The server executes UsersPage() — Server Components, then Client Components — just like before. But this time it mainly sends back an RSC Payload, not HTML, containing something like Page → Users Component → props → component tree.

15Why No HTML Is Needed on Navigation

React already exists in the browser, so it reads the RSC Payload and rebuilds the new React tree directly. If the Client Component's JS isn't already downloaded, Next.js fetches it; if it's already cached, that step is skipped entirely. Then Users() runs in the browser, useState() initializes, JSX is returned, and React updates the DOM directly — a completely normal React render, no server-generated markup required.

16Why No Hydration Happens on Navigation

Hydration only happens when React finds HTML it didn't create itself — the case where the server produced <div>1</div> and the browser has to attach itself to markup it never wrote. After a client-side navigation, React itself is the one creating the new DOM nodes and inserting them — there's nothing foreign to attach to, so there's nothing to hydrate.

17The Painter Analogy

Think of React as a painter. On the initial load, there is no painting yet — the server paints the picture first, the browser shows that painting, and React then learns about it (hydration).

On navigation, React is already painting. The server just says "draw this new page," and React paints it itself — there's no need for server-generated HTML, and nothing to hydrate.

18The Rule to Remember

Why do we need two executions only on the initial render?
The server execution generates HTML for the first paint; the browser execution hydrates that HTML. On navigation there's no server-generated HTML to hydrate — React is already running, so it just executes the Client Component and updates the DOM directly.
Why not just skip server execution and download JS first?
You can — that's exactly what a classic SPA does. Next.js avoids it because the user would be stuck waiting for JavaScript before seeing anything useful.
Does a client-side navigation ever produce hydration?
No. Hydration is specifically for attaching React to markup it didn't create. On navigation, React creates the markup itself, so there's nothing foreign to attach to.
