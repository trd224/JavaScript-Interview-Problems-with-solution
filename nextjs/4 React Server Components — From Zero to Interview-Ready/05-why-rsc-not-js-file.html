Why a Page Fetches RSC Instead of a JS File

Why Server Components can't run in the browser, and the full request lifecycle when navigating to one.
Explained in plain language, with real code examples and diagrams.

01How Server Components Actually Execute

Every time you navigate to a route built from Server Components — like a Dashboard page — that component's function body runs again, fresh, on the server, for that specific request. It is never shipped to the browser as code, so it can never run there.

app/dashboard/page.tsx
export default async function Dashboard() {
  const data = await getDashboard(); // runs on the server, every time
  return <DashboardUI data={data} />;
}

02Why the Browser Cannot Execute Server Components

  • The code never gets sent. Next.js simply doesn't include Server Component source in the JS bundle — there's nothing to run.
  • It often can't run there anyway. Server Components regularly use things a browser has no access to — direct database clients, filesystem access, private environment variables.
  • Security. Even if it were technically possible, shipping that code would expose secrets and internal logic to anyone who opens DevTools.
Restaurant analogy: you can't cook the chef's secret recipe at your dining table — you don't have the kitchen's equipment or the ingredients, and honestly, the restaurant doesn't want you to have the recipe card either. You just get the finished dish.

03Why the App Router "Asks the Server" Instead of Running Locally

This is the direct answer to "why does clicking Dashboard fetch RSC instead of just running some dashboard.js file already in the browser." Because Dashboard is a Server Component, there's no version of it living in the browser to run — the only way to get its output is to ask the server to run it and send back the result.

04The RSC Request Lifecycle for a Navigation

User clicks a link to a Server Component route
        │
        ▼
Browser sends: GET /dashboard?_rsc=...
        │
        ▼
Server runs Dashboard() fresh — fetches its own data
        │
        ▼
Produces an RSC Payload describing the new UI
        │
        ▼
Browser receives the payload, React patches the DOM
        │
        ▼
No JS file for "Dashboard" was ever downloaded, because none exists

05What Happens When a Client Component Is Inside That Server Component

Real pages are rarely 100% Server Components. Say Dashboard renders a Client Component chart: 

Dashboard (Server Component)
   │
   ├── Sidebar (Server Component)
   ├── Header  (Server Component)
   └── Chart   (Client Component)

The RSC Payload for this page doesn't inline Chart's rendered HTML the normal way — it includes a client reference: "there's a Client Component here, its code lives in this specific JS file, and here are its props." That's the one and only piece of this whole page that needs any JavaScript downloaded at all.

06The JS Downloading Process, Specifically for Chart

  1. The RSC Payload arrives, containing Chart's client reference.
  2. The browser checks: "do I already have this JS chunk?" If yes (from a previous visit), skip straight to step 4.
  3. If not, the browser downloads just that one chunk — not Sidebar's code, not Header's code, because they don't have any client-side code to begin with.
  4. React hydrates Chart specifically, using the downloaded code and the props from the payload.
Result: Sidebar and Header remain permanently static HTML. Only Chart becomes interactive. This selective hydration is the entire reason Server + Client Component mixing exists — you pay the JS cost only for the pieces that truly need it.
