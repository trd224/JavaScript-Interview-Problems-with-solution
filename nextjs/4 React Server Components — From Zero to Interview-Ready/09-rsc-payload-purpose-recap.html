The Real Purpose of the RSC Payload

Clearing up the "RSC Payload = Hydration" myth, and whether the payload actually generates HTML when you navigate.
Explained in plain language, with real code examples and diagrams.

01The Most Misunderstood Line in App Router

A huge number of developers quietly believe one equation:

RSC Payload = Hydration
This is not correct. Hydration is only one of several jobs the payload does.

The real scope of the RSC Payload is much broader than "the thing that lets React hydrate the page." It's the single data format that also drives navigation, layout preservation, and streaming — this part puts all of that in one place.

02What the Payload Actually Is

The RSC Payload is React's serialized description of the Server Component tree and the locations of Client Components inside it. Think of it as React's internal data format — a way for the client to understand exactly what the server rendered and how it was structured.

It is not HTML. It is not JavaScript. It is not a replacement for either one — it's a third thing that sits alongside them.

03Rebuilding the Component Tree — the Primary Purpose

Say the server executes something like:

Server Component tree
<Layout>
  <Header />
  <Home />
  <Counter />
</Layout>

But the browser only ever sees flattened output like:

HTML received by the browser
<main>
  <h1>Home</h1>
  <button>0</button>
</main>

From that HTML alone, there's no way to know which part came from Layout, which came from Header, which component is a Client Component, or where the boundaries between them even are. The RSC Payload is what preserves all of that structural information after the HTML has already been flattened.

04Locating the Client Components

Suppose one child in that tree is marked with 'use client' — say, a Counter. The payload effectively encodes something like:

Home
├── Header (Server)
├── Counter (Client)
└── Footer (Server)

From this, React knows exactly which node needs to become interactive, which JavaScript needs to load, and which surrounding nodes can be left alone. Without the payload, React would have no reliable way to know where Counter should be mounted.

05Pointing at the Right JS Modules

The payload also carries references to the actual Client Component modules, conceptually like:

Counter
  ↓
Module ID 25
  ↓
counter.js

React uses these module references to know exactly which JavaScript bundle to request — it isn't guessing based on component names, it's following an explicit pointer that was generated at build/render time.

06Powering Client-Side Navigation

This is one of the biggest reasons the payload exists at all. Say you're on / and click a link to /dashboard. Next.js does not request a brand-new HTML document. Instead it requests something like /dashboard?_rsc=..., and the server responds with only the new RSC Payload — no document, no full page.

React then compares the current component tree against the new one it just received, and updates only what actually changed. That diffing step, running entirely off payload data, is why App Router navigation feels instant.

07Preserving Layouts Across Navigation

Imagine a layout shaped like this:

Layout
├── Sidebar
└── Page

You navigate from Home to Dashboard. The sidebar hasn't changed at all. The new RSC Payload effectively tells React:

Sidebar → Reuse
Page    → Replace

React updates only the page content and leaves the sidebar mounted exactly as it was. This layout-preservation behavior is one of the App Router's biggest practical advantages over a full page reload model.

08Streaming With Suspense

If part of a page is slow — wrapped in something like <Suspense fallback={<Loading />}><Products /></Suspense> — the server doesn't have to wait for everything to finish before responding. It can send the payload in pieces:

Header
  ↓
Sidebar
  ↓
Loading...
  ↓
Products (arrives later)

React progressively updates the UI as each new chunk of payload arrives, instead of blocking the whole page on the slowest piece of data.

09What the Payload Is NOT

It does not display HTML — HTML does that.
It does not execute JavaScript — the browser's JS engine does that.
It does not hydrate the page by itself — React's client runtime performs hydration, using the payload as its map.

10The IKEA Instruction Manual Analogy

HTML is the finished chair — you can see it, sit in it, look at it.

JavaScript is the worker — the one who actually builds or updates things.

The RSC Payload is the instruction manual — "Part A connects to Part B," "use screw X," "attach leg C."

The instructions themselves don't build the chair. The worker reads the instructions and does the building. In the same way, the payload doesn't render anything on its own — React's runtime reads it and does the actual work.

React JS
  +
RSC Payload
  ↓
Hydration
  ↓
Navigation
  ↓
UI Updates

11Myth: "The Payload Generates HTML on Navigation"

A very natural but slightly-off way to describe navigation is: "the RSC Payload generates HTML on the client when you navigate from the home page to the dashboard." That's close, but the wording hides an important distinction worth being precise about.

When you click a <Link href="/dashboard">, the browser does not request a new HTML document at all. It requests something like GET /dashboard?_rsc=abc123, and the server responds with an RSC Payload — no HTML document is returned in that response.

12What Really Happens on a Soft Navigation

React receives the new payload, reads a tree conceptually like Dashboard → Sidebar, UserInfo, Chart, and compares it against the tree currently on screen (Home). The flow looks like this:

RSC Payload
  ↓
React reconciles the new component tree
  ↓
React updates the Browser DOM
  ↓
Browser displays Dashboard

Notice who's doing the work: React updates the DOM. The payload is only the data React reads to decide how. The browser does create real DOM nodes representing the new UI, but React isn't assembling an HTML string like <h1>Dashboard</h1> and handing it to a parser — it's issuing direct operations: create this node, set this text, insert here, remove that old node. That's a fundamentally different (and cheaper) operation than parsing a fresh HTML document.

The corrected sentence: the RSC Payload tells React what the new UI should look like — React then updates the existing DOM on the client without downloading a new HTML document.

14The Mental Model to Keep

Does the RSC Payload generate HTML on navigation?
Not directly. It describes the new UI; React's runtime is what actually updates the DOM based on that description — no new HTML document is ever parsed.
Why does this distinction matter?
It's exactly why App Router navigation is fast: no new document request, no HTML parsing — just targeted DOM operations driven by payload data.
Is the RSC Payload the same thing as hydration?
No. Hydration is one consumer of the payload. The payload also drives navigation, layout preservation, module loading, and Suspense streaming.

15One Sentence to Remember Forever

The HTML is for the browser to display the page, the JavaScript is for executing React, and the RSC Payload is for describing the server-rendered React component tree so React can hydrate Client Components, perform client-side navigation, preserve layouts, and stream updates efficiently.
