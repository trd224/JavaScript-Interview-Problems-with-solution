Link Prefetching

How quietly fetches pages before you click, the Router Cache, and dev vs production differences.
Explained in plain language, with real code examples and diagrams.

01Automatic Prefetching

The App Router doesn't wait for you to click a link before it starts working. By default, <Link> components automatically fetch the RSC Payload for their destination ahead of time, so that by the time you actually click, the data is already sitting in memory.

This alone triggers prefetching
<Link href="/dashboard">Dashboard</Link>

02Viewport Detection — How Next.js Decides When to Prefetch

Next.js uses the browser's IntersectionObserver API to detect when a <Link> scrolls into view. The moment it becomes visible on screen, a background prefetch request fires — you don't need to hover or click anything.

03The Router Cache

The Router Cache is an in-memory store, living in the browser, that holds RSC Payloads you've already fetched — whether from an actual navigation or a prefetch. It's what makes repeat visits to a route (like clicking Back) often completely free of any server request.

Simple meaning: think of it as a short-term memory of "pages I've already asked the kitchen about recently" — no need to ask again if the answer is still fresh.

04The Prefetch Lifecycle

1. Link becomes visible in viewport
2. Background prefetch request sent (low priority, doesn't block anything)
3. RSC Payload received and stored in the Router Cache
4. Time passes... (cache entry has a "freshness" window)
5. User clicks the link
6a. Cache still fresh  → instant navigation, ZERO new network request
6b. Cache expired      → a fresh RSC fetch happens, same as a normal click

05Turning Prefetch Off — prefetch={false}

Disabling prefetch for one link
<Link href="/dashboard" prefetch={false}>
  Dashboard
</Link>

You might do this for routes that are rarely visited, extremely expensive to render, or where you specifically want to avoid unnecessary background server calls (for example, a heavy admin page most users never open).

06Navigating Without Prefetch

If prefetching is disabled, or simply hasn't finished yet by the time you click, nothing breaks — Next.js just falls back to fetching the RSC Payload at click time, exactly like Scenario 2 from Part 3. The only difference is the user waits slightly longer, since the fetch starts only after the click instead of beforehand.

07How the Cache Gets Used on Click

SituationWhat happens on click
Prefetched and still freshInstant navigation using cached payload — no network request at all
Prefetched but staleFresh RSC fetch happens, same as a normal click
Never prefetched (prefetch=false, or link never scrolled into view)RSC fetch starts only now, after the click

09Development vs Production Behavior

Important: prefetching and caching behave noticeably differently between npm run dev and a production build. Development mode intentionally limits caching so you always see your latest code changes — don't judge real-world prefetch behavior from dev mode alone.
npm run devnpm run build + npm start
Prefetching is limited/less aggressivePrefetching is fully active for static routes
Router Cache entries expire quickly, by designCache behaves per the configured staleTimes
Every navigation tends to hit the server freshMany navigations are served instantly from cache
Practical tip: if you're testing or studying prefetch/cache behavior, always do it against a production build (npm run build && npm start) — dev mode will give you a misleading picture.
