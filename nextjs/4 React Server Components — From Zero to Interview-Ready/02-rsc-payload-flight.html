The RSC Payload (React Flight)

The data format underneath every Server Component response — Flight, self.__next_f.push(...), and how hydration uses it.
Explained in plain language, with real code examples and diagrams.

01What Is the RSC Payload?

The RSC Payload is a special, compact format that describes your component tree — not as HTML tags, but as a structured description React can read and rebuild in the browser. It's often called Flight data.

Easiest definition: if HTML is "the picture," the RSC Payload is "the instructions for drawing the picture again, piece by piece." React uses those instructions to know exactly which part of the page is which component, without needing to guess from raw HTML.

02The React Flight Protocol

"Flight" is React's internal name for the wire format used to stream Server Component output to the client. It's not JSON exactly, and not HTML — it's a custom, line-by-line, streaming format designed specifically for this job.

A tiny simplified look at Flight-style output
0:["$","div",null,{"className":"abc","children":10}]

Read this like a small program: "$" means "this is a React element," "div" is the tag name, null is the key, and the object holds the props. React can turn this single line back into a real DOM node.

03Why the RSC Payload Exists

  • Hydration needs it. React needs to know exactly which DOM node belongs to which component, so it can attach interactivity without throwing away and rebuilding the page.
  • Navigation reuses it. When you click a link, Next.js fetches a fresh RSC Payload instead of a whole new HTML page — much lighter and faster.
  • It can describe Client Components too. Instead of embedding a Client Component's real code, the payload just says "there's a Client Component here, its code lives in this file, here are its props" — a reference, not the whole component.

04The Internal Payload Structure

A real payload is a series of numbered lines, where each line can reference other lines by their ID — like a small lookup table:

Simplified real-world example
1:"$Sreact.fragment"
2:I[39756,["/_next/static/chunks/abc.js"],"default"]
4:["$","div",null,{"className":"abc","children":10}]
PieceMeaning
1:, 2:, 4:IDs — other lines can reference these by number instead of repeating the data
"$"Marks "this is a React element"
I[...]An "import reference" — points to a Client Component's JS file so the browser knows what to download
The object at the endProps being passed into that element/component

05What Is self.__next_f.push(...)?

This is how Next.js delivers the RSC Payload during a first page load. Instead of sending it as a separate network request, it embeds the payload directly inside the HTML, as a series of small inline <script> tags.

What you'll actually see in "View Page Source"
<script>
  (self.__next_f = self.__next_f || []).push([0])
</script>
<script>
  self.__next_f.push([1, "4:[\"$\",\"div\",null,{\"className\":\"abc\",\"children\":10}]\n"])
</script>

Each push() call adds one more chunk of the payload to an array (self.__next_f). Next.js can send these chunks one at a time as they become ready — which is what lets React start streaming in parts of a slow page before everything has finished loading.

06The Payload Lives Inside the HTML

Restaurant analogy: on your very first visit, the waiter doesn't make two separate trips — one for the food, one for the recipe card. They bring both on the same tray. That's exactly what happens here: on a first page load, there's no separate network request for the RSC Payload — it rides along inside the one HTML document, tucked into these script tags.
This is exactly why, if you check your Network tab after a hard reload, you won't find a request literally called "RSC payload" — look inside the document response itself, in the Response tab, and search for __next_f.
