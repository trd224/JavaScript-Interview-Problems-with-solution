Complete Mental Model

01The End-to-End Request Lifecycle

┌─────────── HARD NAVIGATION (first load / refresh / typed URL) ───────────┐
│ Browser → GET / → Server runs Server Components → HTML + RSC Payload    │
│         → Browser paints HTML → downloads Client Component JS only      │
│         → React hydrates only Client Components                         │
└────────────────────────────────────────────────────────────────────────┘

┌─────────── SOFT NAVIGATION ( / router.push, already loaded) ──────┐
│ Click → fetch /route?_rsc=... → Server runs Server Components again     │
│       → RSC Payload only (no HTML) → React patches DOM in place         │
│       → downloads any NEW Client Component JS chunk if needed           │
└────────────────────────────────────────────────────────────────────────┘

02The Browser's Role

The browser's job is surprisingly limited in this whole system: request pages, paint HTML immediately, download the small amount of JS it actually needs, and let React interpret the RSC Payload. It never decides how a Server Component should render — that decision was already made before anything reached it.

03The Next.js Server's Role

The server is where all the real work happens: running Server Components, fetching data, producing both HTML and the RSC Payload, and deciding what belongs in each response. It's also the only place your database credentials, secret keys, and private business logic ever exist.

04React's Role

React sits on both sides. On the server, it renders your components into HTML and Flight data. On the client, it reads that Flight data, reconciles it against the current DOM, and hydrates whatever needs to become interactive.

05RSC's Role — Recap

A Server Component: runs only on the server, ships zero JavaScript, produces static output that can never change on its own once rendered.

06Hydration's Role — Recap

Hydration: the process of attaching event listeners and state to existing HTML for Client Components only, using the RSC Payload as the map of what goes where.

07Client Components' Role — Recap

A Client Component: runs on the server once (for the initial HTML, on hard navigations) and then again in the browser after hydration — this is the only component type that can use useState, useEffect, or event handlers.

08Server Components' Role — Recap

The default for every file in the App Router. Safe place for database calls, secret keys, and heavy computation that should never reach the browser.

09The Router Cache's Role

An in-browser, in-memory store of RSC Payloads you've already fetched or prefetched — makes repeat navigations (including Back/Forward) often instant, with zero server contact.

10The Prefetch Cache's Role

Populated automatically when a <Link> scrolls into view — silently fetches the destination's RSC Payload ahead of time so clicking feels instant.

11The Browser Cache's Role

Standard HTTP/disk caching for static assets — JS chunks, CSS, fonts, images — separate from the Router Cache, which is specifically about RSC Payloads, not static files.

13Interview Cheat Sheet

Server Component vs Client Component — the one-line difference?
Server: runs only on the server, ships no JS. Client: runs on server once then again in the browser, ships JS, can be interactive.
What does hydration mean?
Attaching event handlers/state to existing server-rendered HTML in the browser, without re-creating the DOM.
What is the RSC Payload?
A serialized (Flight-format) description of the component tree that React uses to hydrate and to update the UI on navigation.
Why don't navigations request a new HTML document?
Because React already owns the page — it only needs the RSC Payload for what changed, not a whole new document to replace everything.
What is ?_rsc= in the Network tab?
A marker + cache-key hash Next.js adds to fetch requests specifically asking the server for an RSC Payload instead of a full HTML page.
Why is there no file literally named after my component in _next/static/chunks?
Because Next.js/Turbopack performs code splitting, tree shaking, and chunk merging — the output is organized by what's used together, not by your source filenames.
Does a Client Component page still use the RSC system?
Yes — the payload just contains a client reference (pointing to its JS chunk) instead of static rendered output.
What decides whether Back/Forward hits the server?
Whether that route's RSC Payload is still fresh in the Router Cache (governed by the staleTimes setting).

14Summary

Every idea in this book comes back to one sentence: Next.js separates "what should this page look like" (decided on the server, described as HTML + an RSC Payload) from "what needs to become interactive" (Client Components, hydrated selectively in the browser). Navigation reuses the same RSC mechanism as the first load, just without the HTML — which is exactly why it feels instant.

15Conclusion

If you remember nothing else: a page reload and a server request are two completely different things. The server can run your Server Components and send back fresh data constantly, without the browser ever reloading the document — and that single idea is the foundation the entire App Router is built on.
