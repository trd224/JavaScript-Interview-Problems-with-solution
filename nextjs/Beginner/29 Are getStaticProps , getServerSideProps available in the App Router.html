29. Are getStaticProps() and getServerSideProps() Available in the App Router?

No. getStaticProps() and getServerSideProps() are only available in the Pages Router.

The App Router does not use getStaticProps(), getServerSideProps(), or getStaticPaths(). Instead, it uses Server Components, fetch caching options, and special exports such as revalidate and generateStaticParams().

Pages Router

pages/posts.js
export async function getStaticProps() {

  return {
    props: {}
  };

}

export async function getServerSideProps() {

  return {
    props: {}
  };

}

These APIs work only inside the Pages Router.

App Router

app/posts/page.js
export default async function Posts() {

  const res = await fetch(
    'https://api.example.com/posts'
  );

  const posts = await res.json();

  return (
    <h1>{posts.length}</h1>
  );

}

There is no getStaticProps() or getServerSideProps() here.

Why Were They Removed?

In the Pages Router, Next.js needed separate APIs:

Purpose API
SSG getStaticProps()
SSR getServerSideProps()
Dynamic SSG getStaticPaths() + getStaticProps()

In the App Router, data fetching is built directly into Server Components.

Instead of choosing special functions, we control rendering behavior through caching options.

App Router Replacement for SSG

Static generation happens automatically when data can be cached.

SSG Style
async function getPosts() {

  const res = await fetch(
    'https://api.example.com/posts'
  );

  return res.json();

}

export default async function Posts() {

  const posts = await getPosts();

  return (
    <h1>{posts.length}</h1>
  );

}

Next.js may generate this page during the build process.

App Router Replacement for SSR

SSR Style
async function getPosts() {

  const res = await fetch(
    'https://api.example.com/posts',
    { cache: 'no-store' }
  );

  return res.json();

}

export default async function Posts() {

  const posts = await getPosts();

  return (
    <h1>{posts.length}</h1>
  );

}

The cache: 'no-store' option forces fresh data on every request.

This is equivalent to SSR.

App Router Replacement for ISR

ISR Style
export const revalidate = 60;

async function getPosts() {

  const res = await fetch(
    'https://api.example.com/posts'
  );

  return res.json();

}

export default async function Posts() {

  const posts = await getPosts();

  return (
    <h1>{posts.length}</h1>
  );

}

This regenerates the page when a request arrives after 60 seconds.

App Router Replacement for getStaticPaths()

generateStaticParams()
export async function generateStaticParams() {

  return [
    { id: '1' },
    { id: '2' },
    { id: '3' }
  ];

}

This replaces getStaticPaths().

Mapping Old APIs to New APIs

Pages Router App Router
getServerSideProps() Server Component + fetch(..., { cache: 'no-store' })
getStaticProps() Server Component + fetch() (default cache)
getStaticPaths() generateStaticParams()
ISR via revalidate Server Component + export const revalidate

Interview Tip

A common misconception is that getStaticProps() and getServerSideProps() still work in the App Router. They do not. The App Router uses Server Components and fetch caching strategies instead of these special data-fetching functions.
One-Line Definition:
No, getStaticProps() and getServerSideProps() are not available in the App Router. Their functionality is replaced by Server Components, fetch caching options, revalidate, and generateStaticParams().
