What is Server-Side Rendering (SSR) in Next.js?

Server-Side Rendering (SSR) is a rendering technique where Next.js generates the HTML for a page on the server for every request before sending it to the browser.

SSR allows us to fetch fresh data on every request and send a fully rendered HTML page to the browser.

Why Do We Need SSR?

Some pages require the latest data every time a user visits them.

Examples:
  • Stock Market Prices
  • Weather Data
  • Live Dashboard
  • News Websites
  • E-commerce Product Inventory

For such pages, generating HTML at build time is not enough because the data changes frequently.

How SSR Works

SSR Flow
User Requests Page
        │
        ▼
Next.js Server
        │
        ▼
Fetch Latest Data
        │
        ▼

        │
        ▼

        │
        ▼
Hydration Happens

Real Example

Suppose our application displays the current Bitcoin price.

User Opens
/bitcoin-price

Next.js performs:

Server
1. Fetch Latest Bitcoin Price
2. Generate HTML
3. Send HTML To Browser

Every user gets the latest data.

SSR in the Pages Router

In the Pages Router, SSR is implemented using getServerSideProps().

pages/products.js
export async function getServerSideProps() {

  const res = await fetch(
    'https://api.example.com/products'
  );

  const products = await res.json();

  return {
    props: { products }
  };

}

export default function Products({ products }) {

  return (
    <h1>
      Total Products: {products.length}
    </h1>
  );

}

What Happens Internally?

Request Flow
Request #1
      │
      ▼
getServerSideProps()
      │
      ▼
Fresh Data
      │
      ▼
HTML Generated

Request #2
      │
      ▼
getServerSideProps()
      │
      ▼
Fresh Data Again
      │
      ▼
New HTML Generated

The function runs for every request.

SSR in the App Router

The App Router does not use getServerSideProps().

Instead, Server Components can directly fetch data.

app/products/page.js
async function getProducts() {

  const res = await fetch(
    'https://api.example.com/products',
    { cache: 'no-store' }
  );

  return res.json();

}

export default async function Products() {

  const products = await getProducts();

  return (
    <h1>
      Total Products: {products.length}
    </h1>
  );

}

Why cache: 'no-store'?

The cache: 'no-store' option tells Next.js:

Meaning
Do Not Cache

Fetch Fresh Data
For Every Request

This behavior is equivalent to SSR.

Advantages of SSR

  • Always fresh data.
  • Excellent SEO.
  • Better social media previews.
  • Faster first content display.
  • Works well for dynamic content.

Disadvantages of SSR

  • Runs on every request.
  • Higher server load.
  • Slower than SSG.
  • Requires server resources.

When Should We Use SSR?

  • Frequently changing data.
  • User-specific dashboards.
  • Live statistics.
  • Stock prices.
  • Weather information.
  • Real-time inventory.

Interview Tip

A common misconception is that SSR means rendering on the server once. In reality, SSR generates fresh HTML on the server for every incoming request.
One-Line Definition:
Server-Side Rendering (SSR) is a rendering technique in Next.js where the server generates fresh HTML for every request before sending it to the browser.
