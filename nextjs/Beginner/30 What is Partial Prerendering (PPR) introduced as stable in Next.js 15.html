30. What is Partial Prerendering (PPR) Introduced as Stable in Next.js 15?

Partial Prerendering (PPR) is a rendering strategy that combines static rendering and dynamic rendering on the same page.

PPR allows us to prerender the static parts of a page at build time while dynamically rendering only the parts that require fresh data.

Why Was PPR Introduced?

Before PPR, we usually had two choices:

  • Generate the entire page statically (SSG).
  • Render the entire page dynamically (SSR).

This created a problem because many pages contain both static and dynamic content.

Example Problem

Suppose our homepage contains:

Homepage
Header
Hero Section
Features Section
Latest Notifications
Current User Data
Footer

Static content: 

Header
Hero Section
Features Section
Footer

Dynamic content: 

Latest Notifications
Current User Data

Before PPR: 

Entire Page = SSR

OR

Entire Page = SSG

There was no built-in way to mix both efficiently.

How PPR Solves This

PPR Page
Static HTML
      │
      ├── Header
      ├── Hero
      ├── Features
      └── Footer

Dynamic Islands
      │
      ├── Notifications
      └── User Profile

The static content is served immediately.

Dynamic sections are streamed and filled in later.

What Happens Internally?

PPR Flow
Build Time
      │
      ▼

      │
      ▼
Deploy

      │
      ▼

      │
      ▼

Example Component Structure

Dashboard Page
export default function Dashboard() {

  return (
    <>
      <Header />
      <Hero />

      <Suspense fallback="Loading...">
        <UserStats />
      </Suspense>

      <Footer />
    </>
  );

}

Static parts: 

Header
Hero
Footer

Dynamic part: 

UserStats

Why Suspense is Important?

PPR relies heavily on React Suspense boundaries.

Example
<Suspense fallback="Loading...">
  <UserStats />
</Suspense>

The Suspense boundary tells Next.js: 

Static Content Can Be Sent Now

UserStats Can Arrive Later

Without Suspense : 

❌ No boundary exists to separate static and dynamic content.
❌ PPR cannot create a static shell around UserStats.
❌ The route falls back to full dynamic rendering (SSR).

SSR vs SSG vs PPR

Feature SSR SSG PPR
Static Content ❌ ✅ ✅
Dynamic Content ✅ ❌ ✅
Page Generated Every Request Build Time Mixed
Performance Good Excellent Excellent
Fresh Data Always No Only Dynamic Parts

Benefits of PPR

  • Faster initial page load.
  • Better user experience.
  • Keeps dynamic content fresh.
  • Reduces server work.
  • Combines benefits of SSG and SSR.

Important Note

A common misconception is that PPR generates the entire page statically. In reality, PPR generates only the static shell and streams dynamic sections separately.

When Should We Use PPR?

  • Dashboards
  • E-commerce Websites
  • News Portals
  • Personalized Homepages
  • Applications with mixed static and dynamic content

Interview Tip

Think of PPR as "SSG for static sections + SSR for dynamic sections" on the same page.
One-Line Definition:
Partial Prerendering (PPR) is a Next.js 15 rendering strategy that prerenders static parts of a page while dynamically rendering and streaming only the sections that require fresh data.
