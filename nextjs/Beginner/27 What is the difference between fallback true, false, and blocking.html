What is the Difference Between fallback: true, false, and 'blocking'?

The fallback option is returned from getStaticPaths() and controls what Next.js should do when a user requests a dynamic page that was not generated during the build process.

The fallback option determines how Next.js handles routes that are missing from the paths array.

Example Setup

Suppose our application contains these products:

Database
Product 1
Product 2
Product 3
Product 4
Product 5

During build we generate only:

/products/1
/products/2

Now a user visits: 

/products/3

The result depends on the fallback value.

fallback: false

Only pages listed in paths are generated.

Complete Example
export async function getStaticPaths() {

  return {
    paths: [
      { params: { id: '1' } },
      { params: { id: '2' } }
    ],
    fallback: false
  };

}

export async function getStaticProps({ params }) {

  return {
    props: {
      productId: params.id
    }
  };

}

export default function Product({ productId }) {

  return (
    <h1>Product {productId}</h1>
  );

}

Use fallback: false when all pages are known during build time.

fallback: true

Unknown pages are generated on demand.

The user first sees a loading state.

Complete Example
import { useRouter } from 'next/router';

export async function getStaticPaths() {

  return {
    paths: [
      { params: { id: '1' } },
      { params: { id: '2' } }
    ],
    fallback: true
  };

}

export async function getStaticProps({ params }) {

  return {
    props: {
      product: {
        id: params.id
      }
    }
  };

}

export default function Product({ product }) {

  const router = useRouter();

  if (router.isFallback) {
    return <h1>Loading...</h1>;
  }

  return (
    <h1>Product {product.id}</h1>
  );

}

Use fallback: true when generating all pages during build would take too long and a temporary loading state is acceptable.

fallback: 'blocking'

Unknown pages are generated on demand, but users do not see a loading state.

Complete Example
export async function getStaticPaths() {

  return {
    paths: [
      { params: { id: '1' } },
      { params: { id: '2' } }
    ],
    fallback: 'blocking'
  };

}

export async function getStaticProps({ params }) {

  return {
    props: {
      product: {
        id: params.id
      }
    }
  };

}

export default function Product({ product }) {

  return (
    <h1>Product {product.id}</h1>
  );

}

Use fallback: 'blocking' when SEO is important and we do not want users to see a loading screen.

Interview Tip

A common misconception is that fallback generates pages during the build process. In reality, fallback only affects pages that were NOT generated during the build process.
One-Line Definition:
fallback: false returns 404 for unknown pages, fallback: true shows a loading page while generating the page, and fallback: 'blocking' waits for page generation before sending the completed HTML.
